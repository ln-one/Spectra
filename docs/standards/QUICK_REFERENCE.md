# 代码规范快速参考

## 📏 行数限制速查表

| 文件类型 | 行数限制 | 示例 |
|---------|---------|------|
| 类型定义 | <100 行 | `types.py`, `models.py` |
| 工具函数 | <150 行 | `utils.py`, `helpers.py` |
| 业务逻辑 | <200 行 | `processor.py`, `generator.py` |
| 服务编排 | <150 行 | `__init__.py`, `service.py` |
| API 路由 | <200 行 | `router.py` |

## 🎯 单一职责原则

**一个文件只做一件事**

```
✅ 好的拆分
generation/
├── types.py         # 只定义数据类型
├── validator.py     # 只做输入验证
├── processor.py     # 只做核心处理
└── formatter.py     # 只做输出格式化

❌ 不好的拆分
generation/
├── part1.py         # 前 100 行（职责混杂）
├── part2.py         # 中 100 行（职责混杂）
└── part3.py         # 后 100 行（职责混杂）
```

## 🔍 何时拆分？

满足**任一条件**时拆分：

1. ⚠️ **行数超标**
   - 类型定义 >100 行
   - 工具函数 >150 行
   - 业务逻辑 >200 行

2. ⚠️ **职责混杂**
   - 一个文件包含多个不相关功能
   - 数据模型和业务逻辑混在一起

3. ⚠️ **难以理解**
   - 需要滚动多次才能理解
   - 函数之间没有明确关系

## 📦 模块化模板

### 简单服务（<200 行）
```
service.py           # 单文件
```

### 中等服务（200-500 行）
```
service/
├── __init__.py      # 主服务（50-100 行）
├── types.py         # 类型定义（<100 行）
└── helpers.py       # 辅助函数（<150 行）
```

### 复杂服务（>500 行）
```
service/
├── __init__.py      # 主服务（编排）
├── types.py         # 数据类型
├── validator.py     # 验证逻辑
├── processor.py     # 处理逻辑
└── utils.py         # 工具函数
```

## 💡 实践建议

1. **先写后拆** - 不要过早优化
2. **按职责拆** - 不要机械切割
3. **保持简洁** - 使用 `__init__.py` 简化导入
4. **同步文档** - 拆分后更新 README

## 🤖 AI 友好性原则

好的代码应该让 AI 能够：
- ✅ 一次性理解文件全貌
- ✅ 准确定位需要修改的位置
- ✅ 修改单个功能不影响其他部分
- ✅ 生成准确的代码补全

## 🎓 设计理念

**为什么限制行数？**
- 降低认知负担
- 减少修改风险
- 提高 AI 准确性

**为什么强调单一职责？**
- 易于理解
- 易于修改
- 易于测试
- 易于复用
