# AI 协作工作流

> 更新时间：2026-03-02
> 目标：让 AI 修改行为可预测、可审计、可复用。

## 1. 工作原则

- 先对齐现状：先读当前代码与文档，不假设“规划已实现”。
- 最小改动优先：每次改动只解决一个清晰目标。
- 证据驱动：文档中的“已实现”必须能对应代码路径。
- 变更闭环：改代码要同步改文档，改文档要同步修入口。

## 2. 标准流程

1. 明确目标与范围
- 产出：本次只改哪些目录、哪些文件类型。

2. 收集现状证据
- 代码证据：依赖文件、核心实现文件、运行配置。
- 文档证据：当前入口文档、索引文档、重复文档。

3. 先改“单一事实源”
- 技术栈：`docs/architecture/tech-stack.md`
- 架构入口：`docs/README.md`、`docs/architecture/README.md`

4. 再改分层文档
- 仅修改与本次目标直接相关的文档。
- 标注状态：`已落地 / 规划中 / 历史文档`。

5. 清理与删除
- 删除重复文档前先确认无有效引用。
- 删除后全局检索死链并修复。

6. 提交前校验
- Mermaid 语法可解析。
- 关键链接存在。
- 至少完成与改动相关的手动校验；`pre-commit` 脚本按需手动运行。

## 3. 文档状态定义

- `已落地`：代码中已有实现。
- `规划中`：方案存在，但未接入主流程。
- `历史文档`：仅保留背景，不作为当前实现依据。

## 4. 常见反模式（禁止）

- 把 ADR 结论直接当成“当前实现”。
- 在多个文档重复维护同一份技术栈表格。
- 未校验 Mermaid 就提交。
- 删除文档后不修引用。

## 5. 推荐提交策略

- 按主题提交：`docs(architecture)`、`docs(standards)`。
- 一次提交只包含一个目标（例如“修 Mermaid + 对齐入口”）。
