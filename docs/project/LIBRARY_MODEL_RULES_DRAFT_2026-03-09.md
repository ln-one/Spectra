# 库模型规则草案

> 日期：2026-03-09
> 目的：将当前已基本定型的产品模型压缩为一份可被竞赛文档、产品文档、数据模型和接口设计共同引用的统一规则稿。

## 1. 系统本体

系统本体不是 `PPT`、导图、摘要或单次对话结果，而是“库”。

这里的“库”本质上是一个课程数据库节点。一个库同时具备：

1. 一套课程知识结构
2. 一组资源引用
3. 一套权限设置
4. 一组生成能力入口
5. 与其他库之间的引用关系

因此可将其定义为：

**库 = 一个可生成、可引用、可协作、可演化的课程知识单元**

## 2. 长期沉淀对象

系统长期沉淀的不是表现结果，而是课程数据库本身。

正式保存在系统中的核心对象包括：

1. 知识点
2. 知识点关系
3. 案例关系
4. 资源索引
5. 结构顺序
6. 引用关系
7. 权限状态
8. 版本状态

`PPT`、教案、导图、摘要、练习、网页、动画都只是库在不同场景下的按需外化结果。

## 3. 创建方式

库支持两种创建路径：

### 3.1 新建元库

1. 不引用任何父库
2. 从零开始建设
3. 作为一条知识演化链的起点

### 3.2 基于父库创建

1. 选择一个已有库作为父库
2. 选择引用模式
3. 在父库基础上继续演化

## 4. 引用模式

### 4.1 跟随（`follow`）

1. 当前库引用上游库
2. 不额外复制完整内容
3. 上游更新后，当前库底层知识源自动跟踪最新状态
4. 导图、摘要、`PPT` 等派生结果不强制实时刷新，而是按需重生成

适合场景：

1. 学生学习库
2. 基于标准课程做轻量个性化使用
3. 低成本复用

### 4.2 拉取（`pinned`）

1. 以某个上游版本为稳定起点
2. 默认不自动跟随后续变化
3. 当前库独立演化

适合场景：

1. 需要稳定起点
2. 个性化重构
3. 正式课程复用
4. 深度定制

## 5. 多引用结构

一个库可以引用多个来源，但需要满足固定结构：

### 5.1 主基底库

1. 最多一个
2. 提供主课程骨架

### 5.2 辅助引用库

1. 可以多个
2. 提供补充案例、题库、实验、跨学科内容或风格模板

因此，一个库的来源结构是：

1. 一个主基底
2. 多个辅助来源
3. 当前库自己的本地内容

## 6. 用户与库的关系

系统不以“教师/学生”作为本体区分，而以“谁维护哪个库、谁引用哪个库、谁能协作、谁能影响正式状态”来定义行为边界。

因此：

1. 教师可以有自己的库
2. 学生可以有自己的库
3. 协作者可以有自己的库
4. 教研组长可以有自己的库

区别主要由权限决定，而不是由对象类型决定。

## 7. 协作规则

协作默认采用“同库协作 + 会话隔离 + 候选变更 + 审核合入”模式。

规则如下：

1. 协作者属于同一个库
2. 共享同一课程数据库底座
3. 每个协作者拥有独立会话与生成上下文
4. 协作者结果先形成候选变更
5. 候选变更不直接写入正式库状态
6. 维护者审核通过后才正式合入

## 8. 冲突优先级

当一个库引用多个来源时，冲突处理优先级为：

1. 当前库本地显式内容优先
2. 主基底库优先
3. 辅助引用库按顺序优先
4. 系统默认模板或公共知识最低优先

补充规则：

1. 辅助库默认用于补充，不覆盖主骨架
2. 更细字段级策略可后续扩展

## 9. 更新传播

### 9.1 跟随模式

1. 上游变化后，底层引用状态自动跟随最新合法状态
2. 表现层结果不强制实时刷新
3. 用户在访问、导出或重生成时，基于最新状态外化结果
4. 对已有结果给出“上游已更新”提醒

### 9.2 拉取模式

1. 上游变化不自动影响当前库
2. 当前库以既定起点独立演化

## 10. 版本规则

版本采用轻量机制，不引入重型 Git 式模型。

规则如下：

1. 每个库有当前版本号
2. 只有正式入库的结构化变化才产生新版本
3. 草稿、对话、预览、临时生成不算正式版本
4. 候选变更合入后产生新版本
5. 导出物必须记录来源库与来源版本

可理解为：

**版本 = 正式库状态变更后的稳定标识**

## 11. 导出物规则

导出物不是系统本体，只是库的外化结果。

典型导出物包括：

1. `pptx`
2. `docx`
3. 思维导图
4. 摘要
5. 网页
6. 动画资源

约束如下：

1. 必须记录来源库
2. 必须记录来源版本
3. 不自动强制更新
4. 需要时可基于最新库状态重新生成

## 12. 发布规则

当前阶段不单独引入“发布版”对象。

现阶段三层结构已足够：

1. 库
2. 版本
3. 导出物

## 13. 权限规则

至少拆分为四类基础能力：

1. `可见`
2. `可引用`
3. `可协作`
4. `可管理`

补充规则：

1. 个人学习库默认不可被外部引用
2. 维护者可手动开启
3. 可按范围配置为仅自己、指定成员、组内或公开

## 14. 网络结构

库之间的引用网络必须满足 `DAG`（有向无环图）约束。

允许：

1. 一个库引用多个其他库
2. 一个库被多个后续库引用

禁止：

1. 任意引用环

## 15. 统一用户流程

统一工作流如下：

1. 创建自己的库
2. 选择是否引用父库
3. 选择跟随或拉取
4. 继续添加辅助库
5. 上传自己的资料
6. 使用完整生成能力
7. 导出所需结果
8. 若为协作者，则提交候选变更
9. 若为维护者，则审核并合入

## 16. 系统价值表达

系统不是发文件的平台，不是只会生成 `PPT` 的工具，也不是单纯的 `RAG` 知识库。

它是：

**一个以“库”为核心单元、以课程数据库为底座、以引用关系形成知识网络、以多模态内容按需外化的教学内容生产与演化系统。**

当前仍可后续细化，但不影响主架构成立的问题包括：

1. 拉取模式复制哪些对象

4. 搜索与推荐时如何在网络中选择可引用库
5. 计费模型如何挂在库能力上
