## 需求分析

### 功能性需求分析

功能性需求分析的重点，在于将业务痛点转化为可实现、可验证、可拆分的系统模块需求。结合当前应用对象与运行环境，`Spectra` 的功能性需求已经形成较清晰的模块化结构。本节围绕“输入组织、内容生成、成果交付、持续复用”四条主线展开，可归纳为六类模块需求。

#### 教学意图理解与会话组织模块

教师侧业务通常先形成教学目标、重难点和课堂思路，再逐步补充教学资料与修订意见。系统需要将这一渐进式工作过程组织为可推进、可回看、可持续修订的业务流程，以降低提示词工程带来的认知屏障与交互摩擦。

在当前作品中，该需求由 `Session` 语义承接。系统围绕一次具体教学任务组织输入数据、上下文信息、阶段状态和后续动作，使教学准备过程具备连续性和可管理性。

#### 多模态教学资料预处理模块

教学准备涉及 PDF、Word、课件、图片等多种资料形态。系统需要实现多模态教学资料的自动化特征提取与语义向量化存储，使输入数据能够进入后续检索增强和内容生成流程，而不仅停留在附件层或静态文件层。

在当前作品中，该需求对应资料上传、解析处理、切分组织和后续知识库链路。系统通过该模块实现教学资料从文件形态向可检索、可引用、可参与生成的数据形态转化。

#### 检索增强与证据组织模块

内容生成的稳定性依赖于检索结果和证据组织质量。系统需要在正式生成之前完成知识召回、相关性筛选和证据组织，为后续内容生成提供可靠的语义基础。

在当前作品中，该需求对应知识库与检索增强模块；在验证层面，则对应正式检索评估数据。该模块既承担功能职责，也构成系统技术有效性的直接证据。

#### 生成与交付管理模块

教学内容生成需要形成包含阶段确认、正式生成、结果预览、修订和标准导出的完整管理流程。系统应提供可控的内容生产链路，以增强结果与教学目标之间的一致性。

在当前作品中，该需求主要由以下能力承接：

- `outline` 确认；
- 正式生成；
- 预览与修订；
- 下载与标准导出。

上述环节共同构成完整的生成与交付管理模块，是系统形成可演示、可复用业务闭环的核心部分。

#### 多成果对象生成模块

一次教学准备通常需要同时生成课件、文档、导图、互动内容和辅助材料等多种成果对象。系统需要围绕同一组输入数据支持多成果输出，避免不同结果类型之间重复构建输入链路。

在当前作品中，该需求对应工作台内的多类能力入口与多种成果展示形态。该模块反映的是“同一数据基础支撑多类成果对象”的业务要求。

#### 成果保存与持续复用模块

教学准备完成后，成果对象仍需进入持续管理流程。系统需要支持成果保存、版本记录、引用关系管理和后续复用，使成果对象能够成为下一轮教学活动的输入基础。

在当前作品中，该需求由 `Artifact`、`Version`、成果绑定和后续引用关系承接。通过该模块，成果对象具备持续利用、比较和再组织条件。

#### 功能需求对应表

| 业务问题 | 功能模块 | 在作品中的体现 |
| --- | --- | --- |
| 教学资源离散化、接入缺乏规范 | 多模态教学资料预处理模块 | 多类资料进入统一链路，参与后续解析、检索与生成 |
| Prompt Engineering 存在认知屏障与交互摩擦 | 教学意图理解与会话组织模块 | 通过 `Session` 逐步推进一次完整业务过程 |
| 内容生成过程可控性不足 | 生成与交付管理模块 | 生成流程包含确认、正式生成、预览和修订 |
| 检索证据不足影响生成稳定性 | 检索增强与证据组织模块 | 知识库与检索模块提供召回与证据支持 |
| 一次备课需产出多种成果对象 | 多成果对象生成模块 | 当前已展示课件、文档、导图、互动等成果 |
| 成果导出后难以持续利用 | 成果保存与持续复用模块 | 引入 `Artifact`、`Version` 等对象实现保存与绑定 |

综上，功能性需求分析表明：教学准备活动中的每类核心问题在当前作品中均已具备明确的功能承接模块，并可通过会话组织、数据处理、检索增强、生成交付和成果管理串联为连续业务闭环。后续系统设计、关键技术和测试验证章节均以本节需求结构作为展开基础。

从评审阅读角度看，本节可进一步压缩为三项核心判断：一是系统已具备围绕教学任务组织输入与上下文的能力，二是系统已具备围绕多模态数据完成检索增强与内容生成的能力，三是系统已具备围绕成果对象完成交付、保存和持续复用的能力。

从竞赛材料表达看，本节同时证明了系统功能需求并非功能堆叠，而是围绕教学准备业务闭环形成的模块化能力体系。
