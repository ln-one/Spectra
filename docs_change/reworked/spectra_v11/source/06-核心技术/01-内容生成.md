## 基于会话推进的内容生成技术

内容生成要解决的具体问题，不是“让模型一次性吐出一份结果”，而是把教师给出的教学目标、课程资料、模板偏好和后续修改意见组织成一条可以逐步推进、可以中间确认、可以继续承接预览与导出的正式工作链。当前系统采用 `Session -> outline -> confirm -> generate -> preview/export -> artifact/version` 的阶段流程，由 Spectra backend 负责会话编排与状态推进，`Diego` 负责 `outline` 与正式内容生成，`Pagevra` 负责预览、渲染与导出。

| 服务分工 | Spectra backend 承接会话编排与状态推进，`Diego` 承接大纲与正式内容生成，`Pagevra` 承接后续预览与导出 |
| 复核证据 | 会话创建约束测试、Studio card 执行映射测试、会话预览 contract 测试 |

### PPT 主生成链

当前最稳定、证据最完整的内容生成主链是 `PPT` 课件生成。它的起点不是直接调用某个模型，而是先由 Studio 页面收集主题、页数、模板、受众层级和资料范围等输入，再把这些输入组织成一次正式会话请求。对于课件卡片，`card_execution_plans.py` 已将其初始绑定到 `POST /api/v1/generate/sessions`，并把 `topic`、`pages`、`template_id`、`style_preset`、`visual_policy`、`rag_source_ids` 等字段纳入已绑定配置。

这条主链的具体执行顺序可以拆成六步：

1. 教师在 Studio 中配置主题、页数、模板和资料来源；
2. 前端通过 `POST /api/v1/generate/sessions` 创建 `session bootstrap`；
3. backend 建立 `Session`，推进 `DRAFTING_OUTLINE -> AWAITING_OUTLINE_CONFIRM -> GENERATING_CONTENT -> RENDERING` 等阶段状态；
4. `Diego` 负责生成 `outline`，并在确认后进入正式课件生成；
5. `Pagevra` 承接 preview / render / export，把正式结果变成可查看、可导出的页面与文件；
6. 结果通过 `artifact_id`、`current_version_id`、`based_on_version_id` 等字段继续进入正式对象链。

这一段必须按服务边界来理解。Spectra backend 当前不是 `PPT` 的正式生成权威，它负责的是会话编排、状态推进、调用组织、结果绑定和接口整形；`Diego` 负责课件正式生成；`Pagevra` 负责预览、渲染和导出。把这三个角色分开之后，这条链就能直接对上接口、页面和结果对象，不需要再用“智能生成引擎”这类大词去概括。

从工程承托看，这条主链借助了几类现成能力：一是 `Diego` 提供的 `outline` 与正式课件生成服务；二是 `Pagevra` 提供的 preview/export 契约；三是现有 OpenAPI 中的 session、preview、export 返回结构，用来把生成结果稳定接回工作台和后续结果对象。这些现成能力已经进入当前正式主链，因此可以写成系统现实；但正文不应进一步写成“单页 slide patch 精修已经完整闭环”，因为当前课件 refine 仍主要由 chat 通道和生成主链局部修改协议共同承托。

如果按实现面继续往下看，课件卡片当前用到的“轮子”并不神秘：前端侧使用 `studio-cards.ts` 和 OpenAPI `types.ts` 组织卡片协议；后端侧使用 `card_execution_plans.py` 把课件卡片绑定到 `session create` 主路径；正式生成运行时由 `Diego runtime sync` 承担；预览和导出则通过 `preview_queries.py` 与 `Pagevra` 的预览导出契约对接。正文把这些名字写出来，评审就能看明白这不是一句“AI 自动生成 PPT”，而是一条由现成服务和已有契约拼起来的正式主链。

### `outline -> confirm -> generate -> preview/export` 的阶段推进

这条内容生成链成立的关键，不在于某个模型一次输出了多少内容，而在于会话被拆成了可管理的中间阶段。当前系统不会把课件生成处理成“提交后直接等待最终文件”，而是先形成 `outline`，让用户在进入正式生成前就能判断方向是否合理。

从状态推进看，当前流程至少包含四个关键阶段：

- `outline` 草拟：先得到课程结构和页面组织草案；
- 用户确认：在进入正式生成前保留中间判断点；
- 正式生成：确认后才进入课件或卡片成果生成；
- 预览与导出：生成后接入 preview/export 与后续保存链。

这种写法的落点不是抽象方法论，而是实际页面和接口。用户在 Studio 里能看到生成配置和卡片执行入口，在 preview 页面能看到正式结果，在 download 和 history 中能继续查看和导出。会话预览 contract 还会带回 `artifact_id`、版本字段和渲染结果，使预览、导出和结果保存属于同一套流程，而不是几个彼此独立的展示页面。

### Studio 卡片协议层

除 `PPT` 主链外，当前内容生成还通过一套统一的 Studio 卡片协议组织多种结果能力。OpenAPI 中已经存在：

- `GET /api/v1/generate/studio-cards`
- `GET /api/v1/generate/studio-cards/{card_id}/execution-plan`
- `POST /api/v1/generate/studio-cards/{card_id}/execution-preview`
- `POST /api/v1/generate/studio-cards/{card_id}/execute`
- `POST /api/v1/generate/studio-cards/{card_id}/refine`

这些接口说明，Studio 当前不只是“把几个按钮摆在同一页”。它已经把卡片目录、执行预览、正式执行、refine、turn 和 placement 之类的动作拆成统一协议层。前端 `studio-cards.ts` 和 `types.ts` 里还能进一步看到，卡片协议已明确区分：

- `readiness`：当前完成度是 `ready`、`foundation_ready` 还是更早阶段；
- `context_mode`：这张卡片依赖 `session`、`artifact` 还是 `hybrid` 上下文；
- `execution_mode`：初始执行是 `session_command`、`artifact_create` 还是 `composite`；
- `refine/turn/placement`：后续动作是否已经有正式绑定。

从系统设计角度看，这一层做的事情很具体：用户都从同一个 Studio 入口发起卡片执行，前端都走同一套 card schema，后端都按 execution plan 分发到不同载体。对正文而言，最重要的不是把每张卡片都写得很成熟，而是把“统一协议已经存在、但不同卡片成熟度不同”说清楚。

### 八张 Studio 卡片的成熟度与执行绑定

当前工作台中的八张卡片，可以按正式载体、初始执行入口、后续 refine/turn/placement 状态和承托层来分层说明。这里使用的是当前 `card_catalog.py`、`card_execution_plans.py` 和 `test_studio_card_exit_criteria.py` 能直接核对的事实，而不是把所有卡片都写成同等完成度。

| 卡片名称 | 当前正式载体 | 初始执行入口 | refine / turn / placement | 现成轮子或承托层 | 当前完成度说明 |
| --- | --- | --- | --- | --- | --- |
| `courseware_ppt` | `session + ppt artifact` | `POST /api/v1/generate/sessions` | `refine=partial`，通过 `chat` 承托 | `Diego` 生成、`Pagevra` 预览导出、session 主链 | 课件主链已打通，最稳定；但不应写成已具备完整单页 patch 闭环 |
| `word_document` | `docx / summary / exercise artifact` | `POST /api/v1/projects/{project_id}/artifacts` | `refine=ready` | `word_template_engine`、artifact create 主路径 | 文档卡已具备正式生成与 replacement refine，配置协议仍可继续细化 |
| `interactive_quick_quiz` | `exercise artifact` | `POST /api/v1/projects/{project_id}/artifacts` | `refine=ready` | 统一 artifact create + structured refine | 题目成果已能正式创建，单题级改写已接入 refine |
| `interactive_games` | `html artifact` | `POST /api/v1/projects/{project_id}/artifacts` | `refine=partial` | `legacy compatibility adapter` | 当前有正式载体，但仍在兼容冻结区，不能写成 fully ready |
| `knowledge_mindmap` | `mindmap artifact` | `POST /api/v1/projects/{project_id}/artifacts` | `refine=ready` | artifact create + 节点级 replacement refine | 导图与节点扩展已具备正式入口，但仍属于卡片级成果，不宜写成独立图谱系统 |
| `demonstration_animations` | `animation_storyboard / html / gif artifact` | `POST /api/v1/projects/{project_id}/artifacts` | `refine=ready`，`placement=partial` | structured refine、placement 二阶段协议 | 动画正式生成已可落到 HTML5/GIF，投放到 PPT 仍是二阶段 partial 动作 |
| `speaker_notes` | `summary artifact`，依赖 PPT source artifact | `POST /api/v1/projects/{project_id}/artifacts` | `refine=ready` | source-binding 协议、summary artifact 承托 | 已能基于 PPT 结果生成逐页讲稿和段落改写 |
| `classroom_qa_simulator` | `summary artifact` | `POST /api/v1/projects/{project_id}/artifacts` | `refine=partial`，`turn=ready` | turn 协议、summary artifact 承托脚本 | 初始预演已具备，正式多轮推进可走 turn 接口，但完整评估回路仍待完善 |

表中的两个状态需要单独说明。`foundation_ready` 表示协议和正式入口已经接入，能够进入当前工作流，但不等于所有交互语义都已经成熟。`partial` 表示这张卡片已经有正式入口，但后续动作仍可能依赖 `chat` 承托、兼容层、二阶段 placement 或受限 refine 协议。

若继续把“现成轮子”写具体，每张卡片当前大致依赖的是以下几类实现：

- `word_document`：依赖 `word_template_engine`，其中 `common.py` 负责文档类型、字段校验和布局常量，`html.py` 负责把分层教案、学生讲义、课后测试题、实验指导书四种变体渲染成可预览 HTML；
- `interactive_quick_quiz`：依赖统一的 artifact create 与 `tool_refine_builder/quiz.py` 结构化改写路径，题量、难度和干扰项配置直接进入正式 payload；
- `interactive_games`：依赖 `game_template_engine` 生成模板游戏 HTML，再由 `interactive_games_legacy_adapter.py` 负责 compatibility zone 归一化，所以这张卡片现在能跑，但仍处于冻结兼容区；
- `knowledge_mindmap`：依赖统一 artifact create 和 `tool_refine_builder/mindmap.py` 的节点级 replacement refine，而不是另起一套独立图谱运行时；
- `demonstration_animations`：依赖 `animation_contract`、`animation_spec` 和 `artifact_generator.animation_runtime`，先把动画请求归一成 storyboard/spec，再补 runtime snapshot，当前正式输出集中在 HTML5 与 GIF；
- `speaker_notes`：依赖 `normalize_speaker_notes_payload(...)` 把逐页讲稿整理成 `speaker_notes.v2` 结构，并为页级、段落级内容生成 anchors，后续段落改写再通过 `tool_refine_builder/speaker_notes.py` 收口；
- `classroom_qa_simulator`：初始脚本以 summary artifact 承托，多轮追问依赖 `POST /api/v1/generate/studio-cards/classroom_qa_simulator/turn` 这条正式接口，而不是只靠聊天页面临时追问。

另一个需要明确的事实是：八张卡片虽然都进入了统一目录，但它们并不是同一成熟度。`PPT` 主链是当前最稳定、最完整的正式主链；`word_document`、`interactive_quick_quiz`、`knowledge_mindmap`、`speaker_notes` 的正式 artifact/create/refine 路径相对更清楚；`interactive_games`、`demonstration_animations` 的部分动作仍带有兼容区或二阶段特征；`classroom_qa_simulator` 则更适合写成“已有正式 turn 协议，但完整回路仍在完善”。

### 当前缺口与边界

当前内容生成已经形成统一工作台、正式会话主链和多卡片协议层，但还不能写成“所有生成能力都已完全成熟”。至少有四个边界需要保留：

- `PPT` 课件微调当前仍主要由 `chat` 承托和生成主链局部修改协议共同承担，不能写成完整 slide 级 patch 系统；
- `interactive_games` 仍处于 legacy compatibility freeze 区，正式载体虽已存在，但不能写成 fully ready runtime；
- `demonstration_animations` 的 placement 仍是二阶段 partial 动作，不应写成“动画已稳定写回所有 PPT 场景”；
- `classroom_qa_simulator` 虽已具备正式 turn 接口，但虚拟学生协议和完整评估回路仍待补齐。

这一节最终只需要说明一件事：`PPT` 主链和 Studio 多卡片协议都已经进入统一工作流，而且每张卡片做到哪一步，都能用页面、接口、执行绑定和 focused tests 说明。评审若要核对这部分能力，可以同时看四处：Studio 是否存在卡片目录与执行入口，OpenAPI 是否保留 execution-preview/execute/refine/turn/placement 契约，execution plan 是否明确区分 ready 与 partial，focused tests 是否证明这些卡片至少具备一条正式入口。
