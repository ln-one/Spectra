> 状态：`current`
> 用途：`internal handoff only`
> 角色：`competition-figure-pack.md` 的配套文字底稿
>
> 这份文档不负责决定视觉风格，而是负责把每张关键图背后的判断、结构、标签口径和图注短句先钉死。
> 后续同学可以据此画正式图，也可以据此给 Word 版补图注、配文和页内说明。

## 1. 使用规则

这份底稿只做四件事：

- 说明每张图回答什么问题
- 说明读者看完后应该形成什么判断
- 说明图里必须出现的结构和标签
- 提供可以直接落在图下方的中文短句

这份底稿不做四件事：

- 不提供视觉装饰建议
- 不决定具体配色
- 不替代系统事实核对
- 不允许把旧 `docx` 的过时内容重新带回主稿

配套关系固定为：

- [全书正式配图制作说明书](/Users/ln1/Projects/Spectra/docs/documentation/competition-figure-pack.md)：给画图和排版同学的制作单
- [A 类概要介绍定制设计说明](/Users/ln1/Projects/Spectra/docs/documentation/a-class-overview-visual-brief.md)：给 A 类概要介绍做页级结构和判断分布
- 本文：给他们的文字源、标签源、图注源

## 2. 全书总口径

全书图文统一服务四条主线：

- `课程知识空间`：系统本体
- `Studio`：产品面
- `控制平面 + 六个正式能力层`：架构主表述
- `课程知识资产网络`：商业价值主表述

图文统一避免：

- 把系统讲成“前端 + 后端 + 几个外挂服务”
- 把产品讲成“PPT 工具”
- 把部署讲成运维手册
- 把测试讲成表格仓库

图注统一倾向：

- 直接下判断
- 不解释自己不说什么
- 不写“如图所示”“本图展示了”
- 一句只负责一个判断

## 3. 开场图文稿

### 3.1 图 0-1 项目一句话总览图

**要回答的问题**

- `Spectra` 到底是什么

**读者应形成的判断**

- 这不是普通 AI 教学工具，而是一套以课程知识空间为本体的系统

**结构建议**

- 中心：`课程知识空间`
- 第一层：`Studio`、`控制平面`、`六个正式能力层`
- 第二层：`PPT / 教案 / 导图 / 动画 / 游戏 / 小测 / 说课 / 学情预演`
- 外层：`知识沉淀`、`版本锚定`、`引用回流`

**标签建议**

- 课程知识空间
- 多模态外化
- 正式状态
- 持续沉淀
- 组织级交付

**图下注句**

- `Spectra` 维护的核心不是一次性导出文件，而是能够持续沉淀、引用和演化的课程知识空间。

### 3.2 图 1-1 旧工具逻辑 vs Spectra 系统逻辑对比图

**要回答的问题**

- 为什么 `Spectra` 不该被理解为普通 AI 工具

**读者应形成的判断**

- 旧工具只解决单次生成，`Spectra` 解决的是课程知识的持续组织、外化和回流

**左侧结构**

- 文件导出
- 单次生成
- 资料不入链
- 成果不沉淀
- 结果不可复用

**右侧结构**

- 课程知识空间
- `Session` 主链
- `Studio` 多模态外化
- `Artifact / Version / Reference` 回流
- 成果可引用、可演化

**图下注句**

- `Spectra` 把教学内容生产从“生成一个文件”提升为“维护一个持续演化的课程知识空间”。

### 3.3 图 2-1 可行性四象限图

**要回答的问题**

- 为什么这个项目值得进入试点、评估和交付阶段

**读者应形成的判断**

- `Spectra` 的可行性不是单点技术可行，而是技术、工程、经济和应用条件同时成立

**四象限标签**

- 技术可行
- 工程可行
- 经济可行
- 应用可行

**图下注句**

- 这项工作已经跨过“概念可讲”的阶段，进入“系统可做、可评、可交付”的区间。
