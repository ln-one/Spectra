# 最小契约草稿：流式事件与教案绑定 PPT v1

> 更新时间：2026-03-30
> 状态：当前讨论稿
> 用途：为当前阶段的“类 Gamma 流式展示”和“教案绑定 PPT”提供最小契约，避免无效分支讨论。

## 1. 现状依据

当前实现里已经存在以下基础：

1. 生成会话事件流已经走 `SSE`，并支持 `cursor` 断点续传。
2. 事件基础模型已经有：
   - `event_id`
   - `event_type`
   - `state`
   - `state_reason`
   - `timestamp`
   - `cursor`
   - `payload`
3. 前端预览页已经实际消费了一组更细事件：
   - `outline.started`
   - `outline.section.generated`
   - `outline.completed`
   - `slide.updated`
   - `slide.modify.started`
   - `slide.modify.processing`
   - `slide.modify.failed`
   - `task.completed`
   - `task.failed`
4. `speaker_notes` 已经正式要求绑定 `source_artifact_id`，且当前源成果类型是 `pptx`。
5. `word_document` 当前前端虽然已经有“绑定参考成果”的 UI，但后端协议还没有正式收口成“必须绑定 PPT”。

因此：

- 流式事件契约不需要从零设计
- 教案绑定 PPT 契约也不应该另起新模型

## 2. 结论先行

当前阶段直接定成：

1. 流式展示采用**结构块级事件流**，不做 token 级协议。
2. 教案默认绑定 `PPT artifact`，不再让教案独立漂浮。
3. 教案入口产品语义改成：
   - 先选择对应 `PPT`
   - 再生成教案
4. 后端正式字段统一使用：
   - `source_artifact_id`

这已经足够支撑当前阶段实现、联调和演示。

## 3. 流式事件最小契约

### 3.1 当前阶段不做的方向

当前阶段不做：

1. token 级逐字流式协议
2. 每种能力都单独发明一套完全不同的事件格式
3. 前后端分别维护两套事件命名体系

### 3.2 当前阶段采用的原则

流式事件只做两层：

1. 通用事件壳
2. 少量明确的结构块事件

这样既能满足演示观感，也不会让实现复杂度失控。

### 3.3 通用事件壳

所有流式事件统一沿用现有基础结构：

```json
{
  "event_id": "evt_xxx",
  "event_schema_version": 1,
  "event_type": "outline.section.generated",
  "state": "DRAFTING_OUTLINE",
  "state_reason": "outline_generating",
  "progress": 35,
  "timestamp": "2026-03-30T12:00:00Z",
  "cursor": "cur_xxx",
  "payload": {}
}
```

### 3.4 当前阶段必须支持的事件类型

本阶段只强制收口以下事件：

#### A. 大纲流式

1. `outline.started`
2. `outline.section.generated`
3. `outline.completed`

#### B. PPT 流式

1. `ppt.started`
2. `ppt.slide.generated`
3. `ppt.completed`

#### C. 教案流式

1. `word.started`
2. `word.section.generated`
3. `word.completed`

#### D. 通用任务事件

1. `state.changed`
2. `task.completed`
3. `task.failed`

### 3.5 各事件最小 payload

#### 3.5.1 大纲章节事件

`outline.section.generated`

```json
{
  "section_index": 0,
  "title": "课程导入",
  "outline_version": 1
}
```

最小要求：

- `section_index`
- `title`

#### 3.5.2 PPT 单页事件

`ppt.slide.generated`

```json
{
  "slide_index": 0,
  "slide_id": "slide_001",
  "title": "牛顿第二定律",
  "preview_ready": true,
  "artifact_id": "art_ppt_001"
}
```

最小要求：

- `slide_index`
- `slide_id`
- `title`
- `preview_ready`

说明：

- 当前阶段不要求事件里直接塞整页完整二进制内容
- 只要求前端知道“第几页已经出来并可刷新预览”

#### 3.5.3 教案段落事件

`word.section.generated`

```json
{
  "section_key": "teaching_objectives",
  "section_title": "教学目标",
  "preview_ready": true,
  "artifact_id": "art_word_001"
}
```

最小要求：

- `section_key`
- `section_title`
- `preview_ready`

### 3.6 当前阶段的前端消费规则

前端只需要做到：

1. 收到 `outline.section.generated`，就让大纲块出现
2. 收到 `ppt.slide.generated`，就让 PPT 页出现或刷新
3. 收到 `word.section.generated`，就让教案块出现或刷新
4. 收到 `task.completed`，就触发最终刷新
5. 收到 `task.failed`，就显示失败信息

### 3.7 当前阶段不建议再做的选择

当前不建议再讨论：

1. 要不要 token 级流式
2. 要不要为每个能力做完全不同的事件壳
3. payload 里是否塞入超重内容

结论：

- 统一事件壳
- 结构块级事件
- 前端收到事件后按需刷新对应预览

这就是当前阶段最优解。

## 4. 教案绑定 PPT 最小契约

### 4.1 当前阶段不做的方向

当前阶段不再把教案当成完全独立的生成入口。

不再优先做：

1. 先配很多教案参数，再单独生成
2. 教案与 PPT 没有明确来源关系
3. 教案和 PPT 各走各的主线

### 4.2 当前阶段采用的原则

教案是 `PPT` 的配套工件。

因此当前阶段统一定为：

**先选对应 PPT，再生成教案。**

### 4.3 正式字段

教案绑定 `PPT` 的正式字段统一为：

```json
{
  "source_artifact_id": "a-ppt-001"
}
```

不额外新造：

- `ppt_id`
- `deck_id`
- `parent_artifact_id`

当前阶段统一只用：

- `source_artifact_id`

### 4.4 允许绑定的源类型

当前阶段教案只允许绑定：

- `pptx`

结论：

- 不需要再给“教案到底能不能绑定别的源成果”留选择
- 当前阶段直接收口为 `pptx only`

### 4.5 前端最小交互

教案生成页必须做到：

1. 展示“请选择对应 PPT”
2. 只列出当前项目下可选的 `pptx` artifact
3. 选中后把该值写入：
   - `selectedSourceId`
   - 最终请求中的 `source_artifact_id`

### 4.6 后端最小约束

当前阶段建议将教案生成正式约束为：

1. 若是正式教案生成主路径，则 `source_artifact_id` 为必填
2. 若 `source_artifact_id` 对应的不是 `pptx`，直接拒绝
3. 教案 artifact metadata 中应记录来源 `pptx` 信息

建议最小 metadata：

```json
{
  "source_artifact_id": "a-ppt-001",
  "source_artifact_type": "pptx"
}
```

### 4.7 当前阶段产品文案建议

不要写成：

- 绑定参考成果

建议改成更直接的：

- 选择对应 PPT
- 该教案将基于所选 PPT 生成

这样更符合你想要的产品语义。

### 4.8 当前阶段不建议再做的选择

当前不建议再讨论：

1. 教案要不要支持无源生成
2. 教案要不要支持绑定多个 PPT
3. 教案要不要支持绑定非 PPT artifact

结论：

- 当前阶段全部不要
- 就收口成“一个教案绑定一个 PPT”

## 6. 最后结论

为了避免无效选择，当前阶段直接定成：

1. 流式事件只做结构块级，不做 token 级
2. 强制收口三组主事件：
   - `outline.section.generated`
   - `ppt.slide.generated`
   - `word.section.generated`
3. 教案默认并正式绑定 `PPT artifact`
4. 教案绑定字段统一为 `source_artifact_id`
5. 当前阶段教案只允许绑定 `pptx`

这已经足够支撑你们现在的开发、联调和比赛演示。
