# Spectra 文风规范

本文档用于统一当前 `Spectra` 方案书的文风。目标文风为：克制、专业、内敛、术语稳定、可读但不过度解释。

当前默认策略是：主体内容贴实现，少量总纲句保留辨识度，对象语言保留并翻译。

当前送审稿另有两条固定规则：

- 学校名称不进入封面，也不进入正文；
- 封面只保留最小身份集，不写成报名信息页。

## 1. 使用原则

### 1.1 主干内容优先

正文主体默认回答以下问题：

- 作品里具体做了什么；
- 这个功能在页面、模块、接口、流程中体现在哪；
- 为什么这样设计；
- 当前已经拿到了哪些结果、图片或测试证据。

若一句话不能落到这些内容上，不作为正文主句。

### 1.2 点缀表达限量使用

允许保留少量能与当前系统对象语言和实现对应的总纲句，但只能出现在章首、章末或小结，不进入正文主体。

默认控制原则：

- 一章内不超过 `1-3` 处；
- 后句必须接实现、流程或对象语义；
- 没有证据支撑的表达不写入正文。

### 1.3 对象语言保留并翻译

`Project / Session / Artifact / Version / Reference / CandidateChange / Member` 可以继续使用，但第一次出现时一定配中文解释。

推荐写法：

- `Project`：项目资料与结果统一管理空间
- `Session`：一次具体工作过程或会话流程
- `Artifact`：生成结果或成果对象
- `Version`：结果版本与正式状态锚点
- `Reference`：结果之间的引用与复用关系
- `CandidateChange`：进入后续修改或版本演化的候选变更
- `Member`：协作成员及其权限边界

## 2. 可保留表达

以下表达允许保留，但写入正文时必须配套具体解释：

- `生产即沉淀，沉淀即交付`
- `统一工作台`
- `按需外化结果`
- `结果回到项目空间`
- `一次生成不是终点，而是后续复用的起点`
- `资料进入 - 生成 - 预览 - 保存` 是同一条连续链路
- `结果不是导出后的尾声，而是后续工作的起点`

## 3. 需降调表达

以下表达可以保留核心意思，但必须改写为比赛方案书能接受的语言：

- `课程知识空间`
  改写为：`项目资料与结果统一管理空间`
- `正式知识状态沉淀`
  改写为：`结果保存、版本记录与后续复用`
- `多模态外化`
  改写为：`多种成果生成`
- `平台化延展`
  改写为：`后续扩展方向`
- `治理能力`
  改写为：`管理与协作边界`
- `控制平面`
  改写为：`后端编排与管理层`
- `系统本体`
  改写为：`系统整体设计对象` 或 `系统长期维护的对象`

## 4. 禁用表达

以下说法默认不进入当前送审稿正文：

- `商业逻辑闭环成立`
- `具备进入试点和采购沟通的条件`
- `学校/机构治理已经成立`
- `六个正式能力层`
- `契约整形`
- `权责边界治理`
- `递归`
- `自指`
- `strange loop`
- `普遍结构美学`
- `系统已经跨过工程成立门槛`
- `平台化`
- `成熟平台`
- `平台级能力已经成立`

若确需表达近似含义，改写为可证实、可落地的技术方案语言。

## 5. 各章表达预算

### 表达预算较高

- `01 前言`
- `02 项目综述`
- `05 系统设计`
- `11 结语`

这些章节允许保留少量总纲句，用于维持项目辨识度与术语一致性。

### 表达预算较低

- `04 业务分析`
- `06 核心技术`
- `07 项目测试与成果展示`

这些章节只允许少量点缀。主体保持问题、功能、实现、技术、结果和证据结构。

### 表达预算最低

- `03 可行性分析`
- `08 组织管理`
- `09 商业企划`
- `10 风险管理`

这些章节以稳妥、具体、保守为主，不使用宣言式表达。

## 5.1 身份信息规则

- 封面只保留项目标题、副标题、队伍名称、指导老师和项目成员。
- 团队编号、区域、学校、院系、联系方式等信息不进入封面主视觉区。
- 正文不主动展示学校名称，不把项目说明写成报名表或申报表。
- 成员信息如需出现，只保留姓名与角色，不进入个人敏感信息。

## 6. 当前正式参考源

除原始 `Spectra.docx` 外，当前高层表达默认参考以下文档：

- `docs/project/SYSTEM_PHILOSOPHY_2026-03-19.md`
- `docs/project/需求整理-功能收口清单_2026-03-30.md`
- `docs/architecture/service-boundaries.md`

使用原则：

- 哲学文档只提炼系统判断和总纲句；
- 功能收口文档优先用于校正主链、对象语言和产品重心；
- 任何高层表达都必须能回指到代码、接口、页面、测试或图片；
- 不直接搬运英文哲学原句，不保留高哲学浓度表述。

## 7. 禁止句式

以下句式默认从正文中删除：

- `换句话说`
- `更进一步说`
- `这里最想说明的是`
- `本节的重点是`
- `这一章的重点是`
- `重点不在……而在……`
- `不是为了……而是为了……`
- `这里需要证明的是`
- `如果进一步展开`
- `更直白地说`
- `从作品角度看`
- `对……而言，真正……`
- `这说明……的地方在于`
- `其实`
- `比较`
- `明显地`
- `如果用一句更概括的话`
- `更稳妥地说`
- `本文档将在后文进一步解释`
- `这里可以理解为`
- `从某种意义上说`

如确需承接上下文，优先改为直接判断句，不使用解释腔连接。正文不解释“为什么要写这一段”，而是直接陈述这一段的事实、结构和证据。

## 7.1 解释腔清除规则

- 命中上述句式的段落，默认整段重写，不做局部修词。
- 重写后的段落顺序固定为：事实判断 -> 实现或证据 -> 必要时一句结果收束。
- 作者不再在正文中解释“为什么这样写”“本节承担什么作用”“这一段真正想说明什么”。
- 若一句话只是在说明写作意图，而不是说明系统事实、结构、流程、页面、测试或结果，该句直接删除。

## 8. 段落规则

- 一段只处理一个判断。
- 先写判断，再写证据。
- 结论句短，不自夸，不代替读者下判断。
- 不写“很强、很先进、很完整、很智能”这类自我评价。
- 不保留“解释这一段在干什么”的外壳句。
- 若段落以解释腔起笔，整段重写，不只做局部替换。

## 9. 外部材料并入规则

- 测试报告、运行记录和补充说明默认并入现有章节，不单独起附录。
- 外部材料进入正文前，需改写为技术方案书语言，不直接粘贴原文。
- 服务器内部路径、脚本路径、日志目录、PID 文件名等运行细节不进入送审稿正文。
- 外部材料只保留对页面、流程、测试结果和工程验证有证据价值的部分。
