# 前端设计与开发执行指南

> 更新时间：2026-03-05  
> 目标：指导前端团队分阶段完成交互与视觉升级，先做架构，再做优化，避免一开始陷入细节。

## 1. 适用范围

- 当前项目：Spectra MVP（Next.js 15 + Tailwind + Shadcn/Radix + Zustand + RHF + Zod）
- 适用对象：前端开发、前端设计、联调负责人
- 方法原则：`契约优先`、`少造轮子`、`稳定优先`

## 2. 视觉方向（统一基调）

设计风格参考：
- Things3：克制、清晰、层级自然、弱干扰动效
- Apple Human Interface：一致性、可读性、反馈明确、尊重平台习惯（参考：https://developer.apple.com/design/human-interface-guidelines）

风格关键词：
- 清晰、轻量、专业、可信
- 高信息密度但不拥挤
- 交互反馈明确，不做炫技动画

### 2.1 高级白底风格（参考即可，根据实际情况进行调整）

页面基底（避免“纯白刺眼”和“灰蒙”）：
- App Background：`#F6F7F5`
- Surface/Card：`#FFFFFF`
- Subtle Panel：`#F1F3EF`
- Divider：`#E3E7DF`

文字层级（偏 Things3 质感）：
- Primary Text：`#1F2520`
- Secondary Text：`#596359`
- Tertiary Text：`#7A857A`

点缀色（不单调，但克制使用）：
- Primary Accent（薄荷绿）：`#4C8C6A`
- Info Accent（湖蓝）：`#3F83A3`
- Warm Accent（杏橙，用于重点按钮/徽标）：`#D98A4E`

语义色（状态统一）：
- Success：`#2F8F5B`
- Warning：`#C68A2D`
- Error：`#C45353`

使用规则：
- 常态界面以中性色为主，点缀色只用于“可操作元素”和“关键状态”
- 同一屏最多 2 个点缀色，避免彩色噪音
- 卡片阴影轻量化（弱阴影 + 清晰边界），不做重投影拟物

### 2.2 字体与排版建议

- 中文：`PingFang SC`（系统优先）或 `Noto Sans SC`
- 英文与数字：`Inter`
- 标题字重：`600`
- 正文字重：`400/500`
- 行高：正文 `1.5`，说明文字 `1.4`

排版规则：
- 大标题少而准，不堆粗体
- 用间距建立层级，少用分割线
- 长表单分组展示，避免一屏信息拥堵

## 3. 阶段化实施（先架构，后精修）

## Phase 1：架构落地（必须先完成）

目标：先把主流程跑通，页面关系稳定。

交付物：
- 页面骨架：`项目列表页` + `课程工作台单页`
- 工作台三区：左资料区 / 中对话与预览区 / 右生成与任务区
- 主流程闭环：登录 -> 建课 -> 上传 -> 大纲确认 -> 生成 -> 预览修改 -> 导出
- 异常流程：上传失败、解析失败、生成失败、导出失败（均可恢复）
- 基础响应式：移动端/平板/桌面可用，不出现主流程阻断
- 基础无障碍：键盘可达、焦点可见、表单可读
- 基础表单优化：统一校验、明确报错、关键表单可恢复

约束：
- 不做视觉精修
- 不做复杂动效
- 不扩展视频/动画完整链路（先保留入口）

Phase 1 最小验收（防返工）：
- 响应式：`sm/md/lg` 下主流程均可完成
- 无障碍：核心页面可仅用键盘完成主要操作
- 表单：RHF + Zod 全面接管关键表单，错误提示可定位、可理解

## Phase 2：设计系统化（避免样式失控）

目标：建立可复用 UI 基础，不重复造轮子。

交付物：
- 设计 Token（CSS Variables）：
  - 颜色（主色/中性/语义色）
  - 字体（标题/正文/注释）
  - 间距、圆角、阴影、层级
- 通用组件规范（基于 `components/ui`）：
  - Button、Input、Select、Tabs、Dialog、Toast、Card、Progress、Skeleton
- 页面模板规范：
  - 列表页模板、工作台模板、预览模板、表单模板

约束：
- 优先复用 Shadcn/Radix 组件
- 禁止新增平行组件体系

## Phase 3：体验增强（你要求的重点）

目标：在不破坏主流程的前提下，系统性补齐体验能力。

### 3.1 视觉美感
- 控制色彩数量（1 主色 + 1 强调色 + 中性色体系）
- 强化层级（标题/正文/辅助信息）
- 使用一致的留白节奏与分割规则

### 3.2 响应式（增强）
- 断点建议：`sm/md/lg/xl`
- 移动端优先保证：上传、生成、预览、导出可用
- 三栏在小屏降级为分段视图（Tab/抽屉），不强行并排

### 3.3 暗色模式
- 基于 Token 切换，不写大量条件样式
- 保证对比度与语义色可读性（成功/警告/错误）

### 3.4 无障碍（A11y，增强）
- 键盘可达（Tab 顺序、焦点可见）
- 表单可读（label、aria、错误提示关联）
- 颜色不作为唯一信息载体

### 3.5 动效系统
- 仅保留功能性动效：过渡、反馈、状态变化
- Framer Motion 统一时长/缓动
- 支持 `prefers-reduced-motion`

### 3.6 反馈系统
- 统一反馈层级：Inline -> Toast -> Dialog
- 所有异步任务有状态：`queued/running/succeeded/failed`
- 错误信息必须给出“下一步动作”

### 3.7 表单优化（增强）
- RHF + Zod 统一校验
- 即时校验 + 提交校验分层
- 默认值、自动保存、草稿恢复（关键表单）

### 3.8 性能
- 首屏轻量化：按路由/模块分割
- 非关键区懒加载（预览增强、重组件）
- 骨架屏替代阻塞加载

### 3.9 安全体验（前端侧）
- API 错误不泄露内部细节
- 上传前端校验文件类型/大小并给出明确提示
- 对危险操作（覆盖、回滚、删除）做二次确认

## Phase 4：竞赛打磨与验收

目标：确保可演示、可讲清、可量化。

交付物：
- 3 分钟固定 Demo 路径（一次通过）
- 指标看板：成功率、时延、重试率、引用覆盖率
- 视觉收敛（统一字体、色板、动效节奏）

## 4. 技术选型与复用策略（避免重复造轮子）

必须复用：
- 框架：Next.js App Router
- 状态：Zustand（业务）+ 组件内局部状态
- 组件：Shadcn + Radix
- 表单：React Hook Form + Zod
- 动效：Framer Motion（限量）
- 请求：现有 `lib/api/client.ts` 封装

不建议自研：
- 自定义基础弹窗/下拉/选择器（用 Radix）
- 自定义表单框架（用 RHF + Zod）
- 自定义动效引擎（用 Framer Motion）

## 5. 与 API 契约的协作方式

- 以前端页面状态机驱动 API 设计，不以“后端返回字段拼页面”
- OpenAPI 先定义再联调，前后端并行开发
- 新增字段必须同步：
  - 任务状态字段
  - 版本字段
  - 引用字段

联调顺序：
1. Mock（前端先跑通）
2. 真接口联调
3. 契约测试与回归

## 6. 完成定义（DoD）

以下全部满足才算完成：
- 主流程全链路可跑通且可恢复
- 设计系统 Token 与核心组件统一
- 响应式、暗色模式、无障碍基本达标
- 动效、反馈、表单、性能、安全体验均有最小实现
- 代码符合 `standards/frontend.md` 规范
- 无高优先级体验阻塞问题
