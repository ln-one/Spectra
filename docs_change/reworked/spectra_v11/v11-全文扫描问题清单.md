## 总判断

当前 `v11` 的主要问题不是“完全没内容”，而是正文里仍然同时存在三种写法：

- 一部分章节已经开始贴代码、接口和测试；
- 一部分章节还停留在“能力判断”和“结构判断”；
- 一部分章节虽然写了细节，但图、术语和连接句仍然带明显 AI 腔或概念腔。

最虚的区域集中在三块：

1. `06-核心技术`
2. `05-系统设计`
3. `09-商业企划`

其中 `06` 章属于“重写级”，`05` 章属于“删图并图 + 去平台化 + 改成动作链”，`09` 章属于“整体降调 + 降商业口径”。

## 分章问题清单

### 第 1 章 前言

`问题类型`
- 术语解释仍保留较多服务名，老师首次阅读会感觉门槛高。
- “项目说明”与“编写目的”偏总结，作品指向还可以再实一点。

`为什么虚`
- 术语虽然集中解释了，但 `Diego`、`Pagevra`、`Ourograph` 这类名字如果不立刻接“它在作品里干什么”，仍会像系统内部黑话。
- 前言的任务本该是降门槛，现在还略偏“系统介绍”。

`应改成什么写法`
- 所有术语采用“大白话 + 括号保留原名”的顺序。
- 项目说明优先回答“我们现在作品里能看到什么”，少回答“系统哲学是什么”。

`图表处理建议`
- `保留`：无图，维持现状。

### 第 2 章 项目综述

`问题类型`
- [系统定位与亮点](/Users/ln1/Projects/Spectra/docs_change/reworked/spectra_v11/source/02-项目综述/03-系统定位与亮点.md) 仍然偏“抽象亮点罗列”。
- 亮点段落里“界面、流程、对象、证据四层一致性”这种说法还是太像总结稿。

`为什么虚`
- 写的是“系统有什么亮点”，不是“亮点在作品里怎么被看见”。
- 四层一致性这种说法不够学生作品，不够老师口味。

`应改成什么写法`
- 亮点改成 `3-4` 个可见亮点，每个只写：
  - 对应页面
  - 对应流程
  - 对应结果
  - 对应测试或图
- 少用“定位、亮点、价值判断”，多用“在作品里具体体现为”。

`图表处理建议`
- `保留`：无图，主要改文字。

- 合规与推广基础只写已经存在的边界控制，不提前写机构级治理。

`图表处理建议`
- `保留`：无图，主要补证据句。

## 需要统一替换的高风险写法

以下表达在全文中应作为重点扫描词，后续优先替换：

- “系统已经形成……机制”
- “系统具备……能力”
- “也正因此”
- “换句话说”
- “这一步非常重要”
- “本节要说明的事实很简单”
- “更合适的结论是”
- “平台服务层 / 核心业务层 / 平台化延展 / 长期治理”
- “正式知识状态沉淀 / 课程知识空间 / 多模态外化”

推荐统一替换方向：

- 机制 -> 流程 / 做法 / 当前实现
- 能力 -> 页面动作 / 接口动作 / 模块动作
- 平台化 -> 后续扩展方向
- 沉淀 -> 保存与后续复用
- 外化 -> 多种成果生成
