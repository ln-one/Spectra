# Spectra 组内 Brainstorm 讨论稿

> 用途：给组员快速对齐当前主线，不要求一次性达成所有细节，只要求先对“大方向是否成立”达成共识。
> 使用建议：讨论时先看“核心判断”，再看“典型场景”，最后看“待拍板问题”。

## 0. 为什么这次讨论值得认真对待

这次 brainstorming 的意义，不只是给比赛文档找一个更好看的说法，而是在决定我们到底是在做一个“演示型 AI 工具”，还是在做一个“真正可能长成产品的平台底座”。

如果这条主线成立，它的价值不只是：

1. 更好地满足题目要求
2. 让方案讲起来更完整
3. 让演示更有说服力

更重要的是，它把很多原本分散的问题统一起来了：

1. 为什么多模态不是堆功能，而是统一生成源的问题
2. 为什么下游访问者不该只是被动接收文件（学生为常见示例）
3. 为什么参与者不能粗暴直写主状态
4. 为什么这个系统能从教师工具走向平台（示例叙事）

也就是说，这不是“又补了一个亮点”，而是在补整个系统的底层解释力。

如果组里能在这一轮对主线达成共识，我们后面的文档、演示、实现和分工都会顺很多。这个方向一旦立住，团队讨论就不再只是“再加什么功能”，而是“围绕一个更强的系统模型去推进”。

## 1. 我们现在到底在做什么

我们不是在做一个“更快生成 PPT 的工具”。

我们真正想做的是：

1. 教师本来就在做备课和生成 PPT
2. 系统在这个过程中自动沉淀课程数据库
3. 多模态内容不是本体，只是数据库的按需外化
4. 上游维护者（教师为常见示例）分享的不是单个文件，而是一个动态课程空间
5. 下游访问者不是只看文件，而是围绕这个空间继续学习、生成和整理（学生为常见示例）

一句话：

**系统本体是课程数据库和项目空间，不是单个 PPT。**

## 2. 为什么这个方向比“做一个 AI 课件生成器”更强

普通 AI 课件工具的问题：

1. 生成一次就结束
2. 结果是静态文件
3. 更新很难同步
4. 下游访问者只能被动接收
5. 教师还要额外上传和转发

我们的方向：

1. 教师生成 PPT 的过程本身就沉淀数据库
2. 导图、摘要、练习、网页、动画都是按需生成
3. 上游更新后（教师为常见示例），知识源自动更新
4. 下游用户可以基于上游空间继续生成自己的内容
5. 系统可以从单点工具升级为平台

这里真正值得有信心的地方在于：这条路不是靠“说得高级”成立，而是靠结构更统一、工程更合理、商业更完整成立。它不是更花哨，而是更稳。

## 3. 当前核心叙事

### 3.1 第一层

真正沉淀的是数据库，不是导图。

### 3.2 第二层

多模态的关键不在模态数量，而在于是否有统一生成源。

### 3.3 第三层

上游维护者（教师为常见示例）分享的不是 PPT 文件，而是课程空间。

### 3.4 第四层

学生先是访问者，只有在需要持续整理时才创建自己的个人空间（学习空间为示例）。

### 3.5 第五层

系统内部不必写死成教育专用结构，仍然可以沿用 `project + session` 做通用实现。

## 4. 现在定下来的模型

## 4.1 内部主对象

内部继续用：

1. `project`
2. `session`
3. `reference`
4. `artifact`
5. `version`

原因：

1. 和现有 API 兼容
2. 更通用
3. 不把系统锁死在单一教学场景

## 4.2 对外表达

对外可以讲：

1. 课程空间
2. 个人空间（学习空间为示例）
3. 课程资料库

## 4.3 我们说的“库”怎么理解

“库”现在更适合当 brainstorming 词，不一定是最终代码命名。

它更像：

**一个可生成、可引用、可协作、可演化的项目空间。**

## 5. 典型场景

## 5.1 教师场景（示例）

1. 教师创建一个项目空间
2. 上传 PDF、PPT、视频、图片、教案等资料
3. 通过对话说明教学目标、重点难点、讲授逻辑
4. 系统生成 PPT、教案等结果
5. 系统同时把知识结构沉淀进课程数据库
6. 教师修改后，数据库继续更新
7. 上游维护者分享的是课程空间访问入口

## 5.2 下游访问者场景（示例）

1. 下游访问者被邀请进入上游分享的课程空间（学生为常见示例）
2. 下游访问者先只是访问者
3. 下游访问者可以直接查看最新内容
4. 下游访问者可以按需生成导图、摘要、练习
5. 如果下游访问者想长期整理自己的资料，再创建自己的个人空间

关键规则：

**下游用户不创建自己的空间，就不能污染上游权威空间（教师为空间示例）。**

## 5.3 参与者场景

1. 参与者进入同一个项目空间
2. 参与者用自己的 `session` 工作
3. 对话与草稿彼此隔离
4. 参与者提交候选变更
5. 维护者审核合入

关键规则：

**参与者默认不直接改正式状态，而是通过候选变更进入审核。**

## 6. 引用模型

## 6.1 创建项目空间时

有两种起点：

1. 从零开始
2. 基于已有空间创建

## 6.2 基于已有空间创建时

两种模式：

1. `follow`
2. `pinned`

### `follow`

- 跟随上游最新合法状态
- 不复制完整底座
- 表现层按需刷新

### `pinned`

- 固定上游某个稳定起点
- 后续独立演化
- 适合更深度定制

## 6.3 多引用

一个空间可以有：

1. 一个主基底引用
2. 多个辅助引用

辅助引用可以提供：

1. 案例
2. 题库
3. 风格
4. 实验
5. 跨学科内容

## 6.4 冲突优先级

如果多个来源冲突，按这个顺序：

1. 当前项目本地内容优先
2. 主基底引用优先
3. 辅助引用按顺序优先
4. 系统默认模板最低优先

## 7. 更新机制

### 上游更新后（教师为常见示例）

1. 课程数据库更新
2. 下游访问者访问到的是最新知识源（学生为常见示例）
3. 已经生成过的导图、摘要、PPT 不强制实时覆盖
4. 系统提示“上游已更新，可刷新生成”

所以不是：

- 手动 merge
- 手动 pull
- 所有东西实时乱跳

而是：

**知识源自动跟踪，表现层按需刷新。**

## 8. 版本管理怎么理解

不是 Git 那套重型模型。

现在定的是轻量版：

1. 只有正式写入项目状态的结构化变化才算新版本
2. 对话、草稿、预览不算正式版本
3. 导出物只记录自己来自哪个版本

一句话：

**需要轻量版本标识，不需要重型版本系统。**

## 9. 现在这套模型为什么成立

### 工程上

1. 和现有 `project/session` 规划兼容
2. 不需要推翻当前 API
3. 可以分阶段实现

### 产品上

1. 教师流程自然
2. 下游访问者体验自然
3. 协作关系清楚
4. 权威性边界清楚

### 商业上

1. 教师是高频用户
2. 下游访问者可以逐步转化为客户（学生为常见示例）
3. 学校和教研组有采购理由
4. 存储、协作、引用、私有空间、高级生成都有后续商业化空间

## 10. 这轮 brainstorm 最希望组员一起判断的问题

### A. 大方向是否认同

1. 我们要做的是“课程空间 + 数据库”，而不是单次生成工具
2. 多模态是按需外化，不是预生成一堆结果

### B. 下游访问者路径是否自然

1. 下游访问者先作为访问者
2. 需要长期整理时再创建自己的个人空间

### C. 协作路径是否自然

1. 参与者通过独立 `session` 工作
2. 通过候选变更审核合入

### D. 引用模型是否合理

1. `follow / pinned`
2. 主基底引用 + 多辅助引用
3. 固定冲突优先级

### E. 哪些先做，哪些后做

1. 比赛版本要体现到什么程度
2. 下一阶段真实实现先补哪些能力

## 11. 建议的讨论顺序

1. 先讨论“系统本体到底是什么”
2. 再讨论“下游访问者怎么进来”
3. 再讨论“参与者怎么工作”
4. 再讨论“引用和更新机制”
5. 最后讨论“哪些先实现，哪些先写进演进设计”

## 12. 最后一句总纲

如果要把这次 brainstorm 浓缩成一句话，就是：

**Spectra 想做的不是一个只会生成 PPT 的 AI 工具，而是一个以项目空间为核心、以课程数据库为底座、以多模态内容按需外化、以引用与协作为扩展机制的教学内容生产系统。**

如果再进一步说清楚这套设计为什么值得大家投入：

**它让我们不只是“完成一个比赛项目”，而是在把一个本来可能停留在 demo 层的点子，推成一个真正有产品逻辑、有平台潜力、有后续扩展空间的系统方向。**
