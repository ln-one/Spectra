# 6 关键技术与实现

## 6.1 核心语义模型实现（Project/Session/Artifact/Version/Reference）

本项目构建了**以 Project（课程库）为核心、Session（工作会话）为驱动、Artifact（产物）为输出、Version（版本）为锚点、Reference（引用）为纽带**的五层核心语义模型，通过 Prisma ORM 实现类型安全的数据库建模，确保所有业务实体的语义边界清晰、关系稳定、可追溯性强。

核心实体定义与职责

- **Project（课程库）**：系统的核心业务实体，代表一个完整的课程知识空间，存储课程的基本信息、权限配置、引用关系与正式版本记录。一个 Project 可包含多个 Session、多个 Artifact、多个 Version，是资产沉淀与分享的基本单元。

- **Session（工作会话）**：教师一次完整备课行为的上下文容器，存储当前备课的教学意图、临时资料、生成状态与草稿内容。Session 与 Project 关联，一个 Project 可开启多个并行 Session，不同 Session 之间互不干扰，支持教师同时进行多个教学设计尝试。

- **Artifact（生成产物）**：系统生成的各类内容实体，包括 PPT 课件、Word 教案、思维导图、动画创意等。Artifact 分为临时 Artifact 与正式 Artifact，临时 Artifact 关联 Session，仅在当前会话可见；正式 Artifact 关联 Project 与 Version，作为课程资产长期保存。

- **Version（版本）**：Project 正式状态的时间锚点，记录每次发布的内容快照、更新时间、更新人与变更说明。Version 采用语义化版本号（v1.0.0），支持版本对比、一键回滚与上游更新同步，是实现 "生成即沉淀" 的核心载体。

- **Reference（引用关系）**：Project 之间的知识关联纽带，分为主基底引用（Primary Reference）与辅助引用（Secondary Reference）。主基底引用用于基于父库创建子库，辅助引用用于补充案例、题库等跨学科内容。Reference 支持 DAG 约束检测，避免循环依赖。

数据库实现要点

通过 Prisma Schema 定义核心模型，建立严格的外键约束与索引优化：

- Project 表存储课程元数据（名称、学科、学段、创建者、权限配置），建立创建者索引与学科索引。

- Session 表存储会话状态（当前步骤、临时上下文、生成进度），建立 Project 外键与过期时间索引，自动清理 7 天前的过期会话。

- Artifact 表存储产物信息（类型、格式、存储路径、关联版本），建立 Project 外键与 Version 外键。

- Version 表存储版本记录（版本号、变更说明、创建时间、创建者），建立 Project 外键与版本号唯一索引。

- Reference 表存储引用关系（源 Project、目标 Project、引用类型、优先级），建立源与目标的双向索引，添加 DAG 约束触发器。

## 6.2 SessionFirst 状态机与事件流实现

本项目采用**SessionFirst 状态机架构**，将教师的一次完整备课行为建模为有限状态机，通过事件驱动的方式实现状态流转，确保业务流程的可预测性、可追踪性与可恢复性。

状态机定义与状态流转

Session 状态机定义了 10 个核心状态与 15 个触发事件，状态流转严格遵循教学备课的自然逻辑：

表7 状态机与事件流

| **当前状态** | **触发事件** | **动作执行** | **下一个状态** |
|----|----|----|----|
| INITIALIZED | 用户输入教学主题 | 创建 Session，初始化上下文 | INTENT_UNDERSTANDING |
| INTENT_UNDERSTANDING | 需求模糊 | 主动追问补充信息 | INTENT_UNDERSTANDING |
| INTENT_UNDERSTANDING | 需求明确 | 生成结构化教学意图文档 | MATERIAL_UPLOADING |
| MATERIAL_UPLOADING | 用户上传资料 | 触发解析任务，更新进度 | MATERIAL_PROCESSING |
| MATERIAL_PROCESSING | 解析完成 | 分块向量化，索引入库 | OUTLINE_GENERATING |
| OUTLINE_GENERATING | 大纲生成完成 | 展示大纲预览，等待确认 | OUTLINE_CONFIRMING |
| OUTLINE_CONFIRMING | 教师确认大纲 | 锁定大纲，触发内容生成 | CONTENT_GENERATING |
| OUTLINE_CONFIRMING | 教师修改大纲 | 更新大纲，重新生成 | OUTLINE_GENERATING |
| CONTENT_GENERATING | 内容生成完成 | 展示课件 / 教案预览 | PREVIEWING |
| PREVIEWING | 教师提出修改 | 定位目标模块，局部重生成 | CONTENT_GENERATING |
| PREVIEWING | 教师确认发布 | 创建 Version，落库沉淀 | PUBLISHED |

事件流实现机制

- **事件捕获**：前端通过 WebSocket 或 HTTP POST 发送用户操作事件，事件包含事件类型、Session ID、事件参数三部分。

- **状态持久化**：Session 当前状态与上下文实时存储在 Redis 中，同时异步写入 MySQL 持久化，确保服务重启后状态可恢复。

- **动作执行**：状态机根据当前状态与事件类型，路由到对应的业务服务执行动作，如意图理解、资料解析、内容生成等。

- **状态更新**：动作执行成功后，更新 Session 状态并触发下一个动作；执行失败则进入 ERROR 状态，触发重试或人工干预。

- **进度推送**：通过 WebSocket 实时向教师推送状态变化与生成进度，提升交互体验。

## 6.3 上传解析分块索引全链路实现

构建 "分片上传 - 格式识别 - 多模态解析 - 语义分块 - 向量化 - 索引入库" 的全链路数据处理 pipeline，支持 GB 级大文件的稳定处理与教学资料的高质量知识化。

分片上传与断点续传

- 前端将大文件切割为 5MB 的分片，通过并发请求上传，每个分片携带文件 ID、分片序号与总片数。

- 后端接收分片后写入临时存储，校验分片完整性，所有分片上传完成后合并为完整文件。

- 支持断点续传，前端上传中断后，下次上传时先查询已上传分片列表，仅上传缺失分片。

多模态解析与内容提取

- **格式识别**：通过文件扩展名与 Magic Number 双重识别文件格式，支持 PDF、DOCX、PPTX、XLSX、JPG、PNG、MP4、MP3 等 12 种格式。

- **文档解析**：原生 PDF/Word/PPT 使用 MinerU 解析，精准提取标题层级、段落文本、公式（转换为 LaTeX）、图表、表格；扫描件 PDF 使用 Dualweave 进行 OCR 识别与版面还原。

- **多媒体解析**：图片使用 PaddleOCR 提取文字与公式；视频使用 FFmpeg 抽取关键帧（每 10 秒 1 帧），Whisper 提取字幕与语音内容，生成视频片段语义摘要。

语义分块与向量化

- **智能语义分块**：采用 "标题层级 + 语义相似度" 的混合分块策略，优先按文档原生标题层级分块，对于超长段落再使用语义相似度模型（bge-small-zh）切分，每个分块保留完整的语义上下文，长度控制在 512-1024 token。

- **向量化生成**：使用 bge-large-zh-v1.5 模型生成分块向量，维度为 1024，同时生成分块的关键词与元数据（学科、学段、来源文档、页码范围）。

索引入库与增量更新

- **向量索引**：将分块向量与元数据写入 Qdrant，建立 HNSW 索引，设置 ef_construction=200、M=16，兼顾检索精度与速度。

- **结构化索引**：将分块文本、元数据与 Project/Session 关联写入 MySQL，建立全文检索索引与元数据过滤索引。

- **增量更新**：支持文档的增量更新，修改文档时仅重新处理变更部分，删除旧分块索引，插入新分块索引，避免全量重新处理。

## 6.4 多模态 RAG 检索与引用追溯实现

构建 "混合检索 - 结果重排序 - 上下文拼接 - 引用追溯" 的 RAG 增强生成体系，既保证生成内容的准确性，又实现每个知识点的可追溯性。

混合检索与结果重排序

- **多源检索**：同时检索四个来源的内容：当前 Session 的临时资料、当前 Project 的历史版本、绑定的主基底 / 辅助引用库、公共权威知识库（如新课标、教材解析）。

- **混合检索策略**：结合语义检索（向量相似度）与关键词检索（BM25），语义检索召回 Top-50，关键词检索召回 Top-30，合并后去重。

- **元数据过滤**：根据当前教学意图的学科、学段、章节等元数据，对检索结果进行过滤，仅保留相关内容。

- **交叉编码器重排序**：使用 bge-reranker-large 模型对初步检索结果进行重排序，计算查询与每个分块的相关性得分，筛选 Top-10 最相关内容送入大模型。

上下文拼接与引用标注

- **结构化上下文拼接**：将重排序后的分块按逻辑顺序拼接，每个分块前添加引用标记（\[Ref: 文档 ID - 分块 ID\]），分块后添加来源说明（来自《XX 教材》第 X 章）。

- **上下文长度控制**：采用滑动窗口机制控制总长度，优先保留高相关性分块，总上下文长度不超过大模型的最大输入窗口（如 Qwen-14B 为 8K）。

引用追溯实现

- **生成内容标注**：大模型生成的每个知识点、案例、习题都自动标注对应的引用标记，如 "牛顿第二定律（\[Ref:phy-001-ch3-005\]）"。

- **溯源跳转**：教师点击引用标记可跳转至原始文档的对应位置，查看完整上下文，验证内容准确性。

- **引用关系记录**：将生成内容与引用分块的关联关系写入 Reference 表，支持后续的引用统计与版权管理。

## 6.5 对话编排与上下文注入实现

构建 "意图识别 - 多轮对话管理 - 提示词模板 - 上下文动态注入" 的对话编排体系，实现自然、流畅、智能的人机交互。

意图识别与分类

- 使用微调后的 Qwen-7B 分类模型，将教师的自然语言输入分为 8 类意图：生成请求、修改请求、确认请求、取消请求、资料上传、问题咨询、设置调整、其他。

- 对于生成 / 修改请求，进一步提取结构化参数：目标模块、修改类型（增加 / 删除 / 调整 / 简化）、具体要求。

多轮对话上下文管理

- **对话历史存储**：使用 Redis 存储最近 20 轮对话历史，超过 20 轮时使用大模型对历史对话进行摘要，保留核心信息。

- **上下文窗口优化**：采用 "最近对话 + 核心摘要" 的混合策略，最近 10 轮对话完整保留，更早的对话使用摘要，平衡上下文完整性与长度限制。

- **状态感知**：对话编排器感知当前 Session 状态，根据状态调整回复策略，如在大纲确认阶段优先确认大纲，而非直接生成内容。

提示词模板与动态注入

- **分层提示词模板**：构建三层提示词体系：系统提示词（定义角色、任务、输出格式）、任务提示词（根据当前状态动态生成）、上下文提示词（注入检索结果与对话历史）。

- **教学场景优化**：系统提示词针对教育场景优化，要求生成内容符合课程标准、结构完整、语言符合教师 / 学生身份，禁止生成错误知识点。

- **动态参数注入**：任务提示词动态注入当前教学意图、学科、学段、大纲结构、修改要求等参数，确保生成内容符合教师需求。

主动追问与需求补全

- 设计模糊需求检测规则，当教师输入过于简略（如 "生成一份数学课件"）时，自动触发主动追问："请问是哪个学段、哪个章节的内容？教学重点是什么？"

- 追问采用结构化选项 + 自由输入结合的方式，教师可快速选择，也可详细描述，提升交互效率。

## 6.6 大纲生成确认课件生成两阶段实现

采用 "大纲生成 - 确认 - 内容生成" 的两阶段生成策略 ，避免一次性生成完整课件导致的大改风险，提升生成质量与教师满意度。

第一阶段：大纲生成与确认

- **大纲生成**：基于教学意图与检索结果，先生成结构化的课程大纲，包含：教学目标（知识目标 / 能力目标 / 情感目标）、教学重难点、教学过程（导入 / 新授 / 练习 / 总结 / 作业）、板书设计、教学反思框架。

- **大纲可视化**：大纲以树形结构展示，教师可拖拽调整章节顺序，点击增删章节，修改章节标题与内容要点。

- **大纲预生成预览**：教师点击某个章节，可预生成该章节的 1-2 页示例内容，确认生成方向后再进入下一阶段。

第二阶段：分模块内容生成

- **分模块并行生成**：教师确认大纲后，系统将大纲拆分为多个独立模块（每个模块对应 1-2 页 PPT 或 1 个教案小节），使用 RQ 队列并行生成，提升生成速度。

- **模块间一致性保证**：生成前先提取课程的核心概念、案例、数据，写入共享上下文，所有模块生成时都引用共享上下文，确保内容一致、术语统一。

- **内容校验**：每个模块生成后，自动进行初步校验：知识点准确性（与检索结果对比）、格式完整性（包含标题、正文、图片 / 图表占位符）、合规性（过滤敏感内容），校验不通过则自动重生成。

两阶段策略的优势

- **降低成本**：大纲阶段修改成本低，仅需调整结构，无需消耗大量大模型 token 生成完整内容。

- **提升质量**：教师确认大纲后，生成方向明确，避免生成不符合预期的内容。

- **交互友好**：教师可参与到生成过程中，逐步调整，而非被动接受最终结果。

## 6.7 课件预览、下载与产物落库一致性实现

构建 "实时预览 - 标准化导出 - 产物落库" 的一致性链路，确保预览效果、下载文件与落库内容三者完全一致，避免 "预览好看、下载乱码" 的问题。

实时预览渲染

- **Pagevra 前端渲染引擎**：基于 Web Components 与 Canvas 实现 PPT / 教案的像素级实时预览，渲染引擎与导出引擎使用相同的样式模板与布局算法，保证预览与导出效果 1:1 一致。

- **局部实时刷新**：教师修改某一部分内容时，仅重新渲染对应模块，而非整个课件，预览响应时间小于 1 秒。

- **交互预览功能**：PPT 预览支持翻页、缩放、全屏查看、动画预览；教案预览支持目录跳转、注释查看。

标准化导出链路

- **Artifact 导出引擎**：完全基于开源工具链构建，无需依赖商业办公软件：

  - PPTX/HTML5 导出：使用 Marp CLI，将 Markdown 格式的课件内容转换为 PPTX 与 HTML5，支持自定义主题模板。

  - DOCX/PDF 导出：使用 Pandoc，将 Markdown 格式的教案内容转换为 DOCX 与 PDF，支持自定义页眉页脚与样式。

  - 思维导图导出：使用 Mermaid，将课程知识结构转换为 SVG/PNG 思维导图。

- **导出文件校验**：导出完成后自动打开校验，确保文件可正常打开、格式正确、内容完整，校验失败则触发重试。

产物落库一致性保证

- **临时区与正式区分离**：生成的产物先存入 Session 临时区，仅在教师确认发布后才写入 Project 正式区，避免草稿内容污染正式库。

- **版本绑定**：所有正式产物都与唯一的 Version 绑定，导出文件的属性中嵌入版本号与生成时间，支持后续追溯。

- **内容同源**：预览、下载、落库的内容都来自同一个结构化数据源（Session 临时上下文或 Project 正式版本），确保三者完全一致。

## 6.8 候选变更与版本演进机制实现

构建 "候选变更 - 审核 - 合入 - 版本发布 - 回滚" 的完整版本管理体系，支持多人协作备课与课程库的持续演进，同时保证正式版本的稳定性与可追溯性。

候选变更机制

- **协作者权限**：Project 创建者可邀请协作者，协作者拥有编辑权限，但无法直接发布正式版本，只能提交候选变更。

- **变更提交**：协作者修改内容后，提交候选变更，填写变更说明，系统自动生成变更对比（diff），展示修改前后的内容差异。

- **变更隔离**：候选变更存储在独立的变更分支中，不影响正式版本，其他协作者可查看变更，但不会看到未审核的内容。

审核与合入流程

- **审核权限**：仅 Project 维护者（创建者或指定管理员）拥有审核权限。

- **变更对比**：维护者查看变更对比，可预览修改后的效果，支持在线批注修改意见。

- **审核结果**：审核通过则合入正式版本，生成新的 Version；审核驳回则返回给协作者，说明驳回原因，协作者修改后可重新提交。

版本演进与冲突处理

- **语义化版本号**：采用 v 主版本。次版本。修订号的语义化版本规则：

  - 主版本：重大结构调整，如课程体系重构

  - 次版本：新增内容，如新增章节、案例

  - 修订号：小修改，如错别字修正、格式调整

- **DAG 约束检测**：引用关系满足有向无环图（DAG）约束，禁止循环依赖，合入变更时自动检测，若存在冲突则提示人工解决。

- **冲突解决**：当多个变更同时修改同一内容时，采用 "先合入优先" 原则，后合入的变更需手动解决冲突，支持选择保留哪个版本或合并内容。

版本对比、回滚与上游同步

- **版本对比**：支持任意两个版本的内容对比，高亮显示差异部分，便于查看变更历史。

- **一键回滚**：若新版本出现问题，可一键回滚到任意历史版本，回滚时自动生成新的版本记录，保留回滚日志。

- **上游更新同步**：对于基于父库创建的子库，支持两种同步模式：

  - Follow 模式：父库更新后自动提示子库维护者，可选择同步更新

  - Pinned 模式：子库固定使用父库的某个版本，不受上游更新影响

## 6.9 项目引用、成员权限与协作机制实现

本项目构建了 "引用网络 - 权限矩阵 - 协作流程" 三位一体 的协作体系，支持跨校 / 跨教研组的知识复用与多人协同备课，同时通过严格的权限控制保证课程资产的安全与稳定。

项目引用网络实现

- **引用类型定义**：支持两种引用模式，主基底引用（Primary Reference）用于基于成熟课程库创建派生库，如 "人教版高中数学必修一" 作为主基底，各学校 / 教师可在此基础上创建校本化 / 个性化版本；辅助引用（Secondary Reference）用于补充跨学科资源，如语文课程引用历史课程的背景资料。

- **DAG 约束检测**：引用关系满足有向无环图（DAG）约束，通过数据库触发器与应用层双重检测，禁止循环依赖（如 A 引用 B、B 引用 C、C 引用 A），避免内容更新死循环。

- **引用同步策略**：提供两种同步模式，Follow 模式下父库更新后自动提示子库维护者，可选择一键同步或忽略；Pinned 模式下子库固定使用父库的某个版本，不受上游更新影响，保证教学内容的稳定性。

基于 RBAC 的成员权限模型

- **角色定义**：采用 RBAC（基于角色的访问控制）模型，定义 5 种核心角色：

  - 创建者（Owner）：拥有项目的所有权限，可删除项目、转让所有权、管理成员权限

  - 维护者（Maintainer）：可审核候选变更、发布正式版本、管理协作者权限

  - 协作者（Collaborator）：可编辑内容、提交候选变更，但无法直接发布正式版本

  - 访客（Viewer）：仅可查看正式版本内容，无法编辑

  - 学生（Student）：仅可查看教师分享的特定版本内容，无法访问项目后台

- **权限矩阵**：通过 Prisma Schema 定义角色与权限的映射关系，每个 API 接口都进行权限校验，确保用户只能执行权限范围内的操作。

- **组织级权限**：支持学校 / 机构组织架构，组织管理员可管理组织内所有项目，设置组织级权限模板。

多人协作流程实现

- **候选变更机制**：协作者修改内容后，需提交候选变更（Change Request），填写变更说明，系统自动生成变更对比（diff），高亮显示修改前后的内容差异。

- **审核流程**：维护者查看变更对比，可在线预览修改效果，支持批注修改意见；审核通过则合入正式版本，生成新的 Version；审核驳回则返回给协作者，说明驳回原因。

- **冲突解决**：当多个变更同时修改同一内容时，采用 "先合入优先" 原则，后合入的变更需手动解决冲突，支持选择保留某个版本或合并内容。

- **协作通知**：通过站内信、邮件、微信公众号三种渠道发送协作通知，如变更提交、审核结果、版本更新等，确保协作信息及时传达。

## 6.10 模型路由策略与降级可观测性实现

构建 "智能路由 - 故障降级 - 全链路观测" 的模型管理体系，在保证生成质量的同时，降低成本、提高系统可用性。

多模型智能路由策略

- **路由规则引擎**：自研轻量级路由引擎，基于规则配置实现动态路由，支持以下路由维度：

  - 任务复杂度：简单任务（文本润色、标题生成）使用 Qwen-7B，常规任务使用 Qwen-14B，复杂任务（跨学科教案、长文档解析）使用 Minimax-abab6.5

  - 上下文长度：短上下文（\<8K）使用本地轻量级模型，长上下文（8K-128K）使用支持长窗口的大模型

  - 成本优先级：个人用户优先使用低成本模型，机构用户可选择高质量模型

  - 负载均衡：当某一模型队列积压时，自动将新任务路由到负载较低的模型

- **动态配置**：路由规则通过配置中心动态管理，无需重启服务即可调整，支持 A/B 测试不同模型的效果。

故障降级与高可用保障

- **健康检查**：每 30 秒对所有配置的模型 API 进行健康检查，检测响应时间、错误率、可用性三个指标。

- **故障转移**：当某一模型的健康检查失败（错误率 \> 5% 或响应时间 \> 10s）时，自动将该模型标记为不可用，新任务自动路由到备用模型，同时触发告警通知运维人员。

- **降级策略**：当所有大模型都不可用时，启动降级模式：

  - 暂停复杂生成功能，仅保留简单的文本润色与模板填充

  - 提示用户稍后重试，或提供离线模板下载

  - 保存用户的输入内容，待服务恢复后可继续生成

模型调用可观测性

- **调用统计**：记录每次模型调用的详细信息：模型名称、任务类型、输入 token 数、输出 token 数、响应时间、错误码、费用，按用户、项目、机构维度进行统计。

- **性能监控**：通过 Grafana 搭建模型监控仪表盘，实时展示各模型的 QPS、响应时间、错误率、队列长度、费用趋势。

- **告警机制**：设置多级告警阈值，如错误率 \> 3% 触发警告，\>5% 触发严重告警；响应时间 \> 5s 触发警告，\>10s 触发严重告警，通过邮件、钉钉、企业微信发送告警通知。

- **日志追溯**：每次模型调用都生成唯一的 Trace ID，关联输入、输出、错误信息，支持通过 Trace ID 全链路追溯问题。

## 6.11 队列调度、失败恢复与任务一致性实现

针对课件生成、文档解析等耗时任务，构建 "分级队列 - 智能调度 - 失败重试 - 一致性保障" 的异步任务体系，保证任务高可靠执行。

分级队列与智能调度

- **队列分级**：将任务分为三个优先级队列：

  - 高优先级队列（High）：实时生成任务，如教师当前备课的课件生成，响应时间要求 \< 30s

  - 中优先级队列（Medium）：批量处理任务，如学期课件批量生成，响应时间要求 \< 5min

  - 低优先级队列（Low）：后台统计任务，如使用数据统计、日志分析，响应时间要求 \< 1h

- **调度策略**：

  - 优先级调度：高优先级队列任务优先执行，只有高优先级队列为空时才处理中 / 低优先级队列

  - 公平调度：同一优先级队列内采用 FIFO（先进先出），同时保证每个用户的任务公平分配资源，避免某一用户的大量任务占用全部资源

  - 动态扩缩容：根据队列长度自动调整 Worker 数量，队列积压时自动启动新 Worker，队列空闲时自动关闭多余 Worker

失败恢复与重试机制

- **自动重试**：任务失败时自动重试，根据错误类型设置不同的重试策略：

  - 临时性错误（网络超时、模型限流）：指数退避重试，初始间隔 1s，最大间隔 30s，最多重试 5 次

  - 业务错误（内容违规、参数错误）：不重试，直接标记为失败，返回错误信息给用户

  - 系统错误（数据库故障、服务崩溃）：等待服务恢复后自动重试

- **死信队列**：重试超过最大次数的任务移入死信队列（Dead Letter Queue），触发告警通知运维人员人工处理，支持手动重试或放弃任务。

- **幂等性保证**：所有任务都设计为幂等，相同的任务重复执行不会产生副作用，通过任务 ID 与状态机保证，避免重复生成内容。

任务一致性保障

- **状态持久化**：任务状态（等待中、执行中、成功、失败）实时存储在 Redis 中，同时异步写入 MySQL 持久化，服务重启后可恢复任务状态。

- **事务补偿**：对于多步骤任务（如解析 - 分块 - 索引），采用 Saga 模式实现分布式事务，某一步骤失败时自动执行补偿操作（如删除已解析的分块、回滚索引），保证数据一致性。

- **进度追踪**：实时更新任务进度（0%-100%），前端通过轮询或 WebSocket 获取进度，同时记录每个步骤的执行日志，支持问题排查。

- **最终一致性**：对于非关键路径任务，采用最终一致性模型，允许短暂的不一致，但保证最终状态一致，通过定期的对账任务修复不一致数据。

## 6.12 系统设置中心与运行时配置治理实现

构建 "分层设置 - 动态配置 - 灰度发布 - 权限控制" 的配置治理体系，支持个人、组织、系统三级设置，同时保证配置变更的安全与稳定。

分层设置中心

- **个人设置**：教师可配置个人偏好，如默认模板、生成风格（简洁 / 详细）、通知偏好（站内信 / 邮件 / 微信）、界面主题（浅色 / 深色），设置存储在用户表的 JSONB 字段中，支持跨设备同步。

- **组织设置**：学校 / 机构管理员可配置组织级设置，如统一品牌模板、默认学科 / 学段、成员权限模板、数据保留策略，组织设置覆盖个人设置，保证全校风格统一。

- **系统设置**：运维人员可配置系统级设置，如模型路由规则、队列参数、存储路径、告警阈值，系统设置通过配置中心管理，支持动态更新。

运行时配置治理

- **配置中心**：采用 Apollo 或 Nacos 作为配置中心，实现配置的集中管理、动态更新、版本控制、灰度发布。

- **配置版本管理**：每次配置变更都生成新的版本，记录变更人、变更时间、变更内容，支持一键回滚到任意历史版本。

- **灰度发布**：配置变更支持灰度发布，先发布到 10% 的用户 / 服务器，观察无问题后再全量发布，降低配置变更风险。

- **配置校验**：所有配置变更都进行语法校验与逻辑校验，校验不通过则拒绝变更，避免错误配置导致系统故障。

配置权限与审计

- **配置权限**：不同层级的配置有不同的权限要求，个人设置仅用户自己可修改，组织设置需组织管理员，系统设置需运维人员，通过 RBAC 严格控制。

- **配置审计**：记录所有配置变更的详细日志：变更人、变更时间、变更前值、变更后值、变更原因，支持审计追溯。

- **配置同步**：配置变更后实时推送到所有服务器，服务器收到变更后自动热加载，无需重启服务，保证配置一致性。

## 6.13 健康检查、日志链路与运维可观测性实现

构建 "全链路健康检查 - 结构化日志 - 可视化监控 - 智能告警" 的运维可观测性体系，保证系统稳定运行，问题快速定位。

全链路健康检查

- **服务健康检查**：每个服务都暴露 /health 端点，检查服务自身状态、依赖服务状态，返回健康状态（健康 / 亚健康 / 不健康）与详细信息。

- **基础设施健康检查**：

  - 数据库：检查连接状态、主从延迟、慢查询数量

  - 缓存：检查连接状态、内存使用率、命中率

  - 队列：检查连接状态、队列长度、Worker 数量

  - 存储：检查磁盘使用率、读写速度

  - 模型 API：检查响应时间、错误率、可用性

- **健康检查调度**：使用 Prometheus 每 30 秒采集一次健康检查数据，通过 Grafana 展示健康状态仪表盘，设置告警阈值。

结构化日志与链路追踪

- **结构化日志**：所有日志都采用 JSON 格式，包含 Trace ID、Span ID、时间、服务名、日志级别、用户 ID、项目 ID、请求参数、响应结果、错误信息等字段，便于日志分析与查询。

- **Trace ID 全链路追踪**：每个用户请求都生成唯一的 Trace ID，贯穿前端→后端→数据库→缓存→队列→模型 API，通过 Trace ID 可查询整个请求链路的所有日志，快速定位问题。

- **日志收集与存储**：使用 Filebeat 采集日志，Logstash 清洗日志，Elasticsearch 存储日志，Kibana 查询与可视化日志，搭建 ELK 日志平台。

- **日志分级与保留**：日志分为 DEBUG、INFO、WARN、ERROR 四个级别，DEBUG 日志保留 7 天，INFO/WARN 保留 30 天，ERROR 保留 90 天，定期清理过期日志。

运维监控与智能告警

- **监控仪表盘**：通过 Grafana 搭建多维度监控仪表盘：

  - 系统概览：QPS、响应时间、错误率、在线用户数

  - 服务监控：各服务的 CPU、内存、磁盘、网络使用率

  - 数据库监控：连接数、慢查询、主从延迟、事务数

  - 模型监控：调用次数、响应时间、错误率、费用

  - 任务监控：队列长度、执行成功率、平均执行时间

- **智能告警**：

  - 告警规则：设置多级告警阈值，如 QPS 下降 20% 触发警告，下降 50% 触发严重告警；错误率 \> 3% 触发警告，\>5% 触发严重告警

  - 告警渠道：支持邮件、钉钉、企业微信、短信四种告警渠道，根据告警级别选择不同渠道

  - 告警收敛：避免重复告警，同一问题 5 分钟内只发送一次告警

  - 告警自愈：对于可自动恢复的问题（如队列积压），触发自动扩缩容；对于不可自动恢复的问题，通知运维人员人工处理

## 6.14 微服务边界与跨服务编排实现

当用户量增长到一定规模时，系统可平滑演进为微服务架构，本项目提前规划了 "服务边界 - API 契约 - 跨服务编排 - 数据一致性" 的微服务架构方案。

微服务边界划分

- **服务拆分原则**：按业务域拆分为 7 个核心微服务，每个服务独立部署、独立扩展、独立运维：

  - 用户服务（User Service）：用户注册、登录、权限管理、组织管理

  - 课程服务（Course Service）：课程库管理、版本管理、引用管理、协作管理

  - 知识库服务（Knowledge Service）：文档解析、分块、向量化、检索、索引管理

  - 生成服务（Generation Service）：意图理解、对话编排、大纲生成、内容生成、模型路由

  - 任务服务（Task Service）：异步任务调度、队列管理、失败重试、进度追踪

  - 导出服务（Export Service）：课件 / 教案导出、模板管理、格式转换

  - 通知服务（Notification Service）：站内信、邮件、微信通知、通知模板管理

- **边界定义**：每个服务有明确的职责边界，通过 API 契约对外暴露能力，服务内部数据私有，不直接访问其他服务的数据库。

API 契约与服务治理

- **OpenAPI 契约**：每个服务都定义 OpenAPI 3.0 规范的 API 文档，通过 API 网关统一管理，支持自动生成客户端 SDK。

- **API 网关**：使用 Kong 或 APISix 作为 API 网关，负责请求路由、负载均衡、限流熔断、身份认证、日志记录。

- **服务发现**：使用 Consul 或 etcd 实现服务注册与发现，服务启动时自动注册到注册中心，调用方通过注册中心发现服务地址。

- **限流熔断**：在 API 网关与服务间实现限流熔断，保护后端服务不被过载，当某一服务错误率过高时自动熔断，快速返回失败。

跨服务编排与数据一致性

- **事件驱动架构**：采用事件驱动架构实现跨服务协作，服务之间通过消息队列（Kafka/RabbitMQ）传递事件，如 "课程发布" 事件触发 "通知服务" 发送通知、"统计服务" 更新数据。

- **Saga 分布式事务**：对于需要跨服务保证一致性的操作（如 "创建课程 - 初始化知识库 - 发送通知"），采用 Saga 模式实现分布式事务，定义正向操作与补偿操作，某一步骤失败时自动执行补偿。

- **最终一致性**：对于非关键路径操作，采用最终一致性模型，通过事件重试与对账任务保证最终一致。

- **数据隔离**：每个服务拥有独立的数据库，服务之间不直接访问数据库，只能通过 API 交互，保证数据安全与服务解耦。

## 6.15 本地开发到容器化部署的一致性实现

构建 "本地 Docker Compose - 镜像构建 - CI/CD - 多环境部署" 的全链路一致性体系，保证本地开发环境、测试环境、生产环境完全一致，避免 "本地能跑、线上报错" 的问题。

本地开发环境

- **Docker Compose 一键启动**：提供 docker-compose.dev.yml 文件，一键启动所有依赖服务（PostgreSQL、Redis、Qdrant、MinIO），无需手动安装配置。

- **热重载**：前端使用 Next.js 的 Fast Refresh，后端使用 uvicorn 的 --reload，代码修改后自动热重载，无需重启服务，提升开发效率。

- **环境变量管理**：使用.env 文件管理环境变量，提供.env.example 模板，开发者复制后填写自己的配置即可，敏感信息（如 API 密钥）不提交到代码仓库。

- **本地数据持久化**：通过 Docker Volume 持久化本地开发数据，重启容器后数据不丢失。

镜像构建与多环境配置

- **多阶段镜像构建**：使用 Dockerfile 的多阶段构建，减小镜像体积：

  - 前端：先使用 Node 镜像构建，再将构建产物复制到 Nginx 镜像

  - 后端：先使用 Python 镜像构建依赖，再将代码与依赖复制到 slim Python 镜像

- **镜像标签策略**：采用 "语义化版本 + Git commit hash" 的标签策略，如 v1.0.0-abc123，便于追溯镜像对应的代码版本。

- **多环境配置**：通过环境变量区分开发、测试、生产环境，不同环境使用不同的配置（如数据库地址、模型 API 密钥、日志级别），配置不打包到镜像中，通过 Kubernetes ConfigMap/Secret 或 Docker Compose 环境变量注入。

CI/CD 流水线

- **代码提交触发**：使用 GitHub Actions 或 GitLab CI 搭建 CI/CD 流水线，代码提交到 main 分支时自动触发。

- **流水线步骤**：

  1.  代码检查：运行 ESLint、Pylint 进行代码风格检查

  2.  单元测试：运行前端与后端单元测试，覆盖率要求 \> 80%

  3.  镜像构建：构建前端与后端镜像，推送到镜像仓库

  4.  测试环境部署：自动部署到测试环境，运行集成测试

  5.  生产环境部署：测试通过后，手动确认部署到生产环境

- **蓝绿部署**：生产环境采用蓝绿部署策略，先部署新版本到绿环境，验证无问题后再切换流量到绿环境，旧版本蓝环境保留作为回滚备份。

一致性保证

- **相同镜像**：开发、测试、生产环境使用相同的 Docker 镜像，避免环境差异导致的问题。

- **配置管理**：通过配置中心统一管理多环境配置，配置变更经过测试后再同步到生产环境。

- **测试覆盖**：单元测试、集成测试、E2E 测试覆盖核心功能，保证代码变更不破坏现有功能。

- **基础设施即代码**：使用 Terraform 或 Docker Compose 定义基础设施，基础设施变更也经过代码审查与测试，保证基础设施一致性。
