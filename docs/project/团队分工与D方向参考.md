# 团队分工与 D 方向参考

> 更新时间：2026-03-23
> 状态：当前生效
> 适用阶段：初审准备阶段

本文档是当前阶段团队分工的默认主文档。

它同时吸收了此前“下一阶段分工方案”中的执行约束，用来统一回答三类问题：

- 当前阶段到底优先做什么
- ABCD 各自默认负责什么
- D 可以围绕哪些方向做技术深挖，并如何与 C 协同

如果后续存在分工冲突、职责重复或材料描述不一致，默认以本文档为准。

## 1. 当前阶段目标

当前阶段优先目标是通过初审。

因此，团队工作要围绕以下三点收束：

1. 主链路稳定
2. 竞赛材料对题、简洁、可信
3. 技术亮点有证据，而不是只有概念

当前默认主链路是：

`资料上传 -> 检索/RAG -> 对话理解 -> 会话化生成 -> 大纲确认 -> PPT/Word 生成 -> 预览/修改 -> 导出物进入项目空间`

## 2. 当前阶段总原则

当前阶段按以下原则执行：

1. 以“先过初审”为第一目标。
2. 以题目要求为主，不以系统扩张为主。
3. 以主链路稳定优先于边缘功能扩展。
4. 以可展示、可截图、可写进材料的成果优先。
5. 以低冲突、低返工优先于高并行。

## 3. 当前阶段重点

当前阶段只围绕四件事推进：

### 3.1 题目对齐

要明确回答以下问题：

- 我们是否满足“理解意图”
- 我们是否满足“融合多模态参考”
- 我们是否满足“生成 PPT/Word”
- 我们是否满足“预览、修改、再生成”

### 3.2 主链路稳定

主链路统一为：

`上传资料 -> 对话澄清 -> 生成大纲/内容 -> 生成 PPT/Word -> 预览 -> 修改 -> 导出`

这条链路必须稳定、连贯、可演示。

### 3.3 亮点收束

当前阶段只保留三类对外亮点：

1. 多模态资料融合
2. 会话式生成与迭代优化闭环
3. 可验证的技术质量能力，如检索、引用、记忆一致性

### 3.4 材料准备

初审所需材料应围绕以下内容准备：

- 题目要求对照表
- 主链路流程图
- 系统架构图
- 页面截图和结果样例
- 技术亮点证据

### 3.5 当前阶段直接关联的三项系统工作

除上面的总目标外，当前阶段还有三项已经进入正式主线语义的系统工作，需要明确纳入协作判断：

1. **会话标题与 Run 历史**
   - 这不是装饰性功能，而是会话语义、运行追踪、历史回看和材料截图的一部分。
   - 需要保证标题语义化、Run 历史可追踪、前后端字段一致、用户能理解“当前这次运行是什么”。

2. **系统级业务配置页**
   - 这是业务配置页，不是在线编辑 `.env`。
   - 目标是把常用业务配置收口成后端可持久化、前端可理解、系统可复用的正式能力。

3. **结构化生成流与 PPT 单页局部修改**
   - 这不是可有可无的 refine 按钮增强，而是“生成后可定向修改”的核心竞争力之一。
   - 目标是让用户能明确表达“这一页哪里不好、想怎么改”，并形成稳定的单页修改闭环。

## 4. 角色分工

### A：总控与拍板

A 负责：

- 项目主叙事
- 初审阶段的取舍
- 核心语义和契约语言
- 主链路优先级
- 题目要求对照关系
- 决定材料写什么、不写什么
- 决定哪些能力进入初审主展示、哪些能力暂时降级或隐藏
- C / D 出现冲突时的最终拍板

A 不默认负责：

- C 的主线实现
- D 的技术专题实现
- B 的页面细节补位
- E 的文档代写

A 的阶段产出应包括：

- 一版题目要求对照表
- 一版项目简介结构
- 一版项目亮点清单
- 一版初审材料目录
- 一版“进入展示 / 降级保留 / 暂不展示”的能力清单

### B：前端主力与展示可见层

B 负责：

- 工作台前端
- `Studio / Chat / Sources / Library` 的主链路体验
- 上传、对话、生成、预览、下载这条链路的展示完整性
- 初审需要的界面状态、截图和录屏路径
- 页面层面的演示路径优化
- 与后端业务配置页对应的前端配置界面
- 生成过程中的实时事件体验和等待感优化

B 不负责：

- 倒推后端语义
- 自己定义接口语义
- 长期消化 C / D 未解决的冲突

B 的目标是：

- 让评委一眼看出“项目已经做成了”
- 把后端真实语义准确翻译成用户可理解的前端状态和操作

### C：主线业务后端与主线架构维护

C 负责：

- 主线业务后端
- `session -> outline -> generate -> artifact -> download` 这条链路
- 上传、生成、预览、导出、下载等主链路稳定
- 会话、状态、事件、产物语义一致性
- 项目空间里的导出物、引用、状态和业务语义
- 队列、任务执行和主线接口稳定
- 后端业务配置页和系统设置接口
- 单页定向修改能力落地
- 主线相关的后端架构维护
- 与 B 协商并共同收口系统稳定性与用户面语义清晰度
- 与前端、设计共同收口 Studio 卡片的最终展示质量与交互方案
- 会话标题与 Run 历史相关的主线语义落地
- 系统级业务配置页的后端能力与语义维护
- 结构化生成流与 PPT 单页局部修改能力落地

C 的工作重点是：

- 稳定
- 清晰
- 可交付
- 语义一致

C 当前阶段不优先承担：

- D 的技术专题
- 没有明确收益的探索性改动
- 只扩数量、不提完成度的边缘功能

对 C 的补充要求：

- 不能只写接口，不跑系统。
- 需要自己经常运行系统，自己走主链路。
- 在上传、会话、生成、预览、下载、配置页这些主线流程里主动找问题。
- 如果问题属于自己负责范围，默认自己修。
- 如果问题不属于自己负责范围，及时同步给对应的人。

#### C 的当前新增专项

1. **PPT 生成主链路质量提升**
   - 目标不是只保证 PPT 能生成，而是让结果具备更强展示力，向 Gamma 一类产品的视觉完成度靠近。
   - 重点包括：
     - 更好的页面排版与层次
     - 更稳定的图文混排
     - 合理接入图片、示意图、配图素材
     - 数学公式、符号与结构化内容的可读呈现
     - 更强的页面风格统一性与封面、目录、内容页节奏感
   - 该项属于主链路质量问题，优先级高于继续扩更多边缘能力。

2. **PPT 生成时默认联动教案生成**
   - 默认把“生成 PPT + 自动生成配套教案”作为主链路标准能力来设计。
   - 前端层面保留用户可见选择，但系统默认方案应倾向联动生成。
   - 后端与前端语义、输出物、产物命名和下载路径要保持一致，不能出现 PPT 已完成但教案仍停留在弱占位状态。

3. **Studio 卡片与前端、设计联合收口**
   - 其他卡片能力不再只按后端协议能不能跑来判断完成度，而要与前端、设计共同对齐：
     - 哪些卡片保留为正式展示能力
     - 哪些卡片需要补交互语义
     - 哪些卡片需要重新整理视觉与输出呈现
   - 对初审价值不高、质量又明显不足的卡片，不优先追求“数量”，而优先追求“少数卡片做得像成品”。

4. **与 B 联合保障系统稳定与语义清晰**
   - 当前阶段系统“不稳定、语义不清”本身已经是主问题之一，C 需要与 B 明确协商并共同推进修复。
   - C 负责后端主链路、状态、事件、返回结构、产物语义的稳定与一致。
   - B 负责把这些后端语义在前端展示层、工作流路径、用户提示语里表达清楚。

5. **会话标题与 Run 历史**
   - 需要保证标题生成、标题来源、Run 历史、当前运行状态这几套语义在后端、OpenAPI、前端展示层一致。
   - 该能力直接影响主链路可理解性、历史追踪和材料展示质量，不应视为边缘补充。

6. **系统级业务配置页**
   - 当前阶段由 C 主责后端能力和持久化语义，B 对接前端界面。
   - 重点是把“经常要改、会影响系统行为的业务配置”收成正式页面和正式接口，而不是继续散落在部署配置里。

7. **结构化生成流与 PPT 单页局部修改**
   - 当前已有一定基础，但还要继续补足“带用户明确需求输入的单页定向修改”。
   - 目标不是只有一个泛化 refine，而是让单页修改真正进入正式主链路。

8. **资料解析链升级与 MinerU 落地评估**
   - 当前默认资料解析仍以 `local` 方案为主，MinerU 不作为现阶段默认主链。
   - C 负责继续评估和推进资料解析链升级，包括但不限于：
     - MinerU 在 Docker 环境中的真实落地条件
     - 解析质量、运行成本、依赖复杂度和系统稳定性之间的取舍
     - 是否适合作为后续正式主链，或应保持为可选重型解析链
   - 这项工作先作为后端专题推进，待运行成本、稳定性和可维护性明确后，再决定是否进入默认能力。

### D：技术深挖后端与系统级智能质量提升

D 不作为第二个主线后端使用。

D 负责：

- 检索与重排
- 对话记忆与引用一致性
- 模型路由与 provider 治理
- 上传到索引链路的质量治理
- 评测、基线和质量门禁
- 系统级智能能力提升
- 各条链路的生成质量提升
- 与 C 协商智能能力优化与主链路落地之间的边界

D 的职责不是接零散业务，而是围绕专题做深，把“智能效果”和“质量提升”做成可验证证据。

D 当前阶段建议只做以下方向中的一到两个作为主攻：

1. 检索与重排质量
2. 对话记忆与引用一致性

如有余力，再处理：

3. 上传链路降级信息统一
4. 主链路生成质量提升

D 当前阶段不优先承担：

- 新方向发散
- 零散业务补洞
- 没有基线的主观优化

D 的目标是：

- 把“技术深度”转化成“初审材料里的可信证据”
- 把“模型与智能能力”转化成“真实可感知的生成质量提升”

#### D 的当前新增专项

1. **系统级智能能力提升**
   - 不只关注单点模型效果，而是关注整套系统在真实链路中的智能表现。
   - 范围包括但不限于：
     - 对话理解质量
     - 检索召回与重排质量
     - 大纲生成质量
     - PPT/Word 内容生成质量
     - Studio 各卡片的内容生成质量

2. **各链路生成质量提升**
   - D 负责识别哪些问题属于“模型能力 / prompt / 检索 / 生成策略”层面的质量问题。
   - C 负责把这些质量改进稳定落到主链路交付中。
   - 两者默认通过协商划清边界：
     - D 更偏“智能效果与质量提升”
     - C 更偏“主链路稳定与产品落地”

3. **智能能力优化与主链路交付协同**
   - 涉及主链路质量的大项优化，D 不单独拍板落地方式，需与 C 协商。
   - 目标不是让智能优化脱离产品链路，而是让质量提升最终进入可演示、可交付、可写材料的正式能力。

### E：竞赛文档负责人

E 不做开发。

E 负责：

- `docs/competition/` 的正文整理
- 题目要求对照
- 术语统一
- 图表、截图、材料结构整理
- 把 A、C、D 给出的事实整理成正式材料

E 不负责：

- 自己定义实现
- 自己发明技术叙事
- 脱离代码写材料

## 5. 默认协作规则

### 5.1 默认归属

- 主线后端问题默认给 C
- 技术专题默认给 D
- 前端展示和工作台默认给 B
- 语义、契约和取舍默认由 A 决定
- 竞赛文档默认由 E 收口

### 5.2 冲突处理

如果一个问题同时涉及主线业务和技术专题，不要让 C 和 D 同时改同一层。

默认拆成两段：

1. D 先研究、给样本、给 baseline、给建议
2. C 或 D 再接入主线，谁接由 A 拍板

### 5.3 这阶段不追求极限并发

当前更重要的是低冲突、低返工，而不是强行提高并发度。

能拆开就并行，拆不开就串行。

### 5.4 B、C、D 都必须自己跑系统

B、C、D 都需要：

- 自己经常运行系统
- 自己经常走主链路
- 自己在真实界面和真实流程里发现问题

处理规则如下：

1. 如果问题在自己负责范围内，默认自己修。
2. 如果问题不在自己负责范围内，及时同步给对应的人。
3. 不允许长期只看代码、不跑系统。

## 6. D 的方向参考

### 6.1 检索与重排

这是当前最重要的方向之一。

当前系统已经具备：

- 本地项目检索
- 会话检索
- 已激活引用空间检索

但排序和重排仍有继续提升空间。

建议关注：

- `local_session / local_project / reference_base / reference_auxiliary` 的排序关系
- rerank 策略是否过轻
- query 类型差异对检索结果的影响
- 无命中时的解释质量

主要代码入口：

- `backend/services/rag_service/`
- `backend/routers/chat/message_flow.py`

### 6.2 对话记忆与引用一致性

这是最容易影响“像不像真的教学助手”的方向。

建议关注：

- 多轮对话中的会话隔离
- 资料命中与回答内容是否一致
- citation 和实际内容是否一致
- 无命中时的提示是否自然

主要代码入口：

- `backend/routers/chat/`
- `backend/services/prompt_service/`
- `backend/eval/dialogue_memory_audit.py`

### 6.3 模型路由与 provider 治理

当前仓库里已经有路由和 provider 相关基础设施，但还可以做得更稳。

建议关注：

- 模型选择策略
- fallback 是否透明
- provider 失败后的分类和记录
- 质量、延迟、成本之间的平衡

主要代码入口：

- `backend/services/ai/`
- `backend/eval/router_quality_audit.py`

### 6.4 上传到索引链路的质量治理

当前上传链路已经有 parse、chunk、embedding、index 的基础，但降级和质量信息仍有整理空间。

建议关注：

- parse fallback 是否清楚
- embedding 路由是否正确
- 各阶段 timing 是否可解释
- 降级是否会变成“慢但说不清”

主要代码入口：

- `backend/services/media/`
- `backend/services/file_parser/`

### 6.5 评测、基线与质量门禁

D 的价值不只是“想到可以优化什么”，而是把优化结果做成可复现、可对比、可展示的基线。

建议关注：

- 当前 eval 是否真正贴近主链路
- 评测样本是否覆盖真实问题
- 质量改善是否有前后对比
- 哪些结果适合写进初审材料

主要代码入口：

- `backend/eval/`
- `backend/scripts/`

## 7. 本阶段不建议优先做的事

以下内容当前阶段不建议优先推进：

- 大规模扩展新卡片能力
- 新的抽象系统设计
- 与初审无直接关系的深度重构
- 无法形成展示价值的底层优化
- 没有明确证据产出的开放式研究

## 8. 阶段完成标准

如果当前阶段执行有效，应达到以下结果：

1. 有一份清晰的题目要求对照表。
2. 有一条稳定的主链路可以完整演示。
3. 有一套页面截图和结果样例可直接进入材料。
4. 有 1 至 2 项技术亮点具备可量化证据。
5. ABCDE 各自知道本阶段的主要责任，不再反复重定义分工。

## 9. 一句话总结

当前阶段不是继续把系统做大，而是把已有能力收束成一个强对题、强完成度、强证据的初审项目。
