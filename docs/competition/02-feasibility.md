# 2. 可行性分析

## 2.1 可行性总判断

`Spectra` 的可行性，已经不再是“未来也许能做到”的愿景判断，而是“当前是否值得认真评估、值得推进试点、值得进入采购沟通”的现实判断。答案已经开始明确：它成立。

从系统现实看，`Spectra` 已经同时具备以下成立条件：

| 可行性维度 | 当前判断 | 商业意义 |
| --- | --- | --- |
| 技术可行性 | 成立 | 主链真实存在，关键能力已有正式证据支撑 |
| 工程可行性 | 成立 | 控制平面与六个正式能力层已形成协同体系 |
| 产品可行性 | 成立 | `Studio` 和真实 preview/refine 约束已形成可展示产品面 |
| 经济可行性 | 成立 | 不只提高效率，还能沉淀课程知识资产 |
| 应用可行性 | 成立 | 备课、教研、课程资源治理都是高频真实场景 |
| 风险可控性 | 基本可控 | 系统关键风险已有清晰控制机制 |

因此，`Spectra` 已经具备试点、交付和平台化扩展基础。

## 2.2 技术可行性

### 2.2.1 主链已经形成真实闭环

`Spectra` 当前最重要的技术事实，是已经形成一条完整而清晰的主链：

`资料进入 -> 检索增强与证据组织 -> Diego 生成 -> Pagevra 预览与导出 -> Ourograph 状态绑定 -> 后续复用与演化`

这条主链说明系统已经围绕课程知识空间形成统一结构。每个环节都服务于同一件事：把教学内容生产、标准交付和正式知识状态沉淀组织在一起。

### 2.2.2 生成过程具备生产级可管理性

系统的正式生成过程围绕以下阶段展开：

`Session 建立 -> outline 形成 -> outline 确认 -> generation -> preview/refine -> export -> artifact bind`

这组能力直接带来四个结果：

- 生成过程是阶段化的，而不是神秘的；
- 用户可以参与关键判断，而不是只能被动接受结果；
- 结果可以被修正、重做和绑定，而不是导出即结束；
- 交付物可以回到课程空间，成为长期资产的一部分。

这类可管理性，是系统从“能生成”升级为“适合真实教学内容生产”的关键条件。

### 2.2.3 检索增强已经具备正式证据

检索增强决定了系统是否只是“接入了 RAG”，还是具备真正可依赖的证据组织能力。当前 `Stratumind` 已经有正式评估集结果支撑。

结论级结果表明：

- 在 60 题正式集上，advanced 相比 dense-only，Hit@3 从 `38.3%` 提升至 `80.8%`，Quality 从 `41.5%` 提升至 `79.3%`；
- 在 105 题正式集上，advanced 相比 dense-only，Hit@3 从 `48.6%` 提升至 `83.8%`，Quality 从 `49.2%` 提升至 `81.3%`。

这说明系统在证据命中、证据排序和结果质量上已经形成可量化提升，而不是停留在“看起来接上了检索”的层面。详细指标和测试设计由第 6 章承接，这里只保留足以支撑成立性的结论级证据。

### 2.2.4 多模态资料进入具备持续扩展基础

很多系统看起来支持上传多种资料，但底层只是零散解析逻辑的堆叠，难以长期维护。`Spectra` 当前已经将资料进入、解析和后续交付放在统一链路中承接，多模态能力已经成为平台长期能力的一部分。

这对商业成立性很重要。它决定系统未来可以持续吸收 PDF、文档、图片、视频等课程资料，而不会在规模化场景中迅速失控。

### 2.2.5 正式知识状态已经成立

如果系统只有生成和导出，而没有正式知识状态，它最终仍然只是内容工具。`Spectra` 当前已经把 `Project / Reference / Version / Artifact / CandidateChange / Member` 作为正式对象语言，意味着课程空间不再只是工作上下文，而是长期可治理、可演化、可复用的知识资产基础。

这正是系统从“一次性工具”升级为“课程知识空间系统”的技术前提，也是第 8 章平台想象成立的基础。

## 2.3 工程可行性

### 2.3.1 已形成协同体系

`Spectra` 当前的工程形态，已经形成了：

- 前端工作台与 `Studio` 多模态内容工坊；
- `Spectra Backend` 控制平面；
- 六个正式能力层。

这三部分共同构成一套可持续演进的工程体系。用户看到的是统一产品体验，系统内部则保持生成、渲染、知识状态、上传解析、检索和身份组织的能力边界。

这种结构对交付的意义非常直接：

- 更容易定位问题；
- 更容易局部增强；
- 更不容易因为一个环节迭代而污染全局；
- 更适合长期试点、机构交付和后续平台化扩展。

### 2.3.2 主仓职责已经收束为控制平面

当前主仓主要承担的是：

- 会话与事件生命周期；
- 任务编排；
- 响应整形；
- artifact binding；
- 下载契约；
- 对正式能力层的反腐层适配。

这说明 `Spectra` 已经不再依赖“大后端什么都做”的方式维持系统成立，而是在控制平面层明确了自己的长期职责。这样的结构更接近可交付系统，而不是会随着功能增多而迅速塌陷的实验性工程。

### 2.3.3 关键平台总栈已经形成承重结构

`Spectra` 的工程可行性还来自一个很容易被忽视的事实：它已经形成了一套有明确承重关系的平台总栈。

- 前端工作台：`Next.js + React + TypeScript`
- 控制平面：`FastAPI + Pydantic + Prisma async`
- 数据与运行底座：`PostgreSQL + Redis + RQ + Qdrant + Docker Compose`
- 正式能力层：围绕 generation、render/export、formal state、ingest、retrieval 和 identity 形成异构运行时协同

这组总栈的意义，在于它说明项目已经具备真实系统所需要的前端工作台、控制平面、数据底座、任务执行和异构能力层协同结构。当前系统的工程现实，已经明显高于普通 AI 工具壳。

这里的平台总栈只是第 4 章那张统一系统矩阵的前置判断。它先说明 `Spectra` 已经具备工作台、控制平面、数据底座和异构能力层协同结构；更完整的工程分量，则由第 4 章继续展开。`Spectra` 的工程可行性成立的，不只是“能跑起来”，而是“已经形成了有分工、有深度、有长期演化空间的系统级工程底座”。

### 2.3.4 产品面已经能够承接真实场景

`Studio` 当前已经承接多模态内容工坊的真实产品面。`PPT`、`Word` 教案、导图、动画、互动内容、说课助手和学情预演都围绕同一课程空间组织，并遵守真实 preview/refine 约束。

这说明系统不仅在架构层面成立，也在产品体验层面具备了“可展示、可解释、可试点”的基础。

## 2.4 经济可行性

### 2.4.1 价值不止于节省时间

如果一个系统只能更快生成一份课件，它的商业价值通常会被压缩到效率工具层。`Spectra` 的成立性更强，是因为它的价值不止体现在时间节省，还体现在课程知识资产的沉淀、治理和放大。

对不同客户，这种价值分别表现为：

- 对教师：减少资料整理、检索、排版和反复修改成本；
- 对教研组：提高课程资源复用和共创效率；
- 对学校和机构：把分散文件升级为可治理、可持续更新的课程知识资产网络。

客户购买的是一种长期资产能力。

### 2.4.2 商业机会具备放大空间

`Spectra` 的商业成立性，还来自它不是一个封顶很低的单点工具。系统已经具备从教师效率工具走向课程知识资产平台的自然路径：

- 短期可以从高频备课场景切入；
- 中期可以扩展到课程资产复用、教研协作和机构治理；
- 长期可以承接课程库授权、学习者增值服务和平台化内容能力。

这类放大路径，让项目值得进入更严肃的商业评估，而不是只被当作“效果不错的 AI 功能”。

### 2.4.3 试点与投入路径可控

项目当前具备分阶段落地条件：

1. 先服务教师个人和小团队；
2. 再扩展到教研组与学校试点；
3. 再进入机构治理和平台化能力建设。

项目不需要在第一天就承担极重的组织改造成本，而可以沿着真实价值场景逐步推进。对采购和试点来说，这是一种更可接受、也更可执行的投入路径。

## 2.5 应用可行性

### 2.5.1 场景高频且容易理解

备课、资料整理、课件迭代、教案生成、课程复用和资源治理，都是高频、持续、明确存在的真实场景。用户并不需要被教育“为什么要用它”，而是很容易在真实工作中理解系统的价值位置。

这类场景基础，使系统更适合进入试点，而不是只能停留在展示层。

### 2.5.2 系统覆盖的是长链价值，而非局部优化

`Spectra` 覆盖的是完整教学内容链路：

- 内容进入；
- 内容理解；
- 内容生成；
- 内容交付；
- 内容沉淀；
- 内容复用；
- 内容治理与扩展。

这使它不再只是解决某个局部小痛点，而是在组织一整条内容价值链。对于学校、机构和平台客户而言，这种完整性比单点功能更有采购意义。

### 2.5.3 自然具备平台化扩展路径

从教师个人使用，到教研组协同，再到学校和机构级治理，系统具有非常自然的扩展路径，也具备逐步走向平台化应用的可能。

这种扩展性，是可行性分析里最值得强调的一点。因为它说明 `Spectra` 不只是“现在能用”，而是“现在就值得进入长期建设和试点放大”。

## 2.6 风险可控性

`Spectra` 的风险已经进入可识别、可分层、可治理的状态。生成质量、检索质量、多服务协同、数据安全、核心创新披露和商业推进，都已有明确控制路径。

这对可行性的意义在于：项目是在知道风险所在的前提下，仍然具备推进试点和商业化沟通的条件。详细风险治理框架由第 9 章展开，这里只保留结论级桥接。

## 2.7 本章结论

综合来看，`Spectra` 当前已经具备以下成立基础：

- 技术上，主链真实存在，关键能力可验证，正式能力边界清晰；
- 工程上，控制平面与六个正式能力层已形成协同体系；
- 产品上，`Studio` 和真实 preview/refine 约束已经成立；
- 经济上，系统价值不止于效率工具，而具备课程资产化和平台化机会；
- 应用上，场景高频、链路完整、试点路径明确；
- 风险上，关键风险已进入可治理状态。

因此，`Spectra` 现在已经是一套值得被认真评估、值得进入试点、也值得进入采购沟通的课程知识空间解决方案。第 2 章给出的结论是：它已经跨过成立门槛，接下来讨论的是如何放大价值。
