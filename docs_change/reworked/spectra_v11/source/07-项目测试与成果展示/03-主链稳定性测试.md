## 核心业务主链稳定性测试

本节重点回答“系统是否已具备可连续运行、可追踪复查、可隔离异常的端到端主链稳定性”这一评审问题。围绕“数据接入 - 内容生成 - 结果预览 - 标准导出 - 成果保存”核心业务链路开展的端到端稳定性验证表明，当前主链已具备较高的连续运行稳定性和工程可解释性。

### 主链验证内容

本轮主链验证聚焦以下五类问题：

- 生成模块是否承接课件主生成职责；
- `outline` 与确认是否形成可管理的阶段控制；
- 生成结果是否稳定进入预览与修订链路；
- 导出结果是否继续进入成果保存与绑定流程；
- 前端展示是否严格等待真实结果对象返回。

该组验证覆盖的是完整连续业务过程，而非单一功能接口。只有当数据接入、会话推进、内容生成、结果预览、标准导出和成果保存能够连续衔接时，核心业务主链的工程完整性才可成立。

### 20 主题长时负载压测说明

本轮主测试围绕当前课件主生成链开展 `20` 个主题的连续验证。生成 authority 由 `Diego` 承接，送审稿聚焦其在 Spectra 中的业务链路稳定性表现。各主题重复执行同一条用户路径，以验证系统在连续高负载条件下的确定性和异常收敛特征。

本轮验证要求每个主题均完成以下关键步骤：

1. 接收输入并进入会话流程；
2. 生成 `outline` 并进入确认阶段；
3. 完成正式生成；
4. 进入预览与导出链路；
5. 将成果对象继续保存到后续管理链路中。

该组长时负载验证重点检验三项工程特征：

- 流程完整性：业务步骤是否完整出现；
- 状态确定性：阶段推进逻辑是否清晰、可复查；
- 结果一致性：前端展示、后端主链和成果保存是否保持一致。

### 测试目标与执行方式

根据补充材料，本轮主链验证围绕以下四个目标展开：

- 端到端流程是否可用；
- 多主题连续运行是否稳定；
- 局部失败是否能够被记录、隔离和追踪；
- 参数调整对系统吞吐量与可用性的影响边界是否清楚。

对应执行方式为 `20` 个主题的串行全流程验证。每个主题遵循相同业务路径：输入提示词和主题，生成 `outline`，完成确认，再进入正式生成、编译输出和成果保存链。该设计保证测试结果反映真实业务链路，而非手工单次演示结果。

| 测试项 | 当前做法 | 验证目的 |
| --- | --- | --- |
| 主题规模 | `20` 个不同主题串行执行 | 验证跨主题重复运行能力 |
| 大纲阶段 | 生成后自动确认 | 验证中间阶段可被稳定推进 |
| 执行方式 | 后台批处理长时运行 | 验证长时作业编排能力 |
| 任务控制 | 超时、重试、任务间隔 | 验证异常收敛与节奏控制能力 |
| 结果记录 | 汇总统计与日志留痕 | 验证结果可追踪、可复查 |

### 主链运行结果

根据测试报告中的汇总统计，本轮 `20` 主题全流程测试共运行 `20` 个任务，成功 `19` 个，失败 `1` 个，端到端业务链路成功率（Transaction Success Rate）达到 `95%`。任务在后台连续运行约 `4 小时 20 分钟`，形成完整的系统吞吐量与可用性实测记录。成功样本的平均耗时约 `11` 分钟，中位耗时约 `10` 分钟，`P90` 约 `14` 分钟，最短耗时约 `6` 分钟。

| 指标 | 结果 |
| --- | --- |
| 总任务数 | `20` |
| 成功任务数 | `19` |
| 失败任务数 | `1` |
| 端到端业务链路成功率（Transaction Success Rate） | `95%` |
| 连续运行时长 | 约 `4 小时 20 分钟` |
| 平均耗时 | 约 `11` 分钟 |
| 中位耗时 | 约 `10` 分钟 |
| P90 耗时 | 约 `14` 分钟 |
| 最短耗时 | 约 `6` 分钟 |

通过对上述统计结果的横向对比分析可见，当前主链验证已超出单次演示范畴，进入连续运行验证阶段。数据分布特征显示，大部分成功任务均在稳定时间窗口内完成，说明系统在连续长时负载条件下保持了较高的可用性和较稳定的处理确定性。

### 测试机制与异常收敛

除结果统计外，测试机制本身也构成主链稳定性的重要验证依据。补充材料表明，当前验证并非依赖人工盯守，而是通过一组控制机制保证长时运行的稳定性、确定性和异常收敛能力。

| 机制 | 当前设置 | 作用 |
| --- | --- | --- |
| 串行执行 | 一次仅运行一个主题 | 降低并发噪声，保证结果确定性 |
| 任务级超时 | 为单任务设置时间上限 | 防止异常任务长期占用资源 |
| 失败重试 | 失败后按窗口重试一次 | 提升异常收敛能力 |
| 任务间隔 | 主题之间保留固定间隔 | 降低连续调用波动 |
| 日志落盘 | 记录运行过程与错误阶段 | 支撑复查和定位 |
| 汇总统计 | 输出结构化结果汇总 | 支撑成功率与耗时分布分析 |

唯一失败样本发生在 `SLIDES_GENERATING` 阶段，错误记录为 `SLIDE_LLM_ERROR`。该样本在重试后仍未在任务窗口内收敛，最终由超时机制终止。验证结果表明，局部失败能够被隔离，错误阶段能够被明确定位，后续主题执行未受到级联影响。这一特征说明系统具备基础的自主故障恢复与异常收敛能力。

辅助参数矩阵测试进一步表明，单纯提速并不会稳定提升端到端业务链路成功率（Transaction Success Rate）。数据表现显示，系统吞吐量与可用性的优化依赖并发控制、超时窗口和重试策略协同调整。当前工程策略以稳定可用为优先，再在此基础上进行吞吐优化，符合竞赛验证材料的表达要求。

### 主链稳定性测试结论

结合长时负载压测结果与当前作品展示，本节验证结论可归纳为：

- 核心业务主链已通过验证实现数据接入、内容生成、结果预览、标准导出和成果保存的连续衔接；
- 生成过程中的阶段控制、结果修订和成果回看能力已通过验证形成稳定链路；
- 系统在连续高负载验证中保持较高端到端业务链路成功率（Transaction Success Rate）和稳定时间分布；
- 局部异常具备隔离、记录和收敛能力，主链运行表现出较强的工程确定性。

从竞赛评审视角看，本节承担的是“证明系统已超出单次演示，进入连续运行验证阶段”的职责。

综上，第 7 章的三类证据已形成完整验证体系：界面与成果展示验证统一工作台和多模态教学资源产出矩阵，检索评估验证语义检索链路的有效性，主链稳定性测试验证端到端业务链路成功率（Transaction Success Rate）和长时负载下的工程鲁棒性。

本节判断可压缩为一句话：当前核心业务主链已通过长时负载与异常收敛验证，具备可连续运行、可复查追踪、可保持业务一致性的工程稳定性。
