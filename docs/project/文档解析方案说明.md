# 当前文档解析方案说明

## 1. 方案目标

当前文档解析采用“远端高质量优先 + 后端兜底稳定”的设计，目标是：

- 优先获得高质量解析结果（MinerU）
- 在远端失败时快速降级到本地解析（local）
- 保证主系统始终可完成入库与索引，不因单一 provider 故障卡死

## 2. 核心流程

### 2.1 上传阶段

上传文件后，后端可进入 `defer_parse=true` 的等待远端结果模式，文件状态会先显示为：

- `status=uploading`
- `parse_result.deferred_parse=true`
- `parse_result.state=awaiting_remote_result`

### 2.2 成功路径（远端优先）

远端解析成功后，调用：

- `POST /api/v1/files/{file_id}/parse/mineru`

后端会：

- 用远端文本回填
- 触发统一分块与索引链路
- 更新文件为 `status=ready`

结果特征通常为：

- `parse_result.provider=mineru_remote`（或远端 provider）
- `parse_result.fallback_used=false`
- `parse_result.parse_mode=remote_preparsed`

### 2.3 失败路径（降级兜底）

远端失败或超时后，调用：

- `POST /api/v1/files/{file_id}/parse/fallback`

后端会：

- 强制使用 `local` 解析
- 走统一索引链路
- 最终更新为 `status=ready`（或明确失败态）

结果特征为：

- `parse_result.provider=local`
- `parse_result.provider_used=local`
- `parse_result.provider_attempted=["local"]`
- `parse_result.fallback_used=true`
- `parse_result.capability_status.fallback_used=true`
- `parse_result.capability_status.fallback_target=local`

## 3. 关键语义约定

为避免监控和验收误判，当前统一约定：

- 通过 fallback 接口触发并完成降级时，`fallback_used` 必须为 `true`
- `parse_result.fallback_used` 与 `capability_status.fallback_used` 语义保持一致
- 最终使用的 provider 以 `parse_result.provider` / `provider_used` 为准

## 4. 对外联调接口

核心接口：

- `POST /api/v1/files`
- `POST /api/v1/files/{file_id}/parse/mineru`
- `POST /api/v1/files/{file_id}/parse/fallback`
- `GET /api/v1/projects/{project_id}/files`

联调时重点查看字段：

- `status`
- `parse_progress`
- `parse_result.provider`
- `parse_result.fallback_used`
- `parse_result.stage_timings_ms`
- `parse_result.capability_status`

## 5. 常见排查点

- 文件一直停在 `awaiting_remote_result`：
  - 检查是否触发了回填 `/parse/mineru` 或兜底 `/parse/fallback`
- 调用 `/parse/mineru` 失败：
  - 优先查后端日志 trace_id
  - 关注重建索引阶段（delete/reindex）异常
- fallback 后仍非 local：
  - 检查 worker 是否加载最新代码
  - 检查 fallback 调度是否透传强制 local 参数

## 6. 当前结论

当前方案已形成稳定闭环：

- 远端成功：优先远端结果
- 远端失败：确定性降级 local
- 语义字段一致：可观测、可验收、可追踪

该方案可作为本阶段文档解析默认执行口径。
