# 9. 提交版主稿结构建议

本目录中的文档已经按章节拆分，便于分工撰写与持续迭代。进入正式提交阶段时，建议将其进一步收束为一份统一主稿，以增强整体性、阅读连续性和评审体验。本章用于说明推荐的主稿组织方式。

## 9.1 主稿组织原则

提交版主稿不宜呈现为松散文档集合，而应当体现以下三个原则：

1. 开篇先立核心判断，再展开技术与商业论证。
2. 正文始终围绕同一主线展开，即“库”是系统本体、课程资料库是统一生成源、多模态成果是其外化结果。
3. 题目要求覆盖放在正文之后作为附录说明，用于证明方案完整性，而不打断正文叙事。

## 9.2 推荐主稿顺序

正式提交时，建议按以下顺序合并文档：

1. 封面
2. 执行摘要
3. 前言与项目综述
4. 可行性分析
5. 业务与需求分析
6. 系统架构设计
7. 关键技术实现
8. 项目测试与效果评估
9. 组织管理与商业价值
10. 题目要求覆盖说明
11. 附录

对应到当前目录，推荐映射关系如下：

| 主稿部分 | 对应文档 |
|---|---|
| 执行摘要 | `00-executive-summary.md` |
| 前言与项目综述 | `01-overview.md` |
| 可行性分析 | `02-feasibility.md` |
| 业务与需求分析 | `03-requirements-analysis.md` |
| 系统架构设计 | `04-architecture.md` |
| 关键技术实现 | `05-key-technologies.md` |
| 项目测试与效果评估 | `06-testing-evaluation.md` |
| 组织管理与商业价值 | `07-organization-business.md` |
| 题目要求覆盖说明 | `08-requirements-coverage.md` |

## 9.3 主稿开篇建议

正式提交材料开篇建议采用如下结构：

1. 用一句话给出项目定位。
2. 用一段话说明核心判断，即教师真正需要的是围绕备课、生成、沉淀与分发的一体化系统。
3. 用一段话点明系统本体，即真正长期维护的是“库”，不是单个 `PPT` 文件。
4. 用一段话点明核心创新，即教师正常生成 PPT，课程资料库作为副产物无感沉淀。
5. 用一段话说明多模态的底层逻辑，即课程资料库是统一生成源，不同模态是同一知识结构的不同外化。

这样的开篇结构有助于评审快速建立对项目的正确理解，避免把 `Spectra` 误判为一般性的 AI 生成课件工具。

## 9.4 正文衔接建议

为了保证主稿流畅，章节之间建议采用如下衔接逻辑：

1. 综述章提出问题、定位和核心创新。
2. 可行性章回答“为什么这套方案成立”。
3. 需求章回答“系统到底要解决什么问题，为什么这样拆解”。
4. 架构章回答“系统如何把课程资料库作为中枢组织起来”。
5. 技术章回答“统一生成源、轻量版本和引用复用如何在实现层落地”。
6. 测试章回答“这套闭环如何被验证”。
7. 商业章回答“为什么客户愿意买，为什么组织愿意用”。
8. 覆盖说明回答“为什么原题每一项要求都已被完整吸收进方案”。

## 9.5 附录建议

正式提交版附录可包含以下内容：

1. 术语表：对应 `00-terms.md`
2. 题目要求覆盖说明：对应 `08-requirements-coverage.md`
3. 关键图示汇总：可从架构章、技术章、商业章中抽取
4. 测试证据表：可从测试章中抽取

## 9.6 收束建议

在最终导出为 PDF 前，建议对主稿做三项收束：

1. 删除内部写作指导类内容，不将 `00-writing-guide.md` 和 `99-self-review.md` 放入正式提交正文。
2. 保留题目覆盖说明，但放在正文之后，避免影响主叙事。
3. 保留“库是系统本体、课程资料库是统一生成源”这一核心表达，并避免在不同章节中切换为彼此冲突的解释方式。
