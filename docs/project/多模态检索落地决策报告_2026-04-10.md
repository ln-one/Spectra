# 多模态检索落地决策报告（2026-04-10）

> 状态：当前评估结论
> 目标：给 Spectra 当前阶段的多模态检索落地做工程决策，不做泛技术综述

## 1. 结论先行

当前机器条件是“弱 GPU，可接受远端 API，近期优先提升 `PDF 课件/教案` 的页面级检索效果”。在这个前提下，三条路线的结论如下：

- `适合现在做`：`Qwen3-VL-Embedding` 方向，但只作为 `PDF 页面视觉检索 sidecar`，不替换现有文本 RAG 主链路。
- `适合以后做`：`ColPali / Byaldi`，适合在有独立检索服务或更稳定 GPU 资源后作为专项文档视觉检索方案继续评估。
- `不建议当前主线接入`：`VRAG / VimRAG` 作为整套多模态 agentic RAG 方案，不适合直接接入 Spectra 当前主流程。

一句话判断：

> `VRAG` 值得研究，但不适合直接成为 Spectra 当前阶段的现成轮子；Spectra 近期最稳妥的路线是“文本 RAG 保持主路径，新增 PDF 页面视觉检索增强路径”，而不是把 chat 或 generation 改造成多轮视觉 agent。

## 2. 当前系统可接入层

### 2.1 现有主链路

当前 Spectra 的主链路仍然是：

`upload -> parse -> normalize -> chunk -> embedding -> retrieval -> chat/generation`

对应实现事实：

- 上传入库与索引主线：`backend/services/media/rag_indexing.py`
- RAG 检索入口：`backend/services/ai/rag_context.py`
- RAG 服务编排：`backend/services/rag_service/service.py`
- 检索结果合并与 `Reference` 扩展：`backend/services/rag_service/retrieval.py`
- 课件插图接入：`backend/services/task_executor/ppt_image_insertion.py`
- RAG API 能力层：`backend/services/rag_api_service/core.py`

### 2.2 当前多模态现状

当前系统并不是“真正的视觉检索”：

- 图片上传默认仍以占位文本进入 RAG，而不是页面级/图像级视觉 embedding。
- 视频当前主要走摘要式理解与保底片段，不是帧级视觉索引。
- `EmbeddingService` 已经支持 `text-embedding-v3` 和 `qwen3-vl-embedding` 的模型入口，但主索引流程仍围绕“文本 chunk”组织。
- chat 侧已有按需图片理解能力，但它是单次多模态分析，不是统一视觉检索层。

关键代码事实：

- 图片占位：`backend/services/file_parser/direct_extractors.py`
- 视频理解：`backend/services/media/video.py`
- embedding provider 入口：`backend/services/media/embedding.py`
- chat 图片分析：`backend/services/ai/service.py`

### 2.3 当前不应破坏的边界

本次多模态检索增强不应破坏以下系统语义：

- `Project / Session / Artifact / Reference / Version` 的统一语义
- 现有 chat / generation 的主 contract
- 现有 artifact 下载与 project-space 外化语义
- 现有 RAG 超时与可观测性边界，例如 `CHAT_RAG_TIMEOUT_SECONDS`

结论：

- 最自然的接入点不是“替换 chat agent”，而是新增 `retrieval sidecar`
- 最适合先服务的场景不是视频，而是 `PDF 页面图 / 截图 / 表格 / 图示`
- 最低风险入口是：
  - `rag_service` 检索层
  - `rag_api_service` 能力层
  - `PPT 插图与引用增强` 场景层

## 3. 候选路线对比

### 3.1 `VRAG / VimRAG`

仓库定位：

- `VRAG` 是阿里通义实验路线下的多模态 RAG 框架，包含检索引擎、视觉 embedding、demo agent、以及训练代码。
- `VimRAG` 更偏 API 驱动的视觉 memory/agent demo。
- `VRAG` 本地 demo 明确偏重模型部署路径；仓库 README 直接给出本地 `vLLM + Qwen2.5-VL-7B` 路线，并说明本地 VRAG 需要较强 GPU。

与 Spectra 的关系：

- 可借鉴的部分：
  - 多模态检索 sidecar 的思路
  - 页面图/图片/视频片段统一进入检索语义空间
  - 检索与多轮 reasoning 分层
- 不适配的部分：
  - 自带的 agent loop 会显著增加 chat 延迟和故障面
  - 本地视觉模型与检索服务资源要求高
  - 仓库形态更接近 demo / research framework，而非生产 SDK
  - 自己维护 `FAISS + corpus` 语义，不直接对齐 Spectra 当前 `Project / Session / Reference` 语义

工程判断：

- 不适合拿来替换 Spectra 当前的 RAG 主链路
- 也不适合直接把 `VimRAG/VRAG` agent 逻辑塞入现有 chat 主流程
- 只有其中“独立视觉检索 sidecar”这一层次值得借鉴

当前评分：

- `PDF 页面级收益`：高
- `弱 GPU 适配度`：低
- `集成侵入度`：高
- `对 chat 时延风险`：高
- `当前推荐度`：低

结论：

> `VRAG` 当前更适合作为参考架构，不适合作为 Spectra 近期直接接入的现成轮子。

参考：

- [Alibaba-NLP/VRAG](https://github.com/Alibaba-NLP/VRAG)

### 3.2 `Qwen3-VL-Embedding (+ 可选 reranker)`

方案定位：

- `Qwen3-VL-Embedding` 是统一的多模态 embedding 模型，面向文本、图像、截图、视频等输入。
- 与 `Qwen3-VL-Reranker` 可形成“召回 + 重排”组合。
- 这条路线与 Spectra 当前 DashScope / Qwen 生态最接近。

与 Spectra 的关系：

- 优势：
  - 当前代码已存在 `qwen3-vl-embedding` 的 provider 入口，不需要重新发明 embedding 配置层
  - 语义上更接近“新增一个多模态 recall provider”，而不是重写整套 agent
  - 更容易与现有 `EmbeddingService / RAGService / RAG API` 并存
  - 更适合优先解决 `PDF 页面图、截图、图表、版式` 这类问题
- 约束：
  - 仅有模型入口还不够，仍需要新增“PDF 页面渲染 -> 页面图索引 -> 页面级结果返回”的工程层
  - 若走本地开源模型，依然会吃 GPU
  - 当前更适合走远端 provider 或独立服务，而不是强行在主后端本地推理

工程判断：

- 这是当前最适合 Spectra 的近期路线
- 正确接法不是“替换文本 RAG”，而是“新增 PDF 页面视觉检索 sidecar”
- 可以把视觉结果作为附加检索源返回给 chat / PPT 插图 / 引用展示

当前评分：

- `PDF 页面级收益`：高
- `弱 GPU 适配度`：中
- `与现有系统兼容度`：高
- `集成侵入度`：中
- `当前推荐度`：高

结论：

> 这是 Spectra 当前阶段最值得先做 PoC 的方向。

参考：

- [QwenLM/Qwen3-VL-Embedding](https://github.com/QwenLM/Qwen3-VL-Embedding)

### 3.3 `ColPali / Byaldi`

方案定位：

- `ColPali` 是文档视觉检索路线，核心优势是直接对 PDF 页面图做视觉文档检索，减少对 OCR 与版面恢复的依赖。
- `Byaldi` 是对 `ColPali`/`ColQwen2` 一类 late-interaction 多模态模型的易用封装。

与 Spectra 的关系：

- 优势：
  - 对 `PDF 页面图、截图、图表、复杂版式` 问题很对症
  - 适合“用户问一张图、一页版式、一个表格在哪”的文档视觉召回
  - 与当前文本 RAG 的互补性强
- 约束：
  - `Byaldi` 官方 README 明确标注为 `pre-release`
  - 默认索引为未压缩形态，内存/存储压力更明显
  - late-interaction 检索比普通单向量召回更重
  - 即便项目称旧 GPU 也能跑，编码与索引阶段仍更依赖 GPU，CPU 路线不适合当前主系统
  - 若现在接入，运维复杂度会高于 `Qwen3-VL-Embedding sidecar`

工程判断：

- 从“PDF 页面视觉检索效果”看，这是一条很值得保留的后续路线
- 但以当前机器条件和主线交付压力，不适合作为近期默认方案
- 更适合未来在独立检索服务、单独 GPU、或专项视觉文档库场景下继续评估

当前评分：

- `PDF 页面级收益`：高
- `弱 GPU 适配度`：中低
- `集成侵入度`：中高
- `对主系统运维压力`：高
- `当前推荐度`：中

结论：

> `ColPali / Byaldi` 很适合作为后续增强路线保留，但不应成为当前最先落地的方案。

参考：

- [illuin-tech/colpali](https://github.com/illuin-tech/colpali)
- [AnswerDotAI/byaldi](https://github.com/AnswerDotAI/byaldi)

## 4. 决策矩阵

| 路线 | 当前阶段判断 | 主要原因 | 建议定位 |
| --- | --- | --- | --- |
| `VRAG / VimRAG` | 不建议当前主线接入 | 本地重模型 + agent loop 过重，和现有主链路语义耦合差 | 参考架构 |
| `Qwen3-VL-Embedding` | 适合现在做 | 与现有 Qwen / DashScope 生态最接近，最容易做 sidecar | 近期 PoC 首选 |
| `ColPali / Byaldi` | 适合以后做 | 文档视觉检索潜力高，但当前资源和运维成本偏高 | 后续专项路线 |

最终推荐顺序：

1. `Qwen3-VL-Embedding sidecar`
2. `ColPali / Byaldi` 后续专项评估
3. `VRAG` 仅保留为架构参考，不进入近期主线

## 5. 推荐架构落点

### 5.1 总原则

近期架构不应追求“全链路视觉化”，而应保持：

- `文本 RAG` 继续作为主路径
- `PDF 页面视觉检索` 作为增强路径
- `视频多模态检索` 暂缓，不纳入近期主目标

### 5.2 推荐接法

建议新增一个 `multimodal retrieval provider`，挂在现有 `RAGService` 旁边，而不是替换它。

推荐结构：

1. `PDF` 上传后保留现有文本解析与文本索引
2. 新增 `PDF -> 页面图` 派生流程
3. 页面图进入单独的视觉索引
4. 查询时先走现有文本 RAG
5. 满足条件时再附加页面视觉检索结果
6. 结果统一返回到 chat / PPT 插图 / 来源引用层

这里的“满足条件”优先限定为：

- 用户问题明显指向截图、表格、图示、页面内容
- 文本 RAG 命中弱，且文件来源包含 PDF
- 课件插图或引用增强场景明确需要图片级来源

### 5.3 不建议的接法

当前不建议：

- 用视觉检索替换文本 chunk 检索
- 把多轮 agentic retrieval 放进 chat 主链路
- 把视频帧检索作为同一阶段目标
- 新建一套独立于 `Project / Session / Reference` 语义之外的素材系统

## 6. 近期最小 PoC 边界

PoC 只覆盖：

`PDF -> 页面图/截图 -> 检索 -> 在 chat 或 PPT 生成中引用`

不覆盖：

- 视频检索
- 全库多模态统一索引替换
- 多轮检索 agent
- 生产级 reranker/压缩/多阶段优化

### 6.1 PoC 需要新增的接口/数据形态

建议为视觉检索结果定义统一结果形态，至少包括：

- `file_id`
- `page_num`
- `page_image_id` 或稳定页面资源标识
- `score`
- `thumbnail_url` 或可取回的页面图路径
- `citation` / 来源追踪字段

这类结果不应伪装成普通文本 chunk，而应作为并列结果源返回。

### 6.2 PoC 接入位置

优先接入两处：

1. `chat` 的按需附加来源
2. `PPT` 图片插图与来源命中增强

暂不接入：

- generation 主 prompt 的默认大规模拼接
- session 主状态机
- artifact 主存储契约

### 6.3 PoC 成功标准

若以下三项同时满足，则认为路线成立：

1. 能稳定召回正确 PDF 页面或截图
2. 查询延迟可控，不明显拖垮 chat
3. 接入后不破坏现有文本 RAG、状态语义与来源追踪

## 7. 评测与验收建议

本次结论属于工程评估，不是正式 benchmark 结果。后续 PoC 应至少使用三类真实样本：

1. `纯文本命中容易的 PDF`
2. `依赖版式/截图/图表理解的 PDF`
3. `OCR 文本有限、但图像信息关键的 PDF`

每类样本统一记录：

- `Top-k 是否召回正确页面`
- `平均查询耗时`
- `索引耗时`
- `额外存储占用`
- `是否影响 chat 响应稳定性`
- `是否能为 PPT 插图或来源引用带来明显提升`

建议 PoC 输出不要只写“模型能跑”，而要直接回答：

- 相比当前文本 RAG，页面级命中是否明显改善
- 这份改善是否值得新增一套视觉索引链路

## 8. 最终建议

当前机器条件与产品阶段下，Spectra 最适合先做的不是 `VRAG`，而是：

> 保持现有文本 RAG 主链路不动，在其旁边增加一条面向 `PDF 页面图/截图/图表` 的视觉检索 sidecar，首选沿 `Qwen3-VL-Embedding` 方向做 PoC。

原因归纳为四点：

1. 它最贴近当前产品最缺的能力：复杂 PDF 页面级命中。
2. 它对当前系统侵入最小，不需要重写 chat / generation 主语义。
3. 它比 `VRAG` 更容易被限制在 sidecar 边界内。
4. 它比 `ColPali / Byaldi` 更适合在当前资源条件下先做工程验证。

因此，近期建议定为：

1. 不接 `VRAG` 主方案
2. 先做 `Qwen3-VL-Embedding` 页面视觉检索 PoC
3. 保留 `ColPali / Byaldi` 作为未来专项增强路线
