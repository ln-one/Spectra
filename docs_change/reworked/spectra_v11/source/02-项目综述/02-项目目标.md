## 项目目标

### 要解决的核心问题

`Spectra` 当前阶段主要解决以下四个问题：

1. 让教师在一个工作台中完成资料整理、会话推进和结果生成，减少工具来回切换；
2. 让上传资料进入解析、检索增强和证据组织链路，避免停留在附件层；
3. 让生成过程可确认、可修改、可继续推进，避免一次性黑箱输出；
4. 让生成结果可以被保存、对比、引用和继续复用，避免导出后即与系统失联。

### 对应解决思路

系统当前维护的是一条完整工作主链。`Project` 承接资料与结果，`Session` 展开一次具体工作过程，上传解析与检索增强提供证据基础，生成模块负责大纲、生成与修改推进，预览导出模块负责结果查看与标准交付，状态与版本模块负责保存、引用和后续复用。

系统维护的对象覆盖一次工作过程中形成的资料、上下文、结果与后续关系。“生产即沉淀，沉淀即交付”在这里对应的是当前对象链的实际结构：生成结果会进入结果管理空间，版本与引用关系会保留下来，后续工作可以直接从已有结果继续展开。

围绕这条主链，项目目标可以进一步拆成四个层次。第一层是工作台层目标，即让教师在同一界面中完成资料进入、会话推进、结果查看和导出，不再频繁切换工具。第二层是证据层目标，即让上传资料经过解析、切块、检索增强和证据组织之后进入生成链，而不是停留在附件层。第三层是结果层目标，即让生成结果支持确认、修改、预览、导出和历史查看。第四层是对象层目标，即让结果以 `Artifact`、`Version` 和 `Reference` 的方式回到项目空间，成为后续修改和再次使用的基础。

这四层目标在当前作品中都有对应落点。工作台层体现在 Studio、Chat、Sources 和结果查看界面；证据层体现在上传解析、来源查看和检索评估；结果层体现在 outline、confirm、preview、history、export 和 download；对象层则体现在结果保存、版本记录和引用关系。项目目标直接对应到作品当前已经形成的结构，也为后续章节中的设计、技术和测试提供统一判断基线。
