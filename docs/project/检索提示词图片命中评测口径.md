# 检索、提示词、图片命中评测口径

> 更新时间：2026-03-25
> 状态：当前生效
> 定位：成员 D 当前阶段用于初审前质量验证的统一评测口径

## 1. 目标

本口径用于统一三类质量判断：

1. 检索是否命中
2. 提示词优化是否提升 PPT 结果质量
3. 图片素材是否真正支撑页面讲解

当前阶段不再缺主干骨架，因此这份口径重点服务“质量证据化”，而不是功能存在性证明。

## 2. 检索评测口径

检索相关结论默认必须先区分两组语义：

1. 不传 `rag_source_ids`：`默认库检索`
2. 传 `rag_source_ids`：`严格限源检索`

这两组评测不能混合统计后直接得出统一结论。

原因：

1. 默认库检索代表主链路默认体验
2. 严格限源检索代表受控实验条件
3. 两者回答的问题不同，混算会掩盖“是检索问题、素材不足，还是提示词问题”

检索质量默认使用现有 RAG 评测链路：

- 脚本：`backend/eval/run_eval.py`
- 基线：`backend/eval/baselines/rag-baseline-v1.json`
- 基线工具：`backend/eval/baseline_manager.py`

关键指标：

1. `keyword_hit_rate`
2. `failure_rate`
3. `avg_latency_ms`
4. `explainability_rate`
5. `continuity_rate`
6. `fallback_hit_rate`

检索证据输出要求：

1. 必须能复现
2. 必须有 baseline
3. 必须能说明优化前后是否退化
4. 必须标明属于 `默认库检索` 还是 `严格限源检索`

## 3. 提示词优化评测口径

PPT 提示词优化默认使用以下两条证据链：

1. `backend/eval/ppt_quality_audit.py`
2. `backend/eval/ppt_quality_comparison_audit.py`

对应样本：

1. `backend/eval/ppt_quality_samples.json`
2. `backend/eval/ppt_quality_comparison_samples.json`

对应基线：

- `backend/eval/baselines/ppt-quality-baseline-v1.json`

关键指标：

1. `structure_pass_rate`
2. `information_density_pass_rate`
3. `visual_balance_pass_rate`
4. `expression_pass_rate`
5. `image_match_pass_rate`
6. `overall_pass_rate`
7. `overall_improvement_rate`

提示词优化结论必须同时满足：

1. 当前 baseline 不退化
2. 前后对照样本能体现提升
3. 提升点可被解释为具体 prompt 规则，而不是偶然结果

## 4. 图片命中评测口径

图片命中当前采用“规则 + 样本”双轨判断：

规则来源：

1. `docs/project/PPT图片召回与素材命中优化方案.md`
2. `docs/project/PPT插图策略方案.md`

运行口径：

1. 先按 `默认库检索 / 严格限源检索` 分组
2. 再看 `image_match_pass_rate`
3. 再看样本中的失败原因
4. 再看页面文字是否与图片形成直接讲解对应

图片命中必须同时满足：

1. 图片与本页标题或核心结论直接相关
2. 教师能明确说出“图上看什么”
3. 不是装饰性背景图
4. 不挤压掉本页必要文本空间

## 5. 当前输出要求

成员 D 后续输出质量结论时，默认采用以下最小包：

1. 样本
2. baseline
3. 对照结果
4. 指标说明
5. 结论解释
6. 检索模式标注

不再接受只有主观评价、没有样本和指标支撑的“质量提升结论”。
