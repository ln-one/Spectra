# 7 项目测试与成果展示

## 7.1 测试目标

本项目测试聚焦以下六类目标：

1.验证教师端主流程完整可用，系统能够从需求输入走通到成果输出。

2.验证多模态资料能够被解析、入库、检索并参与生成。

3.验证教师反馈能够驱动局部或整体再生成，形成可持续优化闭环。

4.验证课程成果能够以无感方式沉淀为课程资料库，并支持分享访问。

5.验证库、版本、导出物之间的边界清晰，正式状态与临时结果不会混淆。

6.验证系统在接口、任务、检索和导出层面具备可交付的工程稳定性。

## 7.2 测试方法

本项目采用 \*\*"分层递进 + 全链路覆盖 + 自动化优先"\*\* 的综合测试方法，遵循 "测试左移" 原则，将测试贯穿于需求分析、设计、开发、部署的全生命周期。针对 AI 教育系统的特殊性，在传统软件测试基础上增加 AI 生成质量专项测试，确保系统功能完整、性能稳定、体验流畅、安全合规。

一、分层测试体系

1.  **单元测试**

针对单个函数、类、接口进行测试，验证代码逻辑的正确性。采用自动化测试框架，前端使用 Jest，后端使用 Pytest，要求核心业务代码覆盖率达到 80% 以上。测试重点包括：数据校验、边界条件、异常处理、工具函数正确性。所有单元测试在代码提交前自动执行，不通过则无法合并代码。

2.  **集成测试**

针对模块之间的接口与交互进行测试，验证模块间协作的正确性。采用 API 测试工具 Postman 与自动化测试框架 Pytest，测试重点包括：API 接口正确性、数据传输一致性、依赖服务调用、异常场景处理。集成测试在每次代码合并后自动执行，确保模块集成后功能正常。

3.  **系统测试**

针对整个系统进行端到端测试，验证系统是否满足需求规格说明书的要求。采用黑盒测试方法，结合自动化测试工具 Selenium 与人工测试，测试重点包括：功能完整性、性能稳定性、兼容性、安全性、易用性。系统测试在每个迭代周期结束后执行，是版本发布前的最后一道防线。

4.  **验收测试**

由用户与产品团队共同执行，验证系统是否满足实际业务需求。分为 Alpha 测试（内部验收）与 Beta 测试（外部用户验收），邀请试点学校的教师与学生参与测试，收集真实使用反馈，优化产品体验。验收测试通过后，版本方可正式发布。

二、专项测试方法

1.  **AI 生成质量专项测试**

针对 AI 生成内容的特殊性，建立量化的质量评估体系，从准确性、完整性、合规性、相关性四个维度进行测试。采用 "自动校验 + 人工复核" 的方式，自动校验内容结构完整性与合规性，人工复核知识点准确性与教学适用性。建立测试用例库，覆盖不同学科、不同学段、不同类型的教学内容，每次模型更新后进行回归测试。

2.  **数据安全与合规测试**

针对教育行业数据安全要求，进行数据加密、权限控制、数据脱敏、合规审计等专项测试。测试重点包括：用户身份认证、数据传输加密、存储加密、敏感数据脱敏、权限边界、操作日志审计。邀请第三方安全机构进行渗透测试，确保系统无安全漏洞。

3.  **异常场景测试**

模拟各种异常场景，验证系统的容错能力与恢复能力。包括：网络中断、服务器宕机、数据库故障、大模型 API 不可用、大文件上传失败、并发过载等场景。测试系统在异常情况下是否能够正确处理，是否会出现数据丢失、系统崩溃等问题，以及异常恢复后是否能够正常运行。

## 7.3 系统测试计划

本系统测试计划覆盖所有核心功能与非功能需求，测试周期为每个迭代周期 2 周，其中单元测试与集成测试占 1 周，系统测试占 1 周。测试环境与生产环境保持一致，确保测试结果的真实性与有效性。

测试目标

1.  验证系统所有功能是否符合需求规格说明书的要求，功能完整且正确

2.  验证系统在高并发场景下的性能表现，满足预期的响应时间与吞吐量要求

3.  验证系统在不同浏览器、操作系统、设备上的兼容性，保证用户体验一致

4.  验证端到端业务流程的顺畅性，确保教师备课、学生学习、管理员管理等核心流程无断点

5.  发现并修复系统中的 bug 与缺陷，提升系统的稳定性与可靠性

6.  收集测试过程中的问题与建议，为产品优化提供依据

测试范围

1.  功能范围：用户管理、课程库管理、文件上传解析、RAG 检索、内容生成、预览导出、协作管理、权限管理、系统设置等所有核心功能

2.  非功能范围：性能、兼容性、安全性、易用性、可靠性

3.  业务流程范围：教师备课全流程、学生学习全流程、管理员管理全流程、多人协作备课流程

测试环境

- 硬件环境：4 核 8GB 云服务器，500GB SSD 存储，10Mbps 带宽

- 软件环境：Ubuntu 20.04，Docker 24.x，PostgreSQL 16，Redis 7.2，Qdrant 1.9

- 测试工具：Jest、Pytest、Postman、Selenium、JMeter、Chrome DevTools

### 7.3.1 功能测试

功能测试采用黑盒测试方法，验证每个功能模块是否按照需求正确实现，覆盖所有正常场景与异常场景。

用户管理模块

测试用户注册、登录、退出、密码找回、个人信息修改、头像上传等功能。验证不同角色用户（教师、学生、管理员）的权限边界，确保学生无法访问教师后台，普通用户无法访问管理员功能。测试异常场景：用户名已存在、密码错误次数过多、邮箱格式不正确、验证码过期等。

课程库管理模块

测试课程库创建、编辑、删除、分享、权限设置、版本管理、引用管理等功能。验证课程库列表展示、搜索、筛选功能的正确性，版本对比、回滚功能的准确性，引用关系的建立与同步功能的有效性。测试异常场景：删除被引用的课程库、修改他人的课程库、分享权限设置错误等。

文件上传解析模块

测试不同格式文件（PDF、DOCX、PPTX、XLSX、JPG、PNG、MP4）的上传功能，支持大文件分片上传与断点续传。验证文件解析的准确性，包括标题层级提取、文本提取、公式识别、图表提取、表格提取等。测试异常场景：上传不支持的文件格式、上传超大文件、网络中断后断点续传、文件损坏等。

RAG 检索模块

测试关键词检索、语义检索、混合检索功能的准确性，验证检索结果的相关性与排序合理性。测试元数据过滤功能，支持按学科、学段、章节、来源进行过滤。测试增量更新功能，文件修改后检索结果是否同步更新。测试异常场景：检索不存在的内容、检索敏感内容、大量并发检索等。

内容生成模块

测试大纲生成、课件生成、教案生成、思维导图生成、个性化学习材料生成等功能。验证生成内容的结构完整性、知识点准确性、教学适用性。测试局部修改功能，支持对生成内容的任意部分进行修改与重生成。测试异常场景：生成请求参数错误、大模型 API 不可用、生成内容包含敏感信息等。

预览导出模块

测试 PPT、教案、思维导图的实时预览功能，验证预览效果与导出效果的一致性。测试不同格式（PPTX、DOCX、PDF、HTML5、PNG）的导出功能，验证导出文件的格式正确性、内容完整性、可编辑性。测试异常场景：导出大文件、导出格式不支持、导出过程中网络中断等。

协作管理模块

测试成员邀请、权限设置、候选变更提交、审核、合入等功能。验证多人同时编辑时的冲突处理机制，变更对比功能的准确性，协作通知的及时性。测试异常场景：邀请不存在的用户、修改他人提交的变更、审核不通过的变更合入等。

系统设置模块

测试个人设置、组织设置、系统设置的修改与生效功能。验证设置的跨设备同步性，组织设置对个人设置的覆盖性。测试异常场景：修改系统关键配置、配置参数错误等。

### 7.3.2 性能测试

性能测试采用 JMeter 与 Locust 工具，模拟不同并发用户数下的系统表现，验证系统的响应时间、吞吐量、资源使用率与稳定性。

并发性能测试

模拟 100、500、1000 个并发用户同时访问系统，测试核心接口的响应时间。要求：

- 静态页面加载响应时间 \< 1 秒

- 普通 API 接口响应时间 \< 300 毫秒

- 文件上传接口（5MB 文件）响应时间 \< 5 秒

- 大纲生成响应时间 \< 10 秒

- 单页课件生成响应时间 \< 15 秒

- 完整课件生成（20 页）响应时间 \< 60 秒

吞吐量测试

测试系统在单位时间内能够处理的最大请求数。要求：

- 普通 API 接口吞吐量 \> 1000 QPS

- 文件上传接口吞吐量 \> 50 QPS

- 内容生成接口吞吐量 \> 20 QPS

资源使用率测试

监控系统在不同并发负载下的 CPU、内存、磁盘、网络使用率。要求：

- 100 并发用户下，CPU 使用率 \< 30%，内存使用率 \< 40%

- 500 并发用户下，CPU 使用率 \< 60%，内存使用率 \< 70%

- 1000 并发用户下，CPU 使用率 \< 80%，内存使用率 \< 85%

- 磁盘 IO 使用率 \< 70%，网络带宽使用率 \< 80%

稳定性测试

进行 72 小时连续压力测试，模拟 500 个并发用户持续访问系统，验证系统的长期稳定性。要求：

- 系统无崩溃、无内存泄漏、无死锁

- 平均响应时间波动不超过 20%

- 错误率 \< 0.1%

- 所有核心功能正常运行

极限测试

逐步增加并发用户数，直到系统性能达到瓶颈，找出系统的最大承载能力。测试系统在极限负载下的表现，验证系统的过载保护机制是否有效，是否能够优雅降级，避免系统崩溃。

### 7.3.3 兼容性测试

兼容性测试验证系统在不同浏览器、操作系统、设备、第三方软件上的运行效果，保证用户体验的一致性。

- Safari 14 及以上版本

验证页面渲染、交互操作、功能实现、预览效果在不同浏览器上的一致性，无布局错乱、功能失效、样式异常等问题。

操作系统兼容性测试

测试系统在不同操作系统上的兼容性，包括：

- Windows 7、Windows 10、Windows 11

验证系统在不同操作系统上的访问、上传、下载、导出等功能的正确性，无操作系统相关的 bug。

设备兼容性测试

测试系统在不同设备上的兼容性，包括：

- Microsoft Office 2016、2019、365

验证导出的 PPTX、DOCX、PDF 文件能够在上述软件中正常打开、编辑、保存，无格式错乱、内容丢失、乱码等问题。

### 7.3.4 业务流程测试

业务流程测试采用端到端测试方法，模拟真实用户的使用场景，验证完整业务流程的顺畅性与正确性。

教师备课全流程

测试教师从登录系统到完成备课并分享给学生的完整流程：

1.  教师登录系统，创建新课程库

2.  输入教学主题与要求，上传已有教学资料

3.  系统生成课程大纲，教师确认并修改大纲

4.  系统生成完整课件与教案，教师预览并局部修改

5.  教师发布正式版本，生成分享链接

6.  教师设置访问权限与有效期，将链接发送给学生

验证流程中每个步骤的功能正确性，数据一致性，无断点、无卡顿、无数据丢失。

学生学习全流程

测试学生从获取分享链接到使用学习材料的完整流程：

1.  学生通过分享链接访问课程空间

2.  学生查看教师发布的课件与教案

3.  学生下载课程文件，或在线查看动态内容

4.  学生输入个性化需求，生成思维导图、复习提纲等学习材料

5.  学生将学习材料保存到个人学习空间

验证学生端功能的正确性，与教师端数据的同步性，学生无法修改教师的课程库内容。

管理员管理全流程

测试管理员从登录系统到完成系统管理的完整流程：

1.  管理员登录系统后台

2.  管理用户账号，添加、删除、禁用用户，分配角色与权限

3.  管理组织架构，创建部门，设置组织级设置

4.  查看系统使用统计数据，包括用户数、课程库数、生成次数

5.  配置系统参数，管理模型路由、队列参数、告警阈值

验证管理员功能的正确性，权限控制的有效性，系统配置的生效性。

多人协作备课流程

测试多人协作完成备课的完整流程：

1.  课程库创建者邀请协作者加入项目

2.  协作者编辑课程内容，提交候选变更

3.  创建者查看变更对比，在线批注修改意见

4.  协作者根据批注修改内容，重新提交变更

5.  创建者审核通过变更，合入正式版本，生成新的版本号

6.  所有协作者收到版本更新通知

验证协作流程的顺畅性，变更处理的正确性，通知的及时性，多人同时编辑时的冲突处理机制。

## 7.5 系统效果

### 7.5.1 与传统方式对比

为体现系统的应用价值，将 Spectra 与传统人工备课方式、普通单轮生成工具进行对比，结果如下：

表14 不同生成工具对比

| **评估维度** | **传统人工备课** | **普通单轮生成工具** | **Spectra** |
|:--:|:--:|:--:|:--:|
| 教学需求理解 | 依赖教师手工组织 | 多为单轮提示词输入 | 多轮追问并形成结构化教学需求 |
| 资料利用方式 | 手工阅读与摘录 | 通常难以深度结合本地资料 | 项目级多模态资料解析与 RAG 融合 |
| 结果形态 | 单一 PPT 或文档 | 单次文本或单份文件 | PPT、教案、导图、讲义与课程资料库 |
| 修改机制 | 手工重做 | 多数重新整段生成 | 支持预览后局部再生成 |
| 分享方式 | 群聊、邮件或网盘发文件 | 仍以文件分发为主 | 分享可持续更新的课程资料库 |
| 资产沉淀能力 | 几乎没有 | 局限于单次结果 | 生成即沉淀、沉淀即交付 |

对比结果表明，Spectra 的核心提升不只是“更快生成”，而是把备课流程升级为课程资产生产与分发流程。

### 7.5.2 教师侧效果评估

从教师视角看，系统价值主要体现在以下四点：

在当前验证中，备课过程中的重复整理与切换成本有所下降。

多模态资料得到更统一的利用，已有素材的复用效率有所改善。

修改成本下降，教师能够围绕预览结果持续迭代，而不必从头重做。

一次备课自然沉淀为长期可复用的课程资料库，提高课程资产留存率。

### 7.5.3 学生侧效果评估

从学生视角看，系统价值主要体现在以下三点：

学生访问的是持续更新的课程资料库，而不是一次性文件。

学生能够统一查看导图、PPT、教案、讲义、代码示例和视频索引等多模态资源。

学生可以基于课程资料库生成个人导图、摘要和复习材料，形成个性化学习路径。

### 7.5.4 学校与机构侧效果评估

从学校与机构视角看，系统价值主要体现在以下三点：

教师个人备课成果被沉淀为可管理的校本课程资产。

教研组可以在同一知识框架下复用课程资源，提升协作效率。

学校获得统一的权限管理、资源追踪和课程资料分发能力，提升教育数字化治理水平。
