## 知识空间状态与版本管理技术

状态与版本管理链解决的问题，是把一次生成得到的结果纳入项目空间，而不是让结果在导出之后脱离系统。当前实现围绕 `Project`、`GenerationSession`、`Artifact`、`ProjectVersion`、`ProjectReference` 和 `ProjectMember` 几组对象组织状态、版本、引用和成员边界，使结果能够保存、回看、替换、引用和继续复用。

当前实现可以从对象模型、接口返回和 focused tests 三个层面核对：

| 证据类型 | 当前对应 |
| --- | --- |
| 对象模型 | Prisma 中明确存在 `Project`、`GenerationSession`、`Artifact`、`ProjectVersion`、`ProjectReference`、`ProjectMember` |
| 接口与结果链 | 预览与项目空间相关接口会返回 `artifact_id`、`based_on_version_id`、`current_version_id` 等结果锚点 |
| 运行语义 | 结果创建、结果替换、正式记录复用、共享与引用权限均有 focused tests 覆盖 |
| 服务边界 | `Ourograph` 负责 formal state 语义，Spectra backend 负责消费、编排和绑定 |

从结构上看，这部分技术至少回答了四个问题：

- `Project` 承接什么：承接长期存在的课程项目或资料空间；
- `Session` 承接什么：承接一次具体工作过程和中间状态推进；
- `Artifact` / `Version` 承接什么：承接可保存、可回看、可后续比较的结果状态；
- `Reference` / 成员边界承接什么：承接结果关系、引用关系和协作边界。

成员边界和登录会话由 `Limora` 承接，其认证实现使用 Better Auth 的会话与组织边界能力。当前项目空间、成员关系和结果权限共用同一套身份语义，因此页面、项目空间和结果归属能够保持一致口径。

这些对象并非停留在概念层。Prisma schema 中已经存在相应模型；artifact 相关测试会检查结果创建时是否绑定 `based_on_version_id`，替换模式下是否给旧结果写入 `superseded_by_artifact_id`，以及 Office 结果是否优先复用 authority artifact 而不是重新走本地第二渲染路径。项目语义测试和权限语义测试也覆盖了共享、引用和成员权限边界。这意味着“结果能否回到项目空间继续使用”已经写进当前数据模型和行为规则。

图 6-6 对这条回流契约给出了更直接的图示。执行结果先形成统一状态与结构化 payload，再通过 `Artifact` 绑定获得可追溯身份，随后进入 `Project / Version` 关系，成为后续查看、替换、引用和比较的基础。这样组织之后，结果链不再停留在“生成成功”这一瞬间，而是进入项目空间中的正式语义层。

这部分技术直接决定结果链是否闭合。若结果只能导出为文件而不能重新挂回项目空间，前面的生成与导出就会在交付时中断；若结果没有版本锚点，后续修改和比较就缺少稳定参照；若不同结果与资料之间没有引用关系，复用就只能依赖人工记忆。状态与版本管理把这些后续动作转成系统内可追踪、可验证的关系。

从效果上看，这组对象关系把资料、会话、生成结果和版本锚点放进同一条长期可管理的链路。当前作品中的历史查看、结果回流、后续 refine 和引用语义，都依赖这组关系维持稳定。评审若要判断这部分是否真实存在，可以同时看 Prisma 模型、预览返回中的版本字段，以及 focused tests 对 artifact/version/reference/member 语义的覆盖情况。
