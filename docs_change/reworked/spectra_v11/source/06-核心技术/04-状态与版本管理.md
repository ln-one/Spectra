## 成果对象状态与版本管理技术

这部分只回答一个问题：生成出来的结果如何继续留在系统里，而不是停留在一次下载。当前实现围绕 `Project`、`GenerationSession`、`Artifact`、`ProjectVersion`、`ProjectReference` 和 `ProjectMember` 这几组对象，把结果保存、替换、引用和成员可见范围接到了同一条正式链路上。

对象模型、接口返回和 focused tests 可以从以下三个层面核对：

| 证据类型 | 当前对应 |
| --- | --- |
| 对象模型 | Prisma 中明确存在 `Project`、`GenerationSession`、`Artifact`、`ProjectVersion`、`ProjectReference`、`ProjectMember` |
| 接口返回 | 预览与项目资料管理相关接口会返回 `artifact_id`、`based_on_version_id`、`current_version_id` 等结果标识 |
| 运行语义 | 结果创建、结果替换、正式记录复用、共享与引用权限均有 focused tests 覆盖 |
| 服务边界 | `Ourograph` 负责项目、结果、版本、引用这些正式对象语义，Spectra backend 负责把生成结果绑定到这些对象上 |

这组对象各自承担的事可以直接写清楚：

- `Project`：承接项目范围内的资料和结果归属；
- `GenerationSession`：承接一次具体工作过程和阶段状态；
- `Artifact`：承接一份正式结果对象；
- `ProjectVersion`：承接同一结果的历史版本；
- `ProjectReference`：承接资料和结果之间的引用关系；
- `ProjectMember`：承接成员可见范围和共享边界。

正文不需要把 `Ourograph` 写成一套抽象知识本体。对送审稿而言，最重要的是说明结果保存后的三个动作：

1. 生成结束后，结果会先落到 `Artifact`；
2. 若结果被修改或替换，系统会继续记录 `ProjectVersion` 和 `based_on_version_id / superseded_by_artifact_id`；
3. 若结果被其他资料或成果继续使用，系统通过 `ProjectReference` 记录这层关系。

这三步能直接对上页面行为。用户在 preview、history、replace、download 里看到的不是几张临时页面，而是已经进入正式对象链的结果。相关接口会返回 `artifact_id`、`based_on_version_id`、`current_version_id` 等字段，artifact 相关 tests 会检查替换关系和 authority artifact 复用，项目语义 tests 会检查共享与引用边界。

成员边界与登录会话由 `Limora` 承接，其认证实现使用 Better Auth 的会话与组织能力。这里要说明的不是抽象身份体系，而是一个很实际的结果：谁上传的资料、谁生成的结果、谁能看到这个项目里的成果，当前都落在同一套成员与组织关系上，因此结果对象不会脱离正式归属单独漂在外面。

图 6-6 对应的就是这条保存与复用过程：一次生成执行结束后，系统先得到结构化结果，再通过 `Artifact` 绑定正式身份，随后进入 `ProjectVersion` 与 `ProjectReference` 关系，供后续查看、替换、比较和再次使用。

这一章只需要保留这个判断：当前结果并不只停留在“导出成文件”这一层。它已经进入对象模型、接口返回和 focused tests 能同时核对的正式管理链。对外部评审来说，看到这一点就够了。
