# 1、外挂向量知识库

外挂知识库的本质在于不修改基座模型参数，通过提示词工程将特定知识作为 prompt 中的 context，即召回相关性最高的几个文档，让模型分析这些蕴含知识后，并返回答案。知识库适合要求输出明确且精度高的任务。知识库的质量和模型的回答能力息息相关，所以获取权威的数据、进行有效处理后放入外挂向量知识库非常重要。

# 2、数据获取技术

# - 爬虫

爬虫是获取网页并提取和保存信息的自动化程序。主要包括：获取网页、提取信息、保存数据、自动化实现。[1]

# - robots.txt

robots.txt 是一种用于网站根目录下的文本文件，它通过“爬虫排除标准”（Robots Exclusion Protocol, REP）告诉网络爬虫（也称为机器人或搜索引擎爬虫）哪些页面可以被抓取，哪些不应该被抓取。这个标准并不是由任何正式的互联网标准组织制定的，但它已被广泛接受和实施，成为了搜索引擎索引网站内容时的一种自律行为准则。

- robots.txt文件主要包括以下两种类型的指令：

(1)User-agent:指定这条规则适用于哪些爬虫。使用\*可以代表所有的爬虫。

(2) Disallow: 指示爬虫不被允许访问的 URL 路径。空值（即 Disallow:）表示允许访问所有内容。

此外，robots.txt文件也可以包含Allow指令（在某些爬虫中支持）来明确允许访问某些路径，以及其他扩展指令（如Sitemap），指向网站的Sitemap文件位置，帮助搜索引擎更高效地抓取网站。

- 注：本次数据的获取，完全遵守大众点评的 robots.txt。

# 2、Requests

Requests 是一个 Python 中的 HTTP 库，用于向 Web 服务器发送 HTTP 请求并获取响应。Requests 库的使用方式与 urllib 库类似，但更加简单和灵活。Requests 库支持 HTTP 请求，并且可以自动处理 cookies 和会话，使得操作更加方便。

- Requests 库的主要特点如下：[2]

(1) 简单易用：

Requests 库的使用方式简单明了，可以很容易地编写出高效、可靠的 HTTP 请求。

(2) 自动处理 cookies 和会话：

Requests 库可以自动处理 cookies 和会话，使得在处理 Web 服务器时更加方便和安全。

(3) 支持 HTTPS:

Requests 库支持 HTTPS 请求，可以在安全的环境下发送 HTTP 请求。

(4) 可靠性高：

Requests 库的请求是异步的，这意味着它们可以在后台异步执行，不会阻塞主线程。此外，Requests 库还支持连接池，可以确保每次请求都会使用相同的连接，提高了请求的可靠性。

(5) 多路复用：

Requests库支持多路复用，可以将请求分派给多个异步线程处理，提高了处理效率。

# 3、Session和Cookies

在Web中，Session对象用来存储特定用户Session所需的属性及配置信息。这样当用户在应用程序的页面之间跳转时,存储在Session对象中的变量将不会丢失,会在整个用户Session中一直存在下去。当用户请求来自应用程序的页面时,如果该用户还没有Session,那么Web服务器将自动创建一个Session对象。当Session过期或被放弃后,服务器将终止该Session。

在Web中，Cookie则是指某些网站为了鉴别用户身份进行Session跟踪而存储在用户本地终端上的数据。Cookie和Session需要配合,一个在客户端,一个在服务端,二者共同协作,就实现了登录控制。

# - XPath

XPath表示XML路径语言。它使用非XML语法来提供一种灵活地定位（指向）XML文档的不同部分的方法。它也可以用于检查文档中某个定位节点是否与某个模式（pattern）匹配。XPath使用路径标识符（类似于URL）通过层级结构来导航XML文档。它使用非XML语法，使得它可被用于URI和XML属性值。

# - CSS 选择器

CSS（层叠样式表）用于设置和布置网页——例如，更改内容的字体、颜色、大小和间距，将其拆分为多个列，或添加动画和其他装饰功能。而CSS中选择器就是来指定网页上我们想要样式化的HTML元素。有CSS选择器提供了很多种方法，所以在选择要样式化的元素时，我们可以做到很精细的地步。

# - 正则表达式

正则表达式是一种文本模式，包括普通字符（例如，a到z之间的字母）和特殊字符（称为"元字符")，可以用来描述和匹配字符串的特定模式。正则表达式是一种用于模式匹配和搜索文本的工具。正则表达式提供了一种灵活且强大的方式来查找、替换、验证和提取文本数据。

# 7、获取图片的base64编码

# - base64编码的获取

Base64 是网络上最常见的用于传输 8Bit 字节码的编码方式之一，Base64 就是一种基于 64 个可打印字符来表示二进制数据的方法。可查看 RFC2045～RFC2049，上面有 MIME 的详细规范。Base64，就是包括小写字母 a-z、大写字母 A-Z、数字 0-9、符号 "+"、"/"一共 64 个字符的字符集，（任何符号都可以转换成这个字符集中的字符，这个转换过程就叫做 base64 编码。Base64 编码是从二进制到字符的过程，可用于在 HTTP 环境下传递较长的标识信息。采用 Base64 编码具有不可读性，需要解码后才能阅读。[4] 对于 CSS 的前端网页来说，嵌入 base64 编码后的图片的方式相对固定，即采用 background-image: url(data:image/png;base64,base64 编码);的格式，故而采用正则表达式可以迅速获取 base64 的编码，正则表达式的字符匹配为：'background-image: url\"(data:image/p ng;base64,(\.*)"')';。

# - base64的解码

本项目采用python中的base64包的b64decode方法进行解码，然后保存图片于本地。由于b64decode只能对长度为4的倍数的base64编码进行处理，故而在进行转换之前需要进行预处理。首先，去除编码前后的'换行符'、'缩进'以及'空格'。其次，如果编码长度不为4的倍数，则使用'='进行填充，使其成为4的倍数。

# 计算初始距离：

我们需要通过 open-cv 计算初始距离，过程如下：

(1) 初始化变量 left, left $\leftarrow 0$ 。
(2) 以列为单位，从左到右获取每列的像素，并存于数组中。
(3) 初始化变量 cloumn, cloumn $\leftarrow 0$ 。
(4) 如果 cloumn 列均为透明像素则, left $\leftarrow$ left + 1, 则转到步骤 5, 否则转到步骤 6。
(5) $\text{cloumn} \gets \text{cloumn} + 1$ ，转到步骤4。
(6) 返回 left。

上述过程中，得到的left，即为我们想要算出的初始距离。

# 10、模拟人为拖动

# - 模拟人为拖动的距离计算

(1) 开始 - 流程的起点。
(2) 初始化变量 - 包括总位移 dis, 当前位移 current, 中间阈值 mid (初始加速阶段和减速阶段的界限, 人为设定为 $4 / 5 * \mathrm{dis}$ ), 时间间隔 t, 以及初始速度 v。
(3) 判断是否到达目标位移 - 如果当前位移小于总位移, 则继续计算下一段位移;否则, 转到步骤 6。
(4) 计算加速度 $\mathbf{a}$ -如果当前位移小于中间阈值, 则以较大的加速度 (表示加速) 计算; 如果当前位移大于中间阈值, 则以较小的加速度 (表示减速) 计算, 加速度与接近目标的距离动态相关。
(5) 计算并记录位移 - 根据当前速度和加速度计算本时间段内的位移, 更新总位移,并记录该段位移。回到步骤 3。
(6) 最后调整 - 对于累计的位移与总位移之间的小差异进行调整, 确保精确贴合目标位置。

(7) 结束 - 流程的终点，返回所有计算的位移作为移动轨迹。

# 模拟人为拖动

使用 selenium 中的 ActionChains（动作链）方法中的 click_and_hold 方法，实现点击并按住不放这一效果。然后使用 ActionChains 方法中 move_by_offset 方法实现上面记录的每段位移，从而达到模拟人为拖动的效果。最后 ActionChains 方法中的 release 方法结束动作链。

# 11、测试和异常处理

- 经过测试，存在一些异常问题：

(1) dddcor 项目对于拼图缺口识别准确率为 $90\%$ ，故而不能保证每次都能成功验证。
(2) 模拟人为拖动偶尔还是会被识别出来是机器人操作，然后页面会跳转至异常界面，影响后续爬取数据。

- 故而需要一些异常处理机制：

(1) 刷新机制：如果仅仅是错误识别了缺口位置，而导致没有滑动到对应位置，那么就点击刷新获取新的拼图，使用 CSS 选择器查找，其表达式为[alt="refresh"].
(2) 防异常跳转：如果刷新太多次数或者被识别出来是机器人操作，页面会跳转到异常界面，在这种情况下需要等待 2 分钟，然后刷新页面(refresh)，重新进行一遍上述的自动通过滑动验证码的流程。

# 1、数据处理

获取了原生数据以后，需要对文本进行一定的加工处理，去除无意义的符号和空格以及缩进，删除不合理的数据以及参考意义较小的数据。

- 本项目对数据进行以下部分的处理：

(1) 使用正则表达式，筛选出每个数据的关键信息，去除字段开头和结尾无意义的引号、空格、缩进。
(2) 筛选高质量数据，仅仅保留评级在3星以上的相关数据。
(3) 删除无意义数据，有些地点不在本次城市爬取的范围内，可能是本次城市和其他城市的交界处的数据，应当删除。

# 2、数据分块一QA导入

选择文件导入时，可以选择QA拆分方案。仍然需要使用到句子分词器对文本进行拆分，但长度比直接分段大很多。在导入后，会先调用大模型对分段进行学习，并给出一些问题和答案，最终问题和答案会一起被存储到q中。本项目采用了QA问答

# 6.3 模型的微调

通用LLM是在大型且多样化的数据集上进行培训的，而特定领域的LLM是专门为特定领域设计的，并在满足组织特定要求的专有数据集上进行培训。通用LLM对许多企业都有用，但特定领域的LLM更适合在特定领域内需要准确且适合上下文的输出的情况。这些模型的性能通常优于作为ChatGPT基础的GPT-3.5等广泛模型，具有高度针对性，并且可能更高效。

但是在模型的微调过程中，有很多关键因素是需要我们考虑的，模型应该如何选择？是选择基座模型还是chat对话模型？如何构建数据集？选择什么样的预训练方式？选择什么样的微调方式？都是需要我们考虑的，所以我们下面将围绕这几个方面进行讲解。

# 6.3.1 基座模型的选择

开源的LLM通常都会提供base和chat两种模型，首先明确base模型和chat模型的区别：

- base 模型是在大规模语料上以 LM 这种形式的预训练任务（预测下一个 token）训练得来的；
- chat模型是在base模型上sft+rlhf上微调得来的。

但是选择base还是chat模型通常会影响到最后的模型微调后的效果，通过已有的实验和数据表明，在数据量多的情况下，选择base模型可以更好的进行微调，并且模型不会产生遗忘等效果。并且在base模型上进行微调，并不会破坏掉模型原本的通用能力，从而实现完美的知识注入，我们选取的基座模型是qwen-1.5-7Bbase模型。

# 6.3.4 LORA微调

本项目是我们采取 LORA 微调的方式，首先来介绍一下 LORA 微调。

- 微软和 CMU 的研究者指出，现有的一些 PEFT 的方法还存在这样一些问题：

(1) 由于增加了模型的深度从而额外增加了模型推理的延时，如Adapter方法。
(2) Prompt 较难训练，同时减少了模型的可用序列长度，如 Prompt Tuning、Prefix Tuning、P-Tuning 方法。
(3) 往往效率和质量不可兼得，效果差于 full-finetuning。

有研究者对语言模型的参数进行研究发现：语言模型虽然参数众多，但是起到关键作用的还是其中低秩的本质维度（low intrinsic dimension）。受到该观点的启发，提出了 Low-Rank Adaption(LORA)，设计了如下所示的结构，在涉及到矩阵相乘的模块，引入 A、B 这样两个低秩矩阵模块去模拟 Full-finetune 的过程，相当于只对语言模型中起关键作用的低秩本质维度进行更新。

![](images/a428b79790acfce81dbb0c6341e88cbae260b97554427ddf4778679a933ec30b.jpg)
图6-27 LORA微调

# - LORA 的优点如下：

(1) 相比于原始的Adapter方法“额外”增加网络深度，必然会带来推理过程额外的延迟，该方法可以在推理阶段直接用训练好的A、B矩阵参数与原预训练模型的参数相加去替换原有预训练模型的参数，这样的话推理过程就相当于和Full-finetune一样，没有额外的计算量，从而不会带来性能的损失。
(2) 由于没有使用 Prompt 方式，自然不会存在 Prompt 方法带来的一系列问题。
(3) 该方法由于实际上相当于是用 LoRA 去模拟 Full-finetune 的过程，几乎不会带来任何训练效果的损失。

# 2、模型的测试（二）

对于本项目训练的模型的测试，我们进行Query + Response Relevancy，查询与响应相关性评估。

- Query + Response Relevancy: 是评估用户提出的查询（问题）和模型生成的回应（答案）之间的相关性和准确度的一个过程。这个概念在自然语言处理（NLP）领域特别重要，尤其是在对话系统、搜索引擎、推荐系统等应用中。

测试的逻辑如图所示：首先提出问题，然后根据该问题模型的回答与该问题的来源，从而可以评测是否查询与响应有相关性。

![](images/1fafb7fbf0f325c9a98b71cc3433ef4e6e3e62ce7a8fe5fe8238928ce9de256c.jpg)
图6-51模型测试（二）的逻辑

# 该评测有三个指标：

(1) Passing（通过）：这是一个布尔值，指示回应是否通过了相关性的测试。True表明回应被认为是与查询相关的。
(2) Feedback（反馈）：这通常是一个文本值，提供了对于评估结果的额外信息或反馈。在你的示例中，反馈是YES，意味着回应与查询非常相关。

(3) Score（分数）：这是一个数值，通常在 0 到 1 之间，表示回应的相关性程度。分数越高，表示回应与查询的相关性越强。在你的示例中，分数是 1.0，表明回应与查询完全相关。
