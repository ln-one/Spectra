# 最小契约草稿：公共库输入 v1

> 更新时间：2026-03-30
> 状态：当前讨论稿
> 用途：用于快速对齐“公共库”在当前阶段的最小产品语义与最小实现方式。

## 1. 当前结论

当前阶段，公共库不作为独立复杂系统入口先做完整管理，而是作为一种**特殊输入源**接入生成链路。

产品层理解：

- 上传文件是输入
- 选择公共库也是输入
- 两者一起构成本次生成的输入集合

系统层理解：

- 公共库底层仍然使用现有 `Project reference` 语义实现
- 不另起一套平行“公共库专用协议”

因此，当前统一结论是：

**公共库在产品层是特殊输入，在系统层是引用关系。**

## 2. 本阶段不做的方向

当前阶段不优先做：

1. 先做完整长期公共库管理系统
2. 先把 `Library / 库` 整体产品概念完全重构后再做公共库选择
3. 把公共库选择混进 `rag_source_ids`
4. 做多个 `base` 公共库

## 3. 公共库最小产品语义

### 3.1 公共库是什么

公共库是可被当前项目引用的共享 `Project`。

最小成立条件：

- `visibility = shared`
- `is_referenceable = true`

### 3.2 公共库如何被用户理解

用户不需要先理解复杂的 `reference` 数据模型。

当前阶段的产品表达应尽量简单：

- 这是可以加入本次生成的公共知识库
- 它和上传文件一样，都是输入来源
- 但它比普通文件更特殊，因为它代表一整个可复用知识空间

### 3.3 公共库选择放在哪里

当前阶段优先方案：

- 放在上传按钮后的输入窗口中
- 与上传文件的动作放在同一处

原因：

1. 用户会自然理解为“本次生成要带哪些输入”
2. 更符合演示需求
3. 不必等待 `Library / 库` 整体概念先完全收口

## 4. 公共库最小交互契约

### 4.1 用户最少需要完成的动作

在同一个输入窗口里，用户至少可以：

1. 上传文件
2. 选择一个或多个公共库
3. 为每个公共库指定引用模式
4. 查看本次已选文件和已选公共库

### 4.2 引用模式

公共库必须支持两种模式：

1. `follow`
2. `pinned`

建议前端文案：

- `跟随库更新` -> `follow`
- `固定当前版本` -> `pinned`

### 4.3 base 规则

当前阶段不支持多个 `base` 公共库。

约束：

1. 最多一个主基底公共库
2. 其他都作为辅助公共库

建议前端产品规则：

- 如果用户选择第一个公共库，可允许其作为主基底
- 后续新增公共库默认作为辅助引用
- 若交互复杂，本阶段也可以统一先全部按辅助引用处理，再在后端或后续版本细化主基底入口

## 5. 后端最小实现契约

### 5.1 底层落地方式

公共库选择不新增一套新语义，底层仍然写入当前项目的 `reference`：

- `relation_type = base | auxiliary`
- `mode = follow | pinned`

### 5.2 本次生成前的最小行为

如果用户在输入窗口中选了公共库，则在本次生成真正开始前：

1. 先将这些公共库写为当前项目的引用关系
2. 再走现有生成/RAG 主链路

这样可以最大化复用现有实现：

- RAG 检索已经支持当前项目引用目标自动参与检索
- 不需要重新设计公共库专用检索协议

### 5.3 pinned 的最小约束

若公共库选择为 `pinned`，则必须明确对应的 `pinned_version_id`。

因此：

- 前端若允许用户选 `pinned`
- 就必须能拿到该公共库当前可固定的版本 ID

若本阶段前端拿版本选择太复杂，可采用最小保底：

- `pinned` 默认固定到当前最新版本

但底层仍应落成真实的 `pinned_version_id`，不能只停留在 UI 文字。

## 6. 与现有检索语义的关系

### 6.1 不改现有主排序语义

当前阶段继续沿用现有检索优先级：

1. `local_session`
2. `local_project`
3. `reference_base`
4. `reference_auxiliary`

### 6.2 不混用 rag_source_ids

`rag_source_ids` 当前继续只表示：

- 严格限定文件范围的检索

不表示：

- 选择哪个公共库

结论：

- 公共库选择走 `reference`
- 文件限源走 `rag_source_ids`

两者不要混成一个字段。

## 7. 前端最小字段契约

公共库列表最少需要向前端暴露：

1. `project_id`
2. `name`
3. `description`
4. `visibility`
5. `is_referenceable`
6. 可选的主题或标签信息
7. 可用于 `pinned` 的当前版本信息

本次已选公共库项最少展示：

1. 公共库名称
2. 当前选择的引用模式
3. 是否作为主基底或辅助引用

## 8. 建议的最小请求结构

当前阶段可以在生成前的输入提交中增加一组公共库输入信息，例如：

```json
{
  "project_id": "proj_current",
  "output_type": "ppt",
  "options": {
    "rag_source_ids": ["file_1", "file_2"],
    "public_library_inputs": [
      {
        "project_id": "proj_public_math",
        "relation_type": "base",
        "mode": "follow"
      },
      {
        "project_id": "proj_public_images",
        "relation_type": "auxiliary",
        "mode": "pinned",
        "pinned_version_id": "ver_123"
      }
    ]
  }
}
```

说明：

- 这只是当前阶段建议字段
- 核心目的不是冻结最终 OpenAPI，而是提供一个足够轻量的联调契约

## 9. 当前阶段建议的实现顺序

1. 前端先把公共库作为“上传后的特殊输入”放进输入窗口
2. 后端补最小接收字段
3. 后端在生成前把公共库输入转成真实 `reference`
4. 复用现有 RAG 引用检索逻辑
5. 前端在结果页或输入区展示本次使用了哪些公共库

## 10. 仍待后续细化但不阻塞当前阶段的问题

以下问题可以后续再细化，不阻塞当前阶段最小契约成立：

1. 是否允许用户长期保存“默认公共库组合”
2. 是否允许在 `Library / 库` 主界面中集中管理公共库引用
3. `pinned` 是否允许用户手动选择历史版本，而不是默认最新版本
4. 公共库是否需要更细的分类体系
5. 公共库是否需要专门的封面、说明页或推荐理由展示

## 11. 当前结论

当前阶段，公共库的最小契约已经可以收束为：

1. 公共库是特殊输入源
2. 交互上放在上传后的输入窗口
3. 每个公共库都必须支持 `follow / pinned`
4. 最多一个主基底公共库
5. 底层仍然落到 `Project reference`
6. 不与 `rag_source_ids` 混用

这已经足够支撑当前阶段的实现、联调和演示。
