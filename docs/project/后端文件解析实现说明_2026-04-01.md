## 2. 当前支持的解析方法

当前后端文件解析支持以下几类 provider：

1. `local`
2. `llamaparse`
3. `mineru_cloud`
4. `mineru_api`

其中：

- `local`：本地解析
- `llamaparse`：LlamaParse 云端解析
- `mineru_cloud`：MinerU 官方精准解析云 API
- `mineru_api`：自建 MinerU FastAPI 服务

当前项目实际推荐的 PDF 生产策略是：

- 常规文件默认走 `llamaparse`
- 复杂高质量 PDF 走 `mineru_cloud`
- 任一云端失败最终回退 `local`

## 4. 每种方法是怎么实现的

### 4.1 `local`

#### 接的是什么

- 不接任何远端服务
- 完全在后端本地进程内解析

#### 实现方式

- PDF：`pypdf`
- Word：`python-docx`
- PPT：`python-pptx`

#### 优点

- 速度最快
- 不依赖外网
- 适合作为最终兜底

#### 缺点

- 结构恢复较弱
- 对复杂版式、公式、扫描件能力有限

#### 适用场景

- 本地快速解析
- 云端服务失败时的最终 fallback

### 4.2 `llamaparse`

#### 接的是什么

- LlamaParse 云端 API
- 依赖 `LLAMAPARSE_API_KEY`

#### 实现方式

后端通过 `llamaparse_provider.py` 调云端服务，拿回解析文本。

特点：

- 属于远端解析
- 不需要自己部署解析服务
- 相比 `local` 结构与文本覆盖更好

#### 优点

- 常规 PDF 质量较稳定
- 相比 `local` 文本覆盖更高
- 比 `mineru_cloud` 更适合承担默认云端解析路径

#### 缺点

- 明显慢于 `local`
- 对大文件、部分页段存在兼容性问题，需要拆分策略辅助

#### 适用场景

- 常规 PDF 默认解析
- 日常主链路使用

### 4.3 `mineru_cloud`

#### 接的是什么

- MinerU 官方精准解析云 API
- 依赖 `MINERU_CLOUD_API_TOKEN`
- 这是官方远端云服务，不是本地自建 MinerU

#### 实现方式

当前实现采用官方异步上传解析流程：

1. 调用 `POST /api/v4/file-urls/batch`
2. 获取文件上传链接
3. 将本地文件 `PUT` 到官方返回的上传地址
4. 轮询 `GET /api/v4/extract-results/batch/{batch_id}`
5. 任务完成后下载 `full_zip_url`
6. 从 zip 中提取 `full.md`

对应实现文件：

- [`e:\ZRJ666\666\Service_Competition\Spectra1\backend\services\parsers\mineru_cloud_provider.py`](e:\ZRJ666\666\Service_Competition\Spectra1\backend\services\parsers\mineru_cloud_provider.py)

#### 优点

- 当前三种主方案里文本覆盖最强
- 对复杂版式、图表、公式类 PDF 更有优势

#### 缺点

- 速度慢于 `llamaparse`
- 依赖官方远端服务
- 当前账号有默认额度，但应视为存在额度/限流约束的云服务

#### 适用场景

- 扫描件
- 公式较多 PDF
- 图表较多 PDF
- 版式复杂 PDF
- 对文本完整性要求很高的资料

### 4.4 `mineru_api`

#### 接的是什么

- 自建 MinerU FastAPI 服务
- 不是官方云端精准解析 API

#### 实现方式

后端通过 HTTP 调本地或远端自建 MinerU FastAPI，例如：

- `POST /file_parse`

支持：

- 自动发现 OpenAPI 接口
- `files` 这类 multipart 字段上传

#### 当前项目中的定位

这条链路已经接通，但在当前单机 CPU 环境下太慢，不适合作为默认主路径。

#### 适用场景

- 本地实验
- 自建服务验证
- 非当前主推荐路径

## 5. 当前运行时选择逻辑

### 5.1 环境变量入口

主开关：

- `DOCUMENT_PARSER`

可选值包括：

- `local`
- `mineru`
- `mineru_api`
- `mineru_cloud`
- `llamaparse`
- `auto`

### 5.2 当前 `auto` 路由

当前实现中：

- PDF：优先 `mineru_cloud`
- Word / PPT：优先 `llamaparse`
- 其他：`local`

说明：

- `auto` 是代码层默认路由
- 实际生产策略是否直接使用 `auto`，仍应结合质量与速度策略决定

## 6. 当前推荐的生产使用建议

当前建议：

1. 常规文件默认走 `llamaparse`
2. 复杂高质量 PDF 走 `mineru_cloud`
3. 所有云端解析失败最终回退 `local`

补充说明：

- 大文件拆分目前仍建议作为后续质量优化项单独推进
- 不建议在当前阶段把 `mineru_cloud` 直接设为所有 PDF 的默认路径

## 9. 如何验证解析链路是否真的生效

上传一个新的 PDF 后，查看文件详情接口中的 `parse_result`。

重点看：

- `provider_used`
- `provider_attempted`
- `parser_routing`
- `capability_status`

判断标准：

- 如果 `provider_used = mineru_cloud`，说明官方 MinerU 云 API 已真实生效
- 如果 `provider_used = llamaparse`，说明当前实际走的是 LlamaParse
- 如果 `provider_used = local`，说明最终回退到了本地解析

## 10. 成员实操建议

如果其他成员要复现相同能力，建议按这个顺序：

1. 先理解统一入口：`file_parser/__init__.py`
2. 再看 provider registry：`parsers/registry.py`
3. 再看具体 provider 实现
4. 配置对应的 env
5. 重启容器
6. 上传真实文件验证 `provider_used`

不要只看配置，不看实际返回结果。

真正判断是否接通，必须以文件详情接口里的 `provider_used` 为准。
