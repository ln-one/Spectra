# Spectra 商业方案书写作指导

> Status: `current`
> Role: internal writing guide for rewriting the external commercial proposal from current system reality.

## 1. 基本原则

这次不是“修旧稿”，而是重写一份真正能独立成立的商业方案书：

- 借旧 `Spectra.docx` 的目录骨架和商业方案书节奏
- 以当前系统真实结构为准重写正文
- 采用 `Markdown-first, DOCX-second`

必须记住：

- 文档不是唯一真相源
- 必要时必须回到代码实现、测试和当前服务 README/docs
- 如果旧文档写得漂亮但已经过时，必须重写，不迁就旧叙事
- 默认读者看不到仓库、Markdown 链接和任何源文件
- 长章节不要硬顶在一个文件里；当一章同时承载产品面、架构面、证据面与图示面时，默认拆成内部源稿后再汇总
- 第 5 章和第 8 章都已经采用“对外主章 + 内部源稿”结构；继续深写时先改源稿，再提炼到主章
- 第 4 章现已采用“对外主章 + 内部源稿”结构；系统设计继续深写时，先在 `04-architecture-src/` 收口系统本体、Studio 产品面、控制平面、主链和运行现实，再汇总回主章

这套工作流额外吸收三类成熟方法论，但会按 `Spectra` 的商业主稿需要改写使用：

- audience-first：先回答外部读者、采购方和技术评估者要判断的事，再展开实现
- progressive disclosure：先系统总图，再主链，再分组机制，不把细节一上来全塞给读者
- content separation：对外主稿负责 explanation，内部源稿负责更深的事实、代码锚点与图示控制

同时采用 AI-assisted writing 的操作准则：

- 先建 source pack，再下笔
- 先定章节框架，再写段落
- 先按能力组写，再汇总成主章
- 每写完一组就回到代码、测试、产品面做一次 truth-check
- polished prose 只能出现在最后一层，不能一开始就追求“写得漂亮”

## 2. 当前系统的写作总口径

### 2.1 Spectra

`Spectra` 要写成：

- recursive knowledge system 的产品外壳
- `workflow shell`
- `orchestration kernel`
- `contract surface`

不要写成：

- 大一统后端
- 所有产品能力的 owner
- 纯空心网关

### 2.2 六个正式能力源

后续文档统一按六服务讲系统：

- `Diego`：AI PPT / 课件生成核心
- `Pagevra`：render / preview / export 核心
- `Ourograph`：formal knowledge-state core
- `Dualweave`：ingest / delivery 技术底座
- `Stratumind`：retrieval / RAG 技术核心
- `Limora`：identity / session / organization membership authority

### 2.3 Spectra 与 Ourograph 的关系

必须稳定写成：

- `Ourograph` 是知识状态与知识库语义核心
- `Spectra` 是它的 consumer / workflow shell

不要写成：

- Spectra 是知识库 owner
- Ourograph 只是 Spectra 内部一个 service

## 3. 写法主线

所有正文优先使用这条表达链：

1. 要解决什么问题
2. 系统是什么
3. 为什么这样分层
4. 为实现哪些功能采用了哪些技术
5. 最终做成了什么、效果如何

技术章节不要按技术名词铺开，而是按问题域分组：

1. 知识状态与知识演化
2. 会话编排与生成主链
3. 渲染、预览与导出
4. 上传解析与交付
5. 检索增强与证据组织
6. 身份与组织容器

每组统一写成：

1. 要解决的问题
2. 传统方案为什么不够
3. 采用了什么服务/技术
4. 核心机制或协同方式
5. 结果与系统价值

对超长技术章节，再增加一个内部写作要求：

6. 当前代码、测试、前端产品面或 benchmark 能证明什么
7. 本节 Mermaid 图与图文边界

补充一条章节组织原则：

- 外部读者首先要看懂系统是什么、做成了什么、为什么可信
- 只有在这三个问题已经被回答后，才继续展开实现机制和服务边界
- 如果一节上来就是技术名词、服务名和模块名，通常说明顺序写反了
- 文案势能优先于平均铺陈：每章第一段先定性，第二段先说明这件事为什么重要，再进入机制、事实和证据
- 每章结尾都要回答“为什么这值得继续读、继续买、继续建”，不能只做信息回收
- 能放在段落第一句的强判断，不要放到第三句
- 第 3 章优先写“系统级需求必然性”，不写成功能罗列或需求规格书口气
- 第 5 章优先写“能力问题 -> 技术承接 -> 实现结果”，不写技术名词堆砌或服务说明书口气
- 第 4 章优先写“设计必要性与结构合法性”，不写成服务说明书或普通分层介绍
- 第 6 章优先写“系统成果与证据结论”，不写成评测仓库或 benchmark 堆栈
- 表格左列与列表首词优先使用信息承载词，而不是铺垫性短语
- 图前导语优先回答“这张图解决什么判断问题”，而不是泛泛解释图里有什么
- 第 5/6 章进入最终交稿阶段时，优先压密而不是扩写
- 表格和图已经承载的信息，正文不再平铺复述
- 全稿定稿化阶段优先删回声、删桥句、删作者口气，不再继续堆同义判断

补充一条证据分层原则：

- 第 5 章负责讲关键技术如何支撑产品能力，只保留结论级证据摘要
- 第 4 章负责系统层设计判断，优先回答“系统本体是什么、Studio 是什么、控制平面如何成立、六个正式能力层如何自然落位”
- 第 4 章禁止深入实现层叙述；若一段话的核心是在解释“怎么做、为什么技术上有效、各服务如何协同实现”，默认应进入第 5 章
- 第 6 章优先回答“已经做成了什么”与“为什么这些结论可信”，详细测试设计、测试集和指标表只服务于这个目标
- 第 7 章负责组织管理，不承接测试数据
- `Stratumind` 的正式集细节和完整指标表进入第 6 章；第 5 章只写“正式集结果显示 advanced 链路显著领先”这类摘要
- `Dualweave` 没有正式最新场景报告前，只写场景化评估方法和可观测指标，不写具体提升倍数

补充一条商业长章规则：

- 第 8 章负责讲商业生态，不只是列收费项；
- 商业逻辑继续扩写时，先进入 `08-business-plan-src/`；
- 主章只保留外部读者能直接理解的商业结论、阶段路径、收入层级和购买理由；
- 源稿负责保存课程资产网络、权限关系、商业飞轮、收入层级、市场进入和网络效应的深层推演；
- 不把未来平台生态、交易市场、学习者增值和收益分配写成当前已完整上线的事实。

补充一条 4/5 章去重规则：

- 第 4 章只讲系统设计，不重新展开实现机制；
- 第 5 章只讲关键技术与实现，不重新定义系统本体、总体分层和系统主链；
- 如果一段文字更偏“是什么 / 为什么这样分层 / 系统如何成立”，归第 4 章；
- 如果一段文字更偏“怎么做 / 用什么技术 / 为什么有效”，归第 5 章。

## 4. 对外文档表达要求

### 4.1 必须自成闭环

正文不能依赖：

- 仓库路径
- Markdown 链接跳转
- “读者可回看 README / 测试 / 文件”

必须做到：

- 不看仓库也能理解项目是什么
- 不看代码也能理解做成了什么
- 不看源文件也能看懂核心证据

### 4.2 先价值、后系统、再技术

商业主稿默认表达顺序：

1. 行业问题与客户价值
2. 项目定义与系统能力
3. 架构与能力边界
4. 关键技术如何支撑这些能力
5. 已做成的结果与证据

### 4.3 第 8 章商业生态飞轮规则

`08-business-plan.md` 后续必须从“订阅 + 机构交付”的普通 SaaS 写法，升级为“课程知识资产生态”的平台型商业模式。

核心口径：

- `Spectra` 的商业本体不是 AI 课件生成器，而是课程知识资产网络。
- 每个高质量 `Project / 课程库` 都是可引用、可授权、可外化、可演化、可交易的资产节点。
- `Reference / Version / CandidateChange / Member` 不只是技术对象，也支撑引用、授权、收益、权限和演化。
- 角色不是本体，权限组合和关系才是本体。教师、学生、创作者、学习者、机构管理员都应写成围绕课程库展开的权限组合。

第 8 章需要写出五层收入与增长路径：

1. 工具订阅层：个人教师、知识创作者、培训师、企业讲师购买订阅、额度、高级模型、导出能力和私有空间。
2. 课程资产层：优质课程库可以被引用、follow、fork、订阅、授权或购买，创造者获得收益，平台获得抽成。
3. 学习增值层：学习者围绕课程库购买个人导图、复习讲义、互动练习、错题强化、AI 助教和个性化学习路径等高级能力。
4. 组织平台层：学校、培训机构和企业大学购买私有化部署、校本资源接入、权限治理、品牌化和定制服务。
5. 长期智能网络层：平台积累课程库、引用网络、多模态成果和学习行为，逐步演进为 AI 原生个性化学习基础设施。

商业飞轮写法：

- 创作者低门槛建库；
- 学习者、其他创作者或机构引用、使用、外化课程库；
- 平台通过订阅、增值功能、交易抽成、机构交付和服务费变现；
- 使用越多，课程库、引用关系、学习行为和多模态成果越丰富；
- 数据和知识网络反过来增强 AI 能力，推动更多创作、学习和交易。

对外表达边界：

- 正式商业方案书不要使用攻击性替代表述，也不要把长期战略写成粗暴替代教师。
- 推荐写法是：从教师效率工具，演进为课程知识资产平台，再演进为 AI 原生个性化学习基础设施。
- 重点强调赋能优质创作者、放大课程资产、提升个性化学习、帮助机构治理知识资产。

## 5. 老师反馈对应动作

### 5.1 “没有看到具体做成什么”

所以必须补：

- 真实页面或结果截图
- 真实主链闭环描述
- 真实 artifact 或交付结果
- “做成了什么 / 解决了什么”的明确表述

### 5.2 “图太少”

所以必须做图：

- 总体系统架构图
- 六服务边界图
- 核心对象关系图
- 主链流程图
- 必要的结果图 / 系统截图

图的写法也要遵守成熟规范：

- 一图一事：每张图只回答一个核心问题
- 先整体、再局部：先让读者知道系统在哪，再解释其中一条机制
- 图必须可独立阅读：不依赖口头解释和仓库跳转
- 图上的文字优先短标签，复杂含义放图注和正文解释
- 低披露服务只画位置、关系和价值，不画可直接复刻的内部诀窍

### 5.3 “技术与功能关系不清楚”

所以每个技术段都要显式回答：

- 为实现什么功能
- 为什么选择这个技术
- 这个技术如何支撑这个功能

### 5.4 “第 6 章太碎”

所以关键技术章节必须按问题域分组，不再做技术名词堆砌。

### 5.5 “格式要调好”

所以流程固定为：

- Markdown 阶段先统一结构、标题、图文位置、图题编号
- DOCX 阶段再统一缩进、行距、页码、目录、页眉页脚

## 6. 披露边界

### 6.1 Dualweave

低披露。

可以讲：

- 系统价值
- 边界
- 工程难度
- 效果
- 在系统中的位置

不要讲：

- 专利敏感核心机制
- 可直接复刻的关键策略

### 6.2 Ourograph

低到中披露。

可以讲：

- 为什么 formal knowledge-state 重要
- ontology 的意义和优雅性
- 为什么它是知识库核心
- 为什么 Spectra 因为它而更强

不要讲：

- 可直接复刻 formal kernel 的详细设计
- 过深的操作性内部技巧

### 6.3 其他服务

可相对更充分展开：

- `Stratumind`：可结合 benchmark / report / recommendation 强调技术实力
- `Diego`：可强调生成主链、质量门、事件流、产物闭环
- `Pagevra`：可强调 render / preview / export 和服务边界
- `Limora`：可强调 identity / session / membership authority

## 7. 旧材料使用规则

### 7.1 旧 `Spectra.docx`

只可作为：

- 目录结构参考
- 部分背景/目标/商业/风险叙事素材
- 商业方案书语言节奏参考

不能作为：

- 当前技术真相源
- 当前系统边界来源

### 7.2 默认重写区域

默认整章重写：

- 系统设计
- 关键技术
- 任何旧 generation/render/formal-state/identity 主链叙事
- 任何把 Spectra 写回大一统后端的技术表述

## 8. 推荐写作顺序

先写最影响整体商业气质的：

1. `00-executive-summary`
2. `01-overview`
3. `08-business-plan`
4. `04-architecture`
5. `06-testing-evaluation`

再写其余主稿层：

1. `02-feasibility`
2. `03-requirements-analysis`
3. `05-key-technologies`
4. `07-organization-management`
5. `09-risk-management`
6. `10-conclusion`

最后汇总：

1. `90-submission-manuscript`
2. `91-submission-master`
3. `92-final-submission-draft`
4. `99-self-review`

## 9. 图示清单

建议至少准备这些图：

1. 总体系统架构图
2. 六服务边界图
3. Spectra control plane / orchestration kernel 图
4. `Project / Session / Artifact / Version / Reference / CandidateChange` 对象关系图
5. 上传 / 解析 / 索引 / 检索链路图
6. Session -> Diego -> Pagevra -> Ourograph 生成闭环图
7. 知识回流与版本演化图
8. 商业模式图或价值链图

所有图都必须基于当前系统现实，不得沿用过时链路图。

## 10. 术语与措辞提醒

建议稳定使用：

- `LLM`
- `RAG`
- `workflow shell`
- `orchestration kernel`
- `formal knowledge-state`
- `knowledge evolution`
- `retrieval-enhanced generation`

避免：

- 把旧执行术语写成当前产品模型
- 把局部能力写成全系统 owner
- 把“调用了很多技术”当成“讲清楚了系统”
