

### 2. 技术方案调研报告

#### 2.1 大语言模型 (LLM) 选型：核心推理引擎

针对**通义千问 (Qwen)** 进行了实际测试，重点考察其在"教育大纲逻辑规划"与"指令遵循（JSON 输出）"上的表现。

##### 2.1.1 模型路由方案 (Model Routing)

* **推荐方案**：采用 **LiteLLM** 作为统一路由层
 * 统一接口：`completion()` 函数适配 100+ 模型（Qwen/GPT/Claude）
 * 内置 fallback 和负载均衡
 * 原生支持流式输出（streaming）
 * 成本追踪：内置 token 计数与费用计算

* **不推荐 LangChain**：
 * 过度抽象，学习曲线陡峭
 * 调试困难，多层封装
 * 对于本项目场景，手写 Prompt 链路更可控

#### 2.2 文档解析方案：非结构化数据清洗

教学 PDF 含有大量公式和排版，实测了三种工具：

* **测试 A：MinerU (Magic-PDF)** —— **【选型建议】**
 * **测试表现**：对教材 PDF 的公式识别率很高，能将复杂的跨页表格转为完美的 Markdown。测试方法： https://www.bilibili.com/video/BV1UVnkzKEnk/?share_source=copy_web&vd_source=983756a0c941260116b321a289803d8f
 * **优点**：开源自研，针对中文文档优化，能识别阅读顺序。
 * **成本**：可本地服务器部署，无 API 费用，仅需消耗 GPU 算力。
* **测试 B：Docling (IBM 开源)**
 * 特点：速度快，但LaTeX 转化准确度不如 MinerU。
* **测试 C：多模态大模型直接读图**
 * **测试表现**：准确度高但无法批量化处理，Token 消耗极其恐怖（一页教材约 一万到十万token）。

总结：推荐使用MinerU

#### 2.3 向量数据库 (Vector DB) 选型：长效语义记忆

我们需要存储教材的 Embedding 分段。

| 方案 | 优劣势 | 性能与开发成本 | 建议 |
|:------------ |:--------------------------------------- |:------------------------------- |:-------------------------------- |
| **Milvus** | **优**：分布式架构，适合大规模数据；**劣**：部署非常重，需多组件运行。 | 检索延迟 < 50ms。开发成本中等（需维护服务）。 | **生产环境推荐**。 |
| **ChromaDB** | **优**：极轻量，单机运行；**劣**：高并发性能一般。 | 检索延迟约 100ms。**开发成本极低**（几行代码搞定）。 | **初赛/Demo 推荐**。Spectra 早期开发选它最快。 |

总结：推荐使用**ChromaDB**

#### 2.4 PPT/Word 生成方案：生产力转化引擎

 AI 难以控制“自动坐标排版”，所以优先采用“逻辑映射”方案。

1. **PPT 生成：Marp (Markdown to PPT) —— 【选型建议】**
 * **实操方案**：让 Qwen 输出带特定语法的 Markdown。后端通过 Marp 命令行工具调用预设的 CSS 教学主题（ Spectra 专属主题），一键生成精美 HTML5/PPTX 课件。
 * **优势**：排版 100% 稳定，风格统一，代码量极少。
2. **Word 生成：Pandoc**
 * **实操方案**：万能文档转换器。将 AI 生成的 Markdown 教案一键转为标准的 `.docx` 格式，支持页眉页脚和公式。

#### 2.5 建议调研模块（多模态与路由）

##### 2.5.1 Embedding 模型选型（RAG 核心）

* **候选**：Qwen 3 Embedding

##### 2.5.2 视频与图片处理方案

* **图片 OCR**：采用 **PaddleOCR** 进行文字定位 +LLM进行语义理解。
* **视频解析**：采用 **Faster-Whisper**（本地化语音转文字）+ **OpenCV 关键帧抽取**。通过时间轴将老师的讲解与关键画面对齐，生成“频谱课件”。
* **备选图片 OCR方案**：直接使用多模态大模型

##### 2.5.3 模型路由方案 (Model Routing)

* **逻辑**：编写一个简单的 `Router` 类。
 * 如果是 `Text-only` 或 `Chinese-heavy` 任务，自动分发给 **核心LLM API**；如果是带有 `Video-link` 或复杂 `Image` 识别任务，分发给 **多模态大模型** 接口 或 **具体视频与图片处理方案接口**。这能兼顾成本与性能。
