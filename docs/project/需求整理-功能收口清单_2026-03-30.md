# 功能收口清单

> 更新时间：2026-03-30
> 状态：规划中
> 用途：将阶段性随手记录整理为可执行的产品与实现清单，供比赛收口、研发排期、文档撰写统一使用。

## 1. 文档定位

本文档不是历史记录备忘，而是当前阶段的功能收口清单。

目标是把零散想法收束为与 Spectra 现有系统本体一致的执行方向，避免后续讨论继续停留在碎片化功能层面。

本清单默认遵循项目既有语义：

- `Project` 是统一知识空间
- `Session` 是生成与协作过程
- `Artifact` 是正式产物
- `PPT` 是当前阶段最核心的教学工件

## 2. 当前阶段总目标

当前阶段应将 Spectra 收口为一个面向教师场景的多模态教学共创系统。

系统主链路应明确为：

`教师表达需求 -> 结合项目知识库/公共知识库 -> AI 理解与生成 -> 产出 PPT 为核心工件 -> 配套生成 Word 教案与增强内容 -> 支持流式反馈与迭代修改 -> 结果进入项目空间复用`

当前阶段的总体目标：

1. 以 `PPT` 作为最核心的生成目标和交互中心。
2. 以 `Project` 作为统一知识库语义，复用现有项目空间逻辑支持公共知识库。
3. 以后端能力落地为主，前端只补足必要的可用性验证与演示承载。
4. 统一 UI 语义，减少碎片化卡片和文字噪音，提升教师理解成本与操作效率。
5. 保持模型、Provider、Artifact、Session、事件流语义一致，不引入新的平行体系。

### 2.1 本阶段约束

当前所有需求都必须以“能够在本阶段真正完成”为前提。

这意味着：

- 不是追求所有能力的最终理想形态
- 而是要求核心能力在本阶段形成可交付、可演示、可写进材料的完成版本

当前阶段的默认收口原则：

1. 主链路必须完整。
2. 关键观感必须成立。
3. 保底方案优先于空缺。
4. 可完成优先于理想化扩张。

### 2.2 本阶段完成定义

以下结果应视为本阶段必须完成的正式交付：

1. 可以选择项目库或公共库参与生成。
2. 可以生成正式 `PPT` 工件并下载。
3. 生成过程具备大纲逐块、PPT 逐页的渐进式展示效果。
4. `PPT` 关键页面尽量带图，不足时有明确保底图位方案。
5. 可以生成与 `PPT` 对应的 `Word` 教案。
6. 结果进入项目空间 `Artifact` 闭环。
7. 存在可独立验证后端能力的 `Mock UI`。

以下内容本阶段只要求做到可交付版本，不要求一步到终局：

1. 图片插入可以先做到稳定有图，不要求完全智能。
2. 流式体验可以先做到逐块、逐页，不要求 token 级逐字输出。
3. 教案生成可以先基于有限真实教师样本建立可靠模板，不要求覆盖全部学科场景。
4. 模型选型可以先形成当前阶段最优可用方案，不要求一次得出长期终局答案。
5. 正式前端可以先保证主演示链路清晰稳定，不要求一次覆盖所有能力细节。

## 3. 核心收口方向

### 3.1 PPT 主中心化

PPT 不是多个产物之一，而是主产物。后续功能设计要围绕 PPT 聚合，而不是继续扩散成大量平级卡片。

明确要求：

- 生成流程默认围绕 PPT 展开。
- Word 教案作为与 PPT 配套的从属产物。
- 教案创建时应直接选择对应 PPT，而不是暴露一组松散配置框。
- 小题、思维导图、动画、插图等内容，优先作为增强 PPT 的组成部分接入，而不是作为孤立结果展示。
- 当前阶段应优先把“小题、思维导图、动画、插图”这类结果塞回 `PPT` 主工件中，增强 `PPT`，而不是继续让这些能力各自漂浮。
- PPT 生成结果至少应包含：
  - 封面页
  - 目录页
  - 内容页
  - 结束或总结页
- PPT 工件标题、导出命名、展示命名需要统一规范，避免当前标题不准确、不正式的问题。

### 3.2 知识库体系统一化

不额外创建比赛专用知识库系统，直接复用已有 `Project` 语义。

明确要求：

- 继续以 `Project` 作为知识空间统一载体。
- 新增若干公共知识库 `Project`，用于承载比赛要求中的本地知识库资料。
- 用户在创建或使用自己的项目库时，可以选择是否引用这些公共库。
- 该机制应基于现有项目复用或引用逻辑实现，不单独新建一套平行公共资料库架构。
- 比赛提交材料中的本地知识库，应直接来自这些公共 `Project` 的内容准备与说明。
- 前端还需要继续设计 `Library / 库` 的产品概念，当前实现不能视为已经收口。
- `Library / 库` 不能只被做成一个文件列表或资料页，而应成为用户理解“资料从哪里来、生成依赖什么、公共库如何被引用”的统一入口。
- 前端需要把 `Project`、公共库、资料、图片资源、引用关系之间的关系表达清楚。
- 若 `Library / 库` 的概念在前端不清晰，后续公共库复用、图片资源回灌、生成引用路径都会显得混乱。
- 公共库中的具体内容不应凭空决定，需要由质量与资料侧先研究“哪些知识库值得做、每个公共库放什么”，再由主线后端确保其真实可用。

### 3.3 AI 图像能力双通道化

图片补充不能只依赖现有资料检索，必须同时支持知识库检索图片和模型生成图片。

同时，图像能力不应独立漂浮在系统外，而应尽量与 `3.2` 中的知识库体系统一化方案融合。

这里的目标不只是“支持图片”，而是要保证最终演示时生成结果具备足够强的视觉表现力。

明确要求：

- 支持两种插图来源：
  - 知识库或 RAG 检索已有资料图片
  - 调用模型生成新图片作为补充插图
- 对外演示时，核心生成结果必须尽量带图，不能大量出现纯文字 PPT。
- 若智能选图、图文匹配、自动布局暂时做不到足够稳定，则允许使用更强约束的模板化策略，确保最终结果稳定“有图可看”。
- 可以采用保底策略：
  - 优先检索库内图像资源
  - 检索不足时调用图片生成模型
  - 仍不稳定时强制套入预设图位模板和默认插图策略
- 当前阶段应优先保证“生成结果看起来丰富、完整、可展示”，再逐步提升智能度。
- 图像能力应作为 PPT 插图增强能力的一部分接入，而不是独立漂浮功能。
- 图像资源应尽量复用现有库逻辑管理，而不是额外建设一套平行图库系统。
- 可将图片与其对应的文字说明、适用主题、使用建议等内容一起作为资料项放入 `Project` 或公共知识库 `Project` 中。
- 当图片与说明信息共同进入库后，后续生成流程即可像引用其他知识资料一样引用这些图像资源。
- 这样可以将“图片检索”“图片说明”“图文配套引用”统一纳入现有知识库语义，形成更自然的系统闭环。
- 优先通过统一的图库或插图能力模块封装图像来源，但其底层仍尽量落在 `Project` 资料和引用逻辑之上，避免前端直接感知复杂后端差异。
- 后端需要明确记录图片来源类型：
  - `retrieved`
  - `generated`
  - `hybrid`
- 前端需要让用户清楚区分这是检索图还是 AI 生成图。
- 对于 AI 新生成的高质量图片，可进一步支持“回灌入库”：
  - 生成图片
  - 绑定说明文字和元数据
  - 保存为项目资料或公共库资料
  - 在后续 PPT/教案生成中被再次引用

推荐解释：

- 这是一种更符合 Spectra 本体的做法，因为图片不再是一次性插入物，而是可以沉淀、引用、复用、回流的知识资源。
- 这也能自然满足比赛对“本地知识库”的要求，因为图片及其说明本身就可以成为知识库内容的一部分。

### 3.4 流式响应主体验化

生成与对话不能继续停留在整包等待返回的体验，需要将流式反馈做成明确能力。

这里的目标不是仅提供阶段化反馈，而是要做出接近 Gamma 一类产品的“内容正在被实时构建出来”的观感。

明确要求：

- 生成链路支持基于现有 SSE 能力的流式状态更新。
- 目标体验应尽量接近：
  - 大纲一段一段出现
  - PPT 一页一页生成并进入预览区
  - 教案一段一段补全
  - 其他能力结果也以逐块生成的方式出现
- 若实现难度过高，不强求逐句级 token streaming，但至少要实现“按结构块输出”的流式体验。
- 推荐的最低实现粒度：
  - 大纲按章节或卡片块流式出现
  - PPT 按页流式出现
  - 教案按段落或小节流式出现
  - 小测、导图、动画说明、说课内容等按模块块状出现
- 目标不是只告诉用户“到了哪个阶段”，而是让用户实际看到内容被逐步构建出来。
- 这一体验应尽量覆盖所有关键能力，而不只是 PPT 主生成链路。
- 为了演示效果，允许在真实生成链路之上增加“结构块渐进展示层”，只要不伪造最终结果语义即可。
- 必须保留 RAG 超时、模型调用、渲染阶段的可观测性，不能为了炫酷而完全牺牲系统真实性和可调试性。

推荐解释：

- 比赛展示里，观众最容易感知价值的不是后台调度，而是“内容在眼前长出来”。
- 因此流式体验既是交互优化，也是演示层设计重点。
- 在工程上可以优先做“结构块流式”而不是“token 级流式”，用较低实现复杂度换更强观感。

### 3.5 前端体验减噪与语义重构

当前前端问题不是单一视觉问题，而是信息密度、卡片职责、语义表达三者同时混乱。

明确要求：

- 降低页面文字密度，优先图标化表达高频功能。
- 文字保留但必须更大、更短、更明确。
- 功能命名要去歧义，避免用户不知道按钮或卡片到底做什么。
- 将逻辑不同的能力明显分区，避免全部挤在同一层级。
- 对当前过碎的 AI 推荐内容卡片进行处理：
  - 删除无价值的切片化推荐
  - 或改为聚合型推荐摘要
  - 或改为更适合教师理解的结构化建议
- 本阶段前端只做正式产品界面中必要的可用性承载，不负责 `Mock UI`。
- 需要提供一套与正式复杂前端逻辑隔离的 `Mock UI` 或验证 UI，用于快速验证后端能力可接通、可演示。
- `Mock UI` 明确由后端负责人员实现，前端不接这部分职责。
- 这一约束用于确保后端能力能够被独立验证，避免因为前端排期、联调或复杂状态逻辑影响核心能力质量。

### 3.6 教学场景内容质量强化

内容生成不能停留在有内容，而要强调教学法合理、结构清晰、细节完整。

这里不能只靠主观想象写提示词，必须基于真实教师教案样本进行归纳和约束。

明确要求：

- 教案设计、结构规划、提示词编写，必须向真实教师获取教案样本作为参照依据。
- 需要主动收集一批真实可用的教案材料，而不是只依赖模型自行想象“像教案”的写法。
- 需要从真实教案中总结共性结构，例如：
  - 教学目标如何表述
  - 重难点如何拆分
  - 教学过程如何分段
  - 教学方法如何呈现
  - 活动设计如何落地
  - 时间分配如何安排
  - 板书、作业、延伸任务如何写
- 在总结共性的同时，要保留不同学科、不同课型之间的差异，不要把所有教案压成一种空泛模板。
- 提示词、结构化输出 schema、默认模板，都应建立在这些真实样本归纳结果之上。
- 必须保证最终生成内容有真实可参照的风格来源和结构依据，而不是只追求语言上“看起来像”。
- Word 教案生成必须强化以下维度：
  - 教学目标
  - 教学重点与难点
  - 教学过程
  - 教学方法
  - 课堂活动设计
  - 内容控制与时间分配
  - 课后作业与延伸
- PPT 内容也需要体现教师视角，而不是纯知识点堆砌。
- 生成策略要更贴近教师备课和说课需求。
- 内容组织应支持详细版和简洁版等不同控制粒度。
- 教学法、内容控制、教师要求贴合度要进入提示链路和结构化输出要求。

推荐解释：

- 教案不是普通说明文，而是高度场景化的教学文档。
- 若没有真实教师教案作为参照，模型很容易产出“语言顺滑但不够像真实教学设计”的内容。
- 因此这部分必须先做样本收集和共性归纳，再反推提示词、模板和结构化输出设计。

### 3.7 模型路由与 Provider 治理

当前已有模型路由能力，但还需要上升为更完整的系统级治理，而不是简单选择大小模型。

这里不能只停留在“支持路由”和“支持 fallback”的工程表述，必须进一步研究并沉淀出当前阶段的最佳模型选型方案。

明确要求：

- 模型选择继续遵循配置驱动，不能在业务代码中散落硬编码。
- 需要清晰区分不同任务类型的路由策略，例如：
  - 意图理解
  - 标题润色
  - RAG 深度总结
  - 教案推理
  - PPT 内容生成
  - 图片生成
  - 流式对话
- 必须针对关键任务研究具体模型选择，而不是只写“轻模型 / 重模型”这种抽象分层。
- 需要给出当前阶段的明确选型结论，例如：
  - 哪个模型最适合意图理解
  - 哪个模型最适合长上下文 RAG 整合
  - 哪个模型最适合 PPT 内容生成
  - 哪个模型最适合教案细化
  - 哪个模型最适合图片生成
  - 哪个模型最适合流式对话体验
- 选型不应只基于主观印象，而应结合真实任务样本做对比验证。
- 评估维度至少包括：
  - 生成质量
  - 教师场景贴合度
  - 稳定性
  - 响应速度
  - 成本
  - 流式体验表现
  - 对结构化输出的服从度
- 最终应形成“任务类型 -> 首选模型 -> 备选模型 -> Provider -> fallback 条件”的清晰方案。
- Provider 失败时必须保留清晰观测信息：
  - 使用的 provider
  - 使用的 model
  - 失败原因
  - fallback 行为
- 不能通过偷偷切换语义不同的模型来掩盖失败。
- 图片模型接入后，也应纳入统一 Provider 治理。

推荐解释：

- 模型路由治理的目标不是让系统“能跑”，而是让不同任务都尽量落到最合适的模型上。
- 比赛场景下，最终评判标准不是抽象架构优雅，而是实际生成效果、速度、稳定性和演示观感的综合最优。
- 因此这部分必须沉淀出可以落地执行的具体模型选择方案，而不是只保留原则描述。

## 4. 按模块拆解的实施需求

### 4.1 后端实施重点

1. 扩展生成链路，使 PPT 成为核心输出中心。
2. 新增图片补充能力，支持检索图与生成图双通道。
3. 为 PPT 增加封面、目录等结构模板支持。
4. 教案生成逻辑与 PPT 建立明确关联。
5. 公共知识库通过 `Project` 复用实现，不单独造库。
6. 完善 SSE 或事件流，支撑逐块、逐段、逐页的内容流式响应。
7. 完善 `Artifact` 命名、标题规范与导出语义。
8. 整理模型路由和 Provider 配置治理。
9. 优化推荐内容粒度，避免返回过于碎片化的结果。
10. 将图片资源及其说明信息纳入 `Project` 或公共库资料语义，支持检索引用和生成后回灌。
11. 为核心演示链路建立“保底有图”和“渐进式内容展示”的体验兜底机制。
12. 收集真实教师教案样本，归纳共性结构，并将结果反映到教案提示词、模板和输出约束中。
13. 研究并确定关键任务的最佳模型选型方案，形成可执行的模型路由矩阵。
14. 由后端负责实现独立 `Mock UI`，用于能力验证和演示兜底，前端不承担该部分。
15. 研究并明确当前阶段应建设哪些公共知识库及其内容范围，并确保公共库链路真实可用。

### 4.2 前端实施重点

1. 以 PPT 为中心重新组织主要交互入口。
2. 继续设计并收口 `Library / 库` 的前端概念与展示语义。
3. 让用户能清楚理解个人项目库、公共库、资料、图片资源和生成引用之间的关系。
4. 精简卡片表达，减少文字堆叠。
5. 优化图标、标题和布局层次。
6. 清理或重构碎片化推荐展示。
7. 接通流式状态反馈与阶段提示。

### 4.3 Mock UI 与验证层

1. 做一套轻量、隔离的 `Mock UI` 用于验证后端新能力。
2. `Mock UI` 明确归后端负责人员实现，前端不承担该部分职责。
3. `Mock UI` 应优先服务于能力验证、联调兜底和比赛演示，不追求完整产品化。
4. `Mock UI` 需要尽量绕开正式前端中的复杂状态和交互依赖，确保后端能力可以被直接验证。

## 5. 建议优先级

### 5.1 P0

比赛和主链路必须完成：

1. PPT 主中心化

## 6. 可直接用于汇报或详细方案的能力主题

后续对外叙事可以收束为以下 6 个正式能力主题：

1. `统一知识空间`
   基于 `Project` 机制构建教师个人知识库与公共知识库复用体系。
2. `PPT 核心工件生成`
   以 PPT 为中心生成封面、目录、内容页、总结页，并承载增强教学内容。
3. `配套教案协同生成`
   教案直接绑定对应 PPT，生成更贴近教师需求的详细教学设计。
4. `多源插图增强`
   同时支持本地知识库检索插图与 AI 生成插图，并将图像及说明沉淀为可复用知识资源，提升课件可读性。
5. `流式共创体验`
   基于事件流实现生成过程可视化、阶段化和可迭代反馈。
6. `轻量高效的教师界面`
   优化前端信息结构，减少文字噪音与功能碎片，突出核心任务路径。
