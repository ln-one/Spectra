# 成员D后续工作建议：文件解析联调

日期：2026-04-01

## 1. 目的

明确当前文件解析链路已经完成的部分，以及后续需要成员 `D` 配合确认的质量、选型与策略事项。

本文件只讨论文件解析部分，不覆盖 PPT 主链路其他模块。

## 2. 当前已完成情况

当前后端侧已完成：

1. `local` 解析链路可用
2. `llamaparse` 云端解析链路可用
3. `mineru_cloud` 官方云 API 已完成真实项目接入验证
4. 云端失败时可统一回退到 `local`

当前建议中的解析策略为：

- 常规文件默认走 `llamaparse`
- 复杂高质量 PDF 走 `mineru_cloud`
- 任一云端解析失败最终回退 `local`

当前尚未完全收口的点：

- 大文件拆分策略
- 各类 PDF 何时应从 `llamaparse` 升级到 `mineru_cloud`
- 不同解析路径对后续生成质量的实际收益判断

## 3. 为什么需要 D 配合

根据分工单，`D` 负责：

- 质量优化
- 模型选型
- 样本归纳
- 提示词与效果策略
- 对“什么方案值得进入主流程”给出可执行结论

因此，文件解析这部分不需要 `D` 来做后端接线开发，但需要 `D` 配合确认最终策略口径。

换句话说：

- `C` 负责把链路接通、fallback 做稳
- `D` 负责判断哪条链路在质量上更值得成为默认方案

## 4. 建议 D 后续重点确认的事项

### 4.1 确认默认解析策略

需要 `D` 明确确认以下口径是否成立：

1. 常规 PDF 默认走 `llamaparse`
2. 扫描件、公式多、图表多、版式复杂文档走 `mineru_cloud`
3. 云端失败统一回退 `local`

这里需要 `D` 给出的不是泛泛意见，而是“当前阶段可执行的最终口径”。

### 4.2 确认大文件拆分方案

这是当前最需要 `D` 配合确认的点。

需要 `D` 确认：

1. 大文件是否必须默认拆分后再走 `llamaparse`
2. 多大规模应视为“大文件”
   - 按页数
   - 按文件大小
   - 或两者结合
3. 拆分粒度建议
   - 例如 `1-30 / 31-60`
   - 或更小批次
4. 当某个分段出现 `empty_output` 或明显质量异常时，是否直接分段回退 `local`
5. 是否需要对失败分段继续二分定位，以形成“兼容性问题页段”结论

### 4.3 确认质量收益是否足以支撑 `mineru_cloud`

当前已知情况：

- `mineru_cloud` 文本覆盖和 chunk 粒度最强
- 但耗时也最高

需要 `D` 确认：

1. `mineru_cloud` 带来的文本覆盖提升，是否能明显提升后续生成质量
2. 哪类文档必须使用 `mineru_cloud` 才有明显收益
3. 哪类文档使用 `llamaparse` 已经足够

这个判断会直接影响后续生产策略，而不只是解析模块本身。

### 4.4 确认评测样本与评测口径

建议 `D` 输出一组稳定样本，用于后续持续比较：

1. 普通文本型 PDF
2. 公式较多 PDF
3. 图表较多 PDF
4. 扫描/OCR 压力较大的 PDF
5. 大文件 PDF

并明确评测口径：

- 文本完整度
- chunk 粒度
- 标题/结构恢复
- 公式/图表保真度
- 对后续 RAG 与生成链路的实际帮助

## 5. 对 D 的具体交付建议

建议 `D` 本轮给出以下交付物：

1. **最终解析策略建议**
   - 默认用什么
   - 什么情况下升级到 `mineru_cloud`
   - 大文件是否强制拆分

2. **大文件拆分建议**
   - 触发条件
   - 拆分粒度
   - 分段失败后的回退策略

3. **质量样本结论**
   - 哪类文档 `llamaparse` 已足够
   - 哪类文档必须 `mineru_cloud`

4. **可直接给 C 接入的规则口径**
   - 不要停留在研究结论
   - 要给出可以直接落地的判断规则

## 6. 当前推荐协作方式

建议采用下面的协作边界：

### C 负责

- provider 接线
- fallback 链路
- 项目内真实调用验证
- 配置与运行时稳定性

### D 负责

- 质量收益判断
- 样本归纳
- 大文件拆分口径
- 默认策略是否最终成立的确认

## 7. 一句话结论

> 文件解析链路接入已完成，当前最需要成员 `D` 配合确认的是最终质量策略，尤其是大文件拆分方案、`llamaparse` 与 `mineru_cloud` 的边界条件，以及哪些文档类型值得走更重的高质量解析路径。
