## 知识库构造与知识组织

这一节不单独讲传统意义上的“数据库表设计”，而是讲当前资料数据实际上怎么落库、怎么入索引、怎么继续进入来源查看和后续生成。老师和评审真正关心的，也不是表名本身，而是上传的文件最后落到了哪里、谁在用这些数据、后续页面怎么把它们再取出来。

![](../../assets/media/spectra-deployment-architecture.png){width="7.2in" height="3.8in"}
图 5-11 当前部署架构图

资料进入系统后的第一步是文件接收。前端通过资料上传入口把 PDF、Word、图片等课程资料提交到 `POST /api/v1/files`。这一步不是前端把文件直接塞进向量库，而是先由 backend 记录上传对象、项目归属和会话上下文，再交给 `Dualweave` 处理远端解析入口。文件上传索引测试覆盖了 `pending_remote -> reconcile -> terminal` 的状态推进，说明上传后的资料不是停在附件层，而是真的会继续进入后续索引流程。

解析完成后，资料进入标准化、切块和索引阶段。`backend/services/media/rag_indexing.py` 会把解析结果整理成 chunk，并分别准备原始文本、标准化文本和索引文本。这里的“数据放在哪”按职责分成四块：`Uploads` 保存原始文件和中间文件，PostgreSQL 保存上传记录、项目、会话和结果对象，`Qdrant` 保存向量索引，`Redis + RQ` 负责把索引任务和后台状态推进接起来。这样一来，上传文件、检索块、项目对象和后台任务不会混成一层。

资料进入检索阶段后，`Stratumind` 负责正式召回与重排，Spectra backend 负责把页面请求转成检索请求、设置超时、整理返回结果，并提供来源查看接口。当前接口中，`POST /api/v1/rag/search` 承接检索请求，`GET /api/v1/rag/sources/{chunk_id}` 承接来源查看。也就是说，页面上点开的来源块，并不是一段写死说明文字，而是从当前 chunk、索引和来源元数据里重新取回来的。

从数据分工上看，当前这一套知识组织至少包含四类对象。第一类是上传对象，用来记录资料来源、上传状态和远端解析进度；第二类是解析与切块结果，用来把长资料拆成可检索片段；第三类是向量索引，用来支持后续召回；第四类是项目、会话、结果和版本对象，用来把检索结果继续送入生成、预览和结果保存流程。这样写之后，评审就能看清：这里不是单纯“存文件”，而是把文件变成了后续页面和生成流程都能继续消费的数据。

部署架构图对应的是当前运行现实。统一入口节点承接公网访问，前端应用与 FastAPI 后端构成统一接入节点；`Dualweave`、`Stratumind`、`Diego`、`Pagevra`、`Ourograph` 与 `Limora` 分别接住资料解析、检索、生成、渲染、对象关系和身份边界；PostgreSQL、Redis、Qdrant、Uploads 与成果文件存储共同承接数据和结果。

上传资料会继续进入解析、切块、索引和召回流程；检索结果可以在页面上查看来源，并进入后续生成；结果对象仍由 PostgreSQL 中的项目、会话、成果和版本关系承接。这一节的重点，是把“文件上传之后到底发生了什么”说具体，而不是停留在“系统有知识库”。
