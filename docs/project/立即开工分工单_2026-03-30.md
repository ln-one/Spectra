# 立即开工分工单

> 更新时间：2026-03-30
> 状态：当前生效
> 用途：基于当前已完成拍板结果，直接进入执行，不再停留在抽象讨论。

## 1. 当前结论

当前阶段不再继续做大而全设计，直接分工开干。

已经拍板的事项视为当前统一方向：

1. `PPT` 是核心工件。
2. 核心结果必须尽量有图，不够智能就用保底模板策略。
3. 生成体验要做出类 Gamma 的逐块、逐页出现效果。
4. 公共知识库复用 `Project` 逻辑，不另起平行体系。
5. 教案必须基于真实教师教案样本归纳，不靠空想提示词。
6. 模型路由不仅要治理，还要研究出当前阶段的最佳模型选型方案。
7. `Mock UI` 由后端负责实现，前端不承担该部分。

## 2. 执行原则

### 2.1 直接开工

- 当前阶段优先推进真实可演示结果。
- 当前阶段只接“能够在本阶段真正完成”的需求版本，不接无限扩张目标。
- 不再额外等待一轮完整大设计文档再开始。
- 各角色先按职责边界推进手头任务，并在联调中继续收口。

### 2.4 主动找问题、主动找活

这一条是当前阶段的强约束，不是可选建议。

所有成员都必须：

1. 自己真实使用产品。
2. 自己在真实流程里找问题。
3. 自己判断问题能不能修、质量能不能提。
4. 如果问题在自己职责范围内，默认自己直接修。
5. 如果问题不在自己职责范围内，必须主动同步给对应负责人。

禁止的工作方式：

1. 只盯着别人分给自己的那几项，不主动跑产品。
2. 只看代码、不走真实链路。
3. 发现问题后不处理，也不同步。
4. 认为“这不在我分工里”就可以不管。

当前阶段必须明确：

- 不能只局限在分工单下被动做事。
- 必须自己学会找问题、找活干、推动质量提升。
- 分工是责任边界，不是工作上限。

### 2.2 阶段完成优先于理想完整版

当前阶段的要求不是把所有方向都做到最终形态，而是把关键需求收口成“本阶段可完成、可演示、可交付”的版本。

判断标准：

- 能否在本阶段真实做完
- 能否形成稳定演示结果
- 能否写进比赛材料并经得起追问
- 是否直接服务主链路

因此默认规则是：

- 先完成可交付版本，再追求更智能版本
- 先完成保底方案，再继续拔高质量
- 先保证闭环可演示，再补充扩展能力

### 2.3 仍保留“最小契约先行”

当前阶段不采用重文档式契约先行，但仍保留最小契约先行原则。

意思是：

- 不要求所有工作先写完整方案再开发。
- 但涉及跨人协作、接口联调、状态语义、Artifact 语义、事件流语义的内容，必须先对齐最小约定。
- 最小约定可以非常轻量，只需要明确：
  - 输入输出是什么
  - 状态怎么表示
  - 前后端怎么接
  - Artifact 怎么落
  - 流式事件怎么推

结论：

- `不需要重型契约先行`
- `仍需要最小契约先行`

否则很容易造成多人并行后返工。

## 3. 当前最小演示主链路

本轮优先打通的演示链路：

`选择项目库/公共库 -> 输入需求 -> 大纲逐块出现 -> PPT 逐页出现 -> 每页尽量有图 -> 生成对应教案 -> 产物进入项目空间并可下载`

所有工作默认优先服务这条主链路。

## 3.1 当前阶段完成定义

以下结果在本阶段内必须真实完成：

1. 能选择项目库或公共库参与生成。
2. 能生成以 `PPT` 为核心的正式结果。
3. 生成过程里能看到大纲逐块出现、PPT 逐页出现的展示效果。
4. 最终 `PPT` 核心页面尽量带图；若智能选图不稳定，允许使用保底模板策略。
5. 能生成与 `PPT` 对应的 `Word` 教案结果。
6. 生成结果能进入项目空间 `Artifact` 并可下载。
7. `Mock UI` 能独立验证后端主能力并用于演示兜底。

以下内容本阶段不要求做到最终理想形态，只要求做到可交付版本：

1. 图片插入不要求完全智能，但必须稳定有图。
2. 流式体验不要求 token 级逐句输出，但必须做到逐块或逐页输出。
3. 教案不要求覆盖全部学科，但必须基于真实教师样本建立参照。
4. 模型选型不要求形成长期终局方案，但必须给出当前阶段可执行最佳方案。
5. 正式前端不要求覆盖所有花哨能力，但必须保证主演示路径顺畅。

## 4. 角色分工

## 4.1 B：前端正式界面收口

B 立即开工范围：

1. 将正式前端界面收口为 `PPT` 主中心化展示。
2. 继续设计并收口前端的 `Library / 库` 概念本身，而不只是修现有入口样式。
3. 明确前端里的 `Library / 库` 到底是什么：
   - 是统一资料空间
   - 是公共库与个人项目库的选择入口
   - 是生成链路的来源面板
   - 是图片、文档、资料、引用关系的可见载体
4. 解决当前库概念实现不清、入口不顺、语义不稳定的问题。
5. 降低页面文字密度，改进图标、标题、卡片层级和语义清晰度。
6. 清理或重构碎片化推荐内容展示。
7. 让正式前端接住大纲逐块出现、PPT 逐页出现的结果展示。
8. 打通正式前端里公共库选择、资料引用、生成、预览、下载的主路径。
9. 补齐正式演示所需截图、录屏、界面展示素材。
10. 继续把“教案直接选择对应 PPT，而不是一堆框”的正式前端路径收口清楚。
11. 让正式前端里的生成结果优先围绕 `PPT` 展示，而不是把多个能力散成平级入口。

B 不负责：

- `Mock UI`
- 倒推后端业务语义
- 自定义接口语义

B 额外强制要求：

- 必须持续自己跑正式前端产品。
- 必须自己发现界面、路径、语义和展示上的问题。
- 发现属于前端的问题就直接修；不是前端问题就立即同步给对应负责人。

## 4.2 C：主线后端直接落地

C 立即开工范围：

1. 把 `PPT` 主链路真正收口为可交付能力。
2. 保证生成结果落到项目空间 `Artifact` 闭环，并可稳定下载。
3. 把“保底有图”接成主链路能力：
   - 库内图片优先引用
   - 不足时接图片生成
   - 不稳定时强制模板图位兜底
4. 把图片资源及其说明信息纳入 `Project` / 公共库资料语义，支持后续引用。
5. 打通类 Gamma 的渐进式展示主链路：
   - 大纲逐块输出
   - PPT 逐页输出
   - 教案按段或小节输出
6. 实现后端负责的 `Mock UI`，用于能力验证、联调兜底和演示。
7. 确保 `Library / 库`、公共库引用、资料可见性和生成使用路径是真实可用的。
8. 如果当前库能力、公共库接入、引用链路或生成使用路径不可用，必须直接修复。
9. 把 `PPT` 模板能力补齐到本阶段可演示版本，至少包含：
   - 封面页
   - 目录页
   - 内容页
   - 总结或结束页
10. 修正生成工件标题、展示标题、导出文件名不规范的问题。
11. 把教案链路收口成“直接选择对应 PPT 再生成”，而不是继续暴露松散表单逻辑。
12. 把当前散落的小题、思维导图、动画、插图等能力尽量收束为“增强 PPT”的内容接入方式，而不是独立漂浮结果。
13. 保证会话、状态、事件、Artifact、下载语义一致。

C 本阶段完成标准：

- 不要求把所有相关能力做到最终长期架构。
- 但必须把主链路做成“真实可跑、真实可演示、真实可下载”的完成态。
- 若某项高级智能能力无法在本阶段稳定完成，必须给出保底实现，不能空缺。

C 不等待：

- 不等完整大设计文档。
- 在需要联调的地方，只先补最小契约。

C 额外强制要求：

- 必须持续自己跑主链路和后端能力验证路径。
- 必须自己发现业务逻辑、链路、可用性和闭环问题。
- 发现属于主线后端的问题就直接修；不是自己的问题就立即同步给对应负责人。

## 4.3 D：质量、样本与选型专项

D 立即开工范围：

1. 收集真实教师教案样本。
2. 总结教案共性结构，并反推提示词、模板、输出约束。
3. 负责研究当前阶段应该建设哪些公共知识库，以及每个公共库应包含哪些内容。
4. 负责寻找、筛选、整理适合进入公共库的知识资料，包括文本资料、图片资料及其说明信息。
5. 负责给公共库内容做主题分类、适用范围说明和基础整理，确保可以被前端和生成链路清晰引用。
6. 继续推进图片命中、选图策略、图文匹配和版式风险优化。
7. 研究关键任务的最佳模型选型方案：
   - 意图理解
   - 长上下文 RAG 整合
   - PPT 内容生成
   - 教案细化
   - 图片生成
   - 流式对话
8. 形成清晰的模型路由矩阵：
   - 任务类型
   - 首选模型
   - 备选模型
   - Provider
   - fallback 条件
9. 输出可供 C 接入主链路的稳定策略、baseline 与建议。
10. 研究“哪些生成内容应直接塞入 PPT 才最有价值”，包括：
   - 小题
   - 思维导图
   - 动画或动画说明
   - 插图
11. 为 PPT 封面、目录、内容页的版式与内容密度给出可直接接入的质量建议。
12. 为教案与 PPT 的对应关系给出更贴近教师使用习惯的建议，而不是停留在泛泛配套关系。
13. 负责所有提示词相关工作，包括：
   - 系统提示词
   - 任务提示词
   - 教案提示词
   - PPT 内容提示词
   - 图片生成提示词
   - 流式展示用的结构化生成提示词
14. 负责所有生成质量相关工作，包括：
   - 内容质量
   - 结构质量
   - 图文匹配质量
   - 页面信息密度
   - 推荐结果是否碎片化
   - 教师场景贴合度

D 本阶段完成标准：

- 不要求做无限制研究扩张。
- 必须产出能直接被 C 接入主线的样本、结论、策略和选型结果。
- 必须明确给出本阶段要落成哪些公共库，以及这些公共库中实际放什么。
- 若无法得到绝对最优答案，也必须给出“当前阶段最优可用方案”。

D 的职责归纳：

- 只要问题本质上属于提示词、生成质量、效果优化、模型选型、内容质量判断，默认归 `D`。
- 只要问题本质上属于业务逻辑、主链路打通、接口接入、状态语义、Artifact 闭环、库可用性修复，默认归 `C`。

D 额外强制要求：

- 必须持续自己跑真实生成流程并观察结果质量。
- 必须自己主动发现提示词、生成质量、图片命中、内容结构、模型效果上的问题。
- 发现属于质量与效果的问题就直接推进；不是自己的问题就立即同步给对应负责人。

D 不负责：

- 直接替代 C 做主链路功能落地
- 把实验性想法未经收口直接塞进主流程
