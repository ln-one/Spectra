## 系统整体架构

系统整体由前端工作台、后端编排层和专项服务层组成。用户在前端工作台中上传资料、推进会话、查看结果和下载文件；后端把这些动作接成连续流程；生成、渲染、检索、结果保存和权限等功能由不同服务分别承接。

这一套结构解决的是一个很具体的问题：资料上传后不能停在附件层，生成结果出来后也不能只停在导出层。上传的资料要继续进入检索，生成的结果要继续进入预览、导出和保存，保存后的结果还要能回到项目里继续被查看和引用。因此，这一章不是介绍“系统很大”，而是交代这些动作在当前实现里分别落到哪里。

![](../../assets/media/image1.png){width="7.2in" height="5.3in"}
图 5-1 系统整体架构图

从作品实现看，这张图对应一条清楚的工作过程：资料先通过上传入口进入 `POST /api/v1/files`，随后进入解析与索引；生成阶段由 Studio 发起 `Session` 或 artifact create；结果出来后进入 preview / export；最终结果落到 `Artifact / Version / Reference` 这一组正式对象里。前端看到的是统一工作台和 Studio 相关页面，后端承接的是 `Session`、事件、任务和结果绑定，生成、预览导出、资料检索和结果保存分别由不同服务负责。

![](../../assets/media/spectra-frontend-workbench.png){width="7.2in" height="4.1in"}
图 5-2 前端工作台截图

这张截图对应当前作品中最直接可见的工作台形态。左侧为多种成果入口与历史记录区，中部为会话推进与结果承接区，右侧为教学资料与引用资料区。资料、对话、生成、预览和结果查看在同一界面内连续展开，统一工作台已经落实为当前页面结构。

![](../../assets/media/spectra-system-function-architecture.png){width="7.2in" height="6.0in"}
图 5-3 系统功能架构图

这张图的作用，是把图 5-1 里的动作再拆细一点：页面入口在哪，后端把哪些接口接起来，外部服务各自接住哪一段，数据最终落到哪些对象上。读这一张图时，只需要回答一个问题：用户在页面上点一次上传或生成，系统内部是谁接住了后续动作。

按当前作品现实，整体结构只需要先看三层：

- 前端工作台层：统一承接资料、对话、生成、预览和结果管理等操作；
- 后端编排层：围绕 `Session`、事件和结果绑定组织核心流程，负责把不同能力连接成可推进流程；
- 专项服务层：分别承担生成、渲染导出、知识库检索、结果状态和身份边界等职责。

这一拆分可以直接对应到当前作品。前端工作台对应 Studio、Chat、Sources 和结果查看界面；后端编排层对应 `generate`、`preview`、`artifact`、`rag` 等接口和应用服务；专项服务层对应 `Diego`、`Pagevra`、`Dualweave`、`Stratumind`、`Ourograph`、`Limora` 等外部能力源。用户在界面上看到的是一块连续工作区，系统内部则是按接口和服务分工，把上传、检索、生成、预览、导出和保存接成一条可追踪的动作链。

当前结构稳定承接五类内容：项目资料、工作过程、结果对象、结果版本和结果之间的引用关系。`Project` 对应项目资料与结果统一管理范围，`Session` 对应一次具体工作过程，`Artifact` 对应生成结果对象，`Version` 对应结果的稳定版本记录，`Reference` 则承接不同资料和结果之间的复用关系。这组对象界定了系统长期管理的核心内容。

这一章只想先说明一个事实：前端页面、后端接口和专项服务已经把教师备课里最常见的上传、生成、预览、导出和保存接到了一起。后续各节再分别展开模块、流程和对象关系。
