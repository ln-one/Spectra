## 基于多级上下文编排的生成式语义构建技术

该技术链承接教学目标、资料证据、卡片配置和修订要求的统一编排职责，实现从教学意图表达、结构生成、正式生成到后续优化的连续语义构建流程。当前系统采用 `Session -> outline -> confirm -> generate -> refine` 的阶段链路，将生成过程组织为具备状态边界、上下文约束和结果承接能力的可管理流程。生成主链由 `Session` 状态推进承接，`Diego` 承接大纲与正式内容生成，预览、导出和成果绑定继续挂接到同一成果链。

从竞赛评审视角看，本节重点说明三项内容：生成过程如何被状态机化组织，强约束上下文如何抑制语义漂移，成果对象如何保持与后续交付链路的一致挂载。

当前实现可从页面入口、接口契约、服务分工和工程验证路径四个层面核对：

| 工程验证路径 | 当前对应 |
| --- | --- |
| 页面入口 | Studio 生成配置、卡片执行入口、结果预览页 |
| 接口契约 | `POST /api/v1/generate/sessions`、`POST /api/v1/generate/studio-cards/{card_id}/execution-preview`、`POST /api/v1/generate/studio-cards/{card_id}/execute` |
| 服务分工 | Spectra backend 承接会话编排与状态推进，`Diego` 承接大纲与正式生成，`Pagevra` 承接后续预览与导出 |
| 验证基准 | 会话创建约束测试、Studio card 执行映射测试、会话预览 contract 测试 |

从技术流程看，该链路包含四个稳定环节：

- 教学意图与约束条件收集；
- 中间结构生成与阶段确认；
- 正式内容生成；
- 后续优化与成果对象回流。

生成请求并非由单一 prompt 直接触发，而是由会话状态、资料证据、卡片配置和执行上下文共同组装。系统通过多级上下文编排方式将教学目标、重难点、资料证据和阶段状态写入统一执行请求，再由 `outline`、确认和正式生成几个阶段逐步推进。`Session` 承接一次具体业务过程，既保存当前阶段，又约束下一步允许执行的动作，形成稳定的状态机控制逻辑。

接口契约已经将这条链路固化到当前实现中。会话创建接口承接 bootstrap 后的正式启动路径；Studio card 执行接口将执行预览、正式执行和后续优化拆分为独立动作；预览 contract 将 `artifact_id`、`based_on_version_id` 和渲染结果统一返回。该契约设计使生成过程具备幂等化调用边界和清晰的结果承接路径。

从工程质量角度看，该技术链体现出三项关键特征：其一，教学资料和阶段状态共同形成强约束上下文，抑制大模型生成过程中的语义漂移；其二，生成过程通过显式状态迁移进行控制，保证阶段切换的可解释性；其三，正式生成结果与成果对象链保持一致挂载，保证后续预览、导出和版本管理的原子性衔接。

综上，当前系统已形成基于多级上下文编排的生成式语义构建技术链。评审若需核对该能力，可直接检查三类事实：Studio 页面是否存在明确阶段推进，OpenAPI 是否保留执行预览与正式执行的分离契约，以及工程验证路径是否证明生成结果能够继续进入预览、导出和成果保存流程。

本节可归纳为一句判断：系统已经将生成过程从单次模型调用提升为具备状态控制、上下文约束和成果承接能力的工程化语义构建链。
