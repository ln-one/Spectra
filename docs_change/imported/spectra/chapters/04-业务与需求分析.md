# 3. 业务与需求分析

## 3.1 业务需求建模

传统教育数字化工具虽解决了单点内容生成需求，但尚未实现全链路资产化管理。Spectra 的业务需求建模基于课程知识空间的持续生产、沉淀、交付、回流与演化，系统定位为围绕课程知识空间运行的业务级解决方案。

::: {custom-style="Body Text"}
核心业务特征：
:::

- 教师需要统一承接真实备课过程的完整业务链路，而非零散工具的组合；

- 学校和机构需要可长期治理、复用和演化的课程知识资产，而非分散文件；

- 学习者需要持续更新、可外化、可组织的学习内容来源，而非一次性静态文件；

- 平台型系统需要在一次正常的内容生产行为中同时完成正式沉淀、标准交付和后续复用准备，实现数据一致性与资产化逻辑的统一。

本章分析这些业务需求如何推导出课程知识空间系统的架构形态。

## 3.2 用户对象与业务问题

### 3.2.1 教师侧：备课全链路的资产化管理需求

教师是系统的第一高频使用者。传统备课工具虽解决了单点生成需求，但整条备课链路仍处于割裂状态，导致资产化程度低、复用率不足。

  ------------------------------------------------------------------------------------------------------------------------------
  **业务问题**     **需求本质**                                             **系统设计要求**
  ---------------- -------------------------------------------------------- ----------------------------------------------------
  资料分散         旧课件、文档、图片、视频和参考材料难以统一利用           多模态资料进入同一知识链，实现一次输入、全流程复用

  意图表达不完整   教师常常只有教学目标和课堂想法，无法一次性写出完备指令   支持会话化澄清与结构化组织

  生成不可控       单次输出很难稳定符合课程逻辑                             支持 outline 确认、分阶段生成与修改

  修改成本高       局部调整经常触发整体重做                                 将修改纳入可管理的 Session 主链

  成果难沉淀       文件导出后很难继续复用、追踪和演化                       生成结果回流至课程知识空间，形成资产化闭环
  ------------------------------------------------------------------------------------------------------------------------------

  : 表 3-1

::: {custom-style="Body Text"}
教师侧需求推导出系统须同时具备会话组织、资料吸收、证据组织、生成主链、真实预览和正式沉淀能力。
:::

### 3.2.2 学习者侧：持续更新的学习资源访问需求

学习者虽非主要生产者，但长期承接课程知识空间的价值。传统模式下学习者只能获取一次性静态文件，缺乏持续更新机制。

::: {custom-style="Body Text"}
学习者侧核心需求：
:::

- 持续访问和使用权威课程空间中的更新内容；

- 围绕同一课程空间获得导图、复习材料、互动内容等不同外化结果；

- 在保持来源稳定的前提下，生成更适合个人理解和复习的学习结果；

- 在权限边界内使用课程资产，避免在碎片化文件之间切换。

该需求进一步验证系统本体应为课程知识空间，通过数据一致性保障学习结果、复习材料和互动内容的持续更新。

### 3.2.3 学校、教研组与机构侧：组织级资产治理需求

采购方和治理方关注系统能否转化为组织资产，实现长期价值积累。

  --------------------------------------------------------------------------------------------------------
  **组织问题**   **需求本质**                               **系统设计要求**
  -------------- ------------------------------------------ ----------------------------------------------
  教师减负       备课、资料整理和格式加工成本过高           通过标准化流程降低重复劳动成本

  课程资产沉淀   教师成果长期散落在本地文件和聊天群         形成课程知识空间、版本和正式状态管理

  教研复用       优质内容难以被后续引用和演化               支持 Reference 和空间网络，实现资产化复用

  权限治理       教师、学习者、组织和机构之间需要明确边界   具备身份、成员和组织治理能力

  长期交付       客户需要可持续运营的系统                   具备导出、绑定、证据和边界清晰的长期交付能力
  --------------------------------------------------------------------------------------------------------

  : 表 3-2

::: {custom-style="Body Text"}
组织侧需求推导出系统须从架构设计阶段即考虑治理、复用、版本和成员边界，确保课程数据库不退化为难以管理的文件集合。
:::

## 3.3 核心业务领域模型（Domain Model）分析

系统若仅围绕任务、消息和导出文件运转，将退化为一次性内容工具，无法支撑沉淀、复用、治理和商业化目标。Spectra 的业务领域模型以课程知识空间为核心，通过正式对象语言实现数据一致性与资产化逻辑的统一。

  -----------------------------------------------------------------------------------------------
  **领域对象**      **业务必要性**               **资产化意义**
  ----------------- ---------------------------- ------------------------------------------------
  Project           承载长期存在的课程知识空间   统一管理资料、成果、版本、引用和成员关系

  Session           组织一次完整工作展开过程     承接对话、生成、修改和状态事件的完整记录

  Artifact          将外化结果纳入正式语义链     使 PPT、Word、导图、动画等结果可被绑定和复用

  Version           区分过程态与正式态           为课程资产提供正式状态锚点，支持版本回溯

  Reference         表达跨空间复用关系           使课程空间之间形成可追溯的知识网络

  CandidateChange   表达修改与演化入口           将内容更新纳入可治理的演化流程

  Member            表达协作与治理边界           为教师、学习者、组织和机构角色提供清晰权限基础
  -----------------------------------------------------------------------------------------------

  : 表 3-3

::: {custom-style="Body Text"}
该领域模型以知识图谱与动态链接库为底层核心，确保系统本体为库与引用关系，为第 8 章课程知识资产平台逻辑和第 9 章治理框架提供业务基础。
:::

## 3.4 核心能力需求

### 3.4.1 会话化意图理解

系统必须能承接教师以自然语言表达教学目标、重点难点、课时安排和风格要求，并将其组织为后续生成可用的结构化输入。

::: {custom-style="Body Text"}
验收重点：
:::

- 能围绕项目和会话承接一次完整工作过程；

- 能保留上下文、资料和教师修改意见；

- 能把一次备课过程组织为可观察、可推进的 Session。

### 3.4.2 多模态资料进入

系统支持 PDF、Word、课件、图片、视频等资料进入统一链路。资料进入后成为知识来源，参与检索增强和生成主链，实现一次输入、全流程复用。

::: {custom-style="Body Text"}
验收重点：
:::

- 资料进入后被系统识别为知识来源；

- 上传、解析和交付阶段具有可追踪语义；

- 资料结果进入检索增强和生成主链。

### 3.4.3 检索增强与证据组织

系统在生成前组织可信证据，降低无依据生成和事实漂移风险。

::: {custom-style="Body Text"}
验收重点：
:::

- 区分基础召回与证据组织；

- 用正式评估结果说明检索增强质量；

- 证据进入生成链，参与内容生产全流程。

### 3.4.4 生成主链

系统必须支持 AI 课件与演示文稿主生成，并让教师参与关键阶段。

::: {custom-style="Body Text"}
验收重点：
:::

- 支持 outline 形成与确认；

- 支持确认后进入正式 generation；

- 支持结果进入预览、修改和交付链；

- 生成结果能作为 Artifact 与课程空间绑定。

### 3.4.5 Studio 多模态内容工坊

系统围绕同一课程空间外化多种内容形态，支持一次输入派生多类交付结果。至少应覆盖：

- PPT

- Word 教案

- 互动小测

- 互动游戏

- 思维导图

- 演示动画

- 说课助手

- 学情预演

验收重点：

- 以 Studio 内容工坊统一呈现各类能力；

- 各类结果共享课程空间和上下文，保障数据一致性；

- 非 PPT 产物不以假预览冒充真实结果。

### 3.4.6 预览、修改与标准交付

系统保证用户看到的预览、经历的修改和最终交付属于同一能力链，确保前后端数据一致性。

::: {custom-style="Body Text"}
验收重点：
:::

- 支持真实 preview；

- 支持 refine 和 history；

- 支持 PPTX / DOCX 标准导出；

- 下载与成果绑定不脱离正式知识状态。

### 3.4.7 正式知识状态与课程空间复用

生成结果回流至课程知识空间，形成资产化闭环。

::: {custom-style="Body Text"}
验收重点：
:::

- Artifact 进入正式状态绑定；

- Version、Reference、CandidateChange 表达演化与引用关系；

- 课程空间作为长期课程知识资产持续增长；

- 学习者侧和组织侧可在权限边界内复用课程空间。

### 3.4.8 身份、组织与成员边界

系统面向长期部署和机构使用，具备完整的身份、组织和成员边界管理能力。

::: {custom-style="Body Text"}
验收重点：
:::

- 身份与会话边界清晰；

- 组织和成员关系不污染课程知识本体；

- 系统具备向学校、教研组和机构扩展的治理基础。

## 3.5 非功能需求

非功能需求决定系统长期交付能力。易用性、稳定性、可追踪性、可演化性和权限边界任一缺位，均会影响系统的生产级可用性。

  --------------------------------------------------------------------------------------------
  **非功能需求**   **具体含义**                                   **业务影响**
  ---------------- ---------------------------------------------- ----------------------------
  易用性           教师无需掌握复杂 prompt 工程                   决定试点门槛和真实采用率

  稳定性           长链路生成、解析、渲染和导出具备阶段状态       决定系统能否进入真实交付

  可追踪性         资料、证据、成果和版本之间有来源关系           决定审核、复用和组织信任

  可演化性         课程空间、版本、候选变更和引用网络可持续增长   决定系统能否成为长期资产

  可复用性         内容可被教师、学习者、教研组按需再利用         决定采购价值是否能放大

  安全与权限       不同用户、课程和组织之间有清晰边界             决定学校和机构场景是否成立
  --------------------------------------------------------------------------------------------

  : 表 3-4

::: {custom-style="Body Text"}
上述非功能需求共同要求系统从架构设计阶段即具备可追踪、可治理、可演化的工程基础。
:::

## 3.6 需求与六能力层的映射关系

Spectra 的业务需求具有系统级特征，由控制平面和六个正式能力层分别承接不同需求域。

  ----------------------------------------------------------------------------------------------
  **需求域**                 **承接层**   **分工依据**
  -------------------------- ------------ ------------------------------------------------------
  教学意图理解与工作流组织   Spectra      作为 workflow shell 组织会话、事件、状态和契约

  AI 课件主生成              Diego        承担 outline 与 generation 主链

  预览与标准导出             Pagevra      承担 preview、render、PPTX / DOCX export

  正式知识状态与演化语义     Ourograph    承担 knowledge-state core

  上传、解析与交付底座       Dualweave    承担 ingest / delivery 底座

  检索增强与证据组织         Stratumind   承担 retrieval、evidence organization

  身份、组织与成员边界       Limora       承担 identity、session、organization/member boundary
  ----------------------------------------------------------------------------------------------

::: {custom-style="Body Text"}
该映射关系表明，系统级业务需求推导出"控制平面 + 六能力层"的架构形态，各能力层职责边界清晰，符合高内聚、低耦合的工程设计原则。
:::

## 3.7 需求优先级

### 3.7.1 P0：系统闭环成立的核心需求

- 会话化意图理解；

- 多模态资料进入；

- 检索增强与证据组织；

- 课件生成与 outline 确认；

- 真实预览、修改与标准导出；

- 正式知识状态绑定；

- 课程空间长期存在。

### 3.7.2 P1：放大交付价值的产品能力

- 更强的质量门控；

- 更丰富的证据组织；

- 更完整的 Studio 多模态外化体验；

- 更好的课程空间访问与展示能力；

- 更稳定的组织协作体验。

### 3.7.3 P2：平台化延展方向

- 学习者侧个性化派生；

- 更深入的学校/机构治理；

- 更多内容形态外化；

- 更大规模的课程知识网络运营。

优先级顺序对应业务成立逻辑：先确保系统闭环成立，再放大交付价值，最后进入平台化运营能力建设。

## 3.8 本章结论

Spectra 的业务需求覆盖教学意图理解、多模态资料吸收、检索增强、生成主链、Studio 多模态外化、真实预览修改、标准导出、正式知识状态、课程空间复用以及身份组织边界。

::: {custom-style="Body Text"}
业务需求的系统级特征推导出课程知识空间系统的架构形态，由控制平面和六个正式能力层协同承接，实现数据一致性与资产化逻辑的统一。
:::
