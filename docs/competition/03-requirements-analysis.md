# 3. 业务与需求分析

## 3.1 系统级需求判断

旧工具对教育场景给出的答案已经不够了。`Spectra` 要承接的需求本体，是建立一套能够持续生产、沉淀、交付、回流并演化课程知识的系统。项目必须被看作一个围绕课程知识空间运行的系统级方案。

核心判断很明确：

- 教师并不缺零散工具，缺的是一条能承接真实备课过程的主链；
- 学校和机构并不缺文件，缺的是能长期治理、复用和演化的课程知识资产；
- 学习者并不只需要一次性文件，长期更需要持续更新、可外化、可组织的学习内容来源；
- 平台型系统若想成立，就必须让一次正常的内容生产行为同时完成正式沉淀、标准交付和后续复用准备。

第 3 章回答的是：为什么这些需求天然要求系统长成课程知识空间系统。

## 3.2 用户对象与业务问题

### 3.2.1 教师侧：真实备课链路需要被统一承接

教师是系统的第一高频使用者。教师真正面对的问题，是整条备课链路长期割裂。只要这种割裂存在，任何单点生成能力都会很快失去价值：

| 业务问题 | 需求本质 | 系统必须回答什么 |
| --- | --- | --- |
| 资料分散 | 旧课件、文档、图片、视频和参考材料难以统一利用 | 多模态资料必须进入同一知识链 |
| 意图表达不完整 | 教师常常只有教学目标和课堂想法，无法一次性写出完备指令 | 系统必须能承接会话化澄清与结构化组织 |
| 生成不可控 | 单次输出很难稳定符合课程逻辑 | 系统必须支持 outline 确认、分阶段生成与修改 |
| 修改成本高 | 局部调整经常触发整体重做 | 系统必须把修改放进可管理的 `Session` 主链 |
| 成果难沉淀 | 文件导出后很难继续复用、追踪和演化 | 系统必须让结果回到课程知识空间 |

教师侧需求因此天然要求系统同时具备会话组织、资料吸收、证据组织、生成主链、真实预览和正式沉淀，而不是只提供一个“生成按钮”。

### 3.2.2 学习者侧：需要持续更新而非一次性文件

学习者不是主要生产者，但会长期承接课程知识空间的价值。其需求也不只停留在“下载老师的文件”：

- 能持续访问和使用权威课程空间中的更新内容；
- 能围绕同一课程空间获得导图、复习材料、互动内容等不同外化结果；
- 能在保持来源稳定的前提下，生成更适合个人理解和复习的学习结果；
- 能在权限边界内使用课程资产，而不是被迫在碎片化文件之间来回切换。

学习者侧需求进一步说明，系统本体必须是课程知识空间，而不是一组相互脱节的导出物。只有空间是本体，学习结果、复习材料和互动内容才能持续更新，而不是每次重新散落成文件。

### 3.2.3 学校、教研组与机构侧：需要长期资产与治理能力

采购方和治理方并不主要关心“单次生成是否惊艳”，他们更关心系统能否转化为组织资产。

| 组织问题 | 需求本质 | 系统必须回答什么 |
| --- | --- | --- |
| 教师减负 | 备课、资料整理和格式加工成本过高 | 系统必须真正减少重复劳动 |
| 课程资产沉淀 | 教师成果长期散落在本地文件和聊天群 | 系统必须形成课程知识空间、版本和正式状态 |
| 教研复用 | 优质内容难以被后续引用和演化 | 系统必须支持 `Reference` 和空间网络 |
| 权限治理 | 教师、学习者、组织和机构之间需要明确边界 | 系统必须具备身份、成员和组织治理能力 |
| 长期交付 | 客户不能采购一个只能演示的系统 | 系统必须具备导出、绑定、证据和边界清晰的长期交付能力 |

组织侧需求因此天然要求系统从一开始就考虑治理、复用、版本和成员边界，而不是把这些问题留到后期再补。否则，所谓“课程数据库”最终仍会退化回一堆难以管理的文件资产。

## 3.3 为什么必须有这套对象语言

如果系统只围绕任务、消息和导出文件运转，它最终仍然会退化成一次性内容工具。要让业务闭环真正成立，`Spectra` 必须具备一套稳定对象语言；没有这套对象语言，所谓沉淀、复用、治理和商业化都只会停留在口号层。

| 对象 | 为什么必须存在 | 业务意义 |
| --- | --- | --- |
| `Project` | 需要一个长期存在的课程知识空间 | 承接资料、成果、版本、引用和成员关系 |
| `Session` | 需要把一次工作展开组织起来 | 承接对话、生成、修改和状态事件 |
| `Artifact` | 需要把外化结果纳入正式语义链 | 让 PPT、Word、导图、动画等结果可被绑定和复用 |
| `Version` | 需要区分过程态和正式态 | 让课程资产具备正式状态锚点 |
| `Reference` | 需要表达跨空间复用关系 | 让课程空间之间形成知识网络 |
| `CandidateChange` | 需要表达修改与演化入口 | 让内容更新进入可治理演化 |
| `Member` | 需要表达协作与治理边界 | 让教师、学习者、组织和机构角色有清晰权限基础 |

这套对象语言之所以必须存在，是因为系统本体不再是导出文件，而是库与引用关系。只有这样，第 8 章的课程知识资产平台逻辑才不是空想，第 9 章的治理框架也才真正有落点。

## 3.4 核心能力需求

### 3.4.1 会话化意图理解

系统必须能承接教师以自然语言表达教学目标、重点难点、课时安排和风格要求，并将其组织为后续生成可用的结构化输入。

验收重点：

- 能围绕项目和会话承接一次完整工作过程；
- 能保留上下文、资料和教师修改意见；
- 能把一次备课过程组织为可观察、可推进的 `Session`。

### 3.4.2 多模态资料进入

系统必须支持 PDF、Word、课件、图片、视频等资料进入统一链路。这里的关键，是资料必须变成知识来源，而不是静态附件。

验收重点：

- 资料进入后能被系统识别为知识来源；
- 上传、解析和交付阶段具有可追踪语义；
- 资料结果能进入检索增强和生成主链。

### 3.4.3 检索增强与证据组织

系统必须在生成前组织可信证据，降低无依据生成和事实漂移。

验收重点：

- 能区分基础召回与证据组织；
- 能用正式评估结果说明检索增强质量；
- 能让证据真正进入生成链，而不只是停留在搜索结果展示。

### 3.4.4 生成主链

系统必须支持 AI 课件与演示文稿主生成，并让教师参与关键阶段。

验收重点：

- 支持 outline 形成与确认；
- 支持确认后进入正式 generation；
- 支持结果进入预览、修改和交付链；
- 生成结果能作为 `Artifact` 与课程空间绑定。

### 3.4.5 Studio 多模态内容工坊

系统不能只输出 PPT，而必须围绕同一课程空间外化多种内容形态。至少应覆盖：

- `PPT`
- `Word` 教案
- 互动小测
- 互动游戏
- 思维导图
- 演示动画
- 说课助手
- 学情预演

验收重点：

- 这些能力以 `Studio` 内容工坊统一呈现；
- 各类结果共享课程空间和上下文；
- 非 PPT 产物不以假预览冒充真实结果。

### 3.4.6 预览、修改与标准交付

系统必须保证用户看到的预览、经历的修改和最终交付属于同一能力链，而不是前台一套、后台一套。

验收重点：

- 支持真实 preview；
- 支持 refine 和 history；
- 支持 `PPTX / DOCX` 标准导出；
- 下载与成果绑定不脱离正式知识状态。

### 3.4.7 正式知识状态与课程空间复用

系统必须让生成结果不止于文件，而能回到课程知识空间。

验收重点：

- `Artifact` 能进入正式状态绑定；
- `Version`、`Reference`、`CandidateChange` 能表达演化与引用；
- 课程空间能作为长期课程知识资产持续增长；
- 学习者侧和组织侧可以在权限边界内复用课程空间。

### 3.4.8 身份、组织与成员边界

系统必须面向长期部署和机构使用保留身份、组织和成员边界。

验收重点：

- 身份与会话边界清晰；
- 组织和成员关系不污染课程知识本体；
- 系统具备向学校、教研组和机构扩展的治理基础。

## 3.5 非功能需求为什么重要

非功能需求决定长期交付是否成立。只要易用性、稳定性、可追踪性、可演化性和权限边界有一项缺位，系统就会重新掉回“演示很亮眼、落地却发虚”的老路。

| 非功能需求 | 具体含义 | 为什么重要 |
| --- | --- | --- |
| 易用性 | 教师不需要学习复杂 prompt 工程 | 决定试点门槛和真实采用率 |
| 稳定性 | 长链路生成、解析、渲染和导出具备阶段状态 | 决定系统能否进入真实交付 |
| 可追踪性 | 资料、证据、成果和版本之间有来源关系 | 决定审核、复用和组织信任 |
| 可演化性 | 课程空间、版本、候选变更和引用网络可持续增长 | 决定系统能否成为长期资产 |
| 可复用性 | 内容可被教师、学习者、教研组按需再利用 | 决定采购价值是否能放大 |
| 安全与权限 | 不同用户、课程和组织之间有清晰边界 | 决定学校和机构场景是否成立 |

这些需求共同说明，`Spectra` 从一开始就必须是一套可追踪、可治理、可演化的系统，而不是先做出结果、再考虑如何长期承接。

## 3.6 需求为什么天然要求六能力层

`Spectra` 的需求不是散点功能，因此也不该由一团模糊的后端承接。当前更合理的结构，是让控制平面和六个正式能力层分别承接不同需求问题。

| 需求问题 | 承接层 | 为什么这样分工 |
| --- | --- | --- |
| 教学意图理解与工作流组织 | `Spectra` | 作为 workflow shell 组织会话、事件、状态和契约 |
| AI 课件主生成 | `Diego` | 承担 outline 与 generation 主链 |
| 预览与标准导出 | `Pagevra` | 承担 preview、render、`PPTX / DOCX` export |
| 正式知识状态与演化语义 | `Ourograph` | 承担 knowledge-state core |
| 上传、解析与交付底座 | `Dualweave` | 承担 ingest / delivery 底座 |
| 检索增强与证据组织 | `Stratumind` | 承担 retrieval、evidence organization |
| 身份、组织与成员边界 | `Limora` | 承担 identity、session、organization/member boundary |

这张映射表真正说明的是：系统需求天然要求“控制平面 + 六能力层”的结构，而不是允许项目退回“单仓库把所有事都做了”的旧形态。需求如果是系统级的，承接它的结构也必须是系统级的。

## 3.7 需求优先级

### P0：必须成立的系统闭环

- 会话化意图理解；
- 多模态资料进入；
- 检索增强与证据组织；
- 课件生成与 outline 确认；
- 真实预览、修改与标准导出；
- 正式知识状态绑定；
- 课程空间长期存在。

### P1：放大交付价值的产品能力

- 更强的质量门控；
- 更丰富的证据组织；
- 更完整的 `Studio` 多模态外化体验；
- 更好的课程空间访问与展示能力；
- 更稳定的组织协作体验。

### P2：平台化延展方向

- 学习者侧个性化派生；
- 更深入的学校/机构治理；
- 更多内容形态外化；
- 更大规模的课程知识网络运营。

这组优先级首先对应业务成立顺序：先让系统闭环成立，再放大交付价值，最后进入平台化运营能力。顺序一旦颠倒，项目就会重新掉回“局部功能很花，整体系统不成立”的老问题。

## 3.8 本章结论

`Spectra` 的需求要求系统同时承接教学意图理解、多模态资料吸收、检索增强、生成主链、`Studio` 多模态外化、真实预览修改、标准导出、正式知识状态、课程空间复用以及身份组织边界。

需求本身是系统级的，它天然要求项目成为一套课程知识空间系统，并由控制平面和六个正式能力层共同承接。第 3 章证明的，正是这套长期系统为什么从一开始就是必要形态。
