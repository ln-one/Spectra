> 状态：`current`
> 用途：`internal handoff only`
> 角色：A 类“项目概要介绍”图文版定制设计说明

## 1. 结论先说

当前这份 [Spectra A 类概要介绍](/Users/ln1/Projects/Spectra/docs/competition/11-a-class-project-overview.md) 作为**文字源稿**已经够了，但作为最终提交件还不够。

原因很简单：

- A 类评委真正会认真看的，是这个 `概要介绍`
- 真实提交样例普遍不是纯文字，而是 `封面 + 图文分页说明`
- 如果这份材料最后还是大段文字，观感会明显吃亏

因此，后续默认路线固定为：

- [Spectra A 类概要介绍](/Users/ln1/Projects/Spectra/docs/competition/11-a-class-project-overview.md)：文字事实源
- [全书正式配图制作说明书](/Users/ln1/Projects/Spectra/docs/documentation/competition-figure-pack.md)：正式配图制作单
- [全书图文文字底稿](/Users/ln1/Projects/Spectra/docs/documentation/competition-visual-source-notes.md)：图注、标签、判断句来源
- 本文：页级结构、图文比例、每页核心判断和最终成稿气质

这四份材料必须成套使用，不允许只拿其中一份自由发挥。

我们真正要借的是三点：

- 有封面，第一页先给项目气质
- 从第二页开始，每页都有明确小节和配图
- 图不只是装饰，而是承担“系统 / 功能 / 特色 / 技术”的说明压力

不借的东西：

- 它的行业内容
- 它的视觉风格
- 它的花哨装饰
- 它的页序
- 它的文案句式

## 3. Spectra 概要介绍最终形态

默认按 `7 页 PDF` 去做，不做一页塞满的纯文字稿，也不允许退化成“1300 字文字稿 + 后面随便插几张图”。

### 第 1 页：封面页

**页面目的**

- 第一眼立项目气质

**必须出现**

- `Spectra`
- 项目全名
- 赛题编号与命题企业
- “项目概要介绍”
- 团队名称 / 成员 / 学校

**可以出现**

- 一张主视觉背景图
- 一句短口号，例如：
  - `生产即沉淀，沉淀即交付`

**禁止出现**

- 技术细节
- 过多正文

**页面定位**

- 只负责立项目气质和赛题归属
- 不承担正文解释任务

### 第 2 页：问题与项目定位

**页面目的**

- 评委快速理解为什么值得做
- 评委快速理解 `Spectra` 是什么

**推荐结构**

- 左上：前言
- 左下：创意描述 / 项目定位
- 右侧或下半：1 张系统逻辑图

**优先配图**

- [图 1-1 旧工具逻辑 vs Spectra 系统逻辑对比图](/Users/ln1/Projects/Spectra/docs/documentation/competition-figure-pack.md)
  - 如果来不及正式画，至少先做一版清爽静态图

**页面判断句**

- 教学内容生产真正缺的，不是再多一个生成工具，而是把生成、沉淀和交付组织成同一条主线。

**文字来源**

- [Spectra A 类概要介绍](/Users/ln1/Projects/Spectra/docs/competition/11-a-class-project-overview.md) 的 `前言` + `创意描述`

## 4. 图文比例建议

A 类 `概要介绍` 最终交付时，默认遵守以下比例：

- 封面页允许主视觉为主
- 正文页每页至少有 `1` 个视觉锚点
- 除封面外，任一页纯文字面积不要超过版面的 `60%`
- `第 3 / 4 / 5` 页必须是图文主力页
- 不允许出现“上半页一大段文字 + 下半页随便放一张小图”的假图文化

一句话底线：

- 这份材料可以文字不多，但不能全靠文字。

后续执行不允许出现以下情况：

- 只看文字稿，不知道该配什么图
- 只看图单，不知道每页要立什么判断
- 只看样例，就开始抄别人页序和行业内容
- 只补图，不控制页面结构

## 6. 文风底线

这份概要介绍最终必须做到：

- 比详细方案更短
- 比执行摘要更像提交件
- 比 PPT 更稳
- 比旧 `docx` 更真

执行时固定规则：

- 每页正文只保留 `1` 到 `2` 段主文
- 不堆技术名词
- 不堆 benchmark 数字
- 不复述六个服务细节
- 不写内部运行与开发机制
- 少用“不是……而是……”，多用正向定义
- 多用“系统已经形成”“当前已经具备”“当前系统承担”这类冷静判断句

## 7. 一句话底线

这份 `概要介绍` 最终应该像一个有完成面的提交件，而不是从详细方案里抠出来的一页说明文字。
