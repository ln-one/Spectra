# 前端联调建议

## 一、会话文件命名问题

### 1. 问题现状

当前联调中存在命名分叉：

- 历史记录标题与真实 PPT 工件标题不一致（出现硬编码文案，如 `PPT Ready`）。
- 部分下载命名仍使用前端兜底格式（如 `artifact-xxxx.ext`），未与工件标题保持一致。
- Word 教案虽已绑定 `source_artifact_id`，但来源 PPT 信息在界面上不够直观。

### 2. 联调目标

命名必须收口为同一语义源，确保用户看到的名称、历史记录名称、下载名称一致可追踪。

### 3. 建议约束

1. 历史记录标题：优先使用 `artifact.metadata.title`，仅在缺失时回退默认文案。
2. 下载文件名：与工件标题一致，不再前端硬编码 `artifact-xxxx.ext`。
3. Word 教案命名：按“源 PPT 标题 + 教案”生成显示与下载名（扩展名为 `.docx`）。
4. Word 来源可视化：在界面显式展示：
   - `source_artifact_id`
   - 来源 PPT 标题（由该 artifact 的 `metadata.title` 解析）

### 4. 联调验收清单（本部分）

- [ ] 生成 PPT 后，历史记录显示名称与该 PPT 的 `metadata.title` 一致。
- [ ] 下载 PPT 时，文件名与历史/工件展示名一致。
- [ ] 生成 Word 后，命名为“源 PPT 标题 + 教案.docx”。
- [ ] Word 区域可看到其绑定的 `source_artifact_id` 与来源 PPT 标题。
- [ ] 以上命名在刷新页面后仍保持一致（非一次性前端态）。

### 5. 风险提示

1. 若后端未写入 `metadata.title`，前端需明确回退策略并打日志，避免静默降级。
2. 若来源 artifact 不可访问（权限/删除），Word 来源展示需给出可理解提示，不应误导为“无来源”。
