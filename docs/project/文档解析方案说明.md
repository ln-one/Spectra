# 当前文档解析方案说明

## 1. 方案目标

当前文档解析采用“远端高质量优先 + 显式失败”的设计，目标是：

- 优先获得高质量解析结果（MinerU）
- 在远端失败或超时时返回明确失败态，而不是走隐式降级链
- 保证主系统可观测、可诊断，不再靠第二语义偷偷把请求糊过去

## 2. 核心流程

### 2.1 上传阶段

上传文件后，后端可进入 `defer_parse=true` 的等待远端结果模式，文件状态会先显示为：

- `status=uploading`
- `parse_result.deferred_parse=true`
- `parse_result.state=awaiting_remote_result`

### 2.2 成功路径（远端优先）

远端解析成功后，调用：

- `POST /api/v1/files/{file_id}/parse/mineru`

后端会：

- 用远端文本回填
- 触发统一分块与索引链路
- 更新文件为 `status=ready`

结果特征通常为：

- `parse_result.provider=mineru_remote`（或远端 provider）
- `parse_result.fallback_used=false`
- `parse_result.parse_mode=remote_preparsed`

### 2.3 失败路径（显式失败）

远端失败或超时后，不再提供单独的 fallback 触发接口。

后端会：

- 保留失败原因与 provider 诊断信息
- 将文件保持在明确的不可用状态，等待重试或新的远端结果
- 不再自动切换到另一套本地解析语义

## 3. 关键语义约定

为避免监控和验收误判，当前统一约定：

- 不再通过单独接口触发解析降级
- 成功就是远端成功；失败就是显式失败
- provider 诊断信息只用于观测，不再暗示产品层“偷偷成功”

## 4. 对外联调接口

核心接口：

- `POST /api/v1/files`
- `POST /api/v1/files/{file_id}/parse/mineru`
- `GET /api/v1/projects/{project_id}/files`

联调时重点查看字段：

- `status`
- `parse_progress`
- `parse_result.provider`
- `parse_result.fallback_used`
- `parse_result.stage_timings_ms`
- `parse_result.capability_status`

## 5. 常见排查点

- 文件一直停在 `awaiting_remote_result`：
  - 检查是否触发了 `/parse/mineru` 回填，或远端回调是否丢失
- 调用 `/parse/mineru` 失败：
  - 优先查后端日志 trace_id
  - 关注重建索引阶段（delete/reindex）异常
- 远端失败后文件未进入明确失败态：
  - 检查索引链路是否吞掉了 provider 错误
  - 检查文件状态更新与错误回写是否一致

## 6. 当前结论

当前方案口径是：

- 远端成功：采用远端结果
- 远端失败：显式失败，不再走隐藏降级
- 语义字段一致：可观测、可验收、可追踪

该方案可作为本阶段文档解析默认执行口径。
