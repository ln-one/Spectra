## 系统整体架构

系统整体架构设计的核心，在于以工程分层方式组织教学准备活动中的数据接入、业务编排、能力调用和成果沉淀过程。结合当前版本实现情况，`Spectra` 采用典型的多层分布式架构，涵盖接入表现层、业务编排层、核心能力中台及持久化存储层。各层通过明确的接口契约和对象语义协同运行，形成面向教学准备活动的全生命周期业务闭环。

从竞赛评审视角看，本节主要回答三个问题：系统按何种工程层次组织、各层分别承接什么职责、整体架构为何能够支撑完整业务闭环。

从系统职责看，该架构重点解决三类问题：第一，如何将多源教学资料、会话上下文和成果对象纳入统一业务链路；第二，如何通过状态驱动的业务编排机制组织内容生成、结果预览和标准交付；第三，如何通过稳定的对象关系模型承接成果保存、版本溯源和持续复用。当前架构以统一工作台为交互入口，以 `Session` 和成果对象为业务主线，以生成、检索、导出和状态管理等核心能力为支撑，具备清晰的工程结构。

图 5-3 将系统进一步分解为可识别的功能层次。访问层承接 PC 端浏览器、移动端浏览器及管理或演示入口；表现层承接首页、登录注册、项目列表、项目详情、生成预览和系统设置等页面；平台服务层承接 API 服务、统一认证、权限控制、会话状态管理、任务编排调度与事件反馈；核心业务层承接项目空间管理、多源数据接入与预处理、知识检索问答、大纲生成确认、课件生成、预览编辑导出、候选变更管理与成果对象管理；数据资源层与外部支撑层分别承接 PostgreSQL、Redis + RQ、Qdrant、Uploads 以及 Diego、Pagevra、Ourograph、Dualweave、Stratumind、Limora 和 DashScope 等能力来源。

从工程分层角度看，当前系统可进一步抽象为四层结构：

- 接入表现层：统一承接多源数据输入、会话推进、结果查看、修订与交付操作；
- 业务编排层：围绕 `Session`、事件和成果绑定组织状态机驱动的核心处理链路；
- 核心能力中台：承接多源异构数据接入与预处理、基于状态机的生成逻辑调度、检索增强、结构化渲染与标准交付、成果状态管理以及权限控制等专项能力；
- 持久化存储层：承接结构化数据、异步任务状态、语义向量索引、上传文件和成果文件等长期数据对象。

上述分层可直接映射到当前作品。接入表现层对应 Studio、Chat、Sources 与结果查看界面；业务编排层对应会话、预览和成果绑定相关接口与服务；核心能力中台对应生成、检索、导出、状态管理和权限治理链路；持久化存储层对应 PostgreSQL、Redis、Qdrant、Uploads 与成果对象存储。各层之间形成清晰的职责边界，保证系统在演示、维护和扩展过程中保持较高的可解释性和一致性。

从长期管理对象看，当前架构稳定承接五类核心实体：项目资料空间、工作过程、成果对象、成果版本以及对象间引用关系。`Project` 对应项目资料与成果对象统一管理空间，`Session` 对应一次具体业务过程，`Artifact` 对应生成成果对象，`Version` 对应稳定版本锚点，`Reference` 对应资料和成果之间的复用关系。该对象体系定义了系统长期维护的业务核心。

综上，`Spectra` 当前版本已形成“统一接入表现层 + 状态机业务编排层 + 核心能力中台 + 持久化存储层”的整体设计。该架构同时支撑教学资料接入、内容生成、结果交付和成果沉淀四类关键活动，为后续模块设计、数据处理流程和数据库设计提供统一的工程基础。

本节可进一步压缩为一句判断：`Spectra` 已形成层次清晰、职责明确、对象稳定的整体架构，可支撑教学准备活动的端到端系统运行。
