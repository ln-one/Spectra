# Spectra 商业方案书术语表

> Status: `current`
> Role: internal terminology contract for the external commercial proposal.

本文件用于统一商业方案书中的核心对象、系统层术语和服务边界表达，避免不同章节出现口径漂移。本文术语以当前系统现实为准；如文档表述与代码现实冲突，应优先回到当前 live code、focused tests、正式 benchmark 和 canonical architecture docs 进行校验。

## 1. 系统与架构术语

### `Spectra`

指本项目面向教学场景的产品外壳、统一工作台和课程知识空间控制平面。当前系统中，`Spectra` 的准确定位是：

- `workflow shell`
- `orchestration kernel`
- `contract surface`

它负责会话生命周期、任务编排、状态协调、能力聚合、下载绑定与对外产品契约，不再被描述为“大一统后端”。

### `Studio`

指教师使用的多模态内容工坊。它不是孤立功能入口，而是同一课程知识空间的外化操作面，覆盖 `PPT`、`Word` 教案、互动小测、互动游戏、思维导图、演示动画、说课助手和学情预演等能力。对外描述时，应把它理解为一组围绕真实教学工作展开的成果能力：既包括课堂主展示内容，也包括教案讲稿、知识结构梳理、互动练习、动态演示、说课支撑和课前预演。

### `Six Formal Capability Authorities`（六个正式能力源）

指当前系统中承担正式产品能力的六个外部能力源：

- `Diego`
- `Pagevra`
- `Ourograph`
- `Dualweave`
- `Stratumind`
- `Limora`

商业方案书中的技术与架构描述，应始终围绕这六者与 `Spectra` 的协同关系展开，但六个服务不是全文第一主角；系统本体仍是库、引用关系和课程知识空间。

### `Anti-Corruption Layer`（反腐层）

指 `Spectra` 在调用六个正式能力源时所保留的适配与编排层。它可以做 payload translation、response normalization、local mirror mapping 等工作，但不应重新定义上游正式语义，也不应悄悄长成第二套 domain owner。

## 2. 核心业务对象

### `Project`

指长期知识空间，也是课程资料、引用关系、版本锚点、外部产物和正式状态演化的承载体。对外可以理解为课程空间、学习空间或课程资料库；对内仍使用更中性的 `Project` 命名，以保持系统可迁移性。

### `Session`

指围绕某个 `Project` 展开的一次局部工作会话，用于承载对话、预览、修改、事件流、运行状态和生成过程。它是工作过程的载体，不是长期正式状态本体。

### `Artifact`

指从当前知识状态中外部化出的结果，如 `PPTX`、`DOCX`、预览页、导图、动画、互动内容或其他派生材料。`Artifact` 不是系统长期语义本体，而是特定场景下的外化成果，应记录其来源项目与来源版本。

### `Version`

指 `Project` 在某一时刻的正式稳定状态锚点。只有正式进入知识状态的结构化变化才产生新版本；草稿、临时预览和过程态对话不单独构成正式版本。

### `Reference`

指一个 `Project` 对另一个 `Project` 的结构化引用关系，是知识网络和复用体系的基础。

### `CandidateChange`

指尚未写入正式知识状态、等待维护者审核或后续确认的候选变更。它是工作过程向正式状态演化的重要入口。

### `Member`

指与某个知识空间或组织关系绑定的成员语义对象，用于承载访问、协作和后续治理边界。当前正式成员与组织身份 authority 由 `Limora` 与 `Ourograph` 协同承接。

## 3. 知识空间与演化术语

### `Knowledge Space`（知识空间）

指围绕课程或项目沉淀的长期知识容器，是本系统比“生成一个文件”更重要的目标对象。它可承载课程逻辑、资料索引、引用关系、版本演化和外化结果。

### `Formal Knowledge-State`（正式知识状态）

指被正式系统语言承认、可绑定版本、可被引用、可形成后续演化的状态集合。它不是工作过程，而是工作过程沉淀之后的正式语义结果。

### `Knowledge Return Flow`（知识回流）

指生成、修改、导出后的结果并不止步于文件输出，而可以继续影响后续知识状态、版本与资产组织的过程。它体现了 `Spectra` 从“生成工具”走向“知识系统”的关键差异。

### `Follow` / `Pinned`

`Follow` 表示一个空间跟随上游空间的最新合法状态；`Pinned` 表示一个空间以某个上游版本为固定起点进行后续独立演化。商业方案书中可用于解释课程空间复用与组织协作价值，但不展开过深 formal kernel 细节。

## 4. 生成、检索与交付术语

### `LLM`（Large Language Model，大语言模型）

指承担自然语言理解、规划、生成和局部优化任务的大语言模型。在本系统中，它主要用于教学意图理解、内容规划、文本生成和局部修改协同。

### `RAG`（Retrieval-Augmented Generation，检索增强生成）

指在生成前先检索与组织相关证据，再将这些证据送入生成链路，从而提高结果的相关性、可信度和可追溯性。

### `Evidence`（证据）

指被检索、重组并送入生成链路的上下文材料。它比“召回几段文本”更强调来源、相关性、排序位置和可引用性。当前系统中，证据组织的正式技术核心由 `Stratumind` 承担。

### `Ingest`（资料接入）

指资料从上传、解析、规范化到进入后续处理链路的全过程。当前系统中，与上传解析和交付相关的底座能力由 `Dualweave` 承担。

### `Preview` / `Export`

`Preview` 指用户在正式导出前查看真实结构化结果的过程；`Export` 指把当前系统结果正式输出为 `PPTX`、`DOCX` 或其他标准交付成果的过程。当前 preview / render / export authority 归 `Pagevra`。

## 5. 六个正式能力源术语

### `Diego`

承担 AI 课件/PPT 主生成的正式能力源，负责大纲生成、大纲确认、逐页生成、事件流与产物返回。

### `Pagevra`

承担预览、渲染与 `PPTX / DOCX` 正式导出的正式能力源，负责把结构化生成结果转化为可展示、可交付的输出。

### `Ourograph`

承担正式知识状态与知识库语义的正式能力源，是 `Project / Reference / Version / Artifact / CandidateChange / Member` 这套 formal semantics 的核心。`Spectra` 使用它，但不拥有它。

### `Dualweave`

承担上传、解析与多阶段交付基础能力的正式能力源。商业方案书中可以强调它的系统价值、位置、工程难度和评估维度，但不应展开专利敏感的核心机制。

### `Stratumind`

承担检索增强、证据组织和 retrieval 相关能力的正式技术核心。商业方案书中可以结合最新正式 benchmark / report 强调其技术实力。

### `Limora`

承担身份、会话、组织与成员边界的正式能力源，使系统不再在单个产品仓库中混合实现身份 authority。

## 6. 写作约束

1. 首次出现英文缩写时，正式稿中应给出中文说明或上下文解释。
2. 不把 `Spectra` 再写成“知识库 owner”或“大一统后端”。
3. 不把 `Ourograph` 写成 `Spectra` 内部一个普通 service；应明确它是知识状态核心。
4. 不把 `Dualweave` 和 `Ourograph` 写到可直接复刻核心设计的细节级别。
5. “库”“知识空间”“课程资料库”必须与“静态文件集合”区分开来，强调其正式状态、可演化性和引用关系。
6. 六个正式能力源必须出现，但不能取代“知识空间 + Studio + 多模态外化”的产品本体。
