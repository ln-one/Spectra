# 下一阶段推进清单

> 适用状态：当前已完成基础 MVP，需要从“可演示”推进到“可稳定迭代”。
> 文档目标：只回答两件事：`要干什么`、`要产出什么`。

## 敏捷产出原则

- 产出以“可运行功能、可验证结果”为主，不以写文档为目标。
- 文档仅保留最低必要：接口变更、运行方式、验收口径。
- 评审看代码、测试、演示结果，文档只做索引与补充。

## 1. 当前阶段判断

- 已完成：核心闭环（认证、项目、对话、上传、RAG、生成、预览、下载）已打通。
- 当前主要问题：稳定性、配置安全、任务可靠性、前后端一致性仍有提升空间。
- 下一阶段目标：将系统从“能跑”提升到“可持续迭代、可试点使用”。

## 2. 要做什么（按优先级）

## P0 必做（先做）

### 2.1 前端去 Mock 化与真实 API 收敛

要做什么：
- 默认走真实 API。
- `NEXT_PUBLIC_MOCK=true` 时才允许 Mock。
- 清理页面/Store/API 层对 Mock 数据的隐性依赖。

要有什么产出：
- 一组 PR：完成 auth/projects/chat/files/generate/preview/rag 全链路真实调用。
- 一套可执行回归（脚本或 E2E）：覆盖“登录 -> 创建项目 -> 上传 -> 生成 -> 预览 -> 下载”。
- 一次联调演示记录（可用 issue/comment 记录，不单独写长文档）。

验收标准：
- 生产构建默认不走 Mock。
- 核心页面在真实后端下可完整运行。

### 2.2 后端安全与环境配置硬化

要做什么：
- CORS 从硬编码 `*` 改为环境变量白名单。
- 统一开发/测试/生产环境变量定义。
- 明确鉴权、限流、错误码口径。

要有什么产出：
- 一组配置改造 PR（`main.py`、`.env.example`、部署配置）。
- 启动即生效的环境变量模板（开发/生产）。
- 一次安全配置自检结果（在 PR 描述中给出即可）。

验收标准：
- 生产环境不允许 `allow_origins=["*"]`。
- 配置变更可通过实际启动与接口请求验证。

### 2.3 异步任务可靠性升级（生成与索引）

要做什么：
- 将关键后台任务从进程内 `BackgroundTasks` 升级为可恢复的队列任务。
- 补齐任务状态流转、失败重试、手动重放能力。

要有什么产出：
- 一组任务基础设施 PR（队列接入、worker、状态持久化）。
- 失败任务可重试/重放的接口或命令。
- 一组稳定性测试结果（重启、失败重试、重复提交）。

验收标准：
- 服务重启后任务状态可追踪。
- 失败任务可重试且有明确错误原因。

## P1 应做（紧接着做）

### 2.4 RAG 质量评估体系

要做什么：
- 建立固定评测集，不再只靠主观体验调参。
- 建立检索与生成质量指标。

要有什么产出：
- 一份可运行评测数据集（存仓库即可，不需要长说明）。
- 一组可复现指标输出（命中率、引用率、延迟、失败率）。
- 一组优化 PR（chunk 策略、召回参数、prompt 结构）。

验收标准：
- 每次优化前后有可比较指标。
- 关键查询可给出稳定来源引用。

### 2.5 解析器可插拔架构（为 MinerU/LlamaParse 做准备）

要做什么：
- 抽象解析器接口，保留当前本地轻量解析为默认实现。
- 为规划解析器提供接入点，不一次性切换全链路。

要有什么产出：
- 一组实现 PR（LocalProvider + 预留 MinerU/LlamaParse Provider）。
- 一组回归测试（PDF/Word/PPT 解析与失败 fallback 行为）。
- 最小接口说明直接写在代码注释/README 小节中。

验收标准：
- 不改上层业务即可切换解析 provider。
- 默认 provider 行为与当前 MVP 一致。

### 2.6 前端交互稳定性与错误体验统一

要做什么：
- 统一 loading/error/empty/retry 状态模型。
- 统一上传、生成、预览修改链路的错误反馈。

要有什么产出：
- 一组 UI/Store 改造 PR（统一状态组件、错误展示、重试入口）。
- 一组关键流程 E2E 用例。
- 一次可演示的异常场景回归结果（上传失败、生成失败、超时）。

验收标准：
- 关键页面错误处理表现一致。
- 失败场景用户有明确下一步操作。

### 2.8 前端界面重设计（视觉与信息架构）

要做什么：
- 对现有 MVP 页面进行一次系统化 UI/UX 重设计（不仅是样式微调）。
- 重构信息层级、页面布局和关键交互入口，保证教学场景下的可读性与操作效率。
- 保持与现有 API/业务流程兼容，不影响核心功能闭环。

要有什么产出：
- 一版可运行的新界面实现（覆盖登录、项目列表、项目工作台、生成、预览/修改、下载）。
- 关键页面原型或截图集（用于评审，不强制单独设计文档）。
- 一组实施 PR（页面重构 + 通用组件更新 + 响应式适配）。

验收标准：
- 核心页面视觉风格一致且层级清晰。
- 关键任务路径点击成本下降（创建项目/上传资料/发起生成/预览修改/导出）。
- 桌面端与移动端均可正常使用，无明显布局断裂。

### 2.8 可观测性补齐（日志/链路/指标）

要做什么：
- 引入 request_id / task_id 链路追踪。
- 统一结构化日志字段与错误分类。

要有什么产出：
- 一组埋点 PR（关键接口、任务处理、RAG 调用）。
- 一个可直接使用的监控面板或查询脚本。
- 最小日志字段约定（写在代码常量/README 小节中）。

验收标准：
- 能通过 request_id 快速定位一次请求全链路。
- 核心失败类型可统计。
