# 成员D文件解析联调建议

## 1. 联调目标（D 负责范围）

本阶段 C 侧已完成解析链路能力接入与语义修复，D 侧联调重点应聚焦在“质量与策略收口”，而不是基础链路可用性。

D 需要回答的核心问题：

- 哪类文件必须走高质量远端解析（MinerU）
- 哪类文件可以接受本地解析（local）
- 失败降级策略是否满足教学场景稳定性与时效要求
- 如何把评测结论沉淀为可执行路由规则

## 2. 当前链路状态（联调前置共识）

当前后端行为已满足：

- 远端解析成功时优先采用远端结果
- 远端失败时可触发后端 fallback 到 local
- fallback 场景语义一致：
  - `provider=local`
  - `fallback_used=true`
  - `capability_status.fallback_used=true`

D 侧联调无需再验证“是否有接口”，应重点验证“结果质量与策略边界”。

## 3. D 侧样本评测任务

建议按以下样本分层进行对比（每类 3-5 份）：

- 普通文本型 PDF
- 公式/图表密集 PDF
- 扫描/OCR 压力 PDF
- 大文件长文档 PDF

每份样本至少对比两条路径：

- MinerU 结果
- local 结果（含 fallback 场景）

## 4. 统一评测指标（D 输出口径）

建议使用统一量化指标，避免主观描述：

- 文本完整度：缺段、漏页、乱码比例
- 结构可用性：标题层级、列表、表格保真度
- chunk 可检索性：切分后语义连贯性与可召回性
- 下游可用性：RAG 命中质量、生成内容可用度
- 性能稳定性：平均耗时、P95 耗时、失败率

## 5. 联调接口与关键字段

联调接口：

- `POST /api/v1/files/{file_id}/parse/mineru`
- `POST /api/v1/files/{file_id}/parse/fallback`
- `GET /api/v1/projects/{project_id}/files`

关键字段：

- `parse_result.provider`
- `parse_result.fallback_used`
- `parse_result.stage_timings_ms`
- `parse_result.capability_status`
- `parse_result.chunk_count` / `indexed_count`

## 6. D 侧需要产出的“可执行策略”

D 的结论应直接可落地为规则，而非仅研究描述。建议最终给出：

- 默认路由策略：
  - 哪些条件优先 MinerU
  - 哪些条件可直接 local
- 超时与降级策略：
  - 远端等待阈值
  - 触发 fallback 条件
- 质量优先策略：
  - 高质量模式触发条件（如扫描件/公式密集）
- 稳定性优先策略：
  - 课堂实时场景的兜底策略

## 7. 验收标准（面向收口）

联调收口建议满足：

- 有样本分层评测结果与明确对比结论
- 有可执行的默认路由与降级阈值
- 关键字段语义一致，不出现监控歧义
- 能给出“生产默认策略 + 特殊场景策略”两套建议

## 8. 与 C 的协作边界

C 负责：

- 接口与执行链路稳定
- fallback 语义一致
- 工程实现与回归修复

D 负责：

- 质量评测设计与样本结论
- 策略阈值与路由建议
- 最终收口口径（默认策略/例外策略）

目标是形成“可复现评测 + 可执行策略 + 可观测语义”三位一体的文件解析方案。
