# PDF解析能力对比与生产建议

日期：2026-04-01

## 1. 目的

记录当前项目在 PDF 解析链路上三种方案的真实对比结果，并给出生产环境下的默认解析策略建议。

本次比较关注三项核心指标：

- `chunk_count`：切块数量，反映文本颗粒度与后续 RAG 可分段程度
- `text_length`：提取文本长度，反映解析覆盖度
- `parse_ms`：解析耗时，反映解析链路成本

## 2. 三种方案分别接的是什么

### 2.1 `local`

接入方式：

- 项目本地解析
- 不依赖外部云服务
- 当前实现主要基于 `pypdf` / `python-docx` / `python-pptx`

特点：

- 最快
- 结构恢复能力最弱
- 适合作为兜底解析方案

### 2.2 `llamaparse`

接入方式：

- 调用 LlamaParse 云端 API
- 依赖 `LLAMAPARSE_API_KEY`
- 属于远端云解析

特点：

- 相比 `local`，结构和文本覆盖度有明显提升
- 速度明显慢于 `local`
- 目前更适合作为常规云端增强解析路径

### 2.3 `mineru_cloud`

接入方式：

- 调用 MinerU 官方精准解析云 API
- 依赖 `MINERU_CLOUD_API_TOKEN`
- 属于远端云解析，不是本地自建 FastAPI

当前项目接入方式：

- 本地文件先通过官方上传接口申请上传链接
- 上传文件后由官方异步创建解析任务
- 项目轮询任务状态
- 下载官方返回的结果压缩包，并提取 `full.md`

特点：

- 三者中解析覆盖度最强
- 速度最慢
- 官方控制台当前显示有默认额度

当前已确认的额度信息：

- 今日解析文件数：`10000`
- 今日优先解析页数：`2000`

说明：

- 当前至少存在可直接使用的默认额度
- 但官方页面未在本次验证中明确给出超额后的价格表
- 因此，`mineru_cloud` 需要视为“远端官方服务，且可能受额度/限流约束”的方案

## 3. 实测结果

同样 PDF 文件解析结果如下：

- `llamaparse`: `chunk_count=31`, `text_length=8737`, `parse_ms=71362.12`
- `local`: `chunk_count=23`, `text_length=6513`, `parse_ms=1735.64`
- `mineru_cloud`: `chunk_count=52`, `text_length=14210`, `parse_ms=115409.33`

## 4. 相对提升幅度

### 4.1 `llamaparse` 相对 `local`

- `chunk_count`：`+34.8%`（`31 vs 23`）
- `text_length`：`+34.2%`（`8737 vs 6513`）
- `parse_ms`：约 `41.1x`（`71.4s vs 1.7s`）

结论：

- 解析覆盖度明显提升
- 速度显著变慢

### 4.2 `mineru_cloud` 相对 `local`

- `chunk_count`：`+126.1%`（`52 vs 23`）
- `text_length`：`+118.2%`（`14210 vs 6513`）
- `parse_ms`：约 `66.5x`（`115.4s vs 1.7s`）

结论：

- 解析覆盖度提升最明显
- 速度成本也最大

### 4.3 `mineru_cloud` 相对 `llamaparse`

- `chunk_count`：`+67.7%`（`52 vs 31`）
- `text_length`：`+62.6%`（`14210 vs 8737`）
- `parse_ms`：约 `1.62x`（`115.4s vs 71.4s`）

结论：

- `mineru_cloud` 的文本覆盖和切块粒度明显优于 `llamaparse`
- 但耗时进一步增加

## 5. 当前生产判断

### 5.1 结论概述

当前三种方案的工程定位应为：

- `local`：最快，作为最终兜底
- `llamaparse`：速度与质量的平衡点，适合作为常规默认解析路径
- `mineru_cloud`：质量最强，但耗时最高，适合作为复杂 PDF 的高质量增强通道

### 5.2 推荐默认策略

建议当前项目默认 PDF 解析路径为：

1. 默认使用 `llamaparse`
2. 对复杂 PDF 使用 `mineru_cloud`
3. 任一云端解析失败时回退 `local`

原因：

- `local` 虽快，但信息覆盖偏少，不适合承担默认高质量解析职责
- `llamaparse` 相比 `local` 已明显提升覆盖度，且速度仍优于 `mineru_cloud`
- `mineru_cloud` 虽然质量最强，但对主链路吞吐不够友好，同时受官方云端额度和远端服务策略约束

## 6. 适用场景建议

### 6.1 优先 `llamaparse`

适合：

- 普通教学 PDF
- 结构相对规整的文档
- 更关注整体吞吐和稳定性的主链路

### 6.2 优先 `mineru_cloud`

适合：

- 扫描件
- 公式较多的 PDF
- 图表较多的 PDF
- 版式复杂、跨栏明显、结构恢复要求高的资料
- 对文本完整性和切块细粒度要求更高的场景

### 6.3 回退 `local`

适合：

- 云端 provider 不可用
- 云端超时
- 返回空输出
- 额度/限流导致云端暂不可用

## 7. 失败样本处理建议

### 7.1 生产策略

- 默认走 `llamaparse`
- 对大文件先拆分
- 若某一分段出现 `provider_error_type=empty_output`，自动回退 `local`

### 7.2 进一步定位

对失败样本可继续做二分定位，例如：

- `p1-30` 再拆为 `p1-15`
- `p16-30`

目标：

- 找到具体失败页段
- 在汇报中明确写出“某页段兼容性问题”，而不是笼统描述为整体失败

## 8. 当前项目建议落点

当前建议不要将 `mineru_cloud` 直接作为全部 PDF 的默认主链路。

建议方案：

1. 生产默认：`llamaparse`
2. 高质量专项增强：`mineru_cloud`
3. 最终兜底：`local`

一句话总结：

> 当前最稳妥的生产策略是：常规 PDF 默认走 `llamaparse`，复杂高质量 PDF 走 `mineru_cloud`，`local` 作为最终回退。
