# Spectra 项目简要汇报（现状与规划）

> 汇报日期：2026-03-03
> 项目：Spectra（多模态 AI 教学助手）

## 1. 项目目标（一句话）

Spectra 通过“对话 + 资料上传 + RAG 检索 + 异步生成”的方式，帮助教师快速生成可下载的教学课件（PPTX）与教案（DOCX），并支持后续预览和修改。

## 2. 当前已实现（截至 2026-03-02）

### 2.1 业务闭环

已打通核心流程：
- 注册/登录
- 创建项目
- 上传资料并入库
- 发起生成任务
- 查询任务进度
- 下载生成结果（PPTX/DOCX）

### 2.2 已落地能力

- 认证系统：`register / login / me / refresh`
- 项目管理：创建、列表、详情、编辑、删除、统计
- 文件模块：上传、批量上传、删除、意图标注
- 对话模块：消息收发、历史消息、语音入口
- RAG 模块：索引、检索、相似内容查询
- 生成模块：课件生成任务、状态查询、版本查询、下载
- 预览模块：结果预览、按指令修改、导出
- 异步任务：基于 Redis + RQ 的队列与 Worker（支持优先级、重试、状态追踪）

## 3. 项目是怎么运转的（主流程）

1. 用户在前端发起项目与输入（文本/文件）。
2. 前端通过统一 API Client 调用后端 `/api/v1/*` 接口。
3. 后端 Router 完成鉴权与参数校验，转到 Service 层处理业务。
4. 文件解析后进入分块与向量化，写入 ChromaDB，供 RAG 检索。
5. 生成请求进入任务队列，Worker 异步执行：检索上下文 -> 调用 LLM -> 产出 Markdown -> 转换为 PPTX/DOCX。
6. 前端轮询任务状态并展示进度，完成后提供预览与下载。

## 4. 目前采用的关键技术

### 前端

- Next.js 15 + React 18 + TypeScript
- Tailwind CSS + Radix/Shadcn UI
- Zustand（状态管理）
- React Hook Form + Zod（表单与校验）

### 后端

- FastAPI + Pydantic v2
- Prisma Client Python（异步）
- SQLite（当前默认）
- JWT 鉴权（PyJWT + passlib）

### AI 与数据

- LiteLLM + DashScope（Qwen）
- ChromaDB（向量检索）
- 本地轻量解析链路：`pypdf / python-docx / python-pptx`
- 课件导出：Marp CLI（PPTX）+ Pandoc（DOCX）

### 异步与工程化

- Redis + RQ（异步任务队列）
- 测试与质量：pytest、Jest、ESLint、Prettier、Black、isort、flake8
- OpenAPI 模块化维护 + CI/CD 自动校验

## 5. 技术选型说明（做什么、为什么用）

### 前端技术

- Next.js（做什么）：承载页面路由、SSR/CSR 混合渲染与工程化构建。  
  为什么用：生态成熟、与 React 体系兼容好，便于快速迭代项目页/工作台页等复杂页面。
- TypeScript（做什么）：提供类型约束，降低接口字段和状态流转错误。  
  为什么用：前后端接口多、状态复杂，类型系统能显著减少联调成本。
- Tailwind + Radix/Shadcn（做什么）：快速搭建统一 UI 与交互组件。  
  为什么用：开发效率高，可维护性强，能在 MVP 阶段快速形成一致的视觉与组件规范。
- Zustand（做什么）：管理认证、上传、生成、对话等跨组件状态。  
  为什么用：比重型方案更轻，学习和接入成本低，适合当前规模。

### 后端与数据技术

- FastAPI（做什么）：承载 REST API、参数校验、依赖注入、自动文档。  
  为什么用：异步支持好，开发效率高，和 Pydantic 配合能快速构建稳定 API。
- Pydantic v2（做什么）：请求/响应数据模型和校验。  
  为什么用：性能和类型体验较好，能统一接口边界。
- Prisma Client Python（做什么）：数据库访问与模型映射。  
  为什么用：数据模型清晰，迁移和维护成本较低，便于后续从 SQLite 迁移到 PostgreSQL。
- SQLite（做什么）：当前默认关系型存储。  
  为什么用：部署成本低，适合 MVP 快速验证；后续再迁移生产数据库。
- ChromaDB（做什么）：存储向量并支持语义检索。  
  为什么用：本地可运行、接入简单，适合当前 RAG MVP 阶段。

### AI 与任务技术

- LiteLLM + DashScope(Qwen)（做什么）：统一模型调用，完成对话与生成。  
  为什么用：LiteLLM 屏蔽多模型差异，DashScope 在当前场景下可用性与能力匹配度高。
- pypdf/python-docx/python-pptx（做什么）：解析上传文档，提取文本供检索与生成使用。  
  为什么用：轻量、稳定、实现成本低，先满足 MVP 主链路。
- Marp CLI + Pandoc（做什么）：把生成内容导出为 PPTX 和 DOCX。  
  为什么用：工具链成熟，可快速交付“可下载成果”。
- Redis + RQ（做什么）：异步执行生成任务，支持排队、重试和状态追踪。  
  为什么用：把耗时任务从请求线程剥离，提升稳定性和可扩展性。

## 6. 未来计划引入/升级的技术（做什么、为什么）

- PostgreSQL（计划）：替换 SQLite 作为生产主库。  
  目的：提升并发能力、可靠性与运维可控性。
- MinerU / LlamaParse（计划）：作为可插拔解析器能力。  
  目的：提升复杂文档（版式、图文混排）解析质量，降低知识抽取损耗。
- Qwen-VL（计划）：增强图片/视频理解能力。  
  目的：支持更完整的多模态教学资料处理。
- Faster-Whisper（计划）：语音识别与术语纠错链路。  
  目的：补齐语音输入能力，提升课堂场景可用性。
- 可观测性体系（计划，日志/链路/指标）：request_id、task_id、结构化日志。  
  目的：提高线上排障效率与稳定性治理能力。

## 7. 模块划分

- 前端模块：认证页、项目页、生成页、预览页、对话页、API 调用层、状态层（stores）
- 后端模块：Auth / Projects / Files / Chat / RAG / Generate / Preview / Download Routers
- 服务模块：AI、RAG、向量库、解析器、任务队列、任务执行、鉴权、数据库访问
- 数据模块：Prisma Schema、SQLite、ChromaDB、上传与生成文件目录
- 文档与契约模块：OpenAPI（`docs/openapi/`）+ 架构文档 + 开发规范
