# 7. 组织管理

## 7.1 平台型系统的治理目标

`Spectra` 是一套围绕课程知识空间持续演进的商业系统。它同时包含教师工作台、`Studio` 多模态内容工坊、`Spectra Backend` 控制平面、六个正式能力层、正式测试证据体系和对外交付材料。

因此，本章要说明的，是这样的平台型系统为什么能够被稳定组织、持续交付和长期治理。治理目标可以概括为三件事：

1. 产品主线持续成立：系统始终围绕课程知识空间、引用关系、成果外化和知识回流展开；
2. 工程边界持续成立：`Spectra` 保持 workflow shell / orchestration kernel，六个正式能力层各自承担清晰职责；
3. 外部交付持续成立：功能、质量、证据、文档和部署方式能够支撑采购方与技术评估者独立判断。

组织治理的核心，是让产品、架构、研发、测试、文档和商业交付都围绕同一个系统本体收敛。第 7 章要证明的，是这套项目不仅做得出来，而且管得住、交得出、跑得稳。

## 7.2 治理模型与职责分层

`Spectra` 更适合采用“产品本体牵引 + 能力组协同 + 证据闭环”的治理模型。各责任面不按孤立页面或孤立服务切分，而按系统长期成立所需的责任层划分。平台型系统一旦没有清晰责任层，后面所有质量、证据和商业表达都会迅速失焦。

| 治理责任层 | 核心职责 | 主要交付物 | 关键约束 |
| --- | --- | --- | --- |
| 产品本体层 | 定义课程知识空间、对象语言、主链闭环和商业价值 | 产品叙事、需求边界、对象模型、主稿总纲 | 防止系统被写回普通 AI 工具 |
| 产品体验层 | 建设统一工作台、`Studio` 工坊、真实 preview/refine 体验 | 页面、交互、卡片能力、结果展示 | 保证用户看到的是完整产品，而不是接口拼接 |
| 控制平面层 | 维护 `Session`、事件流、任务编排、artifact binding、API 聚合 | 主链服务、状态同步、错误表达、下载契约 | 保持主仓是编排中枢，不长出第二套能力源 |
| 正式能力层 | 持续演进 `Diego / Pagevra / Ourograph / Dualweave / Stratumind / Limora` | 微服务能力、接口契约、运行边界、服务文档 | 保证六服务支撑系统，但不被 Spectra 业务污染通用性 |
| 证据与质量层 | 建立测试集、benchmark、主链验证和成果展示证据 | 第 6 章证据、正式评估表、结果截图、适用边界 | 只使用当前正式证据，不让旧阶段数据回流 |
| 交付与文档层 | 将真实系统转化为独立成立的商业方案书与交付材料 | 商业主稿、图示、术语、风险与商业章节 | 文档必须离开仓库仍能自洽、专业、可评估 |

这种治理模型首先建立的是责任闭环：每一层都直接对应系统是否长期成立，而不是只服务于某个短期功能点。

## 7.3 协作原则

### 7.3.1 主链驱动

项目推进必须以主链闭环为中心，而不是以零散功能数量为中心。只要主链没有站稳，局部功能做得再多，也很难转化为外部认可的系统价值。当前最重要的主链包括：

- 资料进入、解析、检索增强和证据组织；
- `Session` 会话展开、outline 确认、生成和 refine；
- 真实 preview、标准导出、下载和 artifact binding；
- 课程空间中的版本演化、引用关系和成果回流；
- 从教师工作台到组织课程知识资产网络的长期沉淀。

每一项研发、测试和文档工作，都需要能解释它服务于哪条主链。如果一项工作不能改善主链清晰度、稳定性或可交付性，它就不应成为当前优先项。

### 7.3.2 边界评审

由于系统已经形成六个正式能力层，后续协作必须持续进行边界评审：

- `Diego` 负责正式课件生成能力，主仓不恢复旧生成核心；
- `Pagevra` 负责 preview / render / export，主仓不恢复本地正式渲染链；
- `Ourograph` 负责知识状态和知识库语义，`Spectra` 只消费其正式语义；
- `Dualweave` 负责资料进入和解析交付链路，对外维持低披露；
- `Stratumind` 负责检索增强和证据组织，正式数据以最新 benchmark 为准；
- `Limora` 负责身份、登录会话、组织和成员边界。

边界评审的作用，是防止系统在快速迭代中重新退化成大而乱的单体。治理如果不能守住边界，后续所有“平台化”“长期交付”都只会停留在口号层。

### 7.3.3 证据先行

商业方案书不能只靠“看起来很先进”的叙事成立。凡是进入正式主稿的关键能力，都应尽量配套至少一种证据：

- 产品面证据：真实页面、真实结果、真实 preview / refine / download 行为；
- 主链证据：从输入到生成、预览、导出、绑定、回流的闭环说明；
- 评估证据：正式测试集、benchmark、对比表和指标解释；
- 工程证据：运行边界、服务协同、故障表达和可部署性说明。

其中，详细测试集、指标和 benchmark 由第 6 章集中承接；第 5 章只保留帮助理解机制的结论级摘要。

## 7.4 交付治理

### 7.4.1 交付阶段

`Spectra` 的交付更适合按“可用、可信、可采购、可扩展”四个阶段推进。这个阶段划分的价值在于：它让项目的外部判断标准从“功能做了多少”转向“系统能否真正进入试点、采购和扩展”。

| 阶段 | 治理目标 | 关键验收点 |
| --- | --- | --- |
| 可用 | 教师能够完成核心备课与多模态外化 | 能创建课程空间、展开 `Session`、生成并预览主要产物 |
| 可信 | 系统结果有证据支撑，主链不靠假结果撑场 | preview 不伪造、正式评估可解释、失败有明确状态 |
| 可采购 | 外部读者不看仓库也能理解价值、能力和风险 | 主稿自洽，成果、测试、商业与风险章节互相支撑 |
| 可扩展 | 能向教研组、学校和机构化场景演进 | 组织边界、课程资产治理和部署路径清晰 |

这四个阶段会在当前主链上同步收敛。尤其是“可信”和“可采购”，必须贯穿研发和文档全过程。治理成熟度的标志，在于每一阶段都有清晰的外部验收含义。

这套阶段划分把项目从“能不能做”推进到“能不能被外部独立评估、试点和采购”，因此对应的是机构级判断，而不是内部协作便利。

### 7.4.2 版本与变更治理

`Spectra` 的项目本体强调版本、引用和候选变更，组织治理也应采用同样思想：

- 重要产品变化需要说明影响的对象：`Project / Session / Artifact / Version / Reference / CandidateChange / Member`；
- 重要架构变化需要说明是否影响六服务边界；
- 重要文档变化需要说明是否改变商业主线、证据口径或披露边界；
- 对外材料的关键数据必须能追溯到当前正式 benchmark 或正式系统验证结论。

这样的治理方式首先避免的是系统表达断裂：代码变了、文档没变、商业表达还停在旧版本，这类问题一旦出现，外部信任就会迅速下降。

### 7.4.3 文档工程化治理

方案书写作本身也必须按工程治理，而不是一次性作文处理。原因很简单：系统太大、链路太长、能力太多，如果文档不工程化，就会很快回到口径漂移、章节互抢、技术与商业互相打架的状态。

因此，文档治理必须坚持：

- 对外主稿保持完整阅读体验；
- 长章节按能力组拆成内部源稿，再汇总到主章；
- 技术章按“业务问题 -> 产品能力 -> 技术组合 -> 核心机制 -> 当前结果”写；
- 测试章集中承接测试集、benchmark、指标表和适用范围；
- 内部控制文档可以保存 source pack 和治理规则，但正式正文不依赖任何外部跳转。

这关系到项目能否长期保持真实、清晰和可信交付。

## 7.5 质量治理框架

### 7.5.1 产品质量

产品质量的核心，是让教师看到一个真实可用的内容工坊，而不是一组拼起来的 AI demo。

重点要求：

- 多模态产物必须被写成同一课程空间的不同外化切面；
- `PPT`、`Word` 教案、导图、动画、互动小测、互动游戏、说课助手和学情预演都应进入统一产品叙事；
- 前端不应使用假预览冒充真实结果；
- 结果必须进入 artifact、history、download 或后续回流语义，而不是只停留在临时展示。

### 7.5.2 技术质量

技术质量的核心，是让系统边界可解释、可维护、可扩展。

重点要求：

- 主仓保持 orchestration kernel，不把外部正式能力重新搬回本地；
- adapter / facade 只做契约翻译，不重新定义上游领域；
- 旧兼容路径和历史残余必须持续降级、隔离或清退；
- 新增能力必须能说清楚属于 kernel organ、transitional local auxiliary，还是正式能力层。

### 7.5.3 证据质量

证据质量的核心，是让商业结论有当前证据支撑。

重点要求：

- `Stratumind` 主展示数据只使用最新正式评估集；
- 旧阶段检索质量数据不进入正式主结论；
- `Dualweave` 在没有正式最新 benchmark 结论前，不写具体提升倍数；
- 指标必须解释业务含义，而不是只堆数字。

这三类质量要求合在一起，才能保证项目既有创新速度，又不牺牲交付可信度。对平台型系统来说，真正危险的是质量口径彼此脱节。

这套质量框架分别约束产品结果、技术边界和正式证据，目的是让系统在快速演进时仍保持外部可信。

## 7.6 治理型沟通与决策

项目决策应采用“问题域优先”的沟通方式。讨论一项变化时，先判断它属于哪类问题：

| 问题类型 | 决策重点 | 典型输出 |
| --- | --- | --- |
| 产品本体问题 | 是否强化课程知识空间和多模态外化 | 对象语言、用户价值、主链调整 |
| 架构边界问题 | 是否影响六个正式能力层和主仓职责 | 服务边界、adapter 规则、接口约束 |
| 体验问题 | 是否改善教师真实工作流 | 页面、状态、预览、交互和结果展示 |
| 证据问题 | 是否能支撑正式商业结论 | 测试集、benchmark、指标解释 |
| 商业问题 | 是否提高采购理由和交付可行性 | 客户分层、价格结构、交付路径 |
| 披露问题 | 是否影响专利、壁垒或可复制风险 | 公开粒度、图文深度、敏感机制控制 |

这种治理型决策机制能避免讨论在“多写一点”“多做一点”之间空转，而是始终回到：系统是否更专业、更可信、更可交付。它要解决的，是项目能否持续做出对外可信的判断。

## 7.7 本章结论

`Spectra` 的组织管理，本质上是在管理一个持续沉淀、持续外化、持续回流和持续演化的课程知识系统。它不适合按普通小工具项目推进，也不适合只按单个页面或单个服务拆任务。

更合理的治理方式，是以课程知识空间为本体，以 `Studio` 多模态内容工坊为产品面，以 `Spectra Backend` 控制平面为组织中枢，以六个正式能力层为能力底座，以正式测试证据和商业方案书作为对外可信表达。

只有在这样的治理框架下，项目才能既保持创新速度，又真正做到边界清楚、质量可控、交付可信和平台方向可持续。第 7 章立住的，是这套系统已经具备长期治理与持续交付的组织基础。
