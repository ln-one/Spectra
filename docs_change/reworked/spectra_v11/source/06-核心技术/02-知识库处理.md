## 基于资料解析与检索增强的知识库处理技术

知识库处理要解决的问题，是把教师上传的 PDF、Word、图片资料变成页面可查、生成可用、结果可追溯的证据块。当前流程包括资料进入、解析与标准化、切块、向量化、检索增强和证据组织，各步骤都能对上当前作品里的页面入口、接口、服务边界和正式评估结果。

![](../../assets/media/image15.png){width="7.0in" height="3.2in"}
图 6-3 检索增强流程图

这部分能力可以从以下证据核对：

| 证据类型 | 当前对应 |
| --- | --- |
| 页面入口 | 资料上传、来源查看、来源图片查看 |
| 接口契约 | `POST /api/v1/files`、`POST /api/v1/rag/index`、`POST /api/v1/rag/search`、`GET /api/v1/rag/sources/{chunk_id}` |
| 服务分工 | `Dualweave` 负责上传编排与远端解析入口，`Stratumind` 负责 dense + sparse 混合检索、规划、重排与证据打包，Spectra backend 负责查询组织、超时控制与响应整形 |
| 复核证据 | 远端解析衔接测试、RAG 指标计算与回归评估、正式评估集结果 |

当前知识库处理包含以下步骤：

- 资料进入：接收 PDF、Word、图片等资料；
- 远端解析：把复杂格式转成后续能继续处理的文本和页面内容；
- 标准化与切块：把长资料整理成更适合召回和引用的知识块；
- 向量化与检索：把 chunk 写入索引，供后续 dense + sparse 召回；
- 证据组织：把检索结果进一步整理成生成阶段可直接消费的 evidence。

在这一流程里，资料先经过上传编排与远端解析入口，解析结果再进入 Spectra 当前的切块、索引和检索流程。上传后的资料会继续参与来源查看、证据组织和生成上下文构建。若按当前实现把链路拆开，可以得到一条更具体的执行路径：

1. `POST /api/v1/files` 接收上传文件，`Dualweave` 负责远端解析入口与复杂格式衔接；
2. `backend/services/media/rag_indexing.py` 对解析结果做标准化，去除控制字符、整理换行，并按当前配置执行切块；
3. backend 将每个 chunk 写入数据库，同时把 `content / content_raw / content_indexed` 等上下文字段整理好后送入 `Stratumind`；
4. `Stratumind` 用远端 embedding 生成 dense 向量，并把 dense 索引与 sparse 侧字段一起写入当前检索存储；
5. 检索时由 `POST /api/v1/rag/search` 进入 backend facade，再转发给 `Stratumind /search/text`；
6. `Stratumind` 在查询侧先做 query rewriting 和 adaptive retrieval planning，再执行 dense + lexical 混合召回；
7. 候选结果按规划结果决定是否启用 query fusion、是否进入 rerank，以及最终采用哪种 evidence packing 方式；
8. backend 把 `results / evidence / planning_trace / rewrite_trace / telemetry` 取回后做响应整形，再用于页面来源查看或生成上下文拼装。

从工程实现角度看，这一流程已经形成稳定分工。资料上传接口接收文件并记录项目、会话和用途信息；`Dualweave` 负责把复杂格式转成可继续处理的内容；`backend/services/media/rag_indexing.py` 负责切块、索引请求组装和上层超时控制；`Stratumind` 承担正式检索执行。文件上传索引测试覆盖了 `pending_remote -> reconcile -> terminal` 的远端解析衔接逻辑，说明这一流程不是纸面设计。

检索链路的关键，不只是“做了向量检索”，而是当前实现已经把 dense、lexical、planning、rerank 和 evidence packing 明确拆开。`Stratumind` 的 README、`planning/planner.go`、`strategy_default.json`、`search_pipeline` 和 `evidencepack/evidence.go` 都能对上这条链。当前生产检索流遵循：

- `Query Rewriting`
- `Adaptive Retrieval Planning`
- `Hybrid Retrieval`
- `Optional Query Fusion`
- `Optional Rerank`
- `Citation-Aware Evidence Packing`
- `Retrieval Feedback Telemetry`

这条链路不是一个黑盒检索接口。查询改写、规划、召回、重排和证据打包是分开的，后续评估也能区分提升到底来自哪一层。

外部资料里已经有一些接近的路线，例如 Lewis 等提出的 RAG 基本范式、Adaptive-RAG 的动态策略切换、HyPA-RAG 的参数自适应检索，以及产业侧常说的 `agentic retrieval`。这里引用这些资料，只是说明当前检索链不是凭空起名，不替代项目自身实现。

### 当前检索链里的自适应规划

这一节不把 `ARP` 当作系统里的正式模块名，只把它当作一行简写，方便说明当前检索链已经具备“根据问题特征换策略”的能力。代码里真正存在的是 `query feature -> bucket -> profile -> 参数落地` 这条链，而不是一个独立命名的课程级调参中心。

从代码现实看，当前已落地的基础主要有三层。第一层是特征感知，`planner.go` 会从 query 中抽取错误码、文件名、符号型 token、中英文特征、定义意图、引用意图、调试意图以及长度桶等信号；第二层是 bucket 匹配与 profile 选择，系统会在 `debug / error_code / filename_lookup / citation / definition / keyword_heavy / length:long` 等 bucket 上命中候选，再映射到 `balanced_default / keyword_heavy / definition_lookup / citation_heavy` 等 profile；第三层是执行参数落地，被选中的 profile 会直接改写 `dense_weight`、`sparse_weight`、候选数、`evidence_mode`、fusion gate、`retrieval_mode` 与 rerank backend。

当前系统已经能根据问题特征调整：

- `dense_weight / sparse_weight`；
- `dense_candidate_k / sparse_candidate_k / rerank_candidate_k`；
- `evidence_mode`；
- `query fusion` 是否启用；
- `retrieval_mode` 是否转为 `light`；
- `rerank` 是否跳过，以及走哪一种 backend。

放到教学场景里看，这些调整对应的就是不同资料、不同问题、不同证据需求不再共用一套固定检索参数。概念解释型内容更偏语义召回和 definition-first 证据；术语密集型内容更依赖 sparse 命中；需要来源支撑的问题更偏 citation-first 证据组织。当前更适合把这一点写成“检索链已有自适应规划基础”，而不是“课程级编排已经完整产品化”。

图 6-3 里的区块可以直接对到当前实现的五个部分。

### 控制面：Query Rewriting 与 Adaptive Planning

图中的控制面对应 `Stratumind` 在正式检索前对查询做的两层处理。第一层是 `Query Rewriting`，会把原始问题整理成更适合检索的 `retrieval_query`，并抽取错误码、文件名、扩展词等显式信号；第二层是 `Adaptive Planning`，根据这些信号把查询打到不同 bucket，再从 strategy registry 中选出更合适的 profile、权重、候选数和 evidence mode。

这一层在代码里主要落在 `planning/planner.go` 和 `strategy_default.json`。例如：

- 出现错误码、文件名或符号型 token 时，更容易命中 `keyword_heavy / file_entity_lookup / debug_error_lookup`；
- 定义型问题更容易命中 `definition_lookup`；
- 需要引用支撑的问题会把 evidence mode 调整为 `citation_first`；
- 在低延迟预算下，还可能打开 `light_retrieval` 或 `skip_rerank` gate。

这一层留下的直接证据就是第 7 章里出现的 `planning_trace`、`matched_buckets`、`selected_profile_name` 等诊断字段。

### 召回面：Dense Retrieval + Sparse Retrieval + Hybrid Fusion

图中的召回面对应当前正式检索的候选构建阶段。`Stratumind` 不是只做单一路径向量检索，而是同时准备 dense 与 sparse 两类候选，再按 profile 中的权重和候选数组织混合召回。当前文档里说的 `dense + lexical hybrid`，对应的就是这一层。

这里的 dense 侧依赖远端 embedding，当前文本向量模型按仓库约定使用 `text-embedding-v4`；sparse 侧承担更强的关键词、文件名、错误码和局部字面命中能力。`strategy_default.json` 中不同 profile 对两侧权重和候选数给了明确配置，例如：

- `balanced_default`：dense / sparse 各占 `0.5`；
- `keyword_heavy`：sparse 权重升到 `0.65`；
- `semantic_heavy`：dense 权重升到 `0.7`。

若某些 bucket 打开了 `query fusion` gate，系统不会只检索一个 query 变体，而会把 `retrieval_query`、`normalized_query`、错误码项、文件名项、扩展词项等多个变体都跑一遍，再通过 `fusion.go` 里的 RRF 方式把不同变体结果融合起来。这里的 `QueryFusionGain / FusionCandidateGain / FusionUniqueChunkGain`，就是召回面扩展候选覆盖度后留下的量化记录。

### 排序面：API Rerank 与 Late Interaction Sidecar

图中的排序面对应候选结果进入精排的阶段。当前默认生产路径是 API rerank，而不是 late interaction。`Stratumind service.go` 里会在 hybrid 候选形成后判断是否执行 rerank；若 backend 为 `api`，就把候选文本送到 `Dualweave /rerank/text`，由 DashScope `qwen3-rerank` 返回新的排序分数，并把结果显式标成 `ranking_stage=rerank`。

这一层做的是候选精排，不是重新召回。当前实现还会保留：

- `base_score`：hybrid 阶段的原始得分；
- `rerank_score`：重排后的得分；
- `RerankBackendApplied / Fallback / DegradeReason`：本轮重排到底是否执行、有没有降级。

补充比较里的 `late_interaction` 和 `late_interaction_jina` 仍然属于这一层，但它们通过 sidecar 做 ColBERT 风格 token 级重排，只用于对比与受控实验。文档必须把这点写清楚，否则评审会误以为“项目正式链已经是 ColBERT 生产化”，这和代码现实不一致。

### 证据面：Evidence Packing 与引用准备

图中的证据面对应 `evidencepack.Build(...)` 这一步。当前系统不会在 rerank 之后直接把 top-k chunk 原样交给生成，而是再做一轮 evidence packing。这个过程的目的，是从候选结果里挑出更适合定义说明、事实支撑、引用准备和补充上下文的证据组合。

`evidencepack/evidence.go` 里可以看到，当前 evidence packing 至少会考虑：

- relevance reward：相关性仍然是主权重；
- source diversity reward：尽量提高来源多样性；
- citation readiness reward：优先挑更适合引用的块；
- redundancy penalty / same source penalty：抑制重复证据；
- conflict penalty：避免相互矛盾的证据同时进入最终包。

最终 evidence package 会把结果拆成 `PrimaryDefinition`、`PrimaryFacts`、`PrimaryCitations`、`SupportingContext` 等角色，并产出 `SelectedChunkIDs`、`CoverageScore`、`SourceDiversityScore`、`CitationReadinessScore` 等指标。第 7 章的 `Evidence Hit` 和 `Evidence MRR`，本质上就是在衡量这一层是否真的挑到了“可用证据”。

### 评估与观测面：Trace / Telemetry / Eval

图中的评估与观测面说明，这条检索链不是跑完就结束，而是把每一层的关键诊断保留下来。当前 backend 在调用 `Stratumind /search/text` 时会要求返回 `rewrite_trace`、`planning_trace` 和 `evidence`，而 `search_pipeline` 又会补充 `telemetry`。因此，系统至少能回答四类问题：

- 本次 query 有没有改写，改写用了哪些规则；
- planner 选了哪个 profile，命中了哪些 bucket；
- fusion 是否启用，候选覆盖度有没有提升；
- rerank 是否执行，证据最后选中了哪些 chunk。

第 7 章的正式评估和补充比较，就是建立在这一层可观测性的基础上。否则正文最多只能说“advanced 更好”，解释不到底是哪一层在起作用。

就当前具体实现而言，`hybrid` 不是笼统口号，而是 `dense + sparse` 同时进入候选召回。`Stratumind` 的 strategy registry 会根据查询桶和 profile 选择不同权重与候选数，而不是所有问题都走同一套参数。例如：

- `balanced_default`：`dense_weight=0.5`、`sparse_weight=0.5`、`dense_candidate_k=20`、`sparse_candidate_k=20`；
- `keyword_heavy`：`dense_weight=0.35`、`sparse_weight=0.65`，更偏向 lexical 命中；
- `definition_lookup`：`dense_weight=0.65`、`sparse_weight=0.35`，更偏向语义召回；
- `citation_heavy`：保留混合召回，同时把 evidence mode 调成更偏引用准备的模式。

这意味着当前 RAG 不是固定权重的一条死链，而是根据 query bucket 在 `balanced_default / keyword_heavy / definition_lookup / citation_heavy` 等 profile 之间切换。像 `citation`、`filename_lookup`、`keyword_heavy` 这类 bucket，还会进一步打开 `query fusion` gate，把不同召回变体的结果再做一次融合。

在重排层，当前正式生产路径默认不是 late interaction，而是 API rerank。`Stratumind` 内部 service 与 `Dualweave` 文档都说明，当前默认生产链会把候选文本送到 `Dualweave /rerank/text`，再由 DashScope `qwen3-rerank` 返回新的排序分数。重排后结果会显式带上 `base_score`、`rerank_score` 和 `ranking_stage=rerank`，因此系统可以区分“原始 hybrid 候选顺序”和“rerank 后顺序”。

补充比较中的 `late_interaction` 与 `late_interaction_jina` 方案，则不是生产默认链，而是 `Stratumind` 的 late-interaction sidecar 实验路径。它们通过 `POST /rerank/late-interaction` 对候选集做 ColBERT 风格 token 级重排，默认小模型是 `answerdotai/answerai-colbert-small-v1`，补充对比里又测了 `jinaai/jina-colbert-v2`。这一点在第 7 章评估表里要明确写出，否则读者容易误解成“当前线上正式路径就是 ColBERT”，这与代码现实不一致。

证据组织层也不是简单返回几个 chunk。backend 当前调用 `Stratumind /search/text` 时会显式要求 `include_evidence`、`include_planning_trace` 和 `include_rewrite_trace`，因此返回结果不止有检索命中项，还包含：

- `planning_trace`：本次查询命中了哪些 bucket、采用了哪种 profile、planner confidence 如何；
- `rewrite_trace`：本次 query rewriting 应用了哪些规则；
- `evidence`：当前 evidence mode 与证据打包结果；
- `telemetry`：当前 selected profile、query buckets、evidence chunk ids 等诊断信息。

后续生成链不是把“若干相似片段”直接塞进 prompt，而是优先使用经过 evidence packing 后更适合引用和组织的证据集合。第 7 章单独给出 `Evidence Hit` 和 `Evidence MRR`，也是为了验证这一层。

这一流程直接带来三个结果：

- 生成前能先找到更相关的内容；
- 结果更容易和已有资料对齐；
- 检索质量可以通过正式评估数据来说明。

Lewis 等关于 RAG 的研究给出了一条公开技术脉络：语言生成若要减少事实漂移，需要显式检索外部证据并让生成过程利用这些证据。Spectra 当前实现沿用这一基本思路，但这里的证据不是停在“召回几个相似 chunk”，而是还要经过 evidence packing，再返回到来源查看页面和生成上下文里。

在“根据问题差异动态调整检索策略”这一点上，Adaptive-RAG、HyPA-RAG 和 Microsoft `agentic retrieval` 也都提供了可对照的公开表达：有的强调按复杂度切换策略，有的强调参数自适应与混合检索联动，有的强调 query planning 和多路执行。Spectra 当前实现与这些思路相近，但落点更偏向教学资料场景下的 query feature、bucket 命中、profile 选择、evidence mode 和生成前证据组织，而不是复刻某一套通用研究框架。

在向量检索与质量调优层面，Qdrant 官方的 Search Engineering 教程进一步提供了混合检索、检索质量评估和召回调优的公开工程方法。Spectra 当前检索流程没有照搬其中某一个教程，而是在正式评估集、对比基线和证据组织上采用同样强调“可测、可比较、可重复验证”的工程思路。

第 7 章的 60 题与 105 题正式集评估显示，`advanced` 流程在 `Hit@3`、`MRR@3`、`Evidence Hit`、`Evidence MRR` 和综合质量指标上都明显优于 `raw_dense_topk` 与 `dense_only` 基线。这里的提升，对应的不是一句泛泛的“RAG 更强”，而是 query rewriting、adaptive planning、dense + lexical hybrid、API rerank 和 evidence packing 这一串具体动作叠加后的结果。

评审若要核对这部分能力，可以同时看五处：文件上传接口是否存在、来源查看是否落到页面、远端解析与检索衔接测试是否存在、正式评估集是否给出了稳定量化结果，以及重排策略补充比较是否明确写出当前实现方案。
