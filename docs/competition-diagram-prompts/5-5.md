# 5-5 真实预览与标准交付图

## 版本

`单版本`

## 默认适配场景

`PPT / Word 通用`

## 图类型

`契约 / 原则图`

## 这张图只回答什么

为什么系统坚持“真实产物、真实预览、标准交付、结果回流”这一条统一结果链，而不接受假预览和伪交付。

## 主阅读路径

先看左侧真实产物，再看中间预览与修改，再看右侧标准交付，最后看底部回流结果。

## 来源与事实锚点

- `docs/competition/05-key-technologies.md`
- preview / render / export / artifact binding 相关实现
- `docs/architecture/api-contract.md`

## 现有图问题检测

- 原图过于简短，容易像一句原则口号
- 缺少“可修改”和“结果回流”两层信息
- `结论`：`保留简洁，补强信息`

## 信息分层设计

- 左：真实产物
- 中：真实预览与修改
- 右：标准交付
- 下：结果回流

## 分组设计

- 主链三段：
  - `真实产物`
  - `真实预览 / 修改`
  - `标准交付`
- 辅助回流：
  - `Artifact / History`

## 密度策略

- `中低密度`
- 这张图可以简单，但不能空；要保留一个“结果回流”尾部来站住系统性

## 画幅与布局约束

- 横向优先
- 主链保持简洁
- 允许底部有一个小的回流区
- 不要画成参数表或 UI 截图拼贴

## 优化后的 Mermaid 骨架

```mermaid
flowchart LR
    A["真实产物"] --> B["真实预览 / 修改"] --> C["标准交付"]
    C --> D["Artifact / History"]
```

## 中文手绘主 Prompt

请重绘一张用于中国高校竞赛答辩或正文的真实预览与标准交付原则图。  
这张图不需要很复杂，但也不能只有一句口号。  
它要表达一条完整结果链：

- `真实产物`
- `真实预览 / 修改`
- `标准交付`
- `Artifact / History`

画面采用简洁的横向主链，最后接一个较轻的回流区。  
必须让人看出系统坚持的是：

1. 先有真实产物  
2. 再有真实预览和可修改过程  
3. 再形成标准交付  
4. 最后结果进入 `Artifact / History`，而不是停留在前端展示层

整体风格专业、高级、低饱和、克制、简约多彩，中文标签短、大、清楚。  
这张图应当简单、稳、可信，像一张原则图，而不是复杂技术图。
