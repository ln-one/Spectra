# Project-Space 演进设计（基于目标 API 完成态）

> 日期：2026-03-09
> 适用范围：在既有 `project + session` 主干上，演进到可引用、可协作、可按需派生的通用空间模型。
> 前提假设：`docs/project/NEXT_STAGE_ACTIONS.md` 与 `docs/project/SESSION_FIRST_MIGRATION_GUIDE_2026-03-06.md` 中规划的目标 API 已完成。

## 1. 文档目的

本文档用于回答两个问题：

1. 现有已规划完成的 `project/session` 架构，如何承接当前产品思路。
2. 下一步如何在不推翻现有 API 与数据边界的前提下，演进到更完整的引用式知识空间模型。

本文档不追求一次性冻结全部实现细节，只冻结核心对象、边界和演进方向。

## 2. 当前基线（假设目标 API 已完成）

在目标 API 完成态下，系统已具备以下稳定主干：

1. `project_id` 作为数据隔离边界。
2. `session_id` 作为一次生成/修改/预览/导出的会话边界。
3. `task_id` 退化为渲染执行兼容标识，不再承载产品主语义。
4. 上传资料、RAG 检索、生成、预览修改、导出可围绕 `project + session` 跑通。

这意味着，现有系统内部已经天然具备两个通用抽象：

1. `Project`：内容容器、权限边界、资料集合、结果归属边界。
2. `Session`：同一容器下的独立工作上下文，用于隔离对话、草稿和生成链路。

这套抽象本身足够通用，不应被“课程库”“教师库”“学生库”之类场景词替代。

## 3. 命名原则

为了同时满足产品表达与后续通用化，命名拆分为两层：

### 3.1 系统内部命名

保留并扩展通用命名：

1. `project`
2. `session`
3. `reference`
4. `artifact`
5. `version`
6. `member`

### 3.2 产品与方案层命名

可使用更贴近教学场景的表达：

1. 课程空间
2. 学习空间
3. 课程资料库
4. 知识空间
5. 库

结论：内部继续用 `project/session`，对外叙事可使用“库/课程空间/资料库”。其中“库”是最完整的产品本体词，“课程空间/学习空间”是不同场景下的表达。

## 4. 核心对象重定义

在下一阶段，系统核心对象可统一重定义如下：

### 4.1 Project

`Project` 是系统内部的主对象，对外可被解释为“库”“课程空间”或“学习空间”。

它承载：

1. 一套结构化知识状态。
2. 一组上传资料与资源索引。
3. 一组引用关系。
4. 一组成员与权限。
5. 一组派生结果与导出物。

如果用产品语言重述，则可定义为：

**库 = 一个可生成、可引用、可协作、可演化的课程知识单元**

### 4.2 Session

`Session` 是同一 `Project` 下的独立工作会话。

它承载：

1. 对话上下文。
2. 当前生成意图。
3. 草稿大纲与草稿结果。
4. 预览修改链路。
5. 协作者个人工作上下文。

`Session` 用于隔离工作过程，不直接等价于一个新的项目空间。

### 4.3 Artifact

`Artifact` 指从项目状态按需外化出的结果：

1. `pptx`
2. `docx`
3. 导图
4. 摘要
5. 练习
6. 网页/动画资源

它们不是系统本体，只是项目状态在某一时刻的外化结果。

### 4.4 Reference

`Reference` 指一个 `Project` 对另一个 `Project` 的引用关系。

它承载：

1. 被引用对象
2. 引用模式
3. 优先级
4. 权限范围
5. 可见性与可引用性约束

## 5. 产品核心判断

下一阶段产品的关键不在于“生成更多模态”，而在于：

1. 以 `Project` 作为统一空间对象。
2. 以课程数据库作为长期沉淀本体。
3. 以 `Session` 作为会话隔离与协作隔离机制。
4. 以 `Reference` 作为跨空间复用机制。
5. 以 `Artifact` 作为按需外化结果。

换句话说，系统本体是库与引用关系，不是单个导出文件。

## 6. 空间创建模型

### 6.1 新建原生空间

用户可直接从零创建一个 `Project`，也就是新建元库。

此时：

1. 没有上游引用
2. 自己上传资料
3. 自己生成内容
4. 自己维护空间状态

### 6.2 基于上游空间创建

用户也可基于已有 `Project` 创建自己的空间，也就是基于父库创建。

此时需选择引用模式：

1. `follow`
2. `pinned`

## 7. 引用模式

### 7.1 Follow

`follow` 表示当前空间跟随上游空间的最新合法状态。

特点：

1. 不复制完整底座
2. 存储成本低
3. 底层知识源自动跟踪
4. 派生结果按需刷新，不强制覆盖既有导出物
5. 不复制上游素材，仅保存引用状态与必要派生结果，避免存储膨胀

适用场景：

1. 下游学习空间（学生为常见示例）
2. 教师基于组内标准上游空间进行个性化教学准备
3. 协作者围绕主空间做轻量工作

### 7.2 Pinned

`pinned` 表示当前空间固定引用上游某个稳定状态，不自动跟随最新变化。

特点：

1. 起点稳定
2. 适合深度改造
3. 后续独立演化
4. 可作为高级功能实现更重的快照能力

适用场景：

1. 个性化重构
2. 稳定复习材料
3. 长周期课程改编

## 8. 多引用模型

一个库不应只允许单一来源。

### 8.1 主基底引用

每个库最多一个主基底引用，用于提供主知识骨架。

### 8.2 辅助引用

每个库可以有多个辅助引用，用于补充：

1. 案例
2. 实验
3. 题库
4. 风格
5. 跨学科内容

### 8.3 冲突优先级

多引用冲突按以下优先级处理：

1. 当前空间本地显式内容优先
2. 主基底引用优先
3. 辅助引用按顺序优先
4. 系统公共模板与默认知识最低优先

补充规则：

1. 辅助引用默认用于补充，不覆盖主骨架
2. 结构化字段优先于自由文本描述

## 9. 下游访问与个人空间

### 9.1 仅访问上游空间

当下游用户仅查看上游分享的库时（学生访问教师空间为常见示例）：

1. 只拥有可见权限
2. 不拥有写入权限
3. 不污染上游权威空间
4. 可按需查看上游空间当前可见内容的外化结果
5. 默认黑盒访问；公开库透明（未来可配置）。下游引用链不穿透上游。

### 9.2 临时按需生成

下游用户可在不创建个人空间的前提下，按需生成：

1. 思维导图
2. 摘要
3. 复习材料
4. 练习

这些结果不回写上游权威空间。

### 9.3 创建个人学习空间

若下游用户希望持续使用、上传资料、保存私有资产，则可基于上游空间创建自己的 `Project`：

1. 默认采用 `follow`
2. 默认私有
3. 默认不可被外部引用

## 10. 协作模型

### 10.1 推荐模式：同空间协作

对于教研组或多人共同维护同一课程库的场景，推荐使用“同空间协作”：

1. 协作者属于同一个 `Project`
2. 共享同一底层知识状态
3. 各自拥有独立 `Session`
4. 对话和草稿隔离

### 10.2 候选变更与审核合入

协作者不直接写正式空间状态。

统一机制为：

1. 协作者在个人 `Session` 中工作
2. 形成候选变更
3. 管理者审核
4. 审核通过后合入正式状态
5. 合入后产生新的正式版本标识

结论：默认统一采用“候选变更 -> 审核合入”，不再区分轻协作与无审核协作。

## 11. 轻量版本管理

系统需要版本，但不需要重型 Git 模型。这里的版本语义是“正式库状态的稳定标识”，而不是草稿历史流水。

### 11.1 什么算新版本

只有正式写入空间状态的结构化变化，才产生新版本。

例如：

1. 新增或删除知识点
2. 修改知识关系
3. 调整引用关系
4. 正式吸收协作者变更
5. 将生成结果确认写回项目状态

### 11.2 什么不算新版本

以下行为不直接形成正式版本：

1. 普通对话
2. 临时预览
3. 临时导图生成
4. 草稿修改
5. 页面查看

### 11.3 版本用途

版本仅承担以下职责：

1. 标识当前正式状态
2. 标识导出物来源状态
3. 比较上游是否发生变化
4. 支撑 `follow/pinned` 语义

## 12. 更新传播规则

### 12.1 Follow 模式

在 `follow` 模式下：

1. 底层引用状态自动跟随上游最新合法状态
2. 派生内容不强制实时覆盖
3. 用户查看或重新生成时，基于最新状态外化结果
4. 对旧导出物与旧结果给出“上游已更新”的提醒

### 12.2 Pinned 模式

在 `pinned` 模式下：

1. 上游变化不自动影响当前空间
2. 当前空间保持固定起点
3. 后续变化在本空间内独立演化

## 13. 权限模型

建议将权限至少拆分为四类：

1. `visible`：是否可见
2. `referenceable`：是否可被引用
3. `collaborative`：是否可协作提交候选变更
4. `manageable`：是否可审核、合入、配置权限

补充规则：

1. 个人学习空间默认不可被外部引用
2. 维护者可手动开启可引用能力
3. 可进一步限定范围：仅自己、指定成员、组内、公开

## 14. 网络结构约束

所有 `Project` 之间的引用关系必须构成 `DAG`。

不允许：

1. A 引用 B
2. B 引用 C
3. C 再引用 A

否则会破坏：

1. 更新传播
2. 权限判断
3. 来源追踪
4. 成本计算

## 15. 与比赛要求的对应关系

这一演进不会偏离比赛要求，反而会更完整地吸收题目要求：

1. `project + session` 支撑互动式意图理解与会话隔离
2. `upload + rag` 支撑多模态资料融合
3. `artifact` 支撑 PPT、教案、导图、网页、动画等按需外化
4. 轻量版本与候选变更支撑反馈、再生成、修改闭环
5. `reference` 支撑课程资料库从单点工具走向可复用、可协作、可扩展的平台形态

## 16. 分阶段落地建议

### Phase 1

在既有 `project/session` 主干上先补：

1. `project_reference`
2. `reference_mode`
3. 轻量 `version`
4. `artifact` 来源版本记录

### Phase 2

补：

1. 下游个人空间创建（学生为常见示例）
2. 个人空间默认私有
3. Follow 模式的上游更新提醒

### Phase 3

补：

1. 同空间协作
2. 候选变更
3. 审核合入

### Phase 4

补：

1. 多引用可视化
2. 引用优先级配置
3. 跨学科空间组合

## 17. 结论

现有 `project + session` 目标 API 路线不需要被推翻。相反，它正好可以作为下一阶段空间化、引用化、可协作化演进的稳定底座。

最终系统可被这样理解：

1. 内部以 `Project` 为主对象，对外即“库”
2. 以 `Session` 隔离工作上下文
3. 以 `Reference` 组织跨空间复用
4. 以 `Artifact` 承载按需外化结果
5. 以轻量 `Version` 支撑跟随、拉取、导出和更新提醒

更产品化的一句话表达为：

**系统以库为核心单元，以课程数据库为底座，以引用关系形成知识网络，以多模态内容按需外化。**

这使系统既能满足当前教学场景，又保留足够的通用化空间。
