### 非功能性需求分析

| 非功能需求 | 具体要求 | 当前体现 |
| --- | --- | --- |
| 易用性 | 教师不需要写复杂提示词也能推进流程 | 通过工作台和会话方式组织操作 |
| 稳定性 | 生成、解析、预览和导出需要有明确状态 | 主链围绕会话、事件和任务推进 |
| 可追踪性 | 资料、结果和版本关系需要可回看 | 结果保存、版本与绑定机制已纳入设计 |
| 可复用性 | 内容能继续引用和修改 | 生成结果可作为后续工作的基础 |
| 安全与权限 | 用户、组织和结果边界清晰 | 独立的身份与权限模块承接边界问题 |

这些非功能需求看起来不像“生成一个结果”那样直观，但它们直接决定了作品是否能稳定演示、是否方便继续修改、是否具备后续扩展基础。比如易用性影响的是教师能否在不写复杂 prompt 的情况下完成操作；稳定性影响的是生成、解析、预览和导出过程中是否有明确状态；可追踪性影响的是资料、结果和版本之间能否说清来源关系。

对当前作品来说，这几类非功能要求至少已经在以下方面得到体现：

- 操作方式上，系统把资料、会话、预览和结果操作放在同一工作台中，降低了使用门槛；
- 状态表达上，生成、解析、预览和导出围绕会话、事件和任务推进；
- 结果管理上，生成结果可以保存、回看、比对并继续复用，而不是导出后就与系统脱离；
- 权限边界上，用户、组织和结果之间的边界已经有相对独立的承接模块。

因此，第 4 章既要说明“系统要做什么”，也要说明“系统做到这些功能时必须具备什么样的工程基础”。这些非功能需求在当前作品中的成立条件和具体落点都需要写清楚。
