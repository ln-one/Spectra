## 多格式文档高保真渲染与转换技术

渲染与导出链承担的任务，是把正式生成结果变成可查看、可继续修改、可导出的交付物。当前系统把预览、细粒度 slide 查看、历史查看和导出统一到同一条结果链中，避免出现“页面预览是一套内容，下载文件又是另一套内容”的断裂。`Pagevra` 负责预览、渲染和导出 authority，Spectra backend 负责会话状态、结果绑定和下载契约整形。

图 6-4 预览与导出相关示意图，说明结果如何进入 preview、history 和 download 流程。

这条链路当前可以从以下几类证据核对：

| 证据类型 | 当前对应 |
| --- | --- |
| 页面入口 | 结果预览页、历史查看、导出与下载入口 |
| 接口契约 | `GET /api/v1/generate/sessions/{session_id}/preview`、`GET /api/v1/generate/sessions/{session_id}/preview/slides/{slide_id}`、`POST /api/v1/generate/sessions/{session_id}/preview/export` |
| 服务分工 | `Pagevra` 承接 preview / render / export，Spectra backend 承接 artifact 绑定、版本关联和下载响应 |
| 复核证据 | 预览 contract API 测试、preview schema 测试、artifact 渲染与结果绑定测试 |

这部分技术在作品中的体现主要是：

- 结果先进入真实预览，再允许继续修改；
- 最终支持标准文档导出；
- 前端不靠占位内容假装结果已经完成。

按业务问题拆分，这里至少涉及三件事：

- 预览必须来自真实结果，而不是本地临时示意；
- 导出结果要和预览保持一致，避免前后脱节；
- 用户的后续修改、历史查看和下载动作，不能脱离正式结果链。

接口 contract 已经把这条链写进当前实现。预览接口返回的不只是静态页面数据，还会同时返回 `artifact_id`、`based_on_version_id`、`current_version_id` 和 `rendered_preview`；slide 级预览接口负责更细粒度的查看；导出接口则把当前结果链继续推进到正式文件。预览 contract 测试会直接检查 `artifact_id`、版本锚点和 `rendered_preview.pages` 是否齐备，这说明当前链路具备接口级可核对性。

从技术链角度展开，这一部分至少包括四个动作：第一，接住正式生成结果并转换为可查看预览；第二，将预览、历史查看和继续修改绑定在同一 `artifact` 上；第三，把结果按正式导出链转换成 `pptx`、`docx` 等交付物；第四，在导出之后继续保留结果与项目空间、版本锚点和后续 refine 的关系。若只做到其中一半，系统就会退化成“只能预览”或“只能导出”的半链路工具。

当前作品在这部分已经给出清楚证据：页面上可见 preview、history 和 download，接口上存在对应 contract，测试上能够验证预览页返回真实渲染数据和 artifact 绑定关系。渲染导出技术的价值因此不只在文件格式本身，而在于它把“结果展示”和“结果交付”收束为同一条链路。第 5 章中的生成与交付主线、第 7 章中的界面展示和主链测试，都以这一点为前提。
