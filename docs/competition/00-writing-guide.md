# Spectra 竞赛文档写作指导

## 1. 获奖项目文档的核心范式

结合参考材料，竞赛型项目文档通常遵循以下结构：

1. 前言与综述：快速说明项目背景、痛点、目标和核心创新点。
2. 可行性分析：从技术、经济、社会与政策等维度论证项目成立。
3. 业务与需求分析：拆解用户对象、应用场景、功能需求、非功能需求和潜在扩展需求。
4. 系统架构设计：给出分层架构、模块职责和关键数据流。
5. 关键技术实现：深入说明模型、RAG、解析链路、生成引擎和优化机制。
6. 测试与成果展示：呈现功能测试、流程测试、质量评估和竞品对比。
7. 组织管理与商业企划：覆盖团队分工、进度安排、SWOT、商业模式与推广路径。

## 2. 本项目推荐文档结构

建议按以下章节编写：

1. 前言与项目综述
2. 可行性分析
3. 业务与需求分析
4. 系统架构设计
5. 关键技术实现
6. 项目测试与效果评估
7. 组织管理与商业价值

本目录已经按该结构拆分成独立 Markdown 文件，便于多人协作和逐章打磨。

## 3. 写作口径要求

### 3.1 关于“智能体”

评委通常更关注“系统是否具备主动性”，而不仅是“调用了大模型”。因此文档中要反复强调以下几点：

- 智能体会主动追问，而不是被动接收模糊指令。
- 智能体会融合对话历史、上传资料和知识库检索结果。
- 智能体会根据教师反馈进行局部重生成，而不是每次都从零生成。

### 3.2 关于“多模态”

“多模态”不能只停留在口号层面，至少要写清楚：

- 支持哪些输入格式：PDF、Word、PPT、图片、视频、语音。
- 每种输入如何被处理：文本提取、结构化切片、关键帧提取、摘要生成、向量化检索。
- 至少两种格式在演示中如何真正参与生成闭环。

### 3.3 关于“创新性”

建议将创新性集中表达为三点：

1. 以教师教学思路为中心，而非以模板填空为中心。
2. 将多模态资料理解、RAG 和课件生成打通为单一工作流。
3. 从静态文件交付走向动态课程资料库与持续迭代。

## 4. 写作时必须避免的问题

1. 不要把目标指标写成“实际测得结果”。
2. 不要只写技术名词，不解释它们在教师场景中的业务价值。
3. 不要只描述“生成 PPT”，忽略“预览修改”和“可追溯”的闭环。

## 5. 术语建议

建议在正式定稿时统一使用以下术语：

- `LLM`：大语言模型
- `RAG`：检索增强生成
- `Multimodal Agent`：多模态智能体
- `Course Knowledge Hub`：课程资料库
- `Iterative Optimization`：迭代式优化

## 6. 与 Spectra 交付系统的对应关系

### 系统核心能力

- 用户注册登录
- 项目创建与管理
- 资料上传与入库
- RAG 检索链路
- PPT/Word 生成任务
- 预览、修改、下载
- 异步任务编排

### 系统增强能力

- 更强的 PDF 版面解析
- 视频关键帧与时间轴语义提取
- 语音识别与术语纠错全链路
- 分享与协作
- 课程资料库化与学生侧二次生成
