# 团队协作指导

> 更新时间：2026-03-24
> 状态：当前生效
> 定位：当前阶段团队分工、职责边界、协作规则的统一指导文档

本文档用于统一当前阶段的团队协作口径。

它回答三类问题：

- 现在这一阶段到底在做什么
- A / B / C / D / E 分别负责什么
- 新任务出现时默认应该归谁

它不承担逐项打勾的执行管理；执行管理统一放到《团队执行清单》。

## 1. 当前阶段目标

当前阶段优先目标是通过初审。

因此，团队工作要围绕以下目标收束：

1. 主链路稳定
2. 竞赛材料对题、简洁、可信
3. 技术亮点有证据，而不是只有概念
4. 现有能力从中间态收口成真正可演示、可交付的正式能力

当前默认主链路是：

`资料上传 -> 检索/RAG -> 对话理解 -> 会话化生成 -> 大纲确认 -> PPT 生成 -> 导出物进入项目空间`

当前阶段的重点现实判断：

- 当前 PPT 生成还不应被视为真正完成的正式能力，更接近临时中间态
- 部分卡片功能仍然缺失或未真正接入完整链路
- `Library / 库` 还需要收口成统一信息源，而不是分散取数
- 因此，这一阶段既要补功能完成度，也要补生成质量

## 2. 总原则

下一阶段按以下原则执行：

1. 以“先过初审”为第一目标。
2. 以题目要求为主，不以系统扩张为主。
3. 以主链路稳定优先于边缘功能扩展。
4. 以可展示、可截图、可写进材料的成果优先。
5. 以低冲突、低返工优先于高并行。
6. 以 `Library / 库` 作为生成链路统一信息源的方向逐步收口。

## 3. 角色分工

### 3.1 A：总控与拍板

A 负责：

- 项目主叙事
- 初审阶段的取舍
- 核心语义和契约语言
- 主链路优先级
- 材料收口
- C / D 出现冲突时的最终拍板

A 不默认负责：

- C 的主线实现
- D 的技术专题实现
- B 的页面补位
- E 的文档代写

### 3.2 B：前端主力

B 负责：

- 工作台前端
- `Studio / Chat / Sources / Library` 的主链路体验
- 上传、对话、生成、预览、下载这条链路的展示完整性
- `Library / 库` 的前端实现与展示收口
- 库与 `Studio / Chat / Sources` 的前端联动
- 卡片前端交互、可演示化和页面状态补齐
- 截图、录屏和界面展示素材

B 不负责：

- 倒推后端语义
- 自己定义接口语义
- 消化 C / D 未解决的语义冲突

### 3.3 C：主线业务后端与功能补完

C 负责：

- 主线业务后端
- `session -> outline -> generate -> artifact -> download` 这条链路
- 项目空间里的导出物、引用、状态和业务语义
- 队列、任务执行和主线接口稳定
- 后端业务配置页和系统设置接口
- 单页定向修改能力落地
- 主线相关的后端架构维护
- 以库为统一信息源的生成链路收口
- PPT、Word、卡片等主线生成功能的正式落地与补完
- 当前缺失卡片能力的功能补齐与主链路接入
- 大纲生成、图片插入等能力的主链路接入

C 的核心边界：

- C 负责“把功能做出来、接完整、能真实使用”。
- 如果某项能力当前还不存在、只是半成品、或没有真正走通导出闭环，默认归 C。
- C 的重点是功能做完整、链路做闭环，不是只做提示词层面的调优。

### 3.4 D：技术深挖后端与质量优化

D 不作为第二个主线后端使用。

D 负责：

- 检索与重排
- 对话记忆与引用一致性
- 模型路由与 provider 治理
- 上传到索引链路的质量治理
- 评测、基线和质量门禁
- 提示词工程与生成质量实验
- 围绕 PPT、大纲、卡片生成效果的质量优化
- 图片召回、素材命中、提示词、检索利用相关的质量提升

D 的核心边界：

- D 负责“把质量做上去、把效果做更好、把方法论和 baseline 做出来”。
- 如果某项能力已经存在，但效果差、结果不稳、提示词不佳、检索命中不好，默认归 D。
- 如果问题是“还没有真实 PPT 生成能力”或“导出链路并未真正完成”，这是 C 的功能任务，不是 D 的任务。

### 3.5 E：竞赛文档负责人

E 不做开发。

E 负责：

- `docs/competition/` 的正文整理
- 题目要求对照
- 术语统一
- 图表、截图、材料结构整理
- 把 A、B、C、D 给出的事实整理成正式材料

E 不负责：

- 自己定义实现
- 自己发明技术叙事
- 脱离代码写材料

## 4. 当前阶段重点边界

以下边界必须明确区分：

- `C` 负责功能补完和主链路落地。
- `D` 负责质量优化和方法验证。
- `B` 负责前端实现、交互、展示和演示路径。
- `A` 负责取舍、标准和最终拍板。
- `E` 负责把已确认事实整理成正式材料。

具体到当前阶段：

- “把当前临时中间态 PPT 收口为真正的 PPT 生成能力”归 `C`
- “完善很多卡片功能缺失的问题”归 `C` 和 `B`
- “PPT 生成质量提升、提示词工程、图片命中优化、大纲去重优化”归 `D`
- “前端实现库”归 `B`

## 5. 默认协作规则

### 5.1 默认归属

- 主线后端问题默认给 C
- 技术专题默认给 D
- 前端展示和工作台默认给 B
- 语义、契约和取舍默认由 A 决定
- 竞赛文档默认由 E 收口

### 5.2 自动分配规则

以后遇到新增任务，如果没有人手动指派，默认按以下规则自动归属：

1. 只要任务直接影响 `session -> outline -> generate -> artifact -> download` 主链路，就先归 C。
2. 只要任务本质上是在“把功能补出来、把能力接完整、把半成品做成成品”，就先归 C。
3. 只要任务属于质量实验、提示词工程、检索优化、评测基线、RAG 命中改进，就先归 D。
4. 只要任务属于页面实现、交互流程、展示效果、库的前端入口、卡片前端体验，就先归 B。
5. 同一个任务同时涉及主线落地和专题研究时，默认 `C 负责功能落地，D 负责质量专题`，不要混写成同一类工作。
6. 同一个任务同时涉及后端语义和前端展示时，默认 `C 定后端语义，B 接前端呈现`。

### 5.3 冲突处理

如果一个问题同时涉及主线业务和技术专题，不要让 C 和 D 同时改同一层。

默认拆成两段：

1. D 先研究、给样本、给 baseline、给建议
2. C 再把成熟方案接入主线

### 5.4 跑系统要求

B / C / D 都需要：

- 自己经常运行系统
- 自己经常走主链路
- 自己在真实界面和真实流程里发现问题

处理规则如下：

1. 如果问题在自己负责范围内，默认自己修。
2. 如果问题不在自己负责范围内，及时同步给对应的人。
3. 不允许长期只看代码、不跑系统。

## 6. 与执行清单的关系

团队执行以《团队执行清单》为准。

但要明确：

- 清单只承载当前已经明确的任务，不承载全部潜在工作
- 清单里的事项必须执行、必须更新状态
- 清单之外，如果在真实联调、跑系统、演示准备中发现新的必要任务，仍然必须继续补充和推进
- 不能把“清单打完勾”误解成“系统已经彻底完成”
