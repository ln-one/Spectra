## 核心业务流程稳定性测试

除检索评估外，系统核心流程也做了工程验证。核心业务流程验证关注“资料进入 - 生成 - 预览 - 导出 - 保存”能否形成一条稳定流程。这一部分直接回答评审最关心的问题：作品是否具备真实可跑的核心流程。

本轮补充材料把相关证据进一步具体化。新增测试采用批量化、长流程、带失败治理机制的真实运行方式，覆盖从提示词输入、大纲生成与确认，到幻灯片生成、编译输出和批处理调度的完整流程。正文保留的是可计算、可复查、可直接纳入工程验证的结果。

### 核心业务流程验证内容

当前核心流程重点验证了以下问题：

- 生成模块是否承担课件主生成职责；
- `outline` 与确认是否形成可管理阶段；
- 生成结果是否能够进入预览与修改流程；
- 导出结果是否能够继续进入结果保存与绑定流程；
- 前端展示是否等待真实结果，而不是用占位内容冒充成功。

这组测试之所以重要，是因为它覆盖的是连续使用过程，而不是单一功能点。某个接口能返回结果，并不等于核心流程已经成立；只有当资料、会话、生成、预览、导出和结果保存能够连续衔接时，才说明作品具备了可演示、可重复验证的完整流程。

### 20 主题压力测试说明

本轮主测试围绕当前课件课件生成流程开展 `20` 个主题的连续验证。生成 authority 由 `Diego` 承担，但送审稿只关注这一流程在 Spectra 中是否能稳定跑通。测试围绕不同教学主题重复执行同一流程，观察系统是否都能完成以下关键步骤：

1. 接收输入并进入会话流程；
2. 生成 `outline` 并支持确认；
3. 完成正式生成；
4. 进入预览与导出流程；
5. 将结果继续保存到后续结果保存与导出流程中。

这组测试数据的核心价值，在于证明核心流程已经具备重复执行能力。系统在多次重复执行中保持较高成功率，说明验证已经超出单点展示阶段。

测试主要验证三件事：

- 流程完整性：会话、生成、预览、导出和保存这些步骤是否都能出现；
- 状态一致性：中间阶段是否有清楚的推进逻辑，而不是前后状态混乱；
- 结果连贯性：前端展示、后端流程和结果保存是否对得上。

执行方式采用后台批处理的连续验证。每个主题都遵循同一条用户路径：输入主题与提示词，生成 `outline`，完成确认，再进入正式生成、编译输出和结果保存流程。这样得到的结论更接近真实使用过程，而不是手工点击式演示。

### 测试目标与执行方式

根据补充材料，本轮核心业务流程验证重点围绕四个问题展开：

- 端到端流程是否可用；
- 多主题连续运行是否稳定；
- 局部失败是否能够被记录、隔离和追踪；
- 参数调整对速度与成功率的影响边界是否清楚。

对应的执行方式为 `20` 个主题的串行全流程测试。每个主题都遵循同一条用户路径：输入提示词和主题，生成 `outline`，完成确认，再进入正式生成、编译输出和结果保存流程。这样得到的结论更接近真实使用过程。

| 测试项 | 当前做法 | 验证目的 |
| --- | --- | --- |
| 主题规模 | `20` 个不同主题串行执行 | 观察跨主题重复运行能力 |
| 大纲阶段 | 生成后自动确认 | 验证中间阶段可被稳定推进 |
| 执行方式 | 后台批处理长时运行 | 验证无人值守条件下的连续性 |
| 任务控制 | 超时、重试、任务间隔 | 观察失败吸收与节奏控制能力 |
| 结果记录 | 汇总统计与日志留痕 | 保障结果可追踪、可复查 |

这组材料同时说明，核心业务流程验证并不把“速度越快越好”作为唯一目标。当前策略优先保证可用性和可解释性，再在此基础上观察吞吐与稳定性的权衡。因此，这组结果更适合用来说明系统的工程完成度，而不是作为单纯的性能竞赛数据。

### 核心业务流程运行结果

根据测试报告中的汇总统计，本轮 `20` 主题全流程测试共运行 `20` 个任务，成功 `19` 个，失败 `1` 个，整体成功率达到 `95%`。任务在后台连续运行约 `4 小时 20 分钟`，能够在无人值守条件下完成整批处理。成功样本的平均耗时约 `11` 分钟，中位耗时约 `10` 分钟，`P90` 约 `14` 分钟，最短耗时约 `6` 分钟。

| 指标 | 结果 |
| --- | --- |
| 总任务数 | `20` |
| 成功任务数 | `19` |
| 失败任务数 | `1` |
| 整体成功率 | `95%` |
| 连续运行时长 | 约 `4 小时 20 分钟` |
| 平均耗时 | 约 `11` 分钟 |
| 中位耗时 | 约 `10` 分钟 |
| P90 耗时 | 约 `14` 分钟 |
| 最短耗时 | 约 `6` 分钟 |

这组结果说明两点。第一，当前核心流程不只在单次演示下成立，而是在多主题、长时运行场景中保持较高可用性。第二，速度分布具备可预期性，大部分成功任务都在可接受时间窗口内完成，这使后续继续优化具有明确参照。

### 测试机制与失败治理

除结果表外，测试机制本身也构成证据。补充材料表明，当前核心业务流程验证并非依赖人工盯守，而是通过一组控制机制保证长时运行的稳定性与可解释性。

| 机制 | 当前设置 | 作用 |
| --- | --- | --- |
| 串行执行 | 一次仅运行一个主题 | 降低并发噪声，保证结论稳定 |
| 任务级超时 | 为单任务设置时间上限 | 防止异常任务无限占用资源 |
| 失败重试 | 失败后按窗口重试一次 | 吸收短时波动，不中断整批任务 |
| 任务间隔 | 主题之间保留固定间隔 | 降低连续调用带来的波动 |
| 日志落盘 | 记录运行过程与错误阶段 | 支撑复查和问题定位 |
| 汇总统计 | 输出结构化结果汇总 | 便于计算成功率与耗时分布 |

唯一失败样本发生在 `SLIDES_GENERATING` 阶段，错误记录为 `SLIDE_LLM_ERROR`。该样本在重试后仍未在任务窗口内收敛，最终被超时机制终止。这里的结论是：局部失败能够被隔离且可追踪。失败任务并未阻断后续主题执行，错误阶段也能够被明确定位，这正是工程验证的价值所在。

补充参数矩阵测试只作为辅助观察使用。当前材料显示，单纯提速并不能稳定提升成功率；更合理的工程结论是，速度与成功率仍需要通过并发控制、超时窗口和重试机制协同调整，而不是依赖单一激进参数。这一结论与本节主测试的设计取向一致：先保证核心流程可用，再讨论吞吐优化。

### 核心业务流程稳定性测试结论

结合原始测试材料和当前作品展示，可以把结论收束为：

- 系统核心流程已经能够把资料进入、生成、预览、导出和保存串起来；
- 生成过程包含确认、修改和结果回看，不是一次性黑箱输出；
- 前端展示、结果操作和后端主流程之间已经形成较清晰的对应关系；
- 当前主链已经具备支撑作品演示和后续优化的稳定基础。

整体来看，第 7 章最能说明作品完成度的是页面、流程和测试数据本身。检索评估证明系统具备资料与证据支撑能力，核心业务流程稳定性测试证明生成、预览、导出和保存这条主线可重复执行，界面与成果展示则证明这些能力已经落到作品层面。三者共同构成当前最有说服力的成果证据。
