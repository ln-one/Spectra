## 团队组织与分工

`Spectra` 围绕资料进入、检索增强、生成、预览导出、结果保存和文档整理共同推进。团队组织因此按工作内容和交付责任划分，而不是只按前端或后端简单二分。

| 分工方向 | 主要内容 | 对应产出 |
| --- | --- | --- |
| 产品与文档 | 目录设计、需求梳理、术语统一、方案书重写 | 章节结构、对外材料、术语与图表说明 |
| 前端与工作台 | Studio 页面、交互流程、结果展示、下载体验 | 工作台页面、预览结果、交互流程 |
| 后端与流程编排 | 会话流程、任务编排、接口组织、状态推进 | API、主流程、事件与结果绑定 |
| 知识库与检索 | 资料处理、切块、Embedding、召回与证据组织 | 检索流程路、评估结果、证据基础 |
| 测试与材料整理 | 指标统计、核心业务流程验证、截图整理、表格收束、结果核对 | 测试表、稳定性结果、截图、证据说明 |

这种分工方式能够直接对应到作品中的页面、流程、模块或测试结果，也便于和正文中的“需求 - 功能 - 实现 - 技术 - 效果”关系保持一致。

当前团队分工围绕核心流程持续调整。前期主要收束工作台、会话流程和结果交付；中期开始补齐资料进入、检索增强和结果保存与复用；后期则把测试、截图、表格和方案书表达纳入同一套交付节奏。整套组织方式服务于同一条交付流程，而不是围绕孤立页面或单个接口展开。

团队组织按照分工方向而不是个人技术标签展开。产品与文档、前端与工作台、后端与流程编排、知识库与检索、测试与材料整理分别对应资料处理、生成推进、结果验证和正式材料收束。分工结构可以直接回到页面、流程、接口和测试结果，因此具备现实依据。

围绕这些分工方向，团队在推进过程中形成了较清楚的责任边界。产品与文档方向负责把目录结构、术语体系、图表口径和正式表述统一起来，避免方案书、页面和测试结果处在不同版本。前端与工作台方向负责把资料区、会话区、结果区和预览导出入口收成同一条用户可见核心流程。后端与流程编排方向负责保持 `Session`、任务推进、结果绑定和导出契约的一致性。知识库与检索方向负责保证资料进入、切块、向量化、召回与证据组织能够落到正式评估结果。测试与材料整理方向则负责把运行结果、截图、表格和正文证据组织成可阅读的送审材料。

这种组织方式的好处在于，每条责任面都能回到具体产出。页面是否完整、接口是否一致、结果是否能保存、图题和表题是否统一、测试数据是否能进入正文，都有明确责任方向承接。外部读者看到的是一份完整方案书，项目推进则依靠这些责任边界保持一致性。

从组织管理角度看，这种分工方式还有两个直接作用。第一，它把“做功能”和“做材料”放进同一套交付安排，避免代码、页面和正式稿件长期分离。第二，它把界面、流程、测试和图表都对应到明确责任面，便于在后续修改时快速定位该由谁补页面、谁补接口、谁补证据和谁补文字。对竞赛送审材料而言，这种分工方式比按技术栈临时拼人更稳。
