## 结果预览与导出技术

渲染与导出要解决的具体问题，是把正式生成结果变成可查看、可继续修改、可导出的交付物。当前系统把预览、细粒度 slide 查看、历史查看和导出统一到同一套结果流程中，避免出现“页面预览是一套内容，下载文件又是另一套内容”的断裂。`Pagevra` 负责预览、渲染和导出，Spectra backend 负责会话状态、结果绑定和下载契约整形。

图 6-4 预览与导出流程图

这一流程当前可以从以下几类证据核对：

| 证据类型 | 当前对应 |
| --- | --- |
| 页面入口 | 结果预览页、历史查看、导出与下载入口 |
| 接口契约 | `GET /api/v1/generate/sessions/{session_id}/preview`、`GET /api/v1/generate/sessions/{session_id}/preview/slides/{slide_id}`、`POST /api/v1/generate/sessions/{session_id}/preview/export` |
| 服务分工 | `Pagevra` 承接 preview / render / export，Spectra backend 承接 artifact 绑定、版本关联和下载响应 |
| 复核证据 | 预览 contract API 测试、preview schema 测试、artifact 渲染与结果绑定测试 |

作品中的对应表现主要是：

- 结果先进入真实预览，再允许继续修改；
- 最终支持标准文档导出；
- 前端不靠占位内容假装结果已经完成。

这一流程涉及三项直接要求：

- 预览来自真实结果；
- 导出结果要和预览保持一致，避免前后脱节；
- 用户的后续修改、历史查看和下载动作，不能脱离正式结果保存与导出流程。

这一段实际依赖的现成契约主要有三类。第一类是 `frontend/lib/sdk/preview.ts` 和 OpenAPI 里的 preview/export 类型，前端就是按这些字段接预览页、slide 明细页和导出动作。第二类是 `backend/services/preview_helpers/rendered_preview.py`、`preview_queries.py` 和 `artifact_rendering.py`，它们负责把 `Pagevra` 返回的结果整理成前端可直接消费的预览结构。第三类是 `Pagevra` 本身的 preview / render / export 能力，它承担真正的渲染和导出，不是 backend 在本地再做一套 Office 渲染。

接口 contract 已经把这一流程写进当前实现。预览接口返回的不只是静态页面数据，还会同时返回 `artifact_id`、`based_on_version_id`、`current_version_id` 和 `rendered_preview`；slide 级预览接口负责更细粒度的查看；导出接口则把当前结果继续推进到正式文件。预览 contract 测试会直接检查 `artifact_id`、版本字段和 `rendered_preview.pages` 是否齐备，这说明当前流程具备接口级可核对性。

工作机制包括四个动作：第一，接住正式生成结果并转换为可查看预览；第二，将预览、历史查看和继续修改绑定在同一 `artifact` 上；第三，把结果按正式导出链转换成 `pptx`、`docx` 等交付物；第四，在导出之后继续保留结果与项目、版本记录和后续 refine 的关系。若只做到其中一部分，系统就会退化成“只能预览”或“只能导出”的半流程工具。

这里也要把边界写清楚：当前正式 Office 渲染权威仍是 `Pagevra`，不是 backend 本地模板链，也不是临时 fallback 文件。正文适合写“借助 Pagevra 预览导出契约打通了交付链”，不适合写“系统本地已经完整掌握了 Office 渲染能力”。

页面上可见 preview、history 和 download，接口上存在对应 contract，测试上能够验证预览页返回真实渲染数据和 artifact 绑定关系。结果展示和结果交付因此落在同一流程中。第 5 章中的生成与交付流程、第 7 章中的界面展示和核心业务流程测试，都以这一点为前提。
