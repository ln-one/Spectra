## 数据库设计

数据库设计的重点，在于通过稳定的实体关系模型承接系统中的业务对象、状态流转和成果复用关系。结合当前版本的系统语义，数据库设计围绕领域驱动设计（DDD）理念展开，通过 `Project`、`Session`、`Artifact`、`Version` 和 `Reference` 等核心实体构建完备的业务对象关联模型，支撑成果对象的全生命周期管理、版本溯源和逻辑关联。

从竞赛评审视角看，本节重点说明两个方面：系统是否具备清晰的业务实体关系模型，数据库设计是否能够直接支撑成果对象的保存、溯源和持续利用。

从 ER 模型逻辑看，数据库设计可归纳为五组核心实体关系：

- `Project`：承接长期存在的教学资料空间和成果对象管理边界；
- `Session`：承接一次具体业务过程中的输入数据、阶段状态和事件推进；
- `Artifact`：承接课件、文档、导图等成果对象；
- `Version`：承接成果对象的稳定版本锚点，支持回看、比较和持续迭代；
- `Reference`：承接资料与成果对象、成果对象与成果对象之间的逻辑复用关系。

上述实体关系共同构成系统的数据核心。一方面，成果对象不会在导出后脱离系统长期管理；另一方面，业务过程状态、成果稳定状态和引用关系能够在数据库层保持清晰分层。该逻辑关系直接支撑结果保存、历史回看、版本比较和持续复用等核心能力。

从对象职责看，`Project` 承接长期工作空间，`Session` 承接过程态数据，`Artifact` 承接成果对象，`Version` 承接成果稳定状态，`Reference` 承接逻辑关联与复用关系。该职责划分保证了系统在数据层能够区分“过程态”和“结果态”，避免短期操作状态直接覆盖长期成果状态，从而保持业务数据一致性。

数据库层同时承接成员边界与协作关系的治理需求。成员角色、成果归属、版本关系和引用关系在数据库模型中保持稳定承接，使后续权限控制、协作扩展和对象治理具备可持续的数据基础。该结构与前文权限组件和成果管理链路形成直接对应。

从部署结构看，统一入口节点承接公网访问，前端应用与 FastAPI 后端构成统一接入节点；Limora、Dualweave、Stratumind、Diego、Pagevra 与 Ourograph 等能力以独立服务接入；PostgreSQL、Redis、Qdrant、Uploads 与成果文件存储共同构成持久化存储区。该部署结构保证业务实体、异步任务、语义向量索引和成果文件能够在运行层面形成稳定支撑。

综上，数据库设计已围绕核心领域实体形成稳定、可扩展、可溯源的 ER 模型逻辑。该模型支撑项目空间管理、会话过程控制、成果对象沉淀、版本溯源和引用关系维护，是系统整体设计从业务层走向持久化实现的关键基础。

本节可归纳为一句判断：数据库层已经形成面向业务对象全生命周期管理的稳定 ER 模型基础。
