# 团队分工与 D 方向参考

> 更新时间：2026-03-22
> 状态：当前生效

本文档用于说明 Spectra 当前阶段的团队分工，并为 D 提供技术深挖方向参考。

它不是严格的人事制度，也不是逐条任务单。它的作用是：

- 减少分工不清带来的返工
- 保持主线稳定
- 给 D 一个可自己展开的技术方向范围
- 让文档同学有清晰的职责边界

## 1. 当前阶段目标

当前阶段优先目标是通过初审。

因此，团队工作要围绕以下三点收束：

1. 主链路稳定
2. 竞赛材料对题、简洁、可信
3. 技术亮点有证据，而不是只有概念

当前默认主链路是：

`资料上传 -> 检索/RAG -> 对话理解 -> 会话化生成 -> 大纲确认 -> PPT 生成 -> 导出物进入项目空间`

## 2. 角色分工

### A：总控与拍板

A 负责：

- 项目主叙事
- 初审阶段的取舍
- 核心语义和契约语言
- 主链路优先级
- C / D 出现冲突时的最终拍板

A 不默认负责：

- C 的主线实现
- D 的技术专题实现
- E 的文档代写

### B：前端主力

B 负责：

- 工作台前端
- `Studio / Chat / Sources / Library` 的主链路体验
- 初审需要的界面状态、截图和录屏路径
- 与后端业务配置页对应的前端配置界面
- 生成过程中的实时事件体验和等待感优化

B 不负责：

- 倒推后端语义
- 自己定义接口语义
- 消化 C / D 未解决的冲突

### C：主线业务后端与主线架构维护

C 负责：

- 主线业务后端
- `session -> outline -> generate -> artifact -> download` 这条链路
- 项目空间里的导出物、引用、状态和业务语义
- 队列、任务执行和主线接口稳定
- 后端业务配置页和系统设置接口
- 单页定向修改能力落地
- 主线相关的后端架构维护

C 的工作重点是：

- 稳定
- 清晰
- 可交付
- 语义一致

对 C 的补充要求：

- 不能只写接口，不跑系统。
- 需要自己经常运行系统，自己走主链路。
- 在上传、会话、生成、预览、下载、配置页这些主线流程里主动找问题。
- 如果问题属于自己负责范围，默认自己修。
- 如果问题不属于自己负责范围，及时同步给对应的人。

当前这阶段，C 还要特别负责三类事情：

1. 后端业务配置页
- 这是业务配置页，不是在线编辑 `.env`。
- 重点是把常用业务配置做成后端可持久化设置。

2. 单页定向修改
- 当前已有局部重绘基础，但还不够像真正可用的单页修改。
- 要补齐“用户明确输入这一页哪里不好、想怎么改”的链路。
- 不是只有一个没有需求输入的 refine 按钮。

3. 主线架构维护
- router 不继续堆业务逻辑
- 主线相关的大文件要持续拆分
- service 边界和主线语义要持续收口

### D：技术深挖后端

D 不作为第二个主线后端使用。

D 负责：

- 检索与重排
- 对话记忆与引用一致性
- 模型路由与 provider 治理
- 上传到索引链路的质量治理
- 评测、基线和质量门禁

D 的职责不是接零散业务，而是围绕专题做深。以上负责仅供参考，具体工作可由D本人自由选择。

### E：竞赛文档负责人

E 不做开发。

E 负责：

- `docs/competition/` 的正文整理
- 题目要求对照
- 术语统一
- 图表、截图、材料结构整理
- 把 A、C、D 给出的事实整理成正式材料

E 不负责：

- 自己定义实现
- 自己发明技术叙事
- 脱离代码写材料

## 3. 默认协作规则

### 3.1 默认归属

- 主线后端问题默认给 C
- 技术专题默认给 D
- 前端展示和工作台默认给 B
- 语义、契约和取舍默认由 A 决定
- 竞赛文档默认由 E 收口

### 3.2 冲突处理

如果一个问题同时涉及主线业务和技术专题，不要让 C 和 D 同时改同一层。

默认拆成两段：

1. D 先研究、给样本、给 baseline、给建议
2. C 或 D 再接入主线，谁接由 A 拍板

### 3.3 这阶段不追求极限并发

当前更重要的是低冲突、低返工，而不是强行提高并发度。

能拆开就并行，拆不开就串行。

### 3.4 B / C / D 都必须自己跑系统

B / C / D 都需要：

- 自己经常运行系统
- 自己经常走主链路
- 自己在真实界面和真实流程里发现问题

处理规则如下：

1. 如果问题在自己负责范围内，默认自己修。
2. 如果问题不在自己负责范围内，及时同步给对应的人。
3. 不允许长期只看代码、不跑系统。

## 4. D 的定位

D 的工作是：围绕技术深度，做出能支持项目亮点和竞赛材料的专题结果。要有实际数据支撑。

D 可以自己决定做哪个专题、怎么做。

但 D 也需要自己跑系统，不能只看脚本和评测结果。

## 5. D 的方向参考

### 5.1 检索与重排

这是当前最重要的方向之一。

当前系统已经具备：

- 本地项目检索
- 会话检索
- 已激活引用空间检索

但排序和重排仍有继续提升空间。

建议关注：

- `local_session / local_project / reference_base / reference_auxiliary` 的排序关系
- rerank 策略是否过轻
- query 类型差异对检索结果的影响
- 无命中时的解释质量

主要代码入口：

- `backend/services/rag_service/`
- `backend/routers/chat/message_flow.py`

### 5.2 对话记忆与引用一致性

这是最容易影响“像不像真的教学助手”的方向。

建议关注：

- 多轮对话中的会话隔离
- 资料命中与回答内容是否一致
- citation 和实际内容是否一致
- 无命中时的提示是否自然

主要代码入口：

- `backend/routers/chat/`
- `backend/services/prompt_service/`
- `backend/eval/dialogue_memory_audit.py`

### 5.3 模型路由与 provider 治理

当前仓库里已经有路由和 provider 相关基础设施，但还可以做得更稳。

建议关注：

- 模型选择策略
- fallback 是否透明
- provider 失败后的分类和记录
- 质量、延迟、成本之间的平衡

主要代码入口：

- `backend/services/ai/`
- `backend/eval/router_quality_audit.py`

### 5.4 上传到索引链路的质量治理

当前上传链路已经有 parse、chunk、embedding、index 的基础，但降级和质量信息仍有整理空间。

建议关注：

- parse fallback 是否清楚
- embedding 路由是否正确
- 各阶段 timing 是否可解释
- 降级是否会变成“慢但说不清”

主要代码入口：

- `backend/services/media/`
- `backend/services/file_parser/`

### 5.5 评测、基线与质量门禁

当前 `backend/eval/` 已经不是空壳，已经具备较多评测资产。

建议 D 把这部分真正变成可用工具，而不是只放在仓库里。

建议关注：

- 哪类改动该跑哪类评测
- baseline 如何维护
- 什么结果才算值得接入主线

主要代码入口：

- `backend/eval/`

## 6. 当前阶段补充说明

除了主线稳定和技术专题之外，当前阶段还需要明确三类工作：

### 6.1 后端业务配置页

- 这是主线能力，归 C 主责，B 配合界面。
- 它解决的是业务配置问题，不是部署级环境变量管理。

### 6.2 实时事件体验补强

- 当前系统已经有生成会话 SSE 事件流。
- 这阶段更准确的目标是补强实时事件体验和等待感，不是从零开始发明流式响应。
- C 负责后端事件和主线语义，B 负责前端体验。

### 6.3 局部修改能力补齐

- 当前已有单页局部重绘基础。
- 但真正可用的能力应该是“带用户明确需求的单页定向修改”。
- 这项工作归 C 主责。
