## 6.4 全链路业务流（End-to-End Business Flow）与知识回流验证

### 6.4.1 生成与交付全链路业务流

当前全链路业务流可概括为：

::: {custom-style="Body Text"}
Session 建立 -\> Diego outline -\> 教师确认 -\> Diego generation -\> Pagevra preview/export -\> Ourograph artifact bind
:::

::: {custom-style="Body Text"}
该链路验证以下关键点：
:::

- 生成经过 outline、confirm、generation 的可管理过程；

- 预览和导出不由前端伪造，而是等待正式产物返回；

- 交付结果可绑定到知识空间的 Artifact；

- Spectra Backend 承担编排和契约整形，不重复实现 generation/render/formal-state 权责边界。

### 6.4.2 知识沉淀全链路业务流

系统验证需覆盖"结果是否能回到库"。若仅有导出文件，缺少正式状态绑定，系统仍为普通生成工具。

::: {custom-style="Body Text"}
当前全链路业务流以 Project / Session / Artifact / Version / Reference / CandidateChange / Member 为对象语言，使生成结果进入长期知识空间。业务意义：
:::

- Project 承接课程数据库；

- Artifact 承接 PPT、教案、导图、动画、互动内容等按需外化结果；

- Version 与 CandidateChange 承接正式状态和后续演化入口；

- Reference 支撑课程空间之间的引用关系；

- Member 支撑后续协作与组织治理。

### 6.4.3 全链路业务流验证结论

经工程化验证，全链路业务流已实现业务逻辑闭环：

- 资料进入后续知识链，参与内容生产全流程；

- 生成经历确认、预览、修改和交付，过程可管理；

- 结果进入正式知识状态，支持长期演化；

- 多模态成果共享同一课程知识空间，数据一致性得到保障。

该验证表明：系统已实现稳定的全链路业务流，不是若干局部能力的松散拼接。

### 6.5.2 指标解释

检索质量通过以下指标进行客观评估：

  ----------------------------------------------------------------
  **指标**       **解释**
  -------------- -------------------------------------------------
  Hit@3          前 3 个结果中是否命中相关内容（查全率维度）

  MRR@3          相关结果在前 3 个位置中的排序质量（查准率维度）

  Evidence Hit   是否命中可支撑回答的证据

  Evidence MRR   支撑证据是否排在更靠前位置

  Keyword        结果是否覆盖关键术语与核心表达

  Quality        综合质量评分，反映结果可用性

  Avg latency    平均响应延迟，用于观察工程可用性
  ----------------------------------------------------------------

  : 表 6-5

::: {custom-style="Body Text"}
Evidence Hit 和 Evidence MRR 是核心指标，教育内容生成要求检索结果具备可解释、可回溯的证据支撑。

## 6.6 Dualweave 与其他正式能力层工程验证

### 6.6.1 Dualweave：资料进入与交付治理

Dualweave 解决的核心问题是：

::: {custom-style="Body Text"}
**同一份资料如何一次进入系统，同时支撑本地持久化、远端交付、阶段状态、失败语义和后续解析工作流。**
:::

::: {custom-style="Body Text"}
当前 Dualweave 已形成面向资料进入链路的工程验证框架。经工程化验证的能力包括：
:::

- local-truth-first：优先建立本地资料真相，避免远端处理波动直接破坏资料进入链路；

- 单次资料进入：同一份资料进入系统后，同时支撑本地持久化、远端交付和后续解析工作流；

- 阶段状态清晰：通过 completed、pending、failed 等状态表达不同交付阶段；

- 失败语义明确：错误可落到 send、workflow、result、replay 等边界；

- 恢复路径可观测：replay、pending duration、remote completion after pending 等指标描述恢复成本；

- 压力行为可解释：backlog high watermark、capacity rejects 等指标暴露系统压力；

- request return 与 happy-path completion 分离：用户请求返回速度和远端完整交付速度分开衡量。

该设计使 Dualweave 成为资料进入与交付治理底座，具备生产级系统的鲁棒性。

### 6.6.2 Diego：生成主链验证

Diego 的验证重点在于：

- 是否承担 PPT / 课件主生成 authority；

- outline 与 confirm 是否形成可管理生成阶段；

- 生成结果是否能进入 Studio preview/refine 链；

- 生成结果是否能继续绑定到正式知识状态。

这说明主生成全链路业务流已实现"可确认、可追踪、可沉淀"的工作流。

### 6.6.3 Pagevra：预览与导出验证

Pagevra 的验证重点是 preview、render、export 是否统一：

- PPTX 与 DOCX 不由 Spectra 主仓本地旧链路伪造；

- preview 与 download 属于同一正式交付契约；

- 前端等待真实产物，不把占位内容包装成成功结果。

该验证表明系统交付能力符合生产级系统标准。

### 6.6.4 Ourograph：知识状态验证

Ourograph 的验证重点是 formal knowledge-state：

- Project 承接长期课程空间；

- Artifact 作为外化结果进入正式状态；

- Version、Reference、CandidateChange 表达演化与引用；

- Spectra 保持 consumer / workflow shell 身份，不重复实现 formal-state 权责边界。

该验证是系统从"生成工具"升级为"课程知识空间系统"的核心依据。

### 6.6.5 Limora：身份与组织边界验证

Limora 的验证重点在于 identity / organization / member boundary：

- 身份与组织语义不回流到 Spectra 主仓；

- 学校、教研组、机构化使用具备独立边界；

- Member 与知识空间协作形成长期治理基础。

经工程化验证，当前系统已具备资料进入、标准交付、正式知识状态和身份治理同时成立的工程基础。
