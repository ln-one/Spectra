# 系统设计

# 5.1 系统整体架构

系统整体由四个部分组成，分别是

# 1、数据获取与处理模块

数据获取与处理这一模块负责收集文旅信息，并对收集到的文旅信息进行数据清洗，格式统一，分块划分，得到按单个商家信息，娱乐信息，酒店信息等划分的数据块来构成语料库，最终将语料库中全部数据送入向量模型中进行数据向量化，存入文旅数据知识库。

# 2、LLM知识库模块

LLM知识库模块，负责接受后端传来的用户的问题，并从文旅知识库中检索出topk个语义相关的数据作为上下文，与LLM本身的prompt进行拼接，从而可以回答用户的问题返回给后端。

# 3、后端管理模块

后端管理模块，负责对用户信息进行管理，对知识库数据进行统计，与LLM知识库模块进行交互。

# 4、人机交互模模块

人机交互模块，提供了一个用户对话的平台，在这个平台用户可以询问文旅相关的知识并得到平台的响应。此外该平台还可以记录用户的上下文信息与历史对话记录。

# 1、Local Documents (本地文档)

- 作为输入源，包含系统所需处理和检索的信息。
- 数据来源是数据处理模块进行爬取文旅信息地文档并结构化文档

# 2、UnstructuredLoader(非结构化加载器)

- 负责读取本地文档内容，并将其加载到系统中进行进一步的处理。

# 3、Text(文本)

- 表示加载进系统的原始文本数据。

# 4、Text Splitter (文本分割器)

- 将整块文本切分成更小的段落或片段，称为文本块（TextChunks），以便进行细粒度的处理和检索。
- 采用特殊符号'----'作为分割符来分割单个美食店铺，娱乐店铺，酒店店铺

# 5、Embedding (嵌入)

- 对文本块进行向量化处理，将文本转换为数值向量形式，以利用在向量空间中进行语义相似性比较。
- 采用向量模型 bge-zh-large-v1.5 进行嵌入

# 6、VectorStore(向量存储)

- 存储文本块的向量表示，以供后续检索和匹配使用。
采用 milvus 进行存储

# 7、Vector Similarity (向量相似度)

- 计算查询向量与存储的文本块向量之间的相似度，以找到最匹配的内容。
采用L2作为向量相似度地衡量方式

# 8、Query (查询)

- 用户提出的问题或要检索的信息。

# 9、Embedding (嵌入)

- 同第5步，但是这里是对用户查询进行向量化。

# 10、Query Vector (查询向量)

- 用户查询的向量表示。

# 11、Related TextChunks(相关文本块)

- 通过计算向量相似度找到的与用户查询最相关的文本块。

# 12、Prompt Template(提示模板)

- 定义如何将检索到的相关文本块转换成模型可理解的提示，为生成回答做准备。
- 提示模板由开发人员自定义

# 13、Prompt (提示)

- 依据提示模板构造出的输入，将被送入语言模型生成答案。

# 14、LLM(大型语言模型)

- 使用大型语言模型来解释提示并生成合适的回答
- 本系统选取 qwen-1.5-7B 来作为对话模型

# 15、Answer(回答)

- 语言模型生成的回答，是对用户查询的直接回复。

# 5.2.1 Langchain

在这份技术文档中，我们会对 LangChain 这一框架进行详细的介绍。LangChain 是一个基于语言模型（Language Model, 简称 LM）的开发框架，它通过串联不同的模型和工具，使得开发者可以构建出能够理解和生成自然语言的应用程序。

# - LangChain 的特点包括:

(1) 上下文感知：能够理解并处理输入数据的上下文信息，这包括但不限于用户的提示指令、示例和背景资料等。这样可以让语言模型更加精准地捕捉到用户的需求。
(2) 推理能力: LangChain 利用语言模型的推理能力来解释和响应用户的输入。这不仅仅是简单地匹配模式或关键字, 而是通过理解输入的内容, 在给定的上下文中做出恰当的回应或决策。

# - LangChain 在本系统中的应用结合了三种类型的模型：

(1)对话模型：处理和维护对话流，确保连贯性和上下文的一致性。
(2) 向量模型：用于理解和编码语义信息，支持模糊匹配和意图识别。
(3) 重排模型：对可能的回答或动作进行排序，选择最佳选项作为响应。

通过LangChain构建的Agent不仅能理解用户的意图，还能自主调用API，并根据上下文自动选择最合适的知识库来回答问题。这个过程通过以下步骤实现：

1、意图理解：通过分析用户的输入，Agent可以判断出用户的是提出问题、执行任务还是请求服务。
2、API调用：Agent根据意图选择合适的API进行调用，以获取必要的数据或执行某些操作。
# 3 知识库选择：Agent会判断哪些知识库对于当前的任务或问题是最相关的，并

从中提取信息以支持回答。

4、回答生成：结合上述信息，Agent 构造出一个合理、准确且符合上下文的回答或响应。

# 5.2.2 数据处理流程

在当前的信息化时代，知识库系统（Knowledge Base System, KBS）已成为各行各业不可或缺的组成部分，特别是在人工智能领域，知识库对于提升大型语言模型（Large Language Models, LLMs）的性能尤为关键。本文将围绕您提供的知识库系统架构图展开讨论，详细分析系统的功能组成、数据来源的多样性、数据操作的灵活性以及强大的检索功能，并在最后提出系统设计和实现的建议。

# 1、系统功能组成与架构设计

知识库系统的核心目的在于为后端的LLMs提供必要的、结构化的外部知识，从而增强模型的理解和生成能力。系统的主要功能可分为以下几个方面：

(1) 数据输入与管理：系统支持三种数据输入方式：手动数据集、文本数据集和表格数据集，以满足不同数据来源和场景的需求。其中，手动数据集允许用户直接在系统中输入和编辑数据；文本数据集通过特定的分隔符号进行分割，将数据划分为合适的分块长度，便于系统处理和检索；表格数据集则要求有明确的数据格式，例如必须包含 index 标识和可以有 content 列，这样的结构化设计有助于提高数据的可读性和可管理性。
(2) 数据维护操作：除了数据输入之外，系统还需要提供完善的数据增删改查（C RUD）功能，以保证知识库内容的时效性和准确性。用户可以依据实际情况更新知识库中的数据，包括添加新的知识点、删除过时的信息、修改错误的内容以及查询需要的数据。
(3) 知识检索：一个强大的检索系统是知识库系统的灵魂。本系统提供了语义检索、全文检索和混合检索等多种检索方式，以满足用户在不同情境下的查询需求。语义检索侧重于理解和匹配用户查询的意图，全文检索则提供基于关键词的广泛搜索，混合检索结合了两者的优点，既考虑了语义的相关性也确保了搜索的全面性。

# 2、数据来源的多样性与适用场景

(1) 手动数据集：适合于知识量较小、更新频率不高或者需要高度定制化的文旅场景。如举办时间较短的活动内容
(2) 文本数据集：适用于有大量文本文件、日志文件的情况，如各地文旅信息的介绍，通过分割符号和分块长度的设定，可以有效地组织和利用非结构化数据。
(3) 表格数据集：适合于需要进行数据分析和数据处理的场景，如店铺名字，店铺地点，店铺评分等，格式的严谨性有助于实现高效的数据操作和检索。

# 3、数据操作的灵活性与效率

为了提高数据管理的灵活性和效率，知识库系统需要实现以下几个方面的功能：

(1) 数据增加：系统应允许用户轻松添加新的数据项，无论是手动输入、批量导入还是通过API接口。
(2) 数据删除：提供方便的数据删除功能，包括单条记录删除和批量删除，以及基于条件的删除操作。
(3) 数据修改：允许用户对知识库中的数据进行修正和更新，以保证信息的准确性和最新性。

(4) 数据查询：强化查询功能，支持多条件查询、模糊查询等，使用户能够快速找到所需信息。

# 4、知识检索的强大功能

知识检索的效率直接影响到用户体验和知识库的实用性。系统设计时应考虑以下几个要素：

(1) 语义检索：借助自然语言处理技术，理解用户的查询意图，返回最相关的结果。
(2) 全文检索：快速扫描整个知识库，根据关键字匹配相关文档。
(3) 混合检索：结合语义和全文检索的优点，提供更准确、更全面的检索结果。

# 5.2.3 向量库的选取以及向量相似性检索技术

# 1、向量库的选取：选取国产开源向量库Milvus

- Milvus 的特点:

(1) 专注于向量数据库的发展，赋能 AI 应用和向量相似度搜索
(2) 支持多种索引索引建立方式以及向量之间的距离计算方式
(3) 高度可拓展性, Milvus 的分布式和高吞吐量特性使其非常适合为大规模矢量数据提供服务
(4)功能丰富，支持各种数据类型、带有属性过滤功能的增强型矢量搜索

# 2、数据库的建立

- 数据库的字段：

(1) Id: 主键，用来表示每个数据的序号。类型：INT64
(2) Embeddings: 存储向量化的数据。类型：FLOAT_VECTOR。维度：dim:1024。

- 数据库的索引方式：HNSW
- 数据库的向量之间的距离计算方式：L2
- 为什么选择向量数据库：

(1) 矢量数据库通过将每个对象表示为矢量嵌入来保持数据的语义。每个嵌入都是高维空间中的一个点。例如，香蕉的矢量（包括文本和图像）位于苹果附近，而不是猫。

![](images/bb76ee3ee3e4d6e5ab874d756fe3df47ba5d865c7b5bae07d5d095c24a4da7af.jpg)
图5-6向量数据库

# (2) 向量数据库的检索方式远远大于传统数据库的检索方式

相较于传统数据库或者搜索引擎通过关键字的做法来搜索，向量数据库用向量嵌入模型计算矢量嵌入并将其附加到所有数据对象上，矢量嵌入捕捉数据的意义和上下文，矢量数据库借助各种索引算法可实现超快查询。

# 3、索引方式的建立

选择 HNSW 作为索引，其目的是在对比多种索引方式后，HNSW 在搜索的速度和搜索的数据量上面都展现了极大的优势，具有超快的搜索速度和出色的召回率。

HNSW 的介绍:HNSW 是一种基于图的近似最近邻搜索方法，由 Yury Malkov 和 Dmitry Yashunin 于 2016 年提出。它是一种高效的向量索引方法，旨在快速找到高维空间中的最近邻元素。

HNSW 的工作机制基于“小世界”网络的概念，即大多数节点可以通过相对较少的跳数被高效地达到。HNSW 构建了一个分层的图结构，每个层次都是一个小世界图。最上层的图含有较少的节点，并覆盖了整个向量空间；而每向下一层，节点数量增多，图的密度也随之增加。搜索时，算法先在上层的稀疏图中快速定位，然后逐层向下，直到达到最底层的密集图，从而找到最近邻。

# 4、数据库的主要参数以及含义：

- 对于 HNSW 索引，比较核心的参数只有两种。

(1) $m$ : 每层的最大连接数, 增加 $m$ 意味着每个点将连接到更多的邻居, 从而使图更加密集, 这可以加快搜索查询速度, 但代价是索引建立时间更长, 内存使用量更大。默认为 16 , 这里我们设置为 32 。
(2) ef_construction: 用于构建图形的动态候选列表的大小。在算法遍历图形的过程中, 这个列表会暂时保存迄今为止发现的最接近的候选点。一旦完成了对某一点的遍历, 就会对该列表进行排序, 并将最接近的前 $m$ 个点保留为相邻点。较高的 ef_construction 值允许算法考虑更多的候选点, 从而提高索引的质量。不过, 这也会减慢索引构建过程, 因为更多的候选点意味着更多的距离计算。默认为 64 ,这里我们设置为 128 。

# 5、向量相似性检索技术

向量相似性检索是一种尖端技术，它在大规模的稠密向量数据集中寻找与指定查询向量在语义上最为接近或相关的前k个结果。这种技术被广泛应用于多种数据类型，如文本、图像、视频和音频等。为了执行相似性搜索，首先需要将这些非结构化数据通过向量模型转换成高维数值向量，即我们所说的嵌入向量。

在生成嵌入向量之后，向量搜索引擎便开始工作，通过比较输入的查询向量与向量存储库中的向量之间的空间距离，来寻找相似性。两个向量在空间中的接近程度越高，它们的相似度就越大，从而表明这两个实体在语义上或特征上的接近程度。

向量相似性搜索的核心概念涉及到索引构建和距离度量两个方面。之前的段落已经对索引做了阐述，这里我们聚焦于距离度量，它是评估向量之间相似度的关键。主要存在三种距离度量方法：L2距离（欧几里得距离）、余弦相似度和内积。

在我们的系统中，我们选用L2距离来衡量两个向量之间的距离，这也是一种广泛应用的方法。L2距离，或称为欧几里得距离，通过计算两个点在各维度上差值的平方和，然后取其平方根来确定。具体的数学表达式为：

$$
L 2 (A, B) = \sqrt {\sum_ {i = 1} ^ {n} \left(A _ {i} - B _ {i}\right) ^ {2}} \tag {1}
$$

其中，A 和 B 分别表示两个向量，n 是向量的维度。通过这种方式，我们能够量化并比较不同数据点之间的相似性，进而在大规模数据集中高效地进行检索和分析。

# 5.2.4 Web前端

Web前端搭建用户对话界面，利用socket在django框架的基础上建立起后端与前端对话、天气、新闻等数据的连接，实现将交互信息实时呈现在web前端页面上。

Web前端总体使用nuxt.js框架，其是基于vue.js的更高级的框架，用于创建服务器端渲染的vue应用，通过它的使用可以简化了路由的配置，改善SEO和首次加载时间，同时利用vue可复用的组件来构建应用界面，每个组件可以拥有自己的状态、模板和逻辑，进一步增强了前端界面的复用性。

使用 asyncData 或 fetch 方法，我们可以轻松地加载和渲染异步数据，无需担心将数据预填充到应用的初始状态中。同时利用自动化路由系统，与直接使用 Vue.js 相比，Nux t.js 自动基于 pages 目录中的 Vue 文件生成应用的路由配置。这种约定大于配置的方式简化了路由的创建和管理，减少了手动创建和维护路由配置的工作量。

# 5.2.5 后台管理系统

利用Django框架，sqlite3数据库来实现网页后台管理系统的搭建，实现对提问和回答数据的监控、用户的增删查改，用户分组、用户权限修改、模型配置等，通过表格和筛选条件实现记录的展示和遴选；前后端整体的连接、模型的对话以及与数据库数据的交互均以后台管理系统为媒介实现。

在Django开发中，其自带一个强大的后台管理工具，这个工具自动生成，无需编写额外的代码。通过简单的配置，开发者可以对数据进行增删查改操作，这对于快速开发后台管理系统尤其有用。同时我们利用Django的特性以非常清晰的方式设计URL。通过使用正则表达式匹配URL并将其映射到相应的视图函数。Django还有对象关系映射特性（ORM），这允许开发者以Python类的形式定义数据模型，这些类随后会被Django转化成数据库表。这种方法抽象了数据库操作，使开发者可以不用编写SQL代码就能进行数据库查询和操作，极大地提高了开发效率。在构建网页后台管理系统时，Django的这些特性被问我们充分利用。例如，使用Django的ORM可以轻松地创建、修改、删除用户和权限模型；利用Django admin可以迅速搭建起管理界面，实现对提问和回答数据的监控等。

Django同时还具有模板系统，支持复杂的数据显示逻辑。开发者可以使用Django模板语言（DTL）来创建动态的HTML页面，该模板语言提供了丰富的过滤器和标签来处理数据和进行条件渲染。同时在表单处理处理方面，Django处理表单的方式既安全又方便。它提供了表单类来自动处理数据验证、输入转换和错误消息显示等任务。利用上述特性，我们通过Django的视图和模板系统，创建了自定义的前端页面来展示数据，并通过Django提供的表单处理和验证机制来实现数据的提交和编辑。

# 5.2.6 Nginx 服务器与 SSL 安全证书

Nginx 是一款开源、高性能的 HTTP 和反向代理服务器，其具有高效的资源利用率、简易的配置和稳定的性能等特点，该项目即利用 Nginx 服务器实现服务器端的部署。

Nginx 使用较少的资源来处理大量的并发连接，这使得它在处理静态内容、反向代理以及负载均衡时表现出众。同时它能够实现反向代理和负载均衡，Nginx 可以作为反向代理服务器使用，帮助提高后端应用服务器的可用性和性能，通过负载均衡分散客户端的请求到多个应用服务器。我们同时也利用它的安全性，为网站提供了强大的安全保障，包括防止 DDoS 攻击的配置选项、SSL/TLS 加密支持等。

在我们的项目部署过程中，Nginx 扮演了至关重要的角色。首先，我们利用 Nginx 的高性能静态文件服务能力，来部署前端静态资源，包括 HTML、CSS 和 JavaScript 文件，这样可以快速响应用户的请求。通过配置 Nginx 作为反向代理服务器，我们将客户端的请求转发到后端的多个应用服务器，这不仅实现了负载均衡，也增加了后端服务的可用性和可扩展性。此外，我们还配置了 Nginx 的 SSL 模块来为网站启用 HTTPS，加强了数据传输过程中的安全性。

部署过程中，我们遵循了最佳实践，优化了 Nginx 的配置文件，比如通过调整 worker_processes 和 worker Connections 来充分利用服务器的硬件资源。同时，我们也密切关注 Nginx 的日志文件，以便及时发现并解决潜在的问题。
